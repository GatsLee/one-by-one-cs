## 핵심 개념

### 1. 디버깅이란?

디버깅(Debugging)은 프로그램에서 **버그(결함)**를 찾아 수정하는 과정이다. 코딩 시간의 50% 이상이 디버깅에 쓰인다. 체계적인 디버깅 전략이 있으면 시간을 크게 절약할 수 있다.

```
디버깅의 4단계:

┌────────────┐     ┌────────────┐     ┌────────────┐     ┌────────────┐
│  1. 재현    │ ──→ │  2. 격리    │ ──→ │  3. 이해    │ ──→ │  4. 수정    │
│ Reproduce  │     │  Isolate   │     │ Understand │     │    Fix     │
│            │     │            │     │            │     │            │
│ 버그를 다시 │     │ 원인 범위를  │     │ 왜 발생     │     │ 고치고     │
│ 발생시킨다  │     │ 좁힌다      │     │ 하는지 이해  │     │ 재발 방지   │
└────────────┘     └────────────┘     └────────────┘     └────────────┘
```

### 2. 전략 1: 변경 사항 추적 ("뭐가 바뀌었지?")

버그가 갑자기 나타났다면, **마지막으로 변경한 것**이 원인일 확률이 가장 높다.

```
"어제까지 됐는데 오늘 안 돼요" → 어제와 오늘 사이에 바뀐 것은?

체크리스트:
  □ 코드를 수정했나?            →  git diff로 변경 확인
  □ 라이브러리를 업데이트했나?    →  package.json / requirements.txt 비교
  □ 환경 설정을 바꿨나?          →  .env, config 파일 확인
  □ OS나 런타임을 업데이트했나?   →  node -v, python --version 확인
  □ 데이터가 달라졌나?           →  입력 데이터 / DB 상태 확인
```

### 3. 전략 2: 이분법 디버깅 (Binary Search Debugging)

코드가 길 때, 한 줄씩 확인하지 말고 **절반씩 잘라서** 문제 범위를 좁히는 방법이다.

```
100줄 코드에서 버그 찾기:

방법 A: 한 줄씩 확인 → 최대 100번 확인
방법 B: 이분법       → 최대 7번 확인 (log₂ 100 ≈ 7)

┌──────────────────────────────────────────────────┐
│                  전체 코드 100줄                    │
│                                                  │
│  1회: [──────────── 50줄 ────────────|── 50줄 ──] │
│       "50번째 줄에서 print → 여기까지 정상?"         │
│       → 정상이면 뒤쪽이 문제, 비정상이면 앞쪽이 문제  │
│                                                  │
│  2회: [── 25줄 ──|── 25줄 ──]                      │
│  3회: [─ 12줄 ─|─ 13줄 ─]                          │
│  4회: [─ 6줄 ─|─ 6줄 ─]                            │
│  ...                                             │
│  7회: 버그 발견!                                    │
└──────────────────────────────────────────────────┘
```

```python
# 이분법 디버깅 실전 예시
def process_pipeline(data):
    step1 = parse_input(data)
    print(f"[DEBUG] step1: {step1}")      # ← 중간 지점에 출력

    step2 = validate(step1)
    step3 = transform(step2)

    print(f"[DEBUG] step3: {step3}")      # ← 다음 중간 지점

    step4 = enrich(step3)
    step5 = format_output(step4)
    return step5
```

### 4. 전략 3: 러버덕 디버깅 (Rubber Duck Debugging)

고무 오리(또는 아무 대상)에게 **코드를 한 줄씩 설명**하는 방법. 설명하다 보면 "아, 여기가 이상한데?"라는 순간이 온다.

```
러버덕 디버깅 과정:

  나: "이 함수는 users 배열을 받아서..."
  나: "각 user의 age를 확인하고..."
  나: "age가 18 이상이면... 잠깐, 18 '이상'인데 >= 아니라 > 를 썼네!"

  ┌──────────┐
  │   🦆     │  ← 고무 오리 (말을 안 해도 됨)
  │  "..."   │     설명하는 과정 자체가 버그를 찾아줌
  └──────────┘
```

실제로는 고무 오리 대신:
- 동료에게 설명하기
- 혼잣말로 설명하기
- 주석으로 각 줄의 의도를 적기
- AI 챗봇에게 "이 코드가 하는 일"을 설명하기

### 5. 전략 4: 최소 재현 예시 (Minimal Reproducible Example)

버그를 재현하는 **가장 짧은 코드**를 만드는 것. 불필요한 코드를 제거하면 원인이 명확해진다.

```
원본 코드 (200줄, 복잡한 앱)
        │
        ▼ 관련 없는 코드 제거
        │
    축소된 코드 (30줄)
        │
        ▼ 더 제거
        │
  최소 재현 (10줄) ← 이 상태에서 버그의 원인이 보인다
```

```python
# 원본: 복잡한 API 서버에서 데이터 처리 에러
# → 최소 재현으로 축소:

data = {"users": [{"name": "Kim", "scores": None}]}

# 버그: scores가 None일 때 sum() 실패
for user in data["users"]:
    total = sum(user["scores"])  # TypeError: 'NoneType' is not iterable
```

### 6. 전략 5: print 디버깅 vs 디버거

```
┌────────────────────────────────────────────────────────────┐
│                print 디버깅 vs 디버거                        │
├──────────────────────┬─────────────────────────────────────┤
│    print 디버깅       │         디버거 (Debugger)            │
├──────────────────────┼─────────────────────────────────────┤
│ 빠르게 시작           │ 설정 필요                            │
│ 어디서든 사용 가능     │ IDE나 도구 필요                       │
│ 실행 후 제거해야 함    │ 코드 수정 불필요                      │
│ 변수 하나씩 확인      │ 모든 변수를 한눈에 볼 수 있음           │
│ 반복 실행 필요        │ 중단점에서 멈추고 탐색 가능             │
│                      │ 한 줄씩 실행(step) 가능               │
├──────────────────────┴─────────────────────────────────────┤
│ 추천: 간단한 버그 → print / 복잡한 버그 → 디버거              │
└────────────────────────────────────────────────────────────┘
```

### 7. 전략 6: 가설 기반 디버깅 (Scientific Method)

과학 실험처럼 **가설을 세우고 검증**하는 체계적 접근:

```
1. 관찰: "로그인 버튼을 누르면 500 에러가 난다"
2. 가설: "DB 연결이 끊어진 것 아닐까?"
3. 실험: DB 상태 확인 → 정상
4. 새 가설: "비밀번호 해싱 함수에 문제가 있나?"
5. 실험: 해싱 함수 단독 테스트 → 에러 발견!
6. 수정: 해싱 라이브러리 버전 업데이트

★ 가설 없이 이것저것 바꾸면 시간만 낭비된다
```

## 연습 문제

### Q1. "어제까지 되던 코드가 오늘 안 된다." 가장 먼저 확인해야 할 것은?

<details>
<summary>정답 보기</summary>
어제와 오늘 사이에 변경된 것을 확인한다. git diff나 git log로 코드 변경 확인, 라이브러리 업데이트 여부, 환경 설정(.env) 변경 여부, 데이터 변경 여부 순으로 점검.
</details>

### Q2. 이분법 디버깅의 장점은 무엇인가?

<details>
<summary>정답 보기</summary>
코드를 절반씩 나누어 확인하므로 O(log n) 시간에 버그 위치를 찾을 수 있다. 100줄 코드라면 최대 7번 확인으로 버그를 찾는다. 한 줄씩 확인하는 것(O(n))보다 훨씬 효율적.
</details>

### Q3. 러버덕 디버깅이 효과적인 이유는?

<details>
<summary>정답 보기</summary>
코드를 타인(또는 오리)에게 설명하는 과정에서 자신의 가정을 재검토하게 된다. "당연히 이렇게 동작할 것이다"라고 넘어갔던 부분에서 실제로는 다르게 동작하는 것을 발견하게 된다.
</details>
