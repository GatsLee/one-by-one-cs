## 핵심 개념

### 1. SOLID 원칙

SOLID는 로버트 C. 마틴(Uncle Bob)이 정리한 **객체지향 설계의 5대 원칙**이다. 유지보수가 쉽고 확장 가능한 소프트웨어를 만드는 데 핵심적이다.

---

#### S — 단일 책임 원칙 (Single Responsibility Principle)

> **클래스는 변경의 이유가 하나만 있어야 한다.**

하나의 클래스가 여러 역할을 수행하면, 한 역할의 변경이 다른 역할에 영향을 미친다.

```
[나쁜 예 — 여러 책임]
┌───────────────────────┐
│       User            │
│  - name               │
│  - email              │
│                       │
│  + save_to_db()       │  ← DB 저장 책임
│  + send_email()       │  ← 이메일 발송 책임
│  + generate_report()  │  ← 보고서 생성 책임
└───────────────────────┘
  DB 변경, 이메일 변경, 보고서 변경 → 모두 User 클래스 수정 필요

[좋은 예 — 단일 책임]
┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│    User      │  │ UserRepo     │  │ EmailService │
│  - name      │  │              │  │              │
│  - email     │  │ + save(user) │  │ + send(user) │
└──────────────┘  └──────────────┘  └──────────────┘
  각 클래스가 하나의 변경 이유만 가짐
```

```python
# === 나쁜 예: 여러 책임을 가진 클래스 ===

class User:
    def __init__(self, name: str, email: str):
        self.name = name
        self.email = email

    def save_to_db(self):
        """DB 저장 (데이터 영속성 책임)"""
        print(f"INSERT INTO users VALUES ('{self.name}', '{self.email}')")

    def send_welcome_email(self):
        """이메일 발송 (통신 책임)"""
        print(f"Sending email to {self.email}")

    def generate_report(self):
        """보고서 생성 (표현 책임)"""
        return f"Report for {self.name}"


# === 좋은 예: 단일 책임으로 분리 ===

class User:
    """사용자 데이터만 담당"""
    def __init__(self, name: str, email: str):
        self.name = name
        self.email = email


class UserRepository:
    """사용자 DB 저장만 담당"""
    def save(self, user: User):
        print(f"INSERT INTO users VALUES ('{user.name}', '{user.email}')")


class EmailService:
    """이메일 발송만 담당"""
    def send_welcome(self, user: User):
        print(f"Sending welcome email to {user.email}")


class ReportGenerator:
    """보고서 생성만 담당"""
    def generate(self, user: User) -> str:
        return f"Report for {user.name}"
```

#### O — 개방-폐쇄 원칙 (Open-Closed Principle)

> **확장에는 열려 있고, 수정에는 닫혀 있어야 한다.**

기존 코드를 수정하지 않고 새로운 기능을 추가할 수 있어야 한다.

```python
# === 나쁜 예: 할인 유형 추가마다 기존 코드 수정 ===

class DiscountCalculator:
    def calculate(self, price: float, discount_type: str) -> float:
        if discount_type == "percent":
            return price * 0.9
        elif discount_type == "fixed":
            return price - 1000
        elif discount_type == "vip":      # ← 새 유형 추가마다 여기를 수정!
            return price * 0.7
        # elif discount_type == "..."    # 계속 추가...


# === 좋은 예: 확장에 열려 있고 수정에 닫힌 설계 ===

from abc import ABC, abstractmethod

class DiscountStrategy(ABC):
    """할인 전략 인터페이스 — 확장 포인트"""
    @abstractmethod
    def apply(self, price: float) -> float:
        pass


class PercentDiscount(DiscountStrategy):
    def __init__(self, percent: float = 10):
        self.percent = percent

    def apply(self, price: float) -> float:
        return price * (1 - self.percent / 100)


class FixedDiscount(DiscountStrategy):
    def __init__(self, amount: float = 1000):
        self.amount = amount

    def apply(self, price: float) -> float:
        return max(0, price - self.amount)


class VIPDiscount(DiscountStrategy):
    """새 할인 유형을 추가해도 기존 코드를 수정하지 않음!"""
    def apply(self, price: float) -> float:
        return price * 0.7


class DiscountCalculator:
    """기존 코드 수정 없이 새 전략 사용 가능"""
    def calculate(self, price: float, strategy: DiscountStrategy) -> float:
        return strategy.apply(price)


# 사용
calc = DiscountCalculator()
print(calc.calculate(10000, PercentDiscount(20)))  # 8000
print(calc.calculate(10000, VIPDiscount()))         # 7000
```

#### L — 리스코프 치환 원칙 (Liskov Substitution Principle)

> **하위 클래스는 상위 클래스를 대체할 수 있어야 한다.**

부모 클래스를 사용하는 곳에 자식 클래스를 넣어도 프로그램이 올바르게 동작해야 한다.

```python
# === 나쁜 예: 리스코프 위반 ===

class Rectangle:
    def __init__(self, width: int, height: int):
        self._width = width
        self._height = height

    @property
    def width(self):
        return self._width

    @width.setter
    def width(self, value):
        self._width = value

    @property
    def height(self):
        return self._height

    @height.setter
    def height(self, value):
        self._height = value

    def area(self):
        return self._width * self._height


class Square(Rectangle):
    """정사각형은 직사각형의 특수 케이스이지만..."""
    def __init__(self, side: int):
        super().__init__(side, side)

    @Rectangle.width.setter
    def width(self, value):
        self._width = value
        self._height = value  # 정사각형이므로 높이도 변경!

    @Rectangle.height.setter
    def height(self, value):
        self._width = value   # 정사각형이므로 너비도 변경!
        self._height = value


# 문제 발생:
def process_rectangle(rect: Rectangle):
    rect.width = 5
    rect.height = 3
    assert rect.area() == 15  # Rectangle이면 통과, Square이면 실패!
    # Square: width=3, height=3 → area=9 (width 설정이 height에 의해 덮어써짐)


# === 좋은 예: 리스코프 준수 ===

from abc import ABC, abstractmethod

class Shape(ABC):
    @abstractmethod
    def area(self) -> float:
        pass

class Rectangle(Shape):
    def __init__(self, width: float, height: float):
        self.width = width
        self.height = height

    def area(self) -> float:
        return self.width * self.height

class Square(Shape):
    """Rectangle을 상속하지 않고 독립적인 Shape"""
    def __init__(self, side: float):
        self.side = side

    def area(self) -> float:
        return self.side ** 2
```

#### I — 인터페이스 분리 원칙 (Interface Segregation Principle)

> **클라이언트는 사용하지 않는 메서드에 의존하지 않아야 한다.**

하나의 범용 인터페이스보다 여러 개의 구체적인 인터페이스가 낫다.

```python
# === 나쁜 예: 뚱뚱한 인터페이스 ===

from abc import ABC, abstractmethod

class Worker(ABC):
    @abstractmethod
    def work(self): pass

    @abstractmethod
    def eat(self): pass

    @abstractmethod
    def sleep(self): pass


class Robot(Worker):
    def work(self):
        print("로봇 작업 중")

    def eat(self):
        raise NotImplementedError("로봇은 먹지 않습니다!")  # 불필요한 메서드!

    def sleep(self):
        raise NotImplementedError("로봇은 자지 않습니다!")  # 불필요한 메서드!


# === 좋은 예: 분리된 인터페이스 ===

class Workable(ABC):
    @abstractmethod
    def work(self): pass

class Eatable(ABC):
    @abstractmethod
    def eat(self): pass

class Sleepable(ABC):
    @abstractmethod
    def sleep(self): pass


class Human(Workable, Eatable, Sleepable):
    def work(self):
        print("사람 작업 중")

    def eat(self):
        print("점심 식사 중")

    def sleep(self):
        print("수면 중")


class Robot(Workable):  # 필요한 인터페이스만 구현
    def work(self):
        print("로봇 작업 중")

# Robot은 eat(), sleep()에 의존하지 않음!
```

#### D — 의존성 역전 원칙 (Dependency Inversion Principle)

> **고수준 모듈이 저수준 모듈에 의존하지 않아야 한다. 둘 다 추상화에 의존해야 한다.**

```
[나쁜 예: 고수준 → 저수준 직접 의존]
OrderService → MySQLDatabase
  └── DB를 PostgreSQL로 바꾸면 OrderService도 수정 필요!

[좋은 예: 둘 다 추상화에 의존]
OrderService → DatabaseInterface ← MySQLDatabase
                                 ← PostgreSQLDatabase
  └── DB를 바꿔도 OrderService는 수정 불필요!
```

```python
# === 나쁜 예: 구체 클래스에 직접 의존 ===

class MySQLDatabase:
    def query(self, sql: str):
        print(f"MySQL 실행: {sql}")


class OrderService:
    def __init__(self):
        self.db = MySQLDatabase()  # 구체 클래스에 직접 의존!

    def get_orders(self):
        self.db.query("SELECT * FROM orders")


# === 좋은 예: 추상화에 의존 ===

class Database(ABC):
    """추상 인터페이스"""
    @abstractmethod
    def query(self, sql: str): pass


class MySQLDatabase(Database):
    def query(self, sql: str):
        print(f"MySQL 실행: {sql}")


class PostgreSQLDatabase(Database):
    def query(self, sql: str):
        print(f"PostgreSQL 실행: {sql}")


class OrderService:
    def __init__(self, db: Database):  # 추상화에 의존 (의존성 주입)
        self.db = db

    def get_orders(self):
        self.db.query("SELECT * FROM orders")


# 사용: DB를 쉽게 교체 가능
service_mysql = OrderService(MySQLDatabase())
service_pg = OrderService(PostgreSQLDatabase())
```

### 2. 생성 패턴 (Creational Patterns)

#### Singleton 패턴

클래스의 인스턴스가 **오직 하나만** 존재하도록 보장한다.

```
[Singleton 구조]
┌─────────────────────┐
│     Singleton        │
│  - _instance = None  │
│                      │
│  + get_instance()    │──► 항상 같은 인스턴스 반환
└─────────────────────┘
```

```python
class DatabaseConnection:
    """
    Singleton 패턴: DB 연결을 하나만 유지.
    Python에서는 __new__를 오버라이드하여 구현.
    """
    _instance = None

    def __new__(cls):
        if cls._instance is None:
            cls._instance = super().__new__(cls)
            cls._instance._initialized = False
        return cls._instance

    def __init__(self):
        if not self._initialized:
            self.connection = "DB 연결 활성"
            self._initialized = True
            print("DB 연결 생성 (최초 1회)")

    def query(self, sql: str):
        print(f"실행: {sql}")


# 사용
db1 = DatabaseConnection()  # "DB 연결 생성"
db2 = DatabaseConnection()  # 출력 없음 (이미 생성됨)
print(db1 is db2)  # True — 같은 인스턴스!
```

**주의:** Singleton은 전역 상태를 만들어 테스트를 어렵게 하므로, 남용하지 않아야 한다.

#### Factory 패턴

객체 생성 로직을 **별도의 팩토리**에 위임한다. 클라이언트가 구체 클래스를 알 필요 없다.

```
[Factory 구조]
                    ┌──────────┐
클라이언트 ────────►│ Factory  │
  "pdf 만들어줘"    │          │
                    └────┬─────┘
                    ┌────┴────┐
               ┌────▼──┐  ┌──▼────┐
               │ PDF    │  │ DOCX  │
               │Exporter│  │Exporter│
               └────────┘  └───────┘
```

```python
from abc import ABC, abstractmethod

class Notification(ABC):
    @abstractmethod
    def send(self, message: str): pass


class EmailNotification(Notification):
    def send(self, message: str):
        print(f"[이메일] {message}")


class SMSNotification(Notification):
    def send(self, message: str):
        print(f"[SMS] {message}")


class PushNotification(Notification):
    def send(self, message: str):
        print(f"[푸시] {message}")


class NotificationFactory:
    """
    Factory 패턴: 알림 유형에 따라 적절한 객체를 생성.
    새 알림 유형을 추가해도 클라이언트 코드는 변경하지 않음.
    """
    @staticmethod
    def create(notification_type: str) -> Notification:
        factories = {
            "email": EmailNotification,
            "sms": SMSNotification,
            "push": PushNotification,
        }
        if notification_type not in factories:
            raise ValueError(f"알 수 없는 유형: {notification_type}")
        return factories[notification_type]()


# 사용: 구체 클래스를 직접 생성하지 않음
notification = NotificationFactory.create("email")
notification.send("안녕하세요!")

notification = NotificationFactory.create("sms")
notification.send("알림입니다!")
```

#### Builder 패턴

복잡한 객체를 **단계별로** 생성한다. 생성자에 매개변수가 많을 때 유용하다.

```python
class HTTPRequest:
    """Builder 패턴으로 생성되는 복잡한 객체"""
    def __init__(self):
        self.method = "GET"
        self.url = ""
        self.headers = {}
        self.body = None
        self.timeout = 30

    def __repr__(self):
        return (f"HTTPRequest(method={self.method}, url={self.url}, "
                f"headers={self.headers}, body={self.body}, timeout={self.timeout})")


class HTTPRequestBuilder:
    """
    Builder 패턴: 복잡한 HTTP 요청을 단계별로 구성.
    메서드 체이닝(Fluent Interface)으로 가독성 향상.
    """
    def __init__(self):
        self._request = HTTPRequest()

    def method(self, method: str) -> 'HTTPRequestBuilder':
        self._request.method = method
        return self  # 자기 자신 반환 → 체이닝 가능

    def url(self, url: str) -> 'HTTPRequestBuilder':
        self._request.url = url
        return self

    def header(self, key: str, value: str) -> 'HTTPRequestBuilder':
        self._request.headers[key] = value
        return self

    def body(self, body: str) -> 'HTTPRequestBuilder':
        self._request.body = body
        return self

    def timeout(self, seconds: int) -> 'HTTPRequestBuilder':
        self._request.timeout = seconds
        return self

    def build(self) -> HTTPRequest:
        """최종 객체 반환"""
        if not self._request.url:
            raise ValueError("URL은 필수입니다")
        return self._request


# 사용: 메서드 체이닝으로 가독성 좋게 객체 생성
request = (HTTPRequestBuilder()
    .method("POST")
    .url("https://api.example.com/users")
    .header("Content-Type", "application/json")
    .header("Authorization", "Bearer token123")
    .body('{"name": "홍길동"}')
    .timeout(10)
    .build())

print(request)
```

### 3. 구조 패턴 (Structural Patterns)

#### Adapter 패턴

호환되지 않는 인터페이스를 연결하여 함께 동작하게 한다.

```
[Adapter 비유: 110V 콘센트에 220V 기기 연결]

220V 기기 → [변환 어댑터] → 110V 콘센트

[코드에서]
새 시스템 → [Adapter] → 레거시 시스템
```

```python
class LegacyPrinter:
    """기존 시스템: print_text 메서드 사용"""
    def print_text(self, text: str):
        print(f"[레거시] {text}")


class ModernPrinter:
    """새 시스템: render 메서드 사용"""
    def render(self, content: str):
        print(f"[모던] {content}")


class PrinterAdapter:
    """
    Adapter 패턴: ModernPrinter의 인터페이스를
    LegacyPrinter의 인터페이스로 변환.
    """
    def __init__(self, modern_printer: ModernPrinter):
        self.modern_printer = modern_printer

    def print_text(self, text: str):
        """레거시 인터페이스로 호출하면 모던 메서드를 실행"""
        self.modern_printer.render(text)


# 사용: 레거시 코드에서 모던 프린터를 사용
def legacy_system(printer):
    """이 함수는 print_text() 메서드를 기대"""
    printer.print_text("Hello World")

legacy_system(LegacyPrinter())                    # 레거시 프린터 직접 사용
legacy_system(PrinterAdapter(ModernPrinter()))     # 어댑터로 모던 프린터 사용
```

#### Decorator 패턴

객체에 **동적으로 기능을 추가**한다. 상속 없이 기능을 확장한다.

```
[Decorator 구조: 커피 주문]

기본 커피 → [우유 추가] → [시럽 추가] → [휘핑 추가]
  3000원      +500원        +300원       +700원
  = 4500원

각 데코레이터가 원본을 감싸면서 기능/가격을 추가
```

```python
class Notifier(ABC):
    @abstractmethod
    def send(self, message: str): pass

    @abstractmethod
    def cost(self) -> float: pass


class BasicNotifier(Notifier):
    """기본 알림 (이메일)"""
    def send(self, message: str):
        print(f"[이메일] {message}")

    def cost(self) -> float:
        return 0  # 이메일은 무료


class NotifierDecorator(Notifier):
    """데코레이터 기본 클래스"""
    def __init__(self, wrapped: Notifier):
        self._wrapped = wrapped

    def send(self, message: str):
        self._wrapped.send(message)

    def cost(self) -> float:
        return self._wrapped.cost()


class SMSDecorator(NotifierDecorator):
    """SMS 기능 추가 데코레이터"""
    def send(self, message: str):
        super().send(message)
        print(f"[SMS 추가] {message}")

    def cost(self) -> float:
        return super().cost() + 10  # SMS 비용 추가


class SlackDecorator(NotifierDecorator):
    """Slack 기능 추가 데코레이터"""
    def send(self, message: str):
        super().send(message)
        print(f"[Slack 추가] {message}")

    def cost(self) -> float:
        return super().cost() + 5  # Slack 비용 추가


# 사용: 데코레이터를 조합하여 기능 확장
notifier = BasicNotifier()                     # 이메일만
notifier = SMSDecorator(notifier)              # + SMS
notifier = SlackDecorator(notifier)            # + Slack

notifier.send("서버 장애 발생!")
print(f"총 비용: {notifier.cost()}원")
# 출력: [이메일] → [SMS 추가] → [Slack 추가], 비용 15원
```

### 4. 행위 패턴 (Behavioral Patterns)

#### Strategy 패턴

알고리즘을 **교체 가능한 전략 객체**로 캡슐화한다. OCP의 대표적 구현이다.

```python
class SortStrategy(ABC):
    @abstractmethod
    def sort(self, data: list) -> list:
        pass

class BubbleSort(SortStrategy):
    def sort(self, data: list) -> list:
        arr = data.copy()
        n = len(arr)
        for i in range(n):
            for j in range(0, n-i-1):
                if arr[j] > arr[j+1]:
                    arr[j], arr[j+1] = arr[j+1], arr[j]
        print("[BubbleSort] 정렬 완료")
        return arr

class QuickSort(SortStrategy):
    def sort(self, data: list) -> list:
        if len(data) <= 1:
            return data
        pivot = data[len(data) // 2]
        left = [x for x in data if x < pivot]
        middle = [x for x in data if x == pivot]
        right = [x for x in data if x > pivot]
        result = self.sort(left) + middle + self.sort(right)
        return result

class TimSort(SortStrategy):
    def sort(self, data: list) -> list:
        print("[TimSort] 정렬 완료")
        return sorted(data)  # Python 내장 정렬은 TimSort


class Sorter:
    """
    Strategy 패턴: 정렬 알고리즘을 런타임에 교체 가능.
    데이터 크기에 따라 다른 전략을 선택할 수 있다.
    """
    def __init__(self, strategy: SortStrategy):
        self._strategy = strategy

    def set_strategy(self, strategy: SortStrategy):
        """전략을 런타임에 변경"""
        self._strategy = strategy

    def sort(self, data: list) -> list:
        return self._strategy.sort(data)


# 사용
data = [64, 34, 25, 12, 22, 11, 90]

sorter = Sorter(BubbleSort())
print(sorter.sort(data))

# 데이터가 크면 QuickSort로 전략 변경
sorter.set_strategy(QuickSort())
print(sorter.sort(data))
```

#### Observer 패턴

객체의 상태 변화를 **여러 관찰자에게 자동 통보**한다. 이벤트 시스템의 기반이다.

```
[Observer 구조]

┌──────────────┐     알림     ┌───────────┐
│   Subject    │────────────►│ Observer A │
│  (발행자)     │             │ (구독자)   │
│              │────────────►│ Observer B │
│  상태 변경!   │             │ ...        │
└──────────────┘             └───────────┘
```

```python
from typing import List, Callable

class EventEmitter:
    """
    Observer 패턴: 이벤트 기반 알림 시스템.
    이벤트가 발생하면 등록된 모든 리스너에게 알림.
    """
    def __init__(self):
        self._listeners: dict[str, List[Callable]] = {}

    def on(self, event: str, callback: Callable):
        """이벤트 구독"""
        if event not in self._listeners:
            self._listeners[event] = []
        self._listeners[event].append(callback)
        return self  # 체이닝 가능

    def off(self, event: str, callback: Callable):
        """이벤트 구독 해제"""
        if event in self._listeners:
            self._listeners[event].remove(callback)

    def emit(self, event: str, data=None):
        """이벤트 발생 → 모든 리스너에게 알림"""
        if event in self._listeners:
            for callback in self._listeners[event]:
                callback(data)


class OrderSystem:
    """주문 시스템: Observer 패턴 적용"""
    def __init__(self):
        self.events = EventEmitter()

    def create_order(self, order_id: str, amount: float):
        print(f"\n[주문 생성] #{order_id}, {amount:,.0f}원")

        # 이벤트 발행 → 구독자들이 알아서 처리
        self.events.emit("order_created", {
            "order_id": order_id,
            "amount": amount
        })


# === 관찰자(구독자) 정의 ===

def send_confirmation_email(data):
    """이메일 확인 알림"""
    print(f"  [이메일] 주문 #{data['order_id']} 확인 메일 발송")

def update_inventory(data):
    """재고 차감"""
    print(f"  [재고] 주문 #{data['order_id']} 재고 업데이트")

def log_order(data):
    """주문 로깅"""
    print(f"  [로그] 주문 #{data['order_id']}, {data['amount']:,.0f}원 기록")

def notify_analytics(data):
    """분석 시스템에 데이터 전송"""
    print(f"  [분석] 주문 #{data['order_id']} 분석 데이터 전송")


# 사용
shop = OrderSystem()

# 구독자 등록 — 느슨한 결합!
shop.events.on("order_created", send_confirmation_email)
shop.events.on("order_created", update_inventory)
shop.events.on("order_created", log_order)
shop.events.on("order_created", notify_analytics)

# 주문 생성 → 모든 구독자에게 자동 알림
shop.create_order("ORD-001", 50000)
shop.create_order("ORD-002", 120000)
```

## 코드 예시

위의 각 섹션에 이미 Python 코드 예시가 포함되어 있다. 아래는 여러 패턴을 조합한 실전 예시다.

### 실전 예시: 결제 시스템 (여러 패턴 조합)

```python
from abc import ABC, abstractmethod
from typing import List, Callable
from dataclasses import dataclass

# === Domain Model ===

@dataclass
class Payment:
    """결제 정보"""
    amount: float
    method: str
    status: str = "pending"


# === Strategy 패턴: 결제 수단 ===

class PaymentMethod(ABC):
    @abstractmethod
    def process(self, amount: float) -> bool:
        pass

class CreditCardPayment(PaymentMethod):
    def process(self, amount: float) -> bool:
        print(f"  [신용카드] {amount:,.0f}원 결제 처리")
        return True

class BankTransfer(PaymentMethod):
    def process(self, amount: float) -> bool:
        print(f"  [계좌이체] {amount:,.0f}원 이체 처리")
        return True

class KakaoPay(PaymentMethod):
    def process(self, amount: float) -> bool:
        print(f"  [카카오페이] {amount:,.0f}원 결제 처리")
        return True


# === Factory 패턴: 결제 수단 생성 ===

class PaymentMethodFactory:
    _methods = {
        "card": CreditCardPayment,
        "bank": BankTransfer,
        "kakao": KakaoPay,
    }

    @classmethod
    def create(cls, method_type: str) -> PaymentMethod:
        if method_type not in cls._methods:
            raise ValueError(f"지원하지 않는 결제 수단: {method_type}")
        return cls._methods[method_type]()


# === Observer 패턴: 결제 이벤트 ===

class PaymentEventEmitter:
    def __init__(self):
        self._handlers: dict[str, List[Callable]] = {}

    def on(self, event: str, handler: Callable):
        self._handlers.setdefault(event, []).append(handler)

    def emit(self, event: str, payment: Payment):
        for handler in self._handlers.get(event, []):
            handler(payment)


# === Decorator 패턴: 결제 검증 체인 ===

class PaymentValidator(ABC):
    @abstractmethod
    def validate(self, payment: Payment) -> bool:
        pass

class BaseValidator(PaymentValidator):
    def validate(self, payment: Payment) -> bool:
        return True

class AmountValidator(PaymentValidator):
    """금액 검증 데코레이터"""
    def __init__(self, wrapped: PaymentValidator, max_amount: float = 1_000_000):
        self._wrapped = wrapped
        self._max_amount = max_amount

    def validate(self, payment: Payment) -> bool:
        if not self._wrapped.validate(payment):
            return False
        if payment.amount > self._max_amount:
            print(f"  [검증 실패] 최대 금액 {self._max_amount:,.0f}원 초과")
            return False
        if payment.amount <= 0:
            print(f"  [검증 실패] 금액은 0보다 커야 합니다")
            return False
        print(f"  [검증 통과] 금액: {payment.amount:,.0f}원")
        return True

class FraudValidator(PaymentValidator):
    """사기 검증 데코레이터"""
    def __init__(self, wrapped: PaymentValidator):
        self._wrapped = wrapped

    def validate(self, payment: Payment) -> bool:
        if not self._wrapped.validate(payment):
            return False
        # 간단한 사기 탐지 로직
        is_suspicious = payment.amount > 500_000
        if is_suspicious:
            print(f"  [사기 검증] 고액 결제 — 추가 인증 필요")
        else:
            print(f"  [사기 검증] 정상")
        return True


# === 메인 서비스 (SRP + DIP) ===

class PaymentService:
    """
    결제 서비스: 여러 패턴을 조합하여 유연하고 확장 가능한 설계.
    - Strategy: 결제 수단 교체
    - Factory: 결제 수단 생성
    - Observer: 결제 이벤트 처리
    - Decorator: 검증 체인
    - DIP: 추상화에 의존
    """

    def __init__(self, validator: PaymentValidator, events: PaymentEventEmitter):
        self.validator = validator
        self.events = events

    def process_payment(self, amount: float, method_type: str) -> Payment:
        payment = Payment(amount=amount, method=method_type)

        print(f"\n{'='*40}")
        print(f"결제 처리: {amount:,.0f}원 ({method_type})")
        print(f"{'='*40}")

        # 1. 검증 (Decorator 체인)
        if not self.validator.validate(payment):
            payment.status = "rejected"
            self.events.emit("payment_rejected", payment)
            return payment

        # 2. 결제 수단 생성 (Factory)
        method = PaymentMethodFactory.create(method_type)

        # 3. 결제 처리 (Strategy)
        success = method.process(amount)

        if success:
            payment.status = "completed"
            self.events.emit("payment_completed", payment)
        else:
            payment.status = "failed"
            self.events.emit("payment_failed", payment)

        return payment


# === 실행 ===

def main():
    # 검증 체인 구성 (Decorator)
    validator = BaseValidator()
    validator = AmountValidator(validator, max_amount=1_000_000)
    validator = FraudValidator(validator)

    # 이벤트 핸들러 등록 (Observer)
    events = PaymentEventEmitter()
    events.on("payment_completed", lambda p: print(f"  [알림] 결제 완료: {p.amount:,.0f}원"))
    events.on("payment_completed", lambda p: print(f"  [로그] 결제 기록 저장"))
    events.on("payment_rejected", lambda p: print(f"  [알림] 결제 거절됨"))

    # 서비스 생성 (DIP — 추상화에 의존)
    service = PaymentService(validator, events)

    # 결제 실행
    service.process_payment(30000, "card")
    service.process_payment(750000, "kakao")
    service.process_payment(1500000, "bank")  # 한도 초과


if __name__ == "__main__":
    main()
```

## 정리

| 개념 | 핵심 내용 | 적용 상황 |
|------|-----------|-----------|
| **SRP** | 클래스는 변경 이유가 하나뿐 | 하나의 클래스가 여러 역할 수행 시 |
| **OCP** | 확장에 열림, 수정에 닫힘 | 새 기능 추가 시 기존 코드 수정 필요할 때 |
| **LSP** | 하위 클래스는 상위를 대체 가능 | 상속 관계 설계 시 |
| **ISP** | 클라이언트에 불필요한 인터페이스 노출 금지 | 거대한 인터페이스를 구현할 때 |
| **DIP** | 고수준이 저수준에 의존하지 않음 | 구체 클래스에 직접 의존할 때 |
| **Singleton** | 인스턴스가 오직 하나만 존재 | DB 연결, 설정 관리 등 |
| **Factory** | 객체 생성을 별도 클래스에 위임 | 조건에 따라 다른 객체 생성 시 |
| **Builder** | 복잡한 객체를 단계별 생성 | 생성자 매개변수가 많을 때 |
| **Adapter** | 호환되지 않는 인터페이스 연결 | 레거시 시스템 통합 |
| **Decorator** | 동적으로 기능 추가 | 기능 조합이 다양할 때 |
| **Strategy** | 알고리즘을 교체 가능한 객체로 | 런타임에 알고리즘 변경 필요 시 |
| **Observer** | 상태 변화 자동 통보 | 이벤트/알림 시스템 |
| **단위 테스트** | 함수/클래스 단위 독립 검증 | 빠르고 많이, Mock 사용 |
| **통합 테스트** | 모듈 간 상호작용 검증 | 실제 또는 인메모리 DB 사용 |
| **E2E 테스트** | 전체 시스템 흐름 검증 | 실제 API + DB + UI |
