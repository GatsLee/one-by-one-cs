## 핵심 개념

**ER(Entity-Relationship) 모델**은 데이터베이스 설계의 첫 단계인 **개념적 데이터 모델링**에서 사용하는 도구다.
현실 세계의 데이터를 엔티티, 속성, 관계로 추상화하여 데이터베이스 스키마를 설계하기 전에
전체 구조를 시각적으로 파악할 수 있게 해준다.

Peter Chen이 1976년에 제안한 이 모델은 오늘날까지 데이터베이스 설계의 표준 도구로 사용된다.

## 엔티티(Entity)와 엔티티 집합

### 엔티티란?

**엔티티(Entity)**는 현실 세계에서 독립적으로 존재하며 식별 가능한 객체를 말한다.
사람, 장소, 사물, 사건 등 데이터로 표현할 수 있는 모든 것이 엔티티가 될 수 있다.

**엔티티 집합(Entity Set)**은 동일한 속성을 공유하는 엔티티들의 모임이다.
예를 들어, '학생'이라는 엔티티 집합에는 개별 학생 엔티티들이 포함된다.

### 강한 엔티티 vs 약한 엔티티

| 구분 | 강한 엔티티(Strong Entity) | 약한 엔티티(Weak Entity) |
|------|---------------------------|-------------------------|
| 정의 | 자체 키 속성으로 독립 식별 가능 | 다른 엔티티에 의존하여 식별 |
| 키 | 자체 기본키 보유 | 부분키 + 소유 엔티티의 키 |
| 존재 | 독립적 존재 | 소유 엔티티 없이 존재 불가 |
| ER 표기 | 단일 직사각형 | 이중 직사각형 |
| 예시 | 학생, 과목, 교수 | 부양가족(직원에 의존) |

```
  강한 엔티티              약한 엔티티

  ┌─────────┐           ╔═════════╗
  │  학 생   │           ║ 부양가족  ║
  └─────────┘           ╚═════════╝
                              │
                        소유 엔티티(직원)에
                        의존하여 식별
```

약한 엔티티는 반드시 **식별 관계(Identifying Relationship)**를 통해
소유 엔티티(Owner Entity)와 연결되어야 한다.

```sql
-- 강한 엔티티: 직원 테이블
CREATE TABLE 직원 (
    직원번호 INT PRIMARY KEY,        -- 자체 기본키
    이름 VARCHAR(50) NOT NULL,
    부서 VARCHAR(30)
);

-- 약한 엔티티: 부양가족 테이블
CREATE TABLE 부양가족 (
    직원번호 INT,                     -- 소유 엔티티의 키
    부양가족이름 VARCHAR(50),          -- 부분키(Partial Key)
    관계 VARCHAR(20),
    생년월일 DATE,
    PRIMARY KEY (직원번호, 부양가족이름),  -- 복합키로 식별
    FOREIGN KEY (직원번호) REFERENCES 직원(직원번호)
        ON DELETE CASCADE              -- 소유 엔티티 삭제 시 함께 삭제
);
```

## 속성(Attribute)

속성은 엔티티나 관계의 특성을 나타내는 데이터 항목이다.

### 속성의 분류

<Mermaid chart="
graph TD
    A[&quot;속성(Attribute)&quot;] --> B[&quot;구조 기준&quot;]
    A --> C[&quot;값 개수 기준&quot;]
    A --> D[&quot;파생 여부&quot;]
    B --> B1[&quot;단순속성&quot;]
    B --> B2[&quot;복합속성&quot;]
    C --> C1[&quot;단일값&quot;]
    C --> C2[&quot;다중값&quot;]
    D --> D1[&quot;저장속성&quot;]
    D --> D2[&quot;유도속성&quot;]
" />

#### 1. 단순 속성 vs 복합 속성

| 구분 | 단순 속성(Simple) | 복합 속성(Composite) |
|------|-------------------|---------------------|
| 정의 | 더 이상 분해 불가 | 하위 속성으로 분해 가능 |
| 예시 | 학번, 나이 | 주소(시, 구, 동), 이름(성, 이름) |
| 저장 | 단일 컬럼 | 하위 속성별 컬럼 또는 단일 컬럼 |

<Mermaid chart="
graph TD
    A[&quot;주소&quot;] --> B[&quot;시&quot;]
    A --> C[&quot;구&quot;]
    A --> D[&quot;동&quot;]
" />

#### 2. 단일값 속성 vs 다중값 속성

| 구분 | 단일값(Single-valued) | 다중값(Multi-valued) |
|------|----------------------|---------------------|
| 정의 | 하나의 값만 가짐 | 여러 값을 가질 수 있음 |
| 예시 | 학번, 이름 | 전화번호, 취미 |
| ER 표기 | 단일 타원 | 이중 타원 |
| 매핑 | 컬럼으로 직접 매핑 | 별도 테이블로 분리 |

```sql
-- 다중값 속성은 별도 테이블로 분리
CREATE TABLE 학생 (
    학번 INT PRIMARY KEY,
    이름 VARCHAR(50)
);

-- 다중값 속성: 전화번호 (여러 개 가능)
CREATE TABLE 학생_전화번호 (
    학번 INT,
    전화번호 VARCHAR(15),
    PRIMARY KEY (학번, 전화번호),
    FOREIGN KEY (학번) REFERENCES 학생(학번)
);
```

#### 3. 저장 속성 vs 유도 속성

| 구분 | 저장 속성(Stored) | 유도 속성(Derived) |
|------|-------------------|-------------------|
| 정의 | 직접 저장되는 값 | 다른 속성에서 계산 가능 |
| 예시 | 생년월일 | 나이(생년월일로 계산) |
| ER 표기 | 단일 타원 | 점선 타원 |
| 저장 여부 | DB에 저장 | 보통 저장하지 않음 |

#### 4. 키 속성(Key Attribute)

엔티티를 유일하게 식별할 수 있는 속성이다.
ER 다이어그램에서 **밑줄**로 표기한다.

```
         ┌──────────────────┐
         │      학 생        │
         └──────────────────┘
          /    |     |    \
      (학번)  (이름) (학과) (학년)
       ~~~~
     키 속성(밑줄)
```

## 관계(Relationship)

### 관계와 관계 집합

**관계(Relationship)**는 두 개 이상의 엔티티 사이의 의미 있는 연관을 나타낸다.
**관계 집합(Relationship Set)**은 동일한 유형의 관계들의 모임이다.

<Mermaid chart="
erDiagram
    STUDENT[&quot;학생&quot;] }o--o{ COURSE[&quot;과목&quot;] : &quot;수강&quot;
" />

### 관계의 차수(Degree)

| 차수 | 이름 | 설명 | 예시 |
|------|------|------|------|
| 1 | 단항(Unary) | 같은 엔티티 집합 내 관계 | 직원 - 관리 - 직원 |
| 2 | 이항(Binary) | 두 엔티티 집합 간 관계 | 학생 - 수강 - 과목 |
| 3 | 삼항(Ternary) | 세 엔티티 집합 간 관계 | 공급자-부품-프로젝트 |

### 역할(Role)

같은 엔티티 집합이 하나의 관계에 두 번 이상 참여할 때, 각 참여를 구분하기 위해 역할을 부여한다.

<Mermaid chart="
erDiagram
    EMPLOYEE[&quot;직원&quot;] ||--o{ EMPLOYEE : &quot;관리(상사-부하)&quot;
" />

```sql
-- 단항 관계: 직원이 다른 직원을 관리
CREATE TABLE 직원 (
    직원번호 INT PRIMARY KEY,
    이름 VARCHAR(50),
    상사번호 INT,                -- 같은 테이블을 참조 (자기 참조)
    FOREIGN KEY (상사번호) REFERENCES 직원(직원번호)
);
```

### 참여 제약(Participation Constraint)

| 유형 | 설명 | ER 표기 | 예시 |
|------|------|---------|------|
| 전체 참여(Total) | 모든 엔티티가 관계에 참여 | 이중선 | 모든 학생은 반드시 학과에 소속 |
| 부분 참여(Partial) | 일부 엔티티만 관계에 참여 | 단일선 | 일부 교수만 연구 프로젝트 참여 |

## 카디널리티(Cardinality)

카디널리티는 관계에 참여하는 엔티티의 수적 비율을 나타낸다.

### 1:1 관계 (일대일)

한 엔티티가 다른 엔티티 하나와만 연관된다.

<Mermaid chart="
erDiagram
    EMPLOYEE[&quot;직원&quot;] ||--|| DEPARTMENT[&quot;부서&quot;] : &quot;관리(1:1)&quot;
" />

```sql
-- 1:1 관계: 전체 참여 쪽에 FK 배치
CREATE TABLE 부서 (
    부서번호 INT PRIMARY KEY,
    부서명 VARCHAR(50)
);

CREATE TABLE 직원 (
    직원번호 INT PRIMARY KEY,
    이름 VARCHAR(50),
    관리부서번호 INT UNIQUE,          -- UNIQUE로 1:1 보장
    FOREIGN KEY (관리부서번호) REFERENCES 부서(부서번호)
);
```

### 1:N 관계 (일대다)

한 엔티티가 여러 엔티티와 연관된다.

<Mermaid chart="
erDiagram
    DEPARTMENT[&quot;학과&quot;] ||--o{ STUDENT[&quot;학생&quot;] : &quot;소속(1:N)&quot;
" />

```sql
-- 1:N 관계: N쪽에 FK 배치
CREATE TABLE 학과 (
    학과번호 INT PRIMARY KEY,
    학과명 VARCHAR(50)
);

CREATE TABLE 학생 (
    학번 INT PRIMARY KEY,
    이름 VARCHAR(50),
    학과번호 INT,                     -- N쪽에 FK
    FOREIGN KEY (학과번호) REFERENCES 학과(학과번호)
);
```

### M:N 관계 (다대다)

양쪽 모두 여러 엔티티와 연관된다.

<Mermaid chart="
erDiagram
    STUDENT[&quot;학생&quot;] }o--o{ COURSE[&quot;과목&quot;] : &quot;수강(M:N)&quot;
" />

```sql
-- M:N 관계: 연결 테이블(Junction Table) 생성
CREATE TABLE 학생 (
    학번 INT PRIMARY KEY,
    이름 VARCHAR(50)
);

CREATE TABLE 과목 (
    과목번호 INT PRIMARY KEY,
    과목명 VARCHAR(50)
);

-- 연결 테이블: 관계 자체가 테이블이 됨
CREATE TABLE 수강 (
    학번 INT,
    과목번호 INT,
    성적 CHAR(2),                     -- 관계의 속성
    수강학기 VARCHAR(10),
    PRIMARY KEY (학번, 과목번호),
    FOREIGN KEY (학번) REFERENCES 학생(학번),
    FOREIGN KEY (과목번호) REFERENCES 과목(과목번호)
);
```

### 카디널리티 비교 요약

| 카디널리티 | FK 배치 | 테이블 수 변화 | 예시 |
|-----------|---------|---------------|------|
| 1:1 | 어느 쪽이든 (전체 참여 쪽 선호) | 변화 없음 | 직원-부서관리 |
| 1:N | N쪽에 FK | 변화 없음 | 학과-학생 |
| M:N | 연결 테이블 생성 | +1 테이블 | 학생-수강-과목 |

## ER 다이어그램 표기법

### 기본 기호

```
  ┌─────────┐
  │ 엔티티   │    직사각형: 강한 엔티티
  └─────────┘

  ╔═════════╗
  ║ 약한엔티티║    이중 직사각형: 약한 엔티티
  ╚═════════╝

      ◇          다이아몬드: 관계
    관 계

     ( )         타원: 속성
    속 성

    (( ))        이중 타원: 다중값 속성

    (. .)        점선 타원: 유도 속성

    (속성)        밑줄: 키 속성
     ~~~~
```

### 선의 종류

```
  ────────     단일선: 부분 참여
  ════════     이중선: 전체 참여
  ─ ─ ─ ─     점선: 유도 속성 연결
```

### 카디널리티 표기

```
  (min, max) 표기법:

  ┌─────┐  (0,N)     ◇     (1,1)  ┌─────┐
  │학 생 │─────── 소속 ──────────│학 과 │
  └─────┘                        └─────┘

  학생: 최소 0, 최대 N개의 학과 (부분 참여)
  학과: 최소 1, 최대 1명의 학과 소속 (전체 참여)
```

## 일반화/특수화(Generalization/Specialization)

### IS-A 관계

**일반화(Generalization)**는 하위 엔티티들의 공통 속성을 추출하여 상위 엔티티를 만드는 상향식 접근이다.
**특수화(Specialization)**는 상위 엔티티를 하위 엔티티로 세분화하는 하향식 접근이다.

<Mermaid chart="
graph TD
    PERSON[&quot;사람
이름, 생년월일&quot;] --> ISA{&quot;IS-A (d)&quot;}
    ISA --> STUDENT[&quot;학생
학번, 학과&quot;]
    ISA --> EMPLOYEE[&quot;직원
직원번호, 급여&quot;]
" />

### 분리(Disjoint) vs 중복(Overlapping)

| 제약 | 설명 | 표기 | 예시 |
|------|------|------|------|
| 분리(Disjoint) | 하위 엔티티 중 하나에만 소속 | d | 사람은 학생 또는 직원 |
| 중복(Overlapping) | 여러 하위 엔티티에 동시 소속 가능 | o | 사람이 학생이면서 직원 |

### 완전(Total) vs 부분(Partial)

| 제약 | 설명 | 예시 |
|------|------|------|
| 완전 특수화 | 상위 엔티티는 반드시 하위 중 하나 | 모든 사람은 학생 또는 직원 |
| 부분 특수화 | 상위 엔티티만으로 존재 가능 | 학생도 직원도 아닌 사람 존재 |

```sql
-- 일반화/특수화의 릴레이션 매핑 방법들

-- 방법 1: 각 엔티티를 별도 테이블로 (가장 일반적)
CREATE TABLE 사람 (
    사람ID INT PRIMARY KEY,
    이름 VARCHAR(50),
    생년월일 DATE
);

CREATE TABLE 학생 (
    사람ID INT PRIMARY KEY,
    학번 VARCHAR(10) UNIQUE,
    학과 VARCHAR(30),
    FOREIGN KEY (사람ID) REFERENCES 사람(사람ID)
);

CREATE TABLE 직원 (
    사람ID INT PRIMARY KEY,
    직원번호 VARCHAR(10) UNIQUE,
    급여 DECIMAL(10,2),
    FOREIGN KEY (사람ID) REFERENCES 사람(사람ID)
);

-- 방법 2: 단일 테이블에 타입 컬럼 (간단하지만 NULL 많음)
CREATE TABLE 사람_통합 (
    사람ID INT PRIMARY KEY,
    이름 VARCHAR(50),
    생년월일 DATE,
    타입 VARCHAR(10),              -- '학생', '직원', '둘다'
    학번 VARCHAR(10),              -- 학생만 해당
    학과 VARCHAR(30),              -- 학생만 해당
    직원번호 VARCHAR(10),           -- 직원만 해당
    급여 DECIMAL(10,2)             -- 직원만 해당
);
```

## ER &rarr; 릴레이션 매핑 규칙

ER 모델을 관계형 데이터베이스 스키마로 변환하는 7가지 규칙:

### 규칙 1: 강한 엔티티 &rarr; 테이블

각 강한 엔티티 집합은 하나의 테이블로 매핑된다.
엔티티의 단순 속성은 테이블의 컬럼이 되고, 키 속성은 기본키가 된다.

### 규칙 2: 약한 엔티티 &rarr; 복합키 테이블

약한 엔티티는 자신의 부분키와 소유 엔티티의 기본키를 합쳐서 복합 기본키로 사용한다.

### 규칙 3: 1:1 관계 &rarr; FK 포함

전체 참여 쪽 테이블에 상대방의 기본키를 외래키로 추가한다.

### 규칙 4: 1:N 관계 &rarr; N쪽에 FK

N쪽 엔티티의 테이블에 1쪽 엔티티의 기본키를 외래키로 추가한다.

### 규칙 5: M:N 관계 &rarr; 연결 테이블

새로운 연결 테이블을 생성하고, 양쪽 엔티티의 기본키를 복합 기본키로 사용한다.

### 규칙 6: 다중값 속성 &rarr; 별도 테이블

다중값 속성은 별도의 테이블로 분리하고, 원래 엔티티의 기본키를 외래키로 포함한다.

### 규칙 7: 복합 속성 &rarr; 하위 속성을 컬럼으로

복합 속성은 하위 속성 각각을 별도의 컬럼으로 매핑한다.

**ER → 릴레이션 매핑 요약:**

| ER 요소 | 릴레이션 매핑 |
|---------|--------------|
| 강한 엔티티 | 테이블 (키 속성 → PK) |
| 약한 엔티티 | 테이블 (부분키 + 소유자 PK → 복합 PK) |
| 1:1 관계 | 전체 참여 쪽에 FK 추가 |
| 1:N 관계 | N쪽 테이블에 FK 추가 |
| M:N 관계 | 연결 테이블 생성 |
| 다중값 속성 | 별도 테이블 생성 |
| 복합 속성 | 하위 속성을 각각 컬럼으로 |
| 유도 속성 | 보통 매핑하지 않음 (계산으로 처리) |

### 요구사항

- 학생은 학번, 이름, 학과에 소속
- 교수는 교번, 이름, 전공 분야를 가짐
- 과목은 과목번호, 과목명, 학점을 가짐
- 학생은 여러 과목을 수강하며 성적을 받음

- 각 학과에는 한 명의 학과장(교수)이 있음

### SQL 구현

```sql
-- 1. 학과 테이블 (강한 엔티티)
CREATE TABLE 학과 (
    학과번호 INT PRIMARY KEY,
    학과명 VARCHAR(50) NOT NULL
);

-- 2. 교수 테이블 (강한 엔티티)
CREATE TABLE 교수 (
    교번 INT PRIMARY KEY,
    이름 VARCHAR(50) NOT NULL,
    전공분야 VARCHAR(50),
    소속학과번호 INT,                  -- 1:N (학과:교수)
    FOREIGN KEY (소속학과번호) REFERENCES 학과(학과번호)
);

-- 3. 학과장 관계 (1:1) - 학과 테이블에 FK 추가
ALTER TABLE 학과
ADD COLUMN 학과장교번 INT UNIQUE,      -- UNIQUE로 1:1 보장
ADD FOREIGN KEY (학과장교번) REFERENCES 교수(교번);

-- 4. 학생 테이블 (강한 엔티티)
CREATE TABLE 학생 (
    학번 INT PRIMARY KEY,
    이름 VARCHAR(50) NOT NULL,
    학년 INT CHECK (학년 BETWEEN 1 AND 4),
    학과번호 INT NOT NULL,             -- 1:N (학과:학생), 전체 참여
    FOREIGN KEY (학과번호) REFERENCES 학과(학과번호)
);

-- 5. 과목 테이블 (강한 엔티티)
CREATE TABLE 과목 (
    과목번호 INT PRIMARY KEY,
    과목명 VARCHAR(100) NOT NULL,
    학점 INT CHECK (학점 BETWEEN 1 AND 4),
    담당교번 INT,                      -- 1:N (교수:과목)
    FOREIGN KEY (담당교번) REFERENCES 교수(교번)
);

-- 6. 수강 테이블 (M:N 연결 테이블)
CREATE TABLE 수강 (
    학번 INT,
    과목번호 INT,
    수강학기 VARCHAR(10) NOT NULL,      -- 예: '2026-1'
    성적 CHAR(2),                       -- 관계의 속성
    PRIMARY KEY (학번, 과목번호, 수강학기),
    FOREIGN KEY (학번) REFERENCES 학생(학번),
    FOREIGN KEY (과목번호) REFERENCES 과목(과목번호)
);

-- 샘플 데이터 삽입
INSERT INTO 학과 VALUES (1, '컴퓨터공학과', NULL);
INSERT INTO 학과 VALUES (2, '전자공학과', NULL);

INSERT INTO 교수 VALUES (101, '김교수', '데이터베이스', 1);
INSERT INTO 교수 VALUES (102, '이교수', '네트워크', 1);
INSERT INTO 교수 VALUES (103, '박교수', '회로설계', 2);

-- 학과장 지정
UPDATE 학과 SET 학과장교번 = 101 WHERE 학과번호 = 1;
UPDATE 학과 SET 학과장교번 = 103 WHERE 학과번호 = 2;

INSERT INTO 학생 VALUES (2021001, '홍길동', 3, 1);
INSERT INTO 학생 VALUES (2021002, '김철수', 3, 1);
INSERT INTO 학생 VALUES (2022001, '이영희', 2, 2);

INSERT INTO 과목 VALUES (301, '데이터베이스', 3, 101);
INSERT INTO 과목 VALUES (302, '컴퓨터네트워크', 3, 102);
INSERT INTO 과목 VALUES (303, '회로이론', 3, 103);

INSERT INTO 수강 VALUES (2021001, 301, '2026-1', 'A+');
INSERT INTO 수강 VALUES (2021001, 302, '2026-1', 'A0');
INSERT INTO 수강 VALUES (2021002, 301, '2026-1', 'B+');
INSERT INTO 수강 VALUES (2022001, 303, '2026-1', 'A0');

-- 학생별 수강 과목과 성적 조회
SELECT s.학번, s.이름, c.과목명, e.성적
FROM 학생 s
JOIN 수강 e ON s.학번 = e.학번
JOIN 과목 c ON e.과목번호 = c.과목번호
WHERE e.수강학기 = '2026-1'
ORDER BY s.학번, c.과목명;
```

## ER 모델링 설계 지침

### 흔한 실수와 해결책

| 실수 | 문제점 | 해결책 |
|------|--------|--------|
| 관계를 엔티티로 잘못 모델링 | 불필요한 테이블 증가 | 속성이 없는 관계는 관계로 유지 |
| 다중값 속성 무시 | 제1정규형 위반 | 별도 테이블로 분리 |
| 유도 속성 저장 | 데이터 불일치 위험 | 쿼리 시 계산하거나 뷰 사용 |
| 카디널리티 오류 | 잘못된 FK 배치 | 업무 규칙을 정확히 파악 |
| 약한 엔티티 미식별 | 불완전한 식별 체계 | 독립 식별 가능 여부 확인 |

## 정리

| 항목 | 내용 |
|------|------|
| ER 모델 | 개념적 데이터 모델링 도구, 엔티티-속성-관계로 현실 세계 추상화 |
| 엔티티 | 독립 식별 가능한 객체; 강한 엔티티(자체 키)와 약한 엔티티(의존 식별) |
| 속성 | 단순/복합, 단일값/다중값, 저장/유도, 키 속성으로 분류 |
| 관계 | 엔티티 간 연관; 차수(단항/이항/삼항), 참여(전체/부분) |
| 카디널리티 | 1:1(FK+UNIQUE), 1:N(N쪽 FK), M:N(연결 테이블) |
| ER 표기법 | 직사각형(엔티티), 다이아몬드(관계), 타원(속성), 이중선(약한/전체참여) |
| 일반화/특수화 | IS-A 관계; 분리/중복, 완전/부분 제약 |
| 매핑 규칙 | 강한→테이블, 약한→복합키, 1:1→FK, 1:N→N쪽FK, M:N→연결테이블 |
| 핵심 원칙 | 업무 규칙을 정확히 반영하고, 매핑 후 정규화 검증 필수 |
