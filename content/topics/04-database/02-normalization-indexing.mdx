## 핵심 개념

### 정규화(Normalization)란?

데이터베이스 테이블을 **중복을 최소화하고 데이터 무결성을 보장**하도록 구조화하는 과정. 큰 테이블을 작은 테이블로 분해하고 관계(FK)로 연결한다.

> **비유**: 노트 한 장에 모든 정보를 적으면 수정할 때 여러 곳을 고쳐야 한다. 주소록, 과목 목록, 성적표를 각각 별도로 관리하면 수정이 쉽다.

### 이상 현상 (Anomaly)

정규화되지 않은 테이블에서 발생하는 문제들:

```
학생_수강 테이블 (비정규화):
┌────────┬────────┬──────────┬───────┬─────────┐
│ 학생ID  │ 학생이름 │ 과목      │ 교수   │ 교수사무실 │
├────────┼────────┼──────────┼───────┼─────────┤
│ 1      │ 김철수  │ 데이터베이스│ 이교수  │ A301    │
│ 1      │ 김철수  │ 알고리즘   │ 박교수  │ B201    │
│ 2      │ 박영희  │ 데이터베이스│ 이교수  │ A301    │
│ 3      │ 최민수  │ 운영체제   │ 정교수  │ C101    │
└────────┴────────┴──────────┴───────┴─────────┘
```

| 이상 현상 | 설명 | 예시 |
|----------|------|------|
| **삽입 이상** | 불필요한 데이터 없이는 삽입 불가 | 새 교수를 추가하려면 학생/과목 정보도 필요 |
| **갱신 이상** | 중복 데이터 중 일부만 수정하면 불일치 발생 | 이교수 사무실을 A301→A302로 바꾸려면 2행 모두 수정 |
| **삭제 이상** | 필요한 정보가 의도치 않게 삭제 | 최민수가 운영체제 수강 취소하면 정교수 정보도 사라짐 |

### 함수 종속 (Functional Dependency)

정규화의 기반이 되는 개념. **X가 Y를 결정한다** (X → Y).

```
학생ID → 학생이름    (학생ID를 알면 학생이름이 결정됨)
교수 → 교수사무실    (교수를 알면 사무실이 결정됨)
{학생ID, 과목} → 성적 (학생ID와 과목을 알면 성적이 결정됨)
```

### 제1정규형 (1NF)

**모든 속성의 값이 원자값(Atomic Value)**이어야 한다. 한 칸에 하나의 값만.

```
[1NF 위반]                          [1NF 만족]
┌────┬────────────┐              ┌────┬────────┐
│ ID │ 전화번호     │              │ ID │ 전화번호  │
├────┼────────────┤              ├────┼────────┤
│ 1  │ 010-1234,  │              │ 1  │ 010-1234│
│    │ 02-555-1234│              │ 1  │ 02-5551 │
└────┴────────────┘              └────┴────────┘
   ↑ 한 칸에 2개 → 위반             ↑ 분리하여 원자값
```

### 제2정규형 (2NF)

1NF를 만족하고, **부분 종속이 없어야** 한다. 즉, 기본키의 일부에만 종속된 속성을 제거.

> 기본키가 복합키(2개 이상)일 때만 문제가 됨.

```
[2NF 위반]
기본키: {학생ID, 과목}

┌────────┬──────┬──────┬───────┐
│ 학생ID  │ 과목  │ 성적  │ 학생이름│
├────────┼──────┼──────┼───────┤
│ 1      │ DB   │ A    │ 김철수 │
│ 1      │ 알고  │ B    │ 김철수 │
│ 2      │ DB   │ A+   │ 박영희 │
└────────┴──────┴──────┴───────┘

학생이름은 {학생ID}만으로 결정됨 → 부분 종속!
{학생ID, 과목} → 성적  (완전 종속 ✓)
{학생ID} → 학생이름     (부분 종속 ✗)

[2NF 만족 — 분해]
학생 테이블:              수강 테이블:
┌────────┬───────┐      ┌────────┬──────┬──────┐
│ 학생ID  │ 학생이름│      │ 학생ID  │ 과목  │ 성적  │
├────────┼───────┤      ├────────┼──────┼──────┤
│ 1      │ 김철수 │      │ 1      │ DB   │ A    │
│ 2      │ 박영희 │      │ 1      │ 알고  │ B    │
└────────┴───────┘      │ 2      │ DB   │ A+   │
                        └────────┴──────┴──────┘
```

### 제3정규형 (3NF)

2NF를 만족하고, **이행 종속이 없어야** 한다. 기본키가 아닌 속성이 다른 비키 속성을 결정하면 안 됨.

```
[3NF 위반]
기본키: 학생ID

┌────────┬───────┬──────┬─────────┐
│ 학생ID  │ 학과   │ 학과장 │ 학과사무실│
├────────┼───────┼──────┼─────────┤
│ 1      │ 컴공   │ 김교수 │ A동 301 │
│ 2      │ 컴공   │ 김교수 │ A동 301 │
│ 3      │ 수학   │ 이교수 │ B동 201 │
└────────┴───────┴──────┴─────────┘

학생ID → 학과 → 학과장   (이행 종속!)
학생ID → 학과 → 학과사무실 (이행 종속!)

[3NF 만족 — 분해]
학생 테이블:              학과 테이블:
┌────────┬───────┐      ┌───────┬──────┬─────────┐
│ 학생ID  │ 학과   │      │ 학과   │ 학과장 │ 학과사무실│
├────────┼───────┤      ├───────┼──────┼─────────┤
│ 1      │ 컴공   │      │ 컴공   │ 김교수 │ A동 301 │
│ 2      │ 컴공   │      │ 수학   │ 이교수 │ B동 201 │
│ 3      │ 수학   │      └───────┴──────┴─────────┘
└────────┴───────┘
```

### BCNF (Boyce-Codd Normal Form)

3NF를 강화한 형태. **모든 결정자가 후보키**여야 한다.

```
[BCNF 위반]
학생이 과목을 수강하고, 각 과목은 한 교수만 담당한다고 가정.
기본키: {학생, 과목}

┌───────┬──────┬──────┐
│ 학생   │ 과목  │ 교수  │
├───────┼──────┼──────┤
│ 김철수 │ DB   │ 이교수│
│ 박영희 │ DB   │ 이교수│
│ 김철수 │ 알고  │ 박교수│
└───────┴──────┴──────┘

교수 → 과목 (교수가 과목을 결정)
하지만 교수는 후보키가 아님! → BCNF 위반

[BCNF 만족 — 분해]
수강:                    담당:
┌───────┬──────┐        ┌──────┬──────┐
│ 학생   │ 교수  │        │ 교수  │ 과목  │
├───────┼──────┤        ├──────┼──────┤
│ 김철수 │ 이교수│        │ 이교수│ DB   │
│ 박영희 │ 이교수│        │ 박교수│ 알고  │
│ 김철수 │ 박교수│        └──────┴──────┘
└───────┴──────┘
```

### 정규화 단계 요약

```
비정규형
  │ 원자값이 아닌 속성 제거
  ▼
1NF (원자값)
  │ 부분 종속 제거
  ▼
2NF (완전 함수 종속)
  │ 이행 종속 제거
  ▼
3NF (이행 종속 없음)
  │ 결정자가 후보키가 아닌 것 제거
  ▼
BCNF (모든 결정자 = 후보키)
```

### 역정규화 (Denormalization)

정규화를 너무 많이 하면 JOIN이 많아져 **성능이 저하**될 수 있다. 의도적으로 중복을 허용하여 조회 성능을 높이는 것이 역정규화.

## 인덱스 (Index)

### 인덱스란?

데이터베이스에서 **검색 속도를 높이기 위한 자료구조**. 책의 색인(Index)과 같다.

> **비유**: 1000페이지 책에서 "트랜잭션"을 찾으려면 처음부터 끝까지 읽어야 하지만(Full Scan), 색인을 보면 바로 페이지를 찾을 수 있다.

### 인덱스 없이 검색

```sql
SELECT * FROM users WHERE name = '김철수';
```

인덱스 없으면: **Full Table Scan** — 모든 행을 순차적으로 확인. O(n).

```
레코드 1: 박영희... 아님
레코드 2: 이민수... 아님
레코드 3: 김철수... 찾음! (하지만 끝까지 더 확인)
...
레코드 100만: 최지은... 완료

→ 100만 행 모두 확인: O(n)
```

### B-Tree 인덱스

대부분의 RDBMS에서 사용하는 기본 인덱스 구조. **Balanced Tree**.

```
B-Tree (차수 3):

              [30 | 60]
             /    |     \
      [10|20]  [40|50]  [70|80|90]
       /|\      /|\        /| | \
      ... ...  ... ...   ... ... ...
      (리프 노드 → 실제 데이터 포인터)
```

**B-Tree 검색 과정 (50 찾기):**
1. 루트: 30 &lt; 50 &lt; 60 → 중간 자식으로
2. 중간 노드: 40 &lt; 50 ≤ 50 → 찾음!

**시간 복잡도: O(log n)** — 100만 건에서도 약 20번 비교로 찾음.

### B+Tree 인덱스

B-Tree의 변형. **실제 데이터는 리프 노드에만** 저장하고, 리프 노드들이 **연결 리스트**로 연결.

```
B+Tree:

              [30 | 60]            ← 내부 노드 (인덱스만)
             /    |     \
      [10|20]  [40|50]  [70|80|90] ← 내부 노드
       / | \    / | \     / | | \
      리프 노드들 (실제 데이터 포인터)
      [10→20→30→40→50→60→70→80→90]  ← 리프가 순서대로 연결

```

| 비교 | B-Tree | B+Tree |
|------|--------|--------|
| 데이터 위치 | 모든 노드에 데이터 | 리프 노드에만 데이터 |
| 리프 연결 | 연결 안 됨 | 연결 리스트로 연결 |
| 범위 검색 | 비효율적 (트리 순회) | 효율적 (리프 순차 탐색) |
| 동등 비교 | 빠름 | 빠름 |
| 내부 노드 크기 | 큼 (데이터 포함) | 작음 (키만) → 더 많은 키 저장 |

> 대부분의 RDBMS(MySQL InnoDB, PostgreSQL)는 **B+Tree**를 사용한다. 범위 검색이 많은 DB 특성에 최적화.

### 인덱스 설계 전략

**인덱스를 걸어야 하는 경우:**
- WHERE 절에 자주 사용되는 컬럼
- JOIN의 ON 조건에 사용되는 컬럼
- ORDER BY, GROUP BY에 사용되는 컬럼
- 카디널리티(고유 값 수)가 높은 컬럼 (이메일, ID 등)

**인덱스를 걸지 않아야 하는 경우:**
- 테이블이 작은 경우 (Full Scan이 오히려 빠름)
- 삽입/수정/삭제가 매우 빈번한 테이블 (인덱스 유지 비용)
- 카디널리티가 매우 낮은 컬럼 (성별: M/F)
- NULL이 많은 컬럼

**복합 인덱스 (Composite Index):**

```sql
CREATE INDEX idx_name_age ON users(name, age);
```

- (name, age) 순서로 정렬됨
- `WHERE name = '김철수'` → 사용 가능
- `WHERE name = '김철수' AND age = 25` → 사용 가능
- `WHERE age = 25` → 사용 **불가** (첫 번째 컬럼이 없으므로)
- **선두 컬럼 원칙**: 복합 인덱스는 왼쪽부터 순서대로 사용

### 인덱스의 비용

인덱스는 **읽기 성능을 올리지만, 쓰기 성능은 떨어뜨린다.**

```
INSERT 시: 테이블에 데이터 삽입 + 인덱스에 키 추가 + 트리 재배치
UPDATE 시: 기존 인덱스 키 삭제 + 새 인덱스 키 추가
DELETE 시: 인덱스에서 키 삭제 (또는 마킹)

저장 공간: 인덱스 자체도 디스크 공간 차지
```

## 코드 예시

### SQL로 정규화 과정 실습

```sql
-- === 비정규화 테이블 ===
CREATE TABLE 학생수강_비정규 (
    학생ID INT,
    학생이름 VARCHAR(50),
    학과 VARCHAR(50),
    학과장 VARCHAR(50),
    과목 VARCHAR(50),
    교수 VARCHAR(50),
    교수사무실 VARCHAR(50),
    성적 CHAR(2)
);

-- === 3NF로 정규화 ===

-- 학과 테이블 (학과 → 학과장)
CREATE TABLE 학과 (
    학과코드 VARCHAR(10) PRIMARY KEY,
    학과명 VARCHAR(50) NOT NULL,
    학과장 VARCHAR(50)
);

-- 학생 테이블 (학생ID → 학생이름, 학과)
CREATE TABLE 학생 (
    학생ID INT PRIMARY KEY,
    학생이름 VARCHAR(50) NOT NULL,
    학과코드 VARCHAR(10),
    FOREIGN KEY (학과코드) REFERENCES 학과(학과코드)
);

-- 교수 테이블 (교수ID → 교수이름, 사무실)
CREATE TABLE 교수 (
    교수ID INT PRIMARY KEY,
    교수이름 VARCHAR(50) NOT NULL,
    사무실 VARCHAR(50)
);

-- 과목 테이블
CREATE TABLE 과목 (
    과목코드 VARCHAR(10) PRIMARY KEY,
    과목명 VARCHAR(50) NOT NULL,
    교수ID INT,
    FOREIGN KEY (교수ID) REFERENCES 교수(교수ID)
);

-- 수강 테이블 ({학생ID, 과목코드} → 성적)
CREATE TABLE 수강 (
    학생ID INT,
    과목코드 VARCHAR(10),
    성적 CHAR(2),
    PRIMARY KEY (학생ID, 과목코드),
    FOREIGN KEY (학생ID) REFERENCES 학생(학생ID),
    FOREIGN KEY (과목코드) REFERENCES 과목(과목코드)
);
```

### 인덱스 생성 및 효과 확인

```sql
-- === 인덱스 없이 조회 ===
EXPLAIN SELECT * FROM users WHERE email = 'kim@example.com';
-- type: ALL (Full Table Scan)
-- rows: 1000000 (전체 행 검색)

-- === 인덱스 생성 ===
CREATE INDEX idx_email ON users(email);

-- === 인덱스 있을 때 조회 ===
EXPLAIN SELECT * FROM users WHERE email = 'kim@example.com';
-- type: ref (인덱스 사용)
-- rows: 1 (1행만 검색)

-- === 복합 인덱스 ===
CREATE INDEX idx_name_age ON users(name, age);

-- 사용 가능
EXPLAIN SELECT * FROM users WHERE name = '김철수';           -- O
EXPLAIN SELECT * FROM users WHERE name = '김철수' AND age > 20; -- O

-- 사용 불가 (선두 컬럼 누락)
EXPLAIN SELECT * FROM users WHERE age > 20;                   -- X

-- === 유니크 인덱스 ===
CREATE UNIQUE INDEX idx_unique_email ON users(email);
-- 중복 값 방지 + 인덱스 효과

-- === 인덱스 삭제 ===
DROP INDEX idx_email ON users;

-- === 인덱스 목록 확인 ===
SHOW INDEX FROM users;
```

### Python으로 인덱스 효과 체험 (SQLite)

```python
import sqlite3
import time
import random
import string

def create_test_db():
    """테스트 데이터베이스 생성 및 데이터 삽입"""
    conn = sqlite3.connect(':memory:')  # 메모리 DB
    cursor = conn.cursor()

    # 테이블 생성
    cursor.execute('''
        CREATE TABLE users (
            id INTEGER PRIMARY KEY,
            name TEXT,
            email TEXT,
            age INTEGER
        )
    ''')

    # 10만 건 데이터 삽입
    print("10만 건 데이터 삽입 중...")
    data = []
    for i in range(100000):
        name = ''.join(random.choices(string.ascii_lowercase, k=8))
        email = f"{name}_{i}@example.com"
        age = random.randint(18, 65)
        data.append((name, email, age))

    cursor.executemany(
        'INSERT INTO users (name, email, age) VALUES (?, ?, ?)', data
    )
    conn.commit()
    print(f"삽입 완료: {len(data)}건")

    return conn


def test_query(conn, query, params=None, label=""):
    """쿼리 실행 시간 측정"""
    cursor = conn.cursor()
    start = time.time()
    if params:
        cursor.execute(query, params)
    else:
        cursor.execute(query)
    result = cursor.fetchall()
    elapsed = time.time() - start
    print(f"  [{label}] {elapsed*1000:.2f}ms, 결과: {len(result)}건")
    return elapsed


# === 테스트 실행 ===
conn = create_test_db()
target_email = "test_50000@example.com"
```

## 정리

| 항목 | 내용 |
|------|------|
| 정규화 | 데이터 중복 최소화, 이상 현상 방지를 위한 테이블 분해 |
| 1NF | 모든 속성이 원자값 |
| 2NF | 1NF + 부분 종속 제거 |
| 3NF | 2NF + 이행 종속 제거 |
| BCNF | 3NF + 모든 결정자가 후보키 |
| 이상 현상 | 삽입/갱신/삭제 이상 — 비정규화 테이블에서 발생 |
| B-Tree | 균형 트리, 검색 O(log n), 대부분의 DB 인덱스 |
| B+Tree | B-Tree 변형, 리프에만 데이터, 범위 검색에 최적화 |
| 인덱스 비용 | 읽기 빠름 / 쓰기 느려짐 (트리 유지 비용) |
| 복합 인덱스 | 여러 컬럼을 조합한 인덱스, 선두 컬럼 원칙 적용 |
