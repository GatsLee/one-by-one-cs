## 핵심 개념

데이터베이스 시스템은 사용자의 SQL 쿼리를 받아 최적의 방법으로 실행한다.
**쿼리 처리(Query Processing)**는 SQL 문을 파싱하고, 최적화하고, 실행하는 전 과정을 말하며,
**쿼리 최적화(Query Optimization)**는 이 과정에서 가장 효율적인 실행 계획을 선택하는 단계다.

동일한 결과를 내는 쿼리라도 실행 방법에 따라 성능 차이가 수십~수백 배에 달할 수 있다.
따라서 쿼리 최적화는 데이터베이스 성능에 결정적인 영향을 미친다.

### 1단계: 파싱(Parsing)

SQL 문의 문법(Syntax)을 검사하고 파싱 트리(Parse Tree)를 생성한다.
테이블명, 컬럼명 등의 유효성도 이 단계에서 확인한다.

### 2단계: 변환(Translation)

파싱 트리를 **관계대수(Relational Algebra)** 표현으로 변환한다.
하나의 SQL 문은 여러 동치(equivalent) 관계대수 표현으로 변환될 수 있다.

```sql
-- 이 SQL 쿼리는
SELECT 이름 FROM 학생 WHERE 학과번호 = 1;

-- 다음 관계대수로 변환됨:
-- π이름(σ학과번호=1(학생))
```

### 3단계: 최적화(Optimization)

여러 동치 표현 중에서 **비용이 가장 낮은** 실행 계획을 선택한다.
이 단계가 전체 쿼리 성능을 좌우한다.

### 4단계: 실행(Execution)

선택된 실행 계획에 따라 실제로 데이터를 읽고 처리하여 결과를 반환한다.

## 쿼리 실행 계획(Query Execution Plan)

### 논리적 계획 vs 물리적 계획

| 구분 | 논리적 계획(Logical Plan) | 물리적 계획(Physical Plan) |
|------|--------------------------|--------------------------|
| 내용 | 어떤 연산을 할 것인가 | 어떻게 연산을 수행할 것인가 |
| 표현 | 관계대수 연산 트리 | 구체적 알고리즘 지정 |
| 예시 | "학생과 수강을 조인" | "Hash Join으로 학생과 수강을 조인" |
| 인덱스 | 고려하지 않음 | 인덱스 사용 여부 결정 |

```
  논리적 계획:                물리적 계획:

  π이름,과목명                 π이름,과목명
      │                          │
    ⋈ (조인)                  Hash Join
    / \                        / \
  학생  수강                Index Scan  Seq Scan
                            (학생.학번)   (수강)
```

## 선택 연산(Selection) 구현

### 전체 스캔(Full Table Scan)

테이블의 모든 행을 순차적으로 읽으면서 조건에 맞는 행을 찾는다.

```
  ┌───┬───┬───┬───┬───┬───┬───┬───┐
  │ 1 │ 2 │ 3 │ 4 │ 5 │ 6 │ 7 │ 8 │  ← 모든 블록을 순서대로 읽음
  └───┴───┴───┴───┴───┴───┴───┴───┘
   ✓       ✓                   ✓      ← 조건 만족하는 행 반환

  비용: O(N) 블록 읽기 (N = 전체 블록 수)
```

- **장점**: 인덱스가 없어도 사용 가능, 대부분의 행을 반환할 때 효율적
- **단점**: 소수의 행만 필요할 때 비효율적

### 인덱스 스캔(Index Scan)

인덱스를 사용하여 조건에 맞는 행의 위치를 빠르게 찾는다.

```
          인덱스 (B+ 트리)
              [50]
             /    \
          [25]    [75]
         /  \    /   \
       [10][30][60] [90]
        │   │   │    │
        ↓   ↓   ↓    ↓
       데이터 블록으로 직접 접근

  비용: O(log N) + 일치하는 행 수
```

- **장점**: 소수의 행을 찾을 때 매우 효율적
- **단점**: 인덱스 유지 비용, 대량 행 반환 시 오히려 느릴 수 있음

### 선택 연산 비용 비교

| 방법 | 비용 | 적합한 경우 |
|------|------|------------|
| 전체 스캔 | O(N) | 선택률 &gt; 15~20% |
| B+ 트리 인덱스 | O(log N + k) | 동등 비교, 범위 검색 |
| 해시 인덱스 | O(1) | 동등 비교만 |
| 비트맵 인덱스 | O(N/word_size) | 낮은 카디널리티 컬럼 |

```sql
-- 인덱스 생성 예시
CREATE INDEX idx_학생_학과번호 ON 학생(학과번호);

-- 인덱스 스캔이 사용되는 쿼리
SELECT * FROM 학생 WHERE 학과번호 = 1;    -- 동등 비교

-- 범위 검색에도 B+ 트리 인덱스 활용
SELECT * FROM 학생 WHERE 학번 BETWEEN 2021001 AND 2021100;

-- 복합 인덱스
CREATE INDEX idx_수강_학번_과목 ON 수강(학번, 과목번호);
```

## 조인 알고리즘

조인 연산은 쿼리 처리에서 가장 비용이 큰 연산이다.
DBMS는 상황에 따라 적절한 조인 알고리즘을 선택한다.

### 1. 중첩 루프 조인(Nested Loop Join)

가장 단순한 조인 방법으로, 외부 테이블의 각 행에 대해 내부 테이블을 전체 스캔한다.

```
  외부 테이블 (R)          내부 테이블 (S)
  ┌───┐                   ┌───┐
  │ r1│ ──────────────→   │ s1│ 비교
  │   │ ──────────────→   │ s2│ 비교
  │   │ ──────────────→   │ s3│ 비교
  ├───┤                   ├───┤
  │ r2│ ──────────────→   │ s1│ 비교
  │   │ ──────────────→   │ s2│ 비교
  │   │ ──────────────→   │ s3│ 비교
  ├───┤                   ├───┤
  │ r3│ ──────────────→   │ s1│ 비교
  │   │ ...               │...│ ...
  └───┘                   └───┘
```

```sql
-- 중첩 루프 조인의 개념 (의사 코드)
-- for each row r in R:        -- 외부 루프
--     for each row s in S:    -- 내부 루프
--         if r.key = s.key:   -- 조인 조건
--             output (r, s)

-- 실제 SQL
SELECT s.이름, c.과목명
FROM 학생 s, 과목 c, 수강 e
WHERE s.학번 = e.학번 AND e.과목번호 = c.과목번호;
```

**비용**: 외부 테이블 블록 수 × 내부 테이블 블록 수 = O(B_R × B_S)

**블록 중첩 루프 조인(Block Nested Loop Join)** 최적화:
- 행 단위가 아닌 블록 단위로 처리하여 디스크 I/O 감소
- 메모리 버퍼를 활용하여 내부 테이블 읽기 횟수 줄임

### 2. 정렬-병합 조인(Sort-Merge Join)

양쪽 테이블을 조인 키로 정렬한 후, 동시에 스캔하며 병합한다.

```
  정렬된 R              정렬된 S
  ┌────────┐           ┌────────┐
  │ key=1  │ ←─ 포인터  │ key=1  │ ← 매칭!
  │ key=3  │           │ key=2  │
  │ key=5  │           │ key=3  │
  │ key=7  │           │ key=5  │
  │ key=9  │           │ key=8  │
  └────────┘           └────────┘

  양쪽 포인터를 동시에 이동하며 매칭
  → 각 테이블을 한 번만 스캔하면 됨
```

```sql
-- 정렬-병합 조인이 효과적인 경우:
-- 1. 조인 키에 이미 인덱스(정렬)가 있을 때
-- 2. 결과가 정렬 순서로 필요할 때
-- 3. 양쪽 테이블 크기가 비슷할 때

SELECT s.이름, e.성적
FROM 학생 s JOIN 수강 e ON s.학번 = e.학번
ORDER BY s.학번;    -- 정렬 결과가 필요 → Sort-Merge 유리
```

**비용**: O(B_R × log B_R + B_S × log B_S + B_R + B_S)
- 정렬 비용 + 병합 비용 (이미 정렬되어 있으면 정렬 비용 생략)

### 3. 해시 조인(Hash Join)

작은 테이블로 해시 테이블을 만들고, 큰 테이블을 스캔하며 매칭한다.

```
  빌드 단계 (Build Phase):
  작은 테이블 R → 해시 함수 적용 → 해시 테이블 생성

  ┌────────┐         해시 테이블
  │ key=1  │ ──h()─→ [버킷 1]: key=1, key=7
  │ key=3  │ ──h()─→ [버킷 3]: key=3, key=9
  │ key=5  │ ──h()─→ [버킷 5]: key=5
  │ key=7  │ ──h()─→ (버킷 1에 추가)
  │ key=9  │ ──h()─→ (버킷 3에 추가)
  └────────┘

  탐색 단계 (Probe Phase):
  큰 테이블 S의 각 행 → 해시 함수 → 해시 테이블에서 매칭 검색

  ┌────────┐
  │ key=1  │ ──h()─→ 버킷 1 탐색 → 매칭 발견!
  │ key=2  │ ──h()─→ 버킷 2 탐색 → 매칭 없음
  │ key=3  │ ──h()─→ 버킷 3 탐색 → 매칭 발견!
  └────────┘
```

```sql
-- 해시 조인이 효과적인 경우:
-- 1. 동등 조인(=)에서만 사용 가능
-- 2. 한쪽 테이블이 메모리에 들어갈 때 최적
-- 3. 정렬이 불필요할 때

-- 대용량 테이블 조인에 해시 조인 활용
SELECT s.이름, c.과목명, e.성적
FROM 학생 s
JOIN 수강 e ON s.학번 = e.학번          -- 동등 조인 → 해시 조인 후보
JOIN 과목 c ON e.과목번호 = c.과목번호;   -- 동등 조인 → 해시 조인 후보
```

**비용**: O(B_R + B_S) — 각 테이블을 한 번씩만 읽으면 됨 (메모리 충분 시)

### 조인 알고리즘 비교

| 알고리즘 | 시간 복잡도 | 조건 | 장점 | 단점 |
|---------|-----------|------|------|------|
| Nested Loop | O(B_R × B_S) | 제한 없음 | 구현 간단, 어떤 조건에도 가능 | 대용량에 매우 느림 |
| Sort-Merge | O(N log N + M log M) | 동등/범위 | 정렬 결과 재활용, 대용량 적합 | 정렬 비용 발생 |
| Hash Join | O(B_R + B_S) | 동등 조인만 | 가장 빠름 (메모리 충분 시) | 범위 조인 불가 |
| Index NL | O(B_R × log B_S) | 내부 테이블 인덱스 | 소량 조인에 효율적 | 인덱스 필요 |

## 비용 추정(Cost Estimation)

쿼리 최적화기는 각 실행 계획의 비용을 추정하여 최적 계획을 선택한다.

### 비용 구성 요소

| 총 비용 (Total Cost) 구성 요소 |
|-------------------------------|
| 디스크 I/O 비용 |
| CPU 비용 |
| 메모리 비용 |
| 네트워크 비용 (분산 DB) |

> **참고**: 대부분 디스크 I/O가 지배적이다.

### 통계 정보(Statistics)

최적화기가 비용을 추정하기 위해 사용하는 정보:

| 통계 항목 | 설명 | 사용처 |
|----------|------|--------|
| n_r | 테이블의 행(튜플) 수 | 결과 크기 추정 |
| b_r | 테이블의 블록 수 | I/O 비용 계산 |
| l_r | 행의 크기(바이트) | 블록당 행 수 계산 |
| V(A, r) | 속성 A의 고유값 수 | 선택률 추정 |
| 히스토그램 | 값의 분포 | 정확한 선택률 추정 |

```sql
-- PostgreSQL에서 통계 정보 확인
SELECT
    relname AS 테이블명,
    reltuples AS 행수,
    relpages AS 페이지수
FROM pg_class
WHERE relname IN ('학생', '수강', '과목');

-- 컬럼별 상세 통계
SELECT
    attname AS 컬럼명,
    n_distinct AS 고유값수,
    most_common_vals AS 빈출값,
    most_common_freqs AS 빈출빈도
FROM pg_stats
WHERE tablename = '학생';

-- 통계 정보 수동 갱신
ANALYZE 학생;
ANALYZE 수강;
```

## 규칙 기반 vs 비용 기반 최적화

| 구분 | 규칙 기반(RBO) | 비용 기반(CBO) |
|------|---------------|---------------|
| 판단 근거 | 미리 정의된 규칙 우선순위 | 통계 기반 비용 추정 |
| 인덱스 | 인덱스 있으면 무조건 사용 | 비용 비교 후 결정 |
| 통계 | 불필요 | 정확한 통계 필수 |
| 유연성 | 고정적 | 데이터 분포에 적응 |
| 현대 DBMS | 거의 사용하지 않음 | 표준 방식 |
| 대표 시스템 | 초기 Oracle | PostgreSQL, MySQL 8.0+, 최신 Oracle |

```
  규칙 기반 (RBO):
  "인덱스가 있으면 항상 인덱스 스캔" → 단순하지만 비효율적일 수 있음

  비용 기반 (CBO):
  "전체 100만 행 중 90만 행 반환? → 전체 스캔이 더 저렴"
  "전체 100만 행 중 10행 반환?   → 인덱스 스캔이 더 저렴"
```

## 쿼리 최적화 전략

### 1. 선택 연산 먼저 (Selection Push-Down)

조인 전에 선택 연산을 먼저 수행하여 중간 결과의 크기를 줄인다.

```
  최적화 전:                    최적화 후:

  π이름,과목명                   π이름,과목명
      │                            │
  σ학과번호=1                      ⋈
      │                          /   \
      ⋈                      σ학과번호=1  과목
    /   \                       │
  학생   과목                   학생

  (조인 후 선택)              (선택 후 조인 → 훨씬 빠름!)
```

```sql
-- 최적화기가 자동으로 수행하지만, 이해가 중요

-- 비효율적 (논리적으로)
SELECT s.이름, c.과목명
FROM 학생 s
JOIN 수강 e ON s.학번 = e.학번
JOIN 과목 c ON e.과목번호 = c.과목번호
WHERE s.학과번호 = 1;
-- → 최적화기: 먼저 학과번호=1인 학생만 필터 후 조인

-- 서브쿼리도 최적화기가 풀어서 처리
SELECT 이름 FROM 학생
WHERE 학번 IN (
    SELECT 학번 FROM 수강 WHERE 과목번호 = 301
);
-- → 최적화기: 세미조인 또는 조인으로 변환할 수 있음
```

### 2. 프로젝션 먼저 (Projection Push-Down)

필요한 컬럼만 미리 선택하여 중간 결과의 크기를 줄인다.

```
  최적화 전:                    최적화 후:

  π이름                         π이름
      │                            │
      ⋈                           ⋈
    /   \                        /   \
  학생   수강                  π학번,이름  π학번
  (전체컬럼)(전체컬럼)           │         │
                               학생      수강
                             (2컬럼만)  (1컬럼만)
```

### 3. 조인 순서 최적화

다중 테이블 조인에서 조인 순서가 성능에 큰 영향을 미친다.

```
  3개 테이블 A, B, C 조인 시 가능한 순서:

  (A ⋈ B) ⋈ C    ← 중간 결과가 작으면 최적
  (A ⋈ C) ⋈ B    ← 중간 결과가 작으면 최적
  (B ⋈ C) ⋈ A    ← 중간 결과가 작으면 최적

  원칙: 선택률이 높은(결과가 적은) 조인을 먼저 수행
```

```sql
-- 조인 순서는 최적화기가 자동 결정하지만,
-- 힌트로 제어할 수도 있음 (DBMS별 상이)

-- PostgreSQL: join_collapse_limit으로 조인 순서 탐색 범위 제어
SET join_collapse_limit = 8;  -- 8개 이하 테이블은 모든 순서 탐색

-- 통계가 정확해야 최적화기가 좋은 결정을 내림
ANALYZE 학생;
ANALYZE 수강;
ANALYZE 과목;
```

## EXPLAIN / EXPLAIN ANALYZE 사용법

### EXPLAIN: 실행 계획 확인

실제로 쿼리를 실행하지 않고 실행 계획만 보여준다.

```sql
-- 기본 실행 계획 확인
EXPLAIN
SELECT s.이름, c.과목명, e.성적
FROM 학생 s
JOIN 수강 e ON s.학번 = e.학번
JOIN 과목 c ON e.과목번호 = c.과목번호
WHERE s.학과번호 = 1;
```

출력 예시:

```
Hash Join  (cost=35.50..72.30 rows=10 width=64)
  Hash Cond: (e.과목번호 = c.과목번호)
  -> Hash Join  (cost=17.50..52.80 rows=10 width=40)
        Hash Cond: (e.학번 = s.학번)
        -> Seq Scan on 수강 e  (cost=0.00..30.00 rows=1000 width=16)
        -> Hash  (cost=17.00..17.00 rows=20 width=28)
              -> Seq Scan on 학생 s  (cost=0.00..17.00 rows=20 width=28)
                    Filter: (학과번호 = 1)
  -> Hash  (cost=13.00..13.00 rows=200 width=28)
        -> Seq Scan on 과목 c  (cost=0.00..13.00 rows=200 width=28)
```

### EXPLAIN ANALYZE: 실제 실행 + 계획

실제로 쿼리를 실행하고 예측값과 실제값을 함께 보여준다.

```sql
-- 실제 실행 통계 포함
EXPLAIN ANALYZE
SELECT s.이름, c.과목명, e.성적
FROM 학생 s
JOIN 수강 e ON s.학번 = e.학번
JOIN 과목 c ON e.과목번호 = c.과목번호
WHERE s.학과번호 = 1;
```

출력 예시:

```
Hash Join  (cost=35.50..72.30 rows=10 width=64)
           (actual time=0.150..0.280 rows=8 loops=1)
  Hash Cond: (e.과목번호 = c.과목번호)
  -> Hash Join  (cost=17.50..52.80 rows=10 width=40)
                (actual time=0.090..0.180 rows=8 loops=1)
        ...
Planning Time: 0.250 ms
Execution Time: 0.350 ms
```

### 실행 계획 주요 노드

| 노드 | 설명 | 특징 |
|------|------|------|
| Seq Scan | 순차 스캔(전체 테이블) | 인덱스 없거나 대량 반환 |
| Index Scan | 인덱스 스캔 | 소량 행 검색에 효율적 |
| Index Only Scan | 인덱스만으로 결과 반환 | 테이블 접근 불필요 |
| Bitmap Index Scan | 비트맵으로 여러 인덱스 결합 | OR 조건, 다중 조건 |
| Hash Join | 해시 기반 조인 | 동등 조인에 효율적 |
| Merge Join | 정렬-병합 조인 | 정렬된 데이터에 효율적 |
| Nested Loop | 중첩 루프 조인 | 소량 데이터에 적합 |
| Sort | 정렬 연산 | ORDER BY, Merge Join 전 |
| Aggregate | 집계 연산 | GROUP BY, COUNT, SUM 등 |

```
  실행 계획 읽는 순서:

  가장 안쪽(들여쓰기 깊은) → 바깥쪽 순서로 읽음

  Hash Join               ← 4. 최종 조인
    → Hash Join           ← 3. 먼저 실행되는 조인
        → Seq Scan 수강   ← 1. 수강 테이블 스캔
        → Hash            ← 2. 학생 해시 테이블 생성
            → Seq Scan 학생
    → Hash                ← 과목 해시 테이블 생성
        → Seq Scan 과목
```

## 실전 쿼리 튜닝 팁

### 1. 적절한 인덱스 생성

```sql
-- WHERE 절에 자주 사용되는 컬럼에 인덱스
CREATE INDEX idx_학생_학과 ON 학생(학과번호);

-- 조인 키에 인덱스 (보통 FK에 생성)
CREATE INDEX idx_수강_학번 ON 수강(학번);
CREATE INDEX idx_수강_과목번호 ON 수강(과목번호);

-- 복합 인덱스: 자주 함께 사용되는 조건
CREATE INDEX idx_수강_학번_학기 ON 수강(학번, 수강학기);

-- 커버링 인덱스: 테이블 접근 없이 인덱스만으로 결과 반환
CREATE INDEX idx_학생_학과_이름 ON 학생(학과번호, 이름);
-- → SELECT 이름 FROM 학생 WHERE 학과번호 = 1; → Index Only Scan
```

### 2. 인덱스 사용을 방해하는 패턴

```sql
-- 나쁜 예: 컬럼에 함수 적용 → 인덱스 사용 불가
SELECT * FROM 학생 WHERE UPPER(이름) = '홍길동';

-- 좋은 예: 함수 인덱스 생성 또는 값 쪽에 적용
CREATE INDEX idx_학생_이름_upper ON 학생(UPPER(이름));

-- 나쁜 예: 암묵적 타입 변환 → 인덱스 사용 불가
SELECT * FROM 학생 WHERE 학번 = '2021001';  -- 학번이 INT인데 문자열 비교

-- 좋은 예: 올바른 타입 사용
SELECT * FROM 학생 WHERE 학번 = 2021001;

-- 나쁜 예: LIKE 앞쪽 와일드카드 → 인덱스 사용 불가
SELECT * FROM 학생 WHERE 이름 LIKE '%길동';

-- 좋은 예: 앞쪽 고정 문자열 → 인덱스 사용 가능
SELECT * FROM 학생 WHERE 이름 LIKE '홍%';

-- 나쁜 예: OR 조건 → 인덱스 비효율적
SELECT * FROM 학생 WHERE 학과번호 = 1 OR 학년 = 3;

-- 좋은 예: UNION으로 분리
SELECT * FROM 학생 WHERE 학과번호 = 1
UNION
SELECT * FROM 학생 WHERE 학년 = 3;
```

### 3. 쿼리 작성 모범 사례

```sql
-- 필요한 컬럼만 선택 (SELECT * 지양)
SELECT 이름, 학과번호 FROM 학생 WHERE 학과번호 = 1;

-- EXISTS를 IN보다 선호 (서브쿼리가 큰 경우)
-- IN 사용
SELECT 이름 FROM 학생 WHERE 학번 IN (SELECT 학번 FROM 수강);

-- EXISTS 사용 (첫 매칭에서 중단 → 더 효율적)
SELECT 이름 FROM 학생 s
WHERE EXISTS (SELECT 1 FROM 수강 e WHERE e.학번 = s.학번);

-- 불필요한 DISTINCT 제거
-- 나쁜 예: PK로 조인하면 중복 없음
SELECT DISTINCT s.이름 FROM 학생 s JOIN 학과 d ON s.학과번호 = d.학과번호;
-- 좋은 예:
SELECT s.이름 FROM 학생 s JOIN 학과 d ON s.학과번호 = d.학과번호;

-- LIMIT으로 불필요한 행 제한
SELECT * FROM 학생 ORDER BY 학번 LIMIT 10;
```

### 4. 통계 정보 최신 유지

```sql
-- 대량 데이터 변경 후 반드시 ANALYZE 실행
INSERT INTO 수강 SELECT ... FROM ...;  -- 대량 삽입
ANALYZE 수강;                           -- 통계 갱신

-- PostgreSQL: 자동 VACUUM/ANALYZE 설정
-- postgresql.conf
-- autovacuum = on
-- autovacuum_analyze_threshold = 50
-- autovacuum_analyze_scale_factor = 0.1
```

## 쿼리 최적화 실전 예제

```sql
-- 문제: "2026년 1학기에 A+ 받은 학생 이름과 과목명을 조회하라"

-- 튜닝 전: 서브쿼리 + 불필요한 정렬
SELECT DISTINCT s.이름,
    (SELECT c.과목명 FROM 과목 c WHERE c.과목번호 = e.과목번호)
FROM 학생 s, 수강 e
WHERE s.학번 = e.학번
AND e.성적 = 'A+'
AND e.수강학기 = '2026-1'
ORDER BY s.이름;

-- 튜닝 후: 명시적 JOIN + 필요한 것만
SELECT s.이름, c.과목명
FROM 수강 e
JOIN 학생 s ON s.학번 = e.학번
JOIN 과목 c ON c.과목번호 = e.과목번호
WHERE e.성적 = 'A+'
  AND e.수강학기 = '2026-1'
ORDER BY s.이름;

-- 지원 인덱스
CREATE INDEX idx_수강_학기_성적 ON 수강(수강학기, 성적);
```

```sql
-- EXPLAIN ANALYZE로 튜닝 결과 확인
EXPLAIN ANALYZE
SELECT s.이름, c.과목명
FROM 수강 e
JOIN 학생 s ON s.학번 = e.학번
JOIN 과목 c ON c.과목번호 = e.과목번호
WHERE e.성적 = 'A+'
  AND e.수강학기 = '2026-1'
ORDER BY s.이름;

-- 결과에서 확인할 점:
-- 1. Seq Scan이 아닌 Index Scan이 사용되는가?
-- 2. 예측 행 수(rows)와 실제 행 수(actual rows)가 비슷한가?
-- 3. 불필요한 Sort 노드가 없는가?
-- 4. Execution Time이 허용 범위 내인가?
```

## 정리

| 항목 | 내용 |
|------|------|
| 쿼리 처리 단계 | 파싱 → 변환(관계대수) → 최적화 → 실행 |
| 실행 계획 | 논리적(어떤 연산) vs 물리적(어떤 알고리즘) |
| 선택 연산 | 전체 스캔 O(N) vs 인덱스 스캔 O(log N) |
| Nested Loop Join | O(n×m), 모든 조건 가능, 소량에 적합 |
| Sort-Merge Join | O(N log N), 정렬된 데이터에 효율적, 범위 조인 가능 |
| Hash Join | O(N+M), 동등 조인에 최적, 메모리 충분 시 가장 빠름 |
| 비용 추정 | 디스크 I/O 중심, 통계 정보(행 수, 고유값 수, 히스토그램) 활용 |
| 최적화 방식 | 규칙 기반(RBO, 구식) vs 비용 기반(CBO, 현대 표준) |
| 최적화 전략 | Selection Push-Down, Projection Push-Down, 조인 순서 최적화 |
| EXPLAIN | 실행 계획 확인; ANALYZE 추가 시 실제 실행 통계 포함 |
| 튜닝 핵심 | 적절한 인덱스 + 정확한 통계 + 올바른 쿼리 패턴 |
