## 핵심 개념

데이터베이스 시스템은 장애(Failure)가 발생하더라도 데이터의 **일관성**과 **영속성**을 보장해야 한다.
**복구(Recovery)**는 장애 발생 후 데이터베이스를 일관된 상태로 되돌리는 메커니즘이며,
**복제(Replication)**는 데이터를 여러 노드에 복사하여 가용성과 성능을 높이는 기법이다.

이 두 가지는 ACID 속성 중 **원자성(Atomicity)**과 **영속성(Durability)**을 실현하는 핵심 기술이다.

## 장애 유형

### 데이터베이스 시스템의 장애 분류

```
                    장애(Failure)
                        │
          ┌─────────────┼─────────────┐
          │             │             │
    트랜잭션 장애    시스템 장애     디스크 장애
    (Transaction)   (System)       (Media)
          │             │             │
    논리적 오류     전원 차단       헤드 충돌
    교착 상태      OS 크래시       디스크 손상
    제약 위반      메모리 오류      컨트롤러 고장
```

| 장애 유형 | 원인 | 영향 범위 | 복구 방법 |
|----------|------|----------|----------|
| 트랜잭션 장애 | 논리 오류, 교착상태, 제약 위반 | 해당 트랜잭션만 | Undo(롤백) |
| 시스템 장애 | 전원 차단, OS 크래시, 하드웨어 오류 | 메모리 내 모든 데이터 | Redo + Undo |
| 디스크 장애 | 디스크 물리적 손상 | 저장된 데이터 전체 | 백업에서 복원 + Redo |

```sql
-- 트랜잭션 장애 예시: 제약 위반으로 롤백
BEGIN;
UPDATE 계좌 SET 잔액 = 잔액 - 1000000 WHERE 계좌번호 = 'A001';
-- 잔액이 음수가 되면 CHECK 제약 위반 → 트랜잭션 자동 롤백
UPDATE 계좌 SET 잔액 = 잔액 + 1000000 WHERE 계좌번호 = 'A002';
COMMIT;

-- 시스템 장애 예시: COMMIT 직전에 전원 차단
-- → 메모리의 변경 내용이 디스크에 기록되지 않았을 수 있음
-- → 복구 시 Redo/Undo 필요
```

## WAL(Write-Ahead Logging)

### WAL 원칙

**WAL(Write-Ahead Logging)**은 데이터베이스 복구의 핵심 원칙이다.

> **"데이터를 디스크에 기록하기 전에, 반드시 로그를 먼저 디스크에 기록한다."**

이 원칙을 지키면, 장애 발생 시 로그를 통해 어떤 변경이 있었는지 항상 추적할 수 있다.

```
  WAL 원칙의 동작 순서:

  1. 로그 레코드를 로그 버퍼에 작성
  2. 로그 버퍼를 디스크(로그 파일)에 플러시 (Write-Ahead!)
  3. 데이터 변경 내용을 데이터 버퍼에 적용
  4. 데이터 버퍼를 디스크(데이터 파일)에 기록 (나중에)

  ┌──────────────────────────────────────────────┐
  │                   메모리                      │
  │                                              │
  │  ┌──────────┐          ┌──────────┐          │
  │  │ 로그 버퍼 │    ②→   │데이터 버퍼│          │
  │  │ ①기록    │          │ ③적용    │          │
  │  └────┬─────┘          └────┬─────┘          │
  │       │                     │                │
  └───────┼─────────────────────┼────────────────┘
          │ ②먼저!              │ ④나중에
          ↓                     ↓
  ┌──────────────┐     ┌──────────────┐
  │   로그 파일    │     │  데이터 파일   │
  │   (디스크)    │     │   (디스크)    │
  └──────────────┘     └──────────────┘
```

### 로그 레코드 구조

각 로그 레코드는 다음 정보를 포함한다:

```
  ┌─────────────────────────────────────────────┐
  │              로그 레코드 구조                  │
  ├─────────┬───────────────────────────────────┤
  │ LSN     │ Log Sequence Number (고유 식별자)  │
  │ TxID    │ 트랜잭션 ID                        │
  │ Type    │ UPDATE / COMMIT / ABORT / BEGIN    │
  │ PageID  │ 변경된 페이지 번호                   │
  │ Offset  │ 페이지 내 오프셋                    │
  │ Before  │ 변경 전 값 (Undo용)                │
  │ After   │ 변경 후 값 (Redo용)                │
  │ PrevLSN │ 같은 트랜잭션의 이전 로그 LSN       │
  └─────────┴───────────────────────────────────┘
```

```sql
-- WAL 로그의 논리적 예시

-- 트랜잭션: 계좌 A에서 B로 50만원 이체
-- LSN 001: [T1, BEGIN]
-- LSN 002: [T1, UPDATE, 계좌A, 잔액, 100만→50만]
-- LSN 003: [T1, UPDATE, 계좌B, 잔액, 200만→250만]
-- LSN 004: [T1, COMMIT]

-- PostgreSQL에서 WAL 관련 설정 확인
SHOW wal_level;              -- minimal, replica, logical
SHOW max_wal_size;           -- WAL 파일 최대 크기
SHOW wal_buffers;            -- WAL 버퍼 크기
SHOW synchronous_commit;     -- 동기 커밋 여부
```

## 로그 기반 복구

### Undo 복구: 미완료 트랜잭션 롤백

커밋되지 않은 트랜잭션의 변경 사항을 **취소**한다.
로그의 Before 값을 사용하여 원래 상태로 되돌린다.

```
  장애 발생 시점의 로그:

  LSN 001: [T1, BEGIN]
  LSN 002: [T1, UPDATE, 계좌A, 100만→50만]    ← Before: 100만
  LSN 003: [T1, UPDATE, 계좌B, 200만→250만]   ← Before: 200만
  ──── 장애 발생! (COMMIT 없음) ────

  Undo 복구:
  LSN 003의 Before 값으로 복원: 계좌B = 200만
  LSN 002의 Before 값으로 복원: 계좌A = 100만

  → T1이 시작되기 전 상태로 복원됨
  → 역순(뒤에서 앞으로)으로 Undo 수행
```

### Redo 복구: 완료된 트랜잭션 재실행

커밋된 트랜잭션의 변경 사항이 디스크에 반영되지 않았을 때 **재실행**한다.
로그의 After 값을 사용하여 변경을 다시 적용한다.

```
  장애 발생 시점의 로그:

  LSN 001: [T1, BEGIN]
  LSN 002: [T1, UPDATE, 계좌A, 100만→50만]     ← After: 50만
  LSN 003: [T1, UPDATE, 계좌B, 200만→250만]    ← After: 250만
  LSN 004: [T1, COMMIT]                        ← 커밋 완료!
  ──── 장애 발생! (디스크에 미반영) ────

  Redo 복구:
  LSN 002의 After 값 적용: 계좌A = 50만
  LSN 003의 After 값 적용: 계좌B = 250만

  → T1의 변경 사항이 디스크에 확실히 반영됨
  → 정순(앞에서 뒤로)으로 Redo 수행
```

### Undo + Redo 조합

실제 시스템 장애 시에는 **Redo와 Undo를 모두** 수행해야 한다.

```
  로그 상태:

  LSN 001: [T1, BEGIN]
  LSN 002: [T1, UPDATE, 계좌A, 100만→50만]
  LSN 003: [T2, BEGIN]
  LSN 004: [T1, UPDATE, 계좌B, 200만→250만]
  LSN 005: [T1, COMMIT]          ← T1은 커밋됨
  LSN 006: [T2, UPDATE, 계좌C, 300만→100만]
  ──── 장애 발생! (T2는 미커밋) ────

  복구 과정:
  1단계 (Redo): 모든 로그를 순서대로 재실행
     → 커밋된 T1의 변경을 확실히 반영

  2단계 (Undo): 미커밋 T2를 역순으로 롤백
     → LSN 006: 계좌C = 300만 (Before 값 복원)

  결과: T1 반영, T2 취소 → 일관된 상태
```

## 체크포인트(Checkpoint)

### 체크포인트의 필요성

장애 복구 시 로그 전체를 스캔하면 시간이 매우 오래 걸린다.
**체크포인트**는 특정 시점에 모든 변경 내용을 디스크에 강제 기록하여 복구 시작점을 만든다.

```
  체크포인트 없이 복구:

  로그 시작 ─────────────────────────── 장애 발생
  |←──── 전체 로그를 스캔해야 함 ────→|


  체크포인트로 복구 시간 단축:

  로그 시작 ────CP1────CP2────CP3──── 장애 발생
                              |←──→|
                         이 구간만 스캔!
```

### 체크포인트 동작 과정

```
  1. 새로운 트랜잭션 시작 일시 중지 (선택적)
  2. 현재 활성 트랜잭션 목록 기록
  3. 모든 더티 페이지(변경된 데이터)를 디스크에 기록
  4. 로그에 체크포인트 레코드 기록
  5. 트랜잭션 재개

  ┌────────────────────────────────────────┐
  │          체크포인트 레코드              │
  ├────────────────────────────────────────┤
  │ Type: CHECKPOINT                      │
  │ Active Transactions: {T3, T5, T7}     │
  │ Dirty Pages Flushed: Yes              │
  │ Timestamp: 2026-03-05 14:30:00        │
  └────────────────────────────────────────┘
```

```sql
-- PostgreSQL에서 체크포인트 설정
SHOW checkpoint_timeout;          -- 체크포인트 간격 (기본 5분)
SHOW checkpoint_completion_target; -- 완료 목표 비율

-- 수동 체크포인트 실행
CHECKPOINT;

-- 체크포인트 관련 설정 변경
ALTER SYSTEM SET checkpoint_timeout = '10min';
ALTER SYSTEM SET max_wal_size = '2GB';
```

### Fuzzy Checkpoint

현대 DBMS는 **퍼지 체크포인트(Fuzzy Checkpoint)**를 사용한다.
트랜잭션을 중단하지 않고 백그라운드에서 더티 페이지를 점진적으로 기록한다.

| 방식 | 트랜잭션 중단 | 성능 영향 | 복구 복잡도 |
|------|-------------|----------|-----------|
| 정적 체크포인트 | 필요 | 높음 | 낮음 |
| 퍼지 체크포인트 | 불필요 | 낮음 | 높음 (ARIES 필요) |

## ARIES 복구 알고리즘

**ARIES(Algorithm for Recovery and Isolation Exploiting Semantics)**는
현대 DBMS에서 가장 널리 사용되는 복구 알고리즘이다.
IBM 연구소에서 개발되었으며, 3단계로 복구를 수행한다.

### ARIES의 3대 원칙

1. **WAL(Write-Ahead Logging)**: 데이터 기록 전에 로그를 먼저 기록
2. **Redo 시 기록 반복(Repeating History)**: 장애 직전 상태를 정확히 재현
3. **Undo 시 로그 기록(Logging during Undo)**: Undo 중에도 로그를 남겨 중복 Undo 방지

### 3단계 복구 과정

```
  ┌─────────────────────────────────────────────────┐
  │              ARIES 복구 3단계                     │
  │                                                 │
  │  1. 분석(Analysis)  → 어디서부터 복구할지 결정      │
  │  2. Redo            → 장애 직전 상태로 복원        │
  │  3. Undo            → 미완료 트랜잭션 롤백         │
  └─────────────────────────────────────────────────┘
```

#### 1단계: 분석(Analysis Phase)

마지막 체크포인트부터 로그 끝까지 스캔하여 정보를 수집한다.

```
  마지막 체크포인트 ────────────── 로그 끝
  |←──────── 분석 구간 ──────────→|

  수집하는 정보:
  ┌───────────────────────────────────┐
  │ Redo 목록 (Dirty Page Table)       │
  │ → 디스크에 미반영된 페이지 목록      │
  │                                   │
  │ Undo 목록 (Active Transaction Table)│
  │ → 장애 시 활성이었던 트랜잭션 목록    │
  │                                   │
  │ RedoLSN                           │
  │ → Redo를 시작할 LSN                │
  └───────────────────────────────────┘
```

#### 2단계: Redo Phase

RedoLSN부터 로그 끝까지 **모든 변경 사항을 순서대로 재실행**한다.
이렇게 하면 장애 직전의 정확한 상태가 재현된다.

```
  RedoLSN ──────────────────────── 로그 끝
  |←──── 순서대로 Redo 수행 ────→|

  모든 UPDATE 로그를 재실행:
  - 커밋된 트랜잭션의 변경: 확실히 반영
  - 미커밋 트랜잭션의 변경: 일단 반영 (다음 단계에서 Undo)
```

#### 3단계: Undo Phase

장애 시 활성이었던(커밋되지 않은) 트랜잭션들을 **역순으로 롤백**한다.

```
  로그 끝 ←──────────────── 활성 트랜잭션 시작
          |←── 역순 Undo ──→|

  Undo 시에도 CLR(Compensation Log Record) 기록:
  → 복구 중 다시 장애가 발생해도 안전하게 재복구 가능
```

### ARIES 전체 흐름 예시

```
  시간 →

  T1: BEGIN ─── UPDATE(A) ─── UPDATE(B) ─── COMMIT
  T2: BEGIN ─── UPDATE(C) ─── UPDATE(D) ────────── (미커밋)
                     │
                 CHECKPOINT
                                              장애!

  로그:
  LSN 001: [T1, BEGIN]
  LSN 002: [T1, UPDATE, A, old→new]
  LSN 003: [T2, BEGIN]
  LSN 004: [T2, UPDATE, C, old→new]
  LSN 005: [CHECKPOINT, active={T1,T2}]
  LSN 006: [T1, UPDATE, B, old→new]
  LSN 007: [T1, COMMIT]
  LSN 008: [T2, UPDATE, D, old→new]
  ──── 장애! ────

  1단계 분석: 체크포인트(LSN 005)부터 스캔
     → Redo 대상: 모든 UPDATE (LSN 002~008)
     → Undo 대상: T2 (미커밋)

  2단계 Redo: LSN 002부터 순서대로 재실행
     → A, B, C, D 모두 변경 적용

  3단계 Undo: T2를 역순 롤백
     → LSN 008: D를 old 값으로 복원 (CLR 기록)
     → LSN 004: C를 old 값으로 복원 (CLR 기록)

  결과: T1 반영, T2 취소 → 일관된 상태
```

## 복제(Replication)

### 복제의 목적

| 목적 | 설명 |
|------|------|
| 고가용성(HA) | 한 노드 장애 시 다른 노드로 서비스 지속 |
| 읽기 성능 향상 | 읽기 요청을 여러 복제본에 분산 |
| 지리적 분산 | 사용자 가까운 곳에서 데이터 제공 |
| 백업 | 실시간 데이터 백업 역할 |

### 마스터-슬레이브(Master-Slave) 복제

**한 대의 마스터**가 쓰기를 담당하고, **여러 슬레이브**가 읽기를 분산한다.
가장 일반적인 복제 구조다.

```
                 쓰기 요청
                    │
                    ↓
             ┌──────────┐
             │  마스터    │
             │ (Primary) │
             └──┬───┬───┘
        복제 ↙   │   ↘ 복제
  ┌──────────┐ ┌──────────┐ ┌──────────┐
  │ 슬레이브1  │ │ 슬레이브2  │ │ 슬레이브3  │
  │(Replica) │ │(Replica) │ │(Replica) │
  └──────────┘ └──────────┘ └──────────┘
       ↑              ↑              ↑
    읽기 요청       읽기 요청       읽기 요청
```

```sql
-- PostgreSQL 마스터-슬레이브 복제 설정

-- 마스터 서버 설정 (postgresql.conf)
-- wal_level = replica
-- max_wal_senders = 3
-- synchronous_standby_names = 'slave1'

-- 슬레이브 서버 설정 (recovery.conf 또는 postgresql.auto.conf)
-- primary_conninfo = 'host=master_ip port=5432 user=replicator'
-- restore_command = 'cp /archive/%f %p'

-- 복제 상태 확인 (마스터에서)
SELECT client_addr, state, sent_lsn, replay_lsn,
       sent_lsn - replay_lsn AS 복제지연
FROM pg_stat_replication;

-- 슬레이브에서 복제 상태 확인
SELECT pg_last_wal_receive_lsn(),
       pg_last_wal_replay_lsn(),
       pg_last_xact_replay_timestamp();
```

| 장점 | 단점 |
|------|------|
| 구조 단순, 구현 용이 | 마스터가 단일 장애점(SPOF) |
| 읽기 성능 확장 용이 | 쓰기 성능은 확장 불가 |
| 데이터 일관성 유지 쉬움 | 마스터 장애 시 페일오버 필요 |

### 마스터-마스터(Multi-Master) 복제

**여러 노드가 모두 쓰기 가능**한 구조다. 쓰기 성능을 분산할 수 있지만
충돌 해결이 복잡하다.

```
             쓰기 요청          쓰기 요청
                │                  │
                ↓                  ↓
         ┌──────────┐       ┌──────────┐
         │ 마스터 1   │ ←──→ │ 마스터 2   │
         │ (읽기/쓰기)│ 양방향 │ (읽기/쓰기)│
         └──────────┘  복제  └──────────┘
              ↑                    ↑
           읽기 요청            읽기 요청
```

**충돌 해결 전략**:

| 전략 | 설명 | 장단점 |
|------|------|--------|
| 타임스탬프 기반 | 마지막 쓰기가 우선(LWW) | 간단하지만 데이터 유실 가능 |
| 애플리케이션 레벨 | 앱에서 충돌 감지 및 해결 | 정확하지만 복잡 |
| CRDT | 충돌 없는 복제 데이터 타입 | 특정 데이터 구조에만 적용 |

### 동기식 vs 비동기식 복제

```
  동기식 복제 (Synchronous):

  클라이언트 → 마스터: COMMIT
  마스터 → 슬레이브: 로그 전송
  슬레이브 → 마스터: 적용 완료 확인     ← 기다림!
  마스터 → 클라이언트: COMMIT 응답


  비동기식 복제 (Asynchronous):

  클라이언트 → 마스터: COMMIT
  마스터 → 클라이언트: COMMIT 응답       ← 바로 응답!
  마스터 → 슬레이브: 로그 전송 (백그라운드)
```

| 구분 | 동기식(Synchronous) | 비동기식(Asynchronous) |
|------|--------------------|-----------------------|
| 데이터 안전성 | 높음 (손실 없음) | 낮음 (지연분 손실 가능) |
| 응답 속도 | 느림 (슬레이브 대기) | 빠름 (즉시 응답) |
| 가용성 | 슬레이브 장애 시 영향 | 슬레이브 장애에 무관 |
| 네트워크 의존 | 높음 | 낮음 |
| 사용 사례 | 금융, 의료 | 소셜미디어, 로그 |

```sql
-- PostgreSQL 동기식 복제 설정
-- postgresql.conf (마스터)
-- synchronous_commit = on
-- synchronous_standby_names = 'FIRST 1 (slave1, slave2)'

-- 반동기식 (semi-synchronous): 로그 전송까지만 보장
-- synchronous_commit = remote_write

-- 비동기식
-- synchronous_commit = off
```

## 복제 지연(Replication Lag)과 최종 일관성

### 복제 지연이란?

비동기 복제에서 마스터의 변경이 슬레이브에 도달하기까지의 시간차를 말한다.

```
  마스터: T1에서 데이터 변경
         │
         │  ← 복제 지연 (수 ms ~ 수 초)
         ↓
  슬레이브: T1 + Δt에서 변경 반영

  이 구간 동안 마스터와 슬레이브의 데이터가 다름!
```

### 복제 지연으로 인한 문제

```
  시나리오: 사용자가 글을 작성한 직후 조회

  1. 사용자 → 마스터: INSERT 글 작성     ✓ 성공
  2. 사용자 → 슬레이브: SELECT 글 조회   ✗ 아직 미반영!
     "방금 쓴 글이 안 보여요!"
```

### 해결 방법

| 방법 | 설명 | 트레이드오프 |
|------|------|------------|
| Read-after-Write 일관성 | 자신이 쓴 데이터는 마스터에서 읽기 | 마스터 부하 증가 |
| 모노토닉 읽기 | 같은 슬레이브에서 계속 읽기 | 부하 분산 제한 |
| 동기식 복제로 전환 | 복제 지연 제거 | 쓰기 성능 저하 |
| 지연 허용 설계 | 최종 일관성(Eventual Consistency) 수용 | UI에서 처리 필요 |

```sql
-- PostgreSQL: 복제 지연 모니터링
-- 마스터에서 실행
SELECT
    application_name,
    client_addr,
    state,
    pg_wal_lsn_diff(sent_lsn, replay_lsn) AS 바이트_지연,
    replay_lag AS 시간_지연
FROM pg_stat_replication;

-- 슬레이브에서 지연 확인
SELECT
    now() - pg_last_xact_replay_timestamp() AS 복제_지연시간;
```

## 장애 복구 시나리오: 자동 페일오버(Failover)

### 페일오버란?

마스터 서버 장애 시 **슬레이브를 자동으로 마스터로 승격**시키는 과정이다.

```
  정상 상태:

  ┌──────────┐  복제  ┌──────────┐
  │ 마스터(A) │ ────→ │ 슬레이브(B)│
  └──────────┘       └──────────┘
       ↑
    읽기/쓰기


  장애 발생 + 페일오버:

  ┌──────────┐       ┌──────────┐
  │ 마스터(A) │  ✗    │ 새마스터(B)│ ← 승격!
  │  (장애)   │       │(읽기/쓰기)│
  └──────────┘       └──────────┘
                          ↑
                       읽기/쓰기


  복구 후:

  ┌──────────┐  복제  ┌──────────┐
  │슬레이브(A)│ ←──── │ 마스터(B)  │
  │ (재합류)  │       │          │
  └──────────┘       └──────────┘
```

### 페일오버 과정

```
  1. 장애 감지 (Health Check)
     └→ 하트비트 실패 감지 (보통 수 초~수십 초)

  2. 마스터 차단 (Fencing)
     └→ 스플릿 브레인 방지를 위해 이전 마스터 격리

  3. 슬레이브 승격 (Promotion)
     └→ 가장 최신 데이터를 가진 슬레이브를 마스터로 승격

  4. 연결 전환 (Redirect)
     └→ 애플리케이션의 연결을 새 마스터로 전환

  5. 이전 마스터 복구 (Rejoin)
     └→ 복구된 이전 마스터를 슬레이브로 재합류
```

### 스플릿 브레인(Split Brain) 문제

네트워크 단절로 인해 **두 노드가 모두 자신이 마스터라고 인식**하는 상황이다.

```
  네트워크 단절:

  ┌──────────┐    ✗    ┌──────────┐
  │ 마스터(A) │  ←─/─→  │ 슬레이브(B)│
  │ "나는     │         │ "A가 죽은  │
  │  마스터!"  │         │  것 같다.  │
  └──────────┘         │  나를 마스터│
                       │  로 승격!"  │
                       └──────────┘

  → 두 개의 마스터! → 데이터 불일치 발생!
```

**해결 방법**:
- **Quorum(정족수)**: 과반수 동의 없이는 마스터 승격 불가
- **Fencing**: STONITH(Shoot The Other Node In The Head) - 이전 마스터 강제 종료
- **리스(Lease)**: 마스터 역할에 만료 시간을 부여

```sql
-- PostgreSQL 슬레이브 승격 (수동)
-- 슬레이브 서버에서 실행
SELECT pg_promote();

-- 또는 커맨드라인
-- pg_ctl promote -D /var/lib/postgresql/data

-- Patroni (자동 페일오버 도구) 상태 확인
-- patronictl list
-- +--------+---------+---------+----+-----------+
-- | Member | Host    | Role    | TL | Lag in MB |
-- +--------+---------+---------+----+-----------+
-- | node1  | 1.1.1.1 | Leader  | 5  | 0         |
-- | node2  | 1.1.1.2 | Replica | 5  | 0         |
-- | node3  | 1.1.1.3 | Replica | 5  | 0.1       |
-- +--------+---------+---------+----+-----------+
```

## 백업 전략

### 백업 유형 비교

```
  전체 백업 (Full Backup):

  월  [████████████████████]  ← 전체 데이터 복사


  증분 백업 (Incremental Backup):

  월  [████████████████████]  ← 전체 백업
  화  [██]                    ← 월 이후 변경분만
  수  [█]                     ← 화 이후 변경분만
  목  [███]                   ← 수 이후 변경분만

  복원: 월 전체 + 화 증분 + 수 증분 + 목 증분 (순서대로 모두 필요)


  차등 백업 (Differential Backup):

  월  [████████████████████]  ← 전체 백업
  화  [██]                    ← 월 이후 변경분
  수  [███]                   ← 월 이후 변경분 (화 포함)
  목  [█████]                 ← 월 이후 변경분 (화+수 포함)

  복원: 월 전체 + 목 차등 (2개만 필요)
```

| 유형 | 백업 크기 | 백업 속도 | 복원 속도 | 복원 복잡도 |
|------|----------|----------|----------|-----------|
| 전체 | 가장 큼 | 가장 느림 | 가장 빠름 | 가장 간단 |
| 증분 | 가장 작음 | 가장 빠름 | 가장 느림 | 가장 복잡 (순서 의존) |
| 차등 | 중간 | 중간 | 중간 | 중간 (전체+마지막 차등) |

### 실전 백업 전략

```
  추천 전략: 전체 + 증분 조합

  일요일: 전체 백업 (주 1회)
  월~토:  증분 백업 (매일)
  실시간: WAL 아카이빙 (지속적)

  ┌───┬───┬───┬───┬───┬───┬───┐
  │ 일 │ 월 │ 화 │ 수 │ 목 │ 금 │ 토 │
  │전체│증분│증분│증분│증분│증분│증분│
  └───┴───┴───┴───┴───┴───┴───┘

  보관 정책:
  - 일별 백업: 7일 보관
  - 주별 백업: 4주 보관
  - 월별 백업: 12개월 보관
```

```sql
-- PostgreSQL 백업 명령어

-- 논리적 백업 (pg_dump): SQL 형태로 내보내기
-- 전체 데이터베이스 백업
-- pg_dump -h localhost -U postgres mydb > backup.sql

-- 특정 테이블만 백업
-- pg_dump -t 학생 -t 수강 mydb > partial_backup.sql

-- 커스텀 형식 (압축, 병렬 복원 가능)
-- pg_dump -Fc mydb > backup.dump

-- 복원
-- pg_restore -d mydb backup.dump


-- 물리적 백업 (pg_basebackup): 데이터 파일 직접 복사
-- pg_basebackup -h master -D /backup/base -Fp -Xs -P

-- WAL 아카이빙 설정
-- postgresql.conf:
-- archive_mode = on
-- archive_command = 'cp %p /archive/%f'

-- PITR(Point-In-Time Recovery): 특정 시점으로 복구
-- recovery.conf:
-- restore_command = 'cp /archive/%f %p'
-- recovery_target_time = '2026-03-05 14:30:00'
```

### 3-2-1 백업 규칙

  ┌─────────────────────────────────────┐
  │          3-2-1 백업 규칙             │

## 복구와 복제의 관계

```
  ┌──────────────────────────────────────────────┐
  │              전체 데이터 보호 체계              │
  │                                              │
  │  트랜잭션 장애 ← WAL + Undo/Redo             │
  │  시스템 장애   ← WAL + ARIES + 체크포인트      │
  │  디스크 장애   ← 백업 + WAL 아카이빙 + PITR   │
  │  서버 장애     ← 복제 + 자동 페일오버          │
  │  재해(재난)    ← 원격 복제 + 클라우드 백업      │
  │                                              │
  │  각 장애 수준에 맞는 보호 메커니즘을 조합       │
  └──────────────────────────────────────────────┘
```

| 장애 수준 | 보호 메커니즘 | RTO(복구시간) | RPO(데이터손실) |
|----------|-------------|-------------|---------------|
| 트랜잭션 장애 | WAL + Undo | 즉시 | 0 |
| 시스템 장애 | ARIES 복구 | 수 초~수 분 | 0 |
| 단일 디스크 | RAID | 즉시 | 0 |
| 서버 장애 | 동기 복제 + 페일오버 | 수 초~수 분 | 0 |
| 서버 장애 | 비동기 복제 + 페일오버 | 수 초~수 분 | 수 초분 손실 가능 |
| 데이터센터 장애 | 원격 복제 | 수 분~수 시간 | 설정에 따라 |
| 논리적 오류 | PITR(시점 복구) | 수 분~수 시간 | 오류 시점까지 |

## 정리

| 항목 | 내용 |
|------|------|
| 장애 유형 | 트랜잭션 장애(롤백), 시스템 장애(메모리 손실), 디스크 장애(데이터 손실) |
| WAL 원칙 | 데이터 기록 전에 로그를 먼저 디스크에 기록 |
| 로그 레코드 | LSN, TxID, Type, Before(Undo용), After(Redo용) |
| Undo 복구 | 미완료 트랜잭션을 역순으로 롤백 (Before 값 사용) |
| Redo 복구 | 완료된 트랜잭션을 순서대로 재실행 (After 값 사용) |
| 체크포인트 | 주기적으로 변경 내용을 디스크에 기록하여 복구 시작점 생성 |
| ARIES | 분석(Analysis) → Redo → Undo 3단계 복구 알고리즘 |
| 마스터-슬레이브 | 마스터(쓰기) + 슬레이브(읽기 분산), 가장 일반적 |
| 마스터-마스터 | 모든 노드 쓰기 가능, 충돌 해결 복잡 |
| 동기 vs 비동기 | 동기(안전, 느림) vs 비동기(빠름, 지연 손실 가능) |
| 페일오버 | 마스터 장애 시 슬레이브를 자동 승격, 스플릿 브레인 주의 |
| 백업 전략 | 전체(간단), 증분(작지만 복원 복잡), 차등(균형), 3-2-1 규칙 |
