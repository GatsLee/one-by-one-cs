## 핵심 개념

### 1. 버전 관리가 필요한 이유

코드를 작성하다 보면 "어제 버전으로 돌리고 싶다", "이 기능 추가하다가 원본이 망가졌다"는 상황이 온다. 버전 관리 시스템(VCS)은 **파일의 변경 이력을 기록**하여 언제든 과거로 돌아갈 수 있게 한다.

```
버전 관리 없이:

  보고서_최종.docx
  보고서_최종_진짜최종.docx
  보고서_최종_진짜최종_수정.docx
  보고서_제발마지막.docx            ← 어느 게 최신?

Git으로 버전 관리:

  보고서.docx
    ├── v1 (3월 1일) "초안 작성"
    ├── v2 (3월 3일) "서론 수정"
    ├── v3 (3월 5일) "표 추가"
    └── v4 (3월 7일) "최종 검토 반영"    ← 이력이 명확
```

### 2. Git의 3가지 영역

Git은 파일을 3가지 영역으로 관리한다. 이 개념이 Git의 핵심이다.

```
┌──────────────┐     git add      ┌──────────────┐    git commit    ┌──────────────┐
│  Working     │ ──────────────→  │   Staging    │ ──────────────→  │  Repository  │
│  Directory   │                  │   Area       │                  │  (.git)      │
│              │                  │  (Index)     │                  │              │
│ 실제 파일     │                  │ 커밋할 변경    │                  │ 확정된 이력    │
│ (편집 중)     │                  │ (대기 중)     │                  │ (영구 저장)    │
└──────────────┘                  └──────────────┘                  └──────────────┘
        │                                                                  │
        └──────────────────────────────────────────────────────────────────┘
                                    git checkout / restore
```

```
흐름 예시:

1. main.py를 수정한다         →  Working Directory에 변경
2. git add main.py           →  Staging Area로 이동
3. git commit -m "버그 수정"  →  Repository에 영구 저장
```

### 3. 기본 명령어

#### 저장소 시작하기

```bash
# 새 프로젝트 시작
mkdir my-project
cd my-project
git init                    # .git 폴더 생성 (여기에 이력 저장)

# 기존 프로젝트 가져오기
git clone https://github.com/user/repo.git
cd repo
```

#### 변경 사항 확인과 저장

```bash
# 현재 상태 확인 (가장 자주 쓰는 명령)
git status

# 출력 예시:
# On branch main
# Changes not staged for commit:
#   modified:   src/app.py          ← 수정됨, 아직 add 안 함
# Untracked files:
#   src/utils.py                    ← 새 파일, 아직 추적 안 함

# 변경 내용 상세 확인
git diff                            # Working과 Staging 비교
git diff --staged                   # Staging과 마지막 커밋 비교

# Staging에 올리기
git add src/app.py                  # 특정 파일만
git add src/                        # 폴더 전체
git add .                           # 모든 변경 (주의: .gitignore 확인)

# 커밋 (스냅샷 저장)
git commit -m "fix: login validation bug"
```

#### 이력 확인

```bash
# 커밋 이력 보기
git log                             # 전체 이력
git log --oneline                   # 한 줄씩 간략하게
git log --oneline -5                # 최근 5개만

# 출력 예시:
# a1b2c3d fix: login validation bug
# e4f5g6h feat: add search feature
# h7i8j9k initial commit
```

### 4. 커밋 메시지 작성법

커밋 메시지는 **"왜" 이 변경을 했는지** 미래의 나(또는 동료)에게 설명하는 것이다.

```
나쁜 커밋 메시지:
  "수정"
  "aaaa"
  "작업 중"
  "asdfasdf"

좋은 커밋 메시지:
  "fix: prevent null pointer in user profile page"
  "feat: add email notification for new orders"
  "docs: update API endpoint documentation"

형식:
  [타입]: 짧은 설명 (50자 이내)

  타입:
    feat     →  새 기능
    fix      →  버그 수정
    docs     →  문서 변경
    refactor →  리팩토링 (동작 변화 없음)
    test     →  테스트 추가/수정
    chore    →  빌드, 설정 변경
```

### 5. .gitignore — 추적 제외 파일

Git이 추적하지 않아야 할 파일을 지정한다.

```bash
# .gitignore 파일 예시
node_modules/          # 의존성 (npm install로 복원 가능)
__pycache__/           # Python 캐시
.env                   # 환경 변수 (비밀번호, API 키)
*.log                  # 로그 파일
.DS_Store              # macOS 시스템 파일
dist/                  # 빌드 결과물
.vscode/               # 개인 IDE 설정
```

```
★ .env 파일은 절대 커밋하지 마라!
  - API 키, 비밀번호, DB 접속 정보가 들어 있음
  - GitHub에 올라가면 누구나 볼 수 있음
  - .env.example (값 없이 키만)을 대신 커밋
```

### 6. 원격 저장소 (Remote)

로컬 Git과 GitHub(원격)을 연결하여 코드를 공유한다.

```
┌──────────────────┐                    ┌──────────────────┐
│   Local (내 PC)   │    git push       │  Remote (GitHub)  │
│                  │ ──────────────→   │                  │
│  .git/           │                   │  origin/main     │
│                  │ ←──────────────   │                  │
│                  │    git pull       │                  │
└──────────────────┘                    └──────────────────┘
```

```bash
# 원격 저장소 연결
git remote add origin https://github.com/user/repo.git

# 로컬 → 원격 (업로드)
git push origin main

# 원격 → 로컬 (다운로드)
git pull origin main

# 원격 상태 확인
git remote -v
```
## 연습 문제

### Q1. Git의 3가지 영역(Working Directory, Staging Area, Repository)의 역할은?

<details>
<summary>정답 보기</summary>
Working Directory는 실제 파일을 편집하는 공간, Staging Area는 다음 커밋에 포함할 변경을 선택하는 공간(git add), Repository는 커밋된 이력이 영구 저장되는 공간(git commit).
</details>

### Q2. .gitignore에 넣어야 하는 파일의 종류 3가지는?

<details>
<summary>정답 보기</summary>
(1) 환경 변수/비밀 정보(.env, credentials), (2) 의존성 폴더(node_modules/, __pycache__/), (3) 빌드 결과물(dist/, build/). 추가로 OS 시스템 파일(.DS_Store), IDE 설정(.vscode/, .idea/)도 포함.
</details>

### Q3. `git add .`와 `git add 파일명`의 차이점과 주의점은?

<details>
<summary>정답 보기</summary>
`git add .`은 현재 디렉토리의 모든 변경을 staging에 올린다. `git add 파일명`은 특정 파일만 올린다. `git add .`을 사용할 때는 .gitignore 설정을 확인하여 .env나 불필요한 파일이 포함되지 않도록 주의해야 한다.
</details>


## 플래시카드

| 질문 | 답변 |
|------|------|
| git init은 무엇을 하나? | 현재 디렉토리에 .git 폴더를 생성하여 Git 저장소를 초기화 |
| git add의 역할은? | Working Directory의 변경을 Staging Area로 이동 |
| git commit의 역할은? | Staging Area의 변경을 Repository에 영구 저장(스냅샷) |
| git status는 무엇을 보여주나? | 현재 브랜치, 수정/추가/삭제된 파일, staged 여부 |
| git diff와 git diff --staged의 차이는? | diff는 Working↔Staging 비교, --staged는 Staging↔마지막커밋 비교 |
| .gitignore의 역할은? | Git이 추적하지 않을 파일/폴더 패턴을 지정 |
| .env 파일을 커밋하면 안 되는 이유는? | API 키, 비밀번호 등이 공개 저장소에 노출될 수 있음 |
| git push와 git pull의 차이는? | push는 로컬→원격 업로드, pull은 원격→로컬 다운로드 |
| git clone과 git init의 차이는? | clone은 기존 원격 저장소를 복사, init은 새 저장소를 생성 |
| 좋은 커밋 메시지의 조건은? | 타입 접두사 + 간결한 설명, "왜" 변경했는지 전달 |
