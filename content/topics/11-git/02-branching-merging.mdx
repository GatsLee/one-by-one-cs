## 핵심 개념

### 1. 브랜치란?

브랜치(Branch)는 **독립적인 작업 공간**이다. 메인 코드에 영향을 주지 않고 새 기능을 개발하거나 버그를 수정할 수 있다.

<Mermaid chart="
gitgraph
  commit id: &quot;A&quot;
  commit id: &quot;B&quot;
  commit id: &quot;C&quot;
  branch feature
  checkout main
  commit id: &quot;D&quot;
  commit id: &quot;E&quot;

" />

> 브랜치 = 평행 타임라인. feature에서 작업해도 main에 영향 없음!

### 2. 브랜치 기본 명령어

```bash
# 브랜치 목록 보기
git branch                    # 로컬 브랜치 목록 (* = 현재 브랜치)
git branch -a                 # 원격 브랜치 포함

# 새 브랜치 만들기
git branch feat/login         # 브랜치 생성 (이동하지 않음)
git checkout -b feat/login    # 생성 + 이동 (한번에)
git switch -c feat/login      # 생성 + 이동 (최신 명령)

# 브랜치 이동
git checkout feat/login       # 기존 방식
git switch feat/login         # 최신 방식

# 브랜치 삭제
git branch -d feat/login      # 머지된 브랜치만 삭제
git branch -D feat/login      # 강제 삭제 (주의!)
```

### 3. 머지(Merge) — 브랜치 합치기

작업이 끝난 브랜치를 main에 합친다.

```bash
# feat/login 작업 완료 후 main에 합치기
git switch main               # main으로 이동
git merge feat/login          # feat/login을 main에 합침
```

**Fast-forward 머지** (분기 없을 때 — 일직선으로 합쳐짐):

### 4. 머지 충돌(Merge Conflict) 해결

두 브랜치가 **같은 파일의 같은 부분**을 다르게 수정했을 때 충돌이 발생한다.

```
충돌 상황:

  main에서:    line 5 →  "Hello World"
  feature에서:  line 5 →  "Hello Korea"

  → Git: "둘 중 어느 걸 쓸지 사람이 정해야 해!"
```

충돌이 나면 Git이 파일에 마커를 삽입한다:

```
<<<<<<< HEAD
Hello World
=======
Hello Korea
>>>>>>> feat/greeting
```

```
충돌 해결 순서:

  1. 충돌 파일 열기  (git status로 확인)
  2. <<<< ==== >>>> 마커 찾기
  3. 원하는 내용으로 수정 (마커도 삭제)
  4. git add 충돌파일
  5. git commit         (머지 커밋 완료)
```

```
해결 예시:

  Before (충돌 마커 있음):
    <<<<<<< HEAD
    Hello World
    =======
    Hello Korea
    >>>>>>> feat/greeting

  After (수동으로 정리):
    Hello Korea          ← 원하는 내용만 남김
```

### 5. Feature Branch Workflow

실무에서 가장 많이 쓰는 브랜치 전략:

<Mermaid chart="
gitgraph
  commit id: &quot;init&quot;
  branch feat/login
  checkout feat/login
  commit id: &quot;login-1&quot;
  commit id: &quot;login-2&quot;
  checkout main
  merge feat/login
  branch feat/search
  checkout feat/search
  commit id: &quot;search-1&quot;
  commit id: &quot;search-2&quot;
  checkout main
  merge feat/search
  branch fix/typo
  checkout fix/typo
  commit id: &quot;typo-fix&quot; type: HIGHLIGHT
" />

**규칙:**
1. main에서 직접 커밋하지 않는다
2. 기능별로 브랜치를 만든다 (feat/xxx, fix/xxx)
3. 작업 완료 후 PR(Pull Request)로 리뷰받고 머지
4. 머지 후 브랜치는 삭제

### 6. Rebase vs Merge

```
┌──────────────────────────────────────────────────────────┐
│               Merge vs Rebase                            │
├────────────────┬─────────────────────────────────────────┤
│ Merge          │ 머지 커밋이 생김, 이력이 분기 형태로 보존  │
│                │ 안전하고 직관적, 초보자에게 추천           │
├────────────────┼─────────────────────────────────────────┤
│ Rebase         │ 커밋을 재배치하여 일직선 이력             │
│                │ 깔끔하지만 공유 브랜치에서는 위험          │
├────────────────┴─────────────────────────────────────────┤
│ ★ 원칙: 혼자 작업하는 브랜치 → Rebase OK                  │
│         여러 사람이 쓰는 브랜치 → Merge만!                 │
└──────────────────────────────────────────────────────────┘
```

**Merge** (머지 커밋이 생기고 분기 이력 보존):

<Mermaid chart="
gitgraph
  commit id: &quot;A&quot;
  commit id: &quot;B&quot;
  branch feat
  checkout feat
  commit id: &quot;C&quot;
  commit id: &quot;D&quot;
  checkout main
  commit id: &quot;E&quot;
  merge feat id: &quot;M&quot;
" />

**Rebase** (일직선, 깔끔한 이력):

## 연습 문제

### Q1. Fast-forward 머지와 3-way 머지의 차이는?

<details>
<summary>정답 보기</summary>
Fast-forward는 main에 추가 커밋이 없을 때 포인터만 이동(머지 커밋 없음). 3-way 머지는 main에도 새 커밋이 있을 때 두 브랜치의 변경을 합쳐 머지 커밋을 생성.
</details>

### Q2. 머지 충돌(Conflict)은 언제 발생하는가?

<details>
<summary>정답 보기</summary>
두 브랜치가 같은 파일의 같은 줄을 다르게 수정했을 때. Git이 자동으로 합칠 수 없어서 사용자가 직접 어떤 내용을 남길지 결정해야 한다.
</details>

### Q3. 공유 브랜치에서 rebase가 위험한 이유는?

<details>
<summary>정답 보기</summary>
Rebase는 커밋 해시를 변경한다. 다른 사람이 이미 pull한 커밋이 rebase로 바뀌면, 그 사람의 로컬과 원격이 불일치하여 복잡한 충돌이 발생한다.
</details>

| 머지 충돌 해결 순서는? | 충돌 파일 열기 → 마커 정리 → git add → git commit |

| Fast-forward 머지란? | main에 추가 커밋이 없을 때 포인터만 앞으로 이동 (머지 커밋 없음) |
| Rebase를 공유 브랜치에서 쓰면 안 되는 이유는? | 커밋 해시가 변경되어 다른 사람의 로컬과 충돌 발생 |

| 머지 전에 main pull을 하는 이유는? | 최신 변경을 반영하여 충돌을 미리 해결하고 안전하게 머지하기 위해 |
