## 핵심 개념

### 1. GitHub과 원격 저장소

GitHub은 Git 저장소를 호스팅하는 플랫폼이다. 코드를 공유하고, 협업하고, 포트폴리오를 만들 수 있다.

```
Git vs GitHub:

  Git     →  버전 관리 도구 (로컬에서 동작)
  GitHub  →  Git 저장소를 호스팅하는 웹 서비스
             (비유: Git = 엔진, GitHub = 자동차)

  비슷한 서비스: GitLab, Bitbucket
```

### 2. Fork vs Clone

```
┌──────────────────────────────────────────────────────────────┐
│                  Fork vs Clone                               │
├─────────────────┬────────────────────────────────────────────┤
│ Clone           │ 원본 저장소를 로컬에 복사                    │
│                 │ 원본에 push 권한이 있을 때 사용              │
│                 │ git clone URL                              │
├─────────────────┼────────────────────────────────────────────┤
│ Fork            │ 원본 저장소를 내 GitHub 계정에 복사           │
│                 │ 원본에 push 권한이 없을 때 사용              │
│                 │ 오픈소스 기여 시 필수                        │
│                 │ GitHub 웹에서 Fork 버튼 클릭                │
└─────────────────┴────────────────────────────────────────────┘
```

```
오픈소스 기여 흐름:

  원본 저장소 (upstream)     내 Fork (origin)       내 로컬
  ┌──────────────┐      ┌──────────────┐      ┌──────────────┐
  │ user/project │ fork │ me/project   │ clone│  ~/project   │
  │              │ ──→  │              │ ──→  │              │
  │              │      │              │ ←──  │              │
  │              │ ←──  │              │ push │              │
  │              │  PR  │              │      │              │
  └──────────────┘      └──────────────┘      └──────────────┘
```

### 3. Pull Request (PR)

Pull Request는 **"내 변경 사항을 검토하고 합쳐주세요"** 라는 요청이다. 코드 리뷰의 핵심 도구.

```
PR 과정:

  1. 브랜치에서 작업 완료
  2. GitHub에 push
  3. PR 생성 (제목, 설명 작성)
  4. 팀원이 코드 리뷰
  5. 리뷰 반영 (추가 커밋)
  6. 승인(Approve) 받으면 머지
  7. 브랜치 삭제
```

#### 좋은 PR 작성법

```
PR 제목: 간결하게, 무엇을 했는지

  ✗ "수정"
  ✗ "작업 완료"
  ✓ "feat: add email verification on signup"
  ✓ "fix: prevent crash when user has no profile image"

PR 설명(Body):

  ## 변경 사항
  - 회원가입 시 이메일 인증 기능 추가
  - 인증 메일 템플릿 생성

  ## 스크린샷 (UI 변경 시)
  [Before/After 이미지]

  ## 테스트
  - [x] 이메일 발송 확인
  - [x] 인증 링크 클릭 시 활성화 확인
  - [x] 만료된 링크 에러 처리 확인
```

### 4. 코드 리뷰

코드 리뷰는 PR에서 다른 사람의 코드를 검토하는 것이다. 버그를 잡고, 코드 품질을 높이고, 지식을 공유한다.

```
코드 리뷰 체크리스트:

  □ 코드가 의도한 대로 동작하는가?
  □ 엣지 케이스를 처리했는가? (null, 빈 배열, 큰 데이터)
  □ 이해하기 쉬운 변수명/함수명인가?
  □ 중복 코드가 있는가?
  □ 보안 문제가 있는가? (SQL 인젝션, XSS 등)
  □ 테스트가 포함되어 있는가?
```

```
리뷰 코멘트 잘 다는 법:

  ✗ "이거 잘못됨"
  ✗ "별로인데"

  ✓ "여기서 user가 null일 수 있을 것 같아요.
     user?.name으로 바꾸면 어떨까요?"

  ✓ "이 로직이 getUsers()에서도 비슷하게 있는데,
     공통 함수로 추출하면 좋을 것 같습니다."

  접두사 관례:
    nit:      →  사소한 제안 (안 고쳐도 됨)
    question: →  이해가 안 되는 부분
    suggestion: → 개선 제안
    blocker:  →  반드시 수정 필요
```

### 5. Issues — 작업 관리

GitHub Issues는 **버그 보고, 기능 요청, 할 일 추적**에 사용한다.

```
Issue 예시:

  제목: 로그인 페이지에서 비밀번호 재설정 링크가 작동하지 않음

  ## 버그 설명
  비밀번호 재설정 링크 클릭 시 404 에러 발생

  ## 재현 방법
  1. /login 페이지 접속
  2. "비밀번호를 잊으셨나요?" 클릭
  3. 404 페이지로 이동됨

  ## 예상 동작
  비밀번호 재설정 폼이 표시되어야 함

  ## 환경
  - 브라우저: Chrome 130
  - OS: macOS 15
```

```
Issue와 PR 연결:

  커밋/PR에서 "Closes #42" 또는 "Fixes #42"를 쓰면
  PR이 머지될 때 Issue가 자동으로 닫힌다.

  git commit -m "fix: password reset link (Closes #42)"
```

### 6. 실무 협업 흐름 요약

```
팀 협업 전체 흐름:

  1. Issue 확인/생성    →  "무엇을 할 것인가?"
  2. 브랜치 생성        →  git switch -c feat/issue-42
  3. 코드 작성 + 커밋   →  작은 단위로 자주 커밋
  4. push              →  git push -u origin feat/issue-42
  5. PR 생성           →  설명 + "Closes #42"
  6. 코드 리뷰         →  팀원 피드백 반영
  7. 머지              →  main에 합침
  8. 브랜치 삭제        →  정리

  이 과정을 반복
```
