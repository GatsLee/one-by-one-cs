## 핵심 개념

### 1. 왜 테스트를 해야 하나?

코드가 "일단 돌아간다"는 것과 "올바르게 동작한다"는 것은 다르다. 테스트는 **코드가 의도한 대로 동작하는지 자동으로 검증**하는 코드다.

```
테스트가 없으면:

  코드 수정 → "기존 기능도 잘 되나?" → 수동으로 전부 확인
  → 시간 오래 걸림 → 귀찮아서 안 하게 됨 → 버그 발견 못함
  → 배포 후 유저가 버그 발견 → 긴급 수정

테스트가 있으면:

  코드 수정 → 테스트 실행 (1분) → 통과 ✓ → 배포
  → 테스트 실패 ✗ → 어디가 깨졌는지 바로 알림 → 수정 후 배포
```

### 2. 테스트가 특히 중요한 경우

```
AI가 생성한 코드:
  - AI가 만든 코드가 "맞는 것 같아 보이지만" 엣지 케이스에서 실패
  - 테스트가 AI 코드의 안전망 역할
  - "AI가 맞는지" 사람이 매번 확인하는 것보다 테스트가 훨씬 빠름

리팩토링할 때:
  - 코드 구조를 바꿨는데 동작은 같아야 할 때
  - 테스트가 "동작이 바뀌지 않았다"를 보장

여러 사람이 함께 작업할 때:
  - 내가 수정한 코드가 다른 사람의 기능을 깨뜨릴 수 있음
  - 테스트가 이를 자동으로 감지
```

### 3. 테스트 피라미드

테스트에는 여러 종류가 있고, 각각 역할과 비용이 다르다.

```
                    /\
                   /  \
                  / E2E \          느리고 비싸지만
                 / Tests \         실제 사용자 경험 검증
                /──────────\
               / Integration\      여러 모듈의 연결 검증
              /    Tests     \
             /────────────────\
            /   Unit Tests     \   빠르고 저렴
           /                    \  개별 함수/클래스 검증
          /──────────────────────\

  아래로 갈수록: 빠름, 저렴, 많이 작성
  위로 갈수록: 느림, 비쌈, 적게 작성
```

```
┌──────────────┬──────────────────────────────────────────────┐
│ 종류          │ 설명                                          │
├──────────────┼──────────────────────────────────────────────┤
│ Unit Test    │ 함수 하나, 클래스 하나를 독립적으로 테스트       │
│ (단위 테스트)  │ 빠름 (밀리초), 가장 많이 작성                   │
├──────────────┼──────────────────────────────────────────────┤
│ Integration  │ 여러 모듈/서비스가 함께 동작하는지 테스트         │
│ (통합 테스트)  │ DB, API 등 외부 의존성 포함                    │
├──────────────┼──────────────────────────────────────────────┤
│ E2E Test     │ 실제 사용자처럼 브라우저에서 전체 흐름 테스트     │
│ (End-to-End) │ 느림 (초~분), 가장 적게 작성                    │
└──────────────┴──────────────────────────────────────────────┘
```

### 4. 무엇을 테스트하고, 무엇을 테스트하지 않나

```
테스트해야 할 것:
  ✓ 비즈니스 로직 (가격 계산, 할인 적용, 점수 산출)
  ✓ 엣지 케이스 (빈 입력, 0, null, 초대형 데이터)
  ✓ 에러 처리 (잘못된 입력에 적절히 에러를 내는가?)
  ✓ API 응답 형식 (클라이언트가 기대하는 구조인가?)
  ✓ 데이터 변환 (입력 → 출력 매핑이 맞는가?)

테스트하지 않아도 되는 것:
  ✗ 프레임워크 내장 기능 (React의 useState가 동작하는지)
  ✗ 외부 라이브러리 내부 동작
  ✗ 단순 getter/setter (로직 없는 코드)
  ✗ 자주 바뀌는 UI 디테일 (버튼 색상, 텍스트)
```

### 5. 테스트의 ROI (투자 대비 효과)

```
테스트 작성 비용:
  - 시간: 기능 코드의 30-50% 추가 시간
  - 학습: 테스트 프레임워크 학습 필요

테스트가 돌려주는 것:
  - 버그 조기 발견 (배포 전 vs 배포 후)
  - 리팩토링 자신감 ("이거 바꿔도 괜찮을까?" → 테스트 돌리면 앎)
  - 문서 역할 (테스트 코드 = "이 함수는 이렇게 쓴다"는 예시)
  - 디버깅 시간 절약 (어디가 깨졌는지 바로 앎)

  ★ 프로젝트가 커질수록 테스트의 가치는 기하급수적으로 증가
```

### 6. 첫 테스트 맛보기

```python
# 함수
def add(a, b):
    return a + b

# 테스트 (pytest)
def test_add():
    assert add(2, 3) == 5
    assert add(0, 0) == 0
    assert add(-1, 1) == 0
```

```javascript
// 함수
function add(a, b) {
  return a + b;
}

// 테스트 (Jest/Vitest)
test("add correctly sums two numbers", () => {
  expect(add(2, 3)).toBe(5);
  expect(add(0, 0)).toBe(0);
  expect(add(-1, 1)).toBe(0);
});
```

```
테스트 코드의 패턴:

  1. 준비 (Arrange)  →  입력 데이터 준비
  2. 실행 (Act)      →  테스트 대상 함수 호출
  3. 검증 (Assert)   →  결과가 기대값과 같은지 확인
```
