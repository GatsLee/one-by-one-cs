## 핵심 개념

### 1. LLM 에이전트란?

LLM 에이전트는 **대규모 언어 모델(LLM)**을 두뇌로 사용하면서, **도구(Tools)**와 **메모리(Memory)**를 결합하여 복잡한 작업을 자율적으로 수행하는 시스템이다. 단순히 텍스트를 생성하는 것을 넘어, 실제 환경과 상호작용하며 목표를 달성한다.

#### 기본 LLM vs 에이전트 비교

```
[기본 LLM]
사용자 → "내일 서울 날씨에 맞는 옷 추천해줘"
    → LLM → "일반적으로 봄에는..." (학습 데이터 기반 추측, 실시간 정보 없음)

[LLM 에이전트]
사용자 → "내일 서울 날씨에 맞는 옷 추천해줘"
    → LLM: "먼저 내일 서울 날씨를 확인해야겠다"      (사고/Think)
    → Tool: weather_api("서울", "2026-03-02")         (행동/Act)
    → 결과: {"temp": 8, "rain_prob": 60%}            (관찰/Observe)
    → LLM: "기온이 낮고 비 올 확률이 높으니 따뜻한 옷과 우산을 추천하자"
    → 답변: "내일 서울은 8도에 비 올 확률 60%입니다.
             따뜻한 겉옷과 우산을 챙기세요."
```

#### 에이전트의 3대 핵심 구성 요소

```
┌───────────────────────────────────────────────────────────┐
│                      LLM 에이전트                          │
│                                                           │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐       │
│  │     LLM     │  │    도구     │  │   메모리     │       │
│  │   (Brain)   │  │   (Tools)   │  │  (Memory)   │       │
│  │             │  │             │  │             │       │
│  │ - 추론      │  │ - 웹 검색   │  │ - 대화 이력  │       │
│  │ - 계획 수립  │  │ - 코드 실행  │  │ - 작업 상태  │       │
│  │ - 판단      │  │ - API 호출   │  │ - 과거 경험  │       │
│  │ - 자연어 이해│  │ - DB 조회   │  │ - 외부 지식  │       │
│  └─────────────┘  │ - 파일 I/O  │  └─────────────┘       │
│                   └─────────────┘                        │
│                                                           │
│  ┌─────────────────────────────────────────────────┐      │
│  │            에이전트 루프 (Agent Loop)              │      │
│  │                                                 │      │
│  │   ┌──────┐    ┌──────┐    ┌──────┐             │      │
│  │   │Think │───→│ Act  │───→│Observe│──┐          │      │
│  │   │(사고) │    │(행동) │    │(관찰) │  │          │      │
│  │   └──────┘    └──────┘    └──────┘  │          │      │
│  │       ↑                              │          │      │
│  │       └──────────────────────────────┘          │      │
│  │                                                 │      │
│  │   종료 조건 충족 시 → 최종 답변 반환              │      │
│  └─────────────────────────────────────────────────┘      │
└───────────────────────────────────────────────────────────┘
```

에이전트 루프는 **Think-Act-Observe** 사이클을 반복한다:

1. **Think (사고)**: 현재 상황을 분석하고 다음 행동을 결정
2. **Act (행동)**: 선택한 도구를 실행하거나 최종 답변을 생성
3. **Observe (관찰)**: 행동의 결과를 확인하고 다음 단계를 판단

이 루프는 **종료 조건**(답변 완성, 최대 반복 횟수, 오류 발생)을 만족할 때까지 계속된다.

### 2. ReAct 패턴 (Reasoning + Acting)

ReAct(2022, Yao et al.)는 LLM이 **추론(Reasoning)**과 **행동(Acting)**을 교차 수행하는 프레임워크다. 순수 Chain-of-Thought(CoT)는 추론만 하고, 순수 Act-only는 근거 없이 행동만 한다. ReAct는 이 둘을 결합하여 **근거 있는 행동**을 가능하게 한다.

#### ReAct vs 순수 CoT vs Act-only 비교

```
[순수 CoT (Chain-of-Thought)]
질문: "BTS 멤버 중 가장 나이가 많은 사람의 출생 연도는?"
Thought 1: BTS 멤버를 떠올려보면... 진이 가장 나이가 많다.
Thought 2: 진의 출생연도는 1992년이다.
Answer: 1992년
→ 문제: 학습 데이터에 의존, 오래된 정보일 수 있음

[Act-only (도구만 사용)]
Action 1: search("BTS 가장 나이 많은 멤버 출생연도")
Observation 1: 진(김석진), 1992년 12월 4일
Answer: 1992년
→ 문제: 검색 결과가 모호하면 다음 행동을 판단하기 어려움

[ReAct (추론 + 행동)]
Thought 1: BTS 멤버 중 가장 나이가 많은 사람을 찾아야 한다.
Action 1:  search("BTS 멤버 나이순 정리")
Observation 1: 진(1992) > 슈가(1993) > 제이홉(1994) > RM(1994) > ...
Thought 2: 진이 1992년생으로 가장 나이가 많다. 확인되었다.
Answer: 진(김석진)의 출생 연도는 1992년입니다.
→ 장점: 추론 과정이 투명하고, 도구로 사실 확인 가능
```

#### ReAct의 Thought / Action / Observation 사이클

```
┌──────────────────────────────────────────────────────┐
│                  ReAct 실행 흐름                       │
│                                                      │
│  사용자 질문                                          │
│      │                                               │
│      ▼                                               │
│  ┌─────────┐                                         │
│  │Thought 1│  "이 질문에 답하려면 X를 먼저 찾아야 한다"│
│  └────┬────┘                                         │
│       ▼                                               │
│  ┌─────────┐                                         │
│  │Action 1 │  search("X 관련 정보")                   │
│  └────┬────┘                                         │
│       ▼                                               │
│  ┌──────────────┐                                    │
│  │Observation 1 │  "X에 대한 검색 결과: ..."          │
│  └────┬─────────┘                                    │
│       ▼                                               │
│  ┌─────────┐                                         │
│  │Thought 2│  "X를 알았으니 이제 Y를 확인해야 한다"    │
│  └────┬────┘                                         │
│       ▼                                               │
│  ┌─────────┐                                         │
│  │Action 2 │  calculate(Y)                            │
│  └────┬────┘                                         │
│       ▼                                               │
│  ┌──────────────┐                                    │
│  │Observation 2 │  "계산 결과: ..."                   │
│  └────┬─────────┘                                    │
│       ▼                                               │
│  ┌─────────┐                                         │
│  │Thought 3│  "충분한 정보를 얻었다. 답변을 구성한다"  │
│  └────┬────┘                                         │
│       ▼                                               │
│  ┌─────────┐                                         │
│  │ Answer  │  최종 답변 반환                           │
│  └─────────┘                                         │
└──────────────────────────────────────────────────────┘
```

**ReAct의 핵심 장점:**
- **투명성**: 추론 과정이 명시적으로 기록되어 디버깅이 용이
- **정확성**: 외부 도구로 사실을 검증하므로 환각(hallucination) 감소
- **유연성**: 중간 결과에 따라 계획을 동적으로 수정 가능
- **추적 가능성**: 각 단계를 로깅하여 에이전트 행동을 분석 가능

### 3. Tool Use / Function Calling

에이전트의 핵심 능력 중 하나는 **도구 사용(Tool Use)**이다. LLM은 자체적으로 웹 검색, 계산, 파일 접근 등을 할 수 없지만, **함수 호출(Function Calling)**을 통해 외부 도구를 실행할 수 있다.

#### Function Calling 흐름

```
┌──────────────────────────────────────────────────────────────┐
│                   Function Calling 실행 흐름                   │
│                                                              │
│  1. 도구 스키마 정의 (개발자)                                  │
│     ┌─────────────────────────────────────┐                  │
│     │ {                                   │                  │
│     │   "name": "get_weather",            │                  │
│     │   "description": "도시의 현재 날씨", │                  │
│     │   "parameters": {                   │                  │
│     │     "city": {"type": "string"},      │                  │
│     │     "unit": {"type": "string"}       │                  │
│     │   }                                 │                  │
│     │ }                                   │                  │
│     └─────────────────────────────────────┘                  │
│                                                              │
│  2. 사용자 메시지 + 도구 목록 → LLM에 전달                     │
│     "서울 날씨 알려줘" + [get_weather, search, ...]           │
│                                                              │
│  3. LLM이 도구 호출 결정                                      │
│     → {"tool": "get_weather", "args": {"city": "서울"}}      │
│                                                              │
│  4. 애플리케이션이 실제 함수 실행                               │
│     → get_weather("서울") → {"temp": 15, "sky": "맑음"}      │
│                                                              │
│  5. 실행 결과를 LLM에 다시 전달                                │
│     → LLM이 자연어 답변 생성                                  │
│     → "서울의 현재 기온은 15도이고 맑습니다."                   │
└──────────────────────────────────────────────────────────────┘
```

#### OpenAI vs Anthropic Function Calling 형식 비교

```
[OpenAI 형식]
tools = [{
    "type": "function",
    "function": {
        "name": "get_weather",
        "description": "주어진 도시의 현재 날씨를 반환",
        "parameters": {
            "type": "object",
            "properties": {
                "city": {"type": "string", "description": "도시 이름"},
            },
            "required": ["city"]
        }
    }
}]

[Anthropic 형식]
tools = [{
    "name": "get_weather",
    "description": "주어진 도시의 현재 날씨를 반환",
    "input_schema": {
        "type": "object",
        "properties": {
            "city": {"type": "string", "description": "도시 이름"},
        },
        "required": ["city"]
    }
}]
```

**도구 스키마가 중요한 이유:**
- LLM은 스키마의 `description`을 읽고 **어떤 도구를 언제 사용할지** 판단
- 파라미터 설명이 명확할수록 LLM이 올바른 인자를 전달할 확률이 높음
- 잘못된 스키마 → 잘못된 도구 호출 → 에이전트 실패

### 4. 메모리 유형 (Memory Types)

에이전트가 효과적으로 작동하려면 다양한 유형의 메모리가 필요하다. 인간의 기억 체계와 유사하게, 에이전트 메모리도 여러 계층으로 구분된다.

```
┌─────────────────────────────────────────────────────────────┐
│                     에이전트 메모리 유형                       │
│                                                             │
│  ┌─────────────────────┐   ┌─────────────────────┐         │
│  │  단기 메모리          │   │  장기 메모리          │         │
│  │  (Short-term)        │   │  (Long-term)         │         │
│  │                     │   │                     │         │
│  │  - 현재 대화 이력    │   │  - 벡터 DB 저장      │         │
│  │  - 컨텍스트 윈도우   │   │  - 과거 대화 요약    │         │
│  │  - 작업 중간 결과    │   │  - 사용자 선호도     │         │
│  │                     │   │  - 학습된 패턴       │         │
│  │  수명: 세션 동안     │   │  수명: 영구 저장     │         │
│  └─────────────────────┘   └─────────────────────┘         │
│                                                             │
│  ┌─────────────────────┐   ┌─────────────────────┐         │
│  │  에피소드 메모리      │   │  작업 메모리          │         │
│  │  (Episodic)         │   │  (Working Memory)    │         │
│  │                     │   │                     │         │
│  │  - 과거 작업 경험    │   │  - 현재 목표         │         │
│  │  - 성공/실패 기록    │   │  - 실행 계획         │         │
│  │  - 유사 상황 참조    │   │  - 중간 추론 결과    │         │
│  │                     │   │  - 활성 컨텍스트     │         │
│  │  수명: 장기 저장     │   │  수명: 작업 동안     │         │
│  └─────────────────────┘   └─────────────────────┘         │
└─────────────────────────────────────────────────────────────┘
```

| 메모리 유형 | 구현 방식 | 사용 시점 | 예시 |
|-----------|---------|---------|------|
| 단기 메모리 | 대화 버퍼, 슬라이딩 윈도우 | 현재 대화 중 | 직전 질문 기억 |
| 장기 메모리 | 벡터 DB (ChromaDB, Pinecone) | 과거 정보 검색 | 사용자 선호도 참조 |
| 에피소드 메모리 | 구조화된 로그 저장 | 유사 작업 수행 시 | "지난번에 이 방법이 성공했다" |
| 작업 메모리 | 상태 변수, 스크래치패드 | 복잡한 추론 중 | 다단계 계산의 중간 결과 |

### 5. 계획 수립 (Planning)

복잡한 작업을 수행하려면 에이전트는 **계획을 세우고 실행**해야 한다. 계획 수립은 큰 목표를 작은 하위 작업으로 분해하는 과정이다.

#### Task Decomposition (작업 분해)

```
[작업 분해 예시]

목표: "한국 AI 스타트업 투자 동향 보고서 작성"

Plan:
├── Step 1: 한국 AI 스타트업 목록 조사
│   ├── 1a: 최근 투자 받은 AI 스타트업 검색
│   └── 1b: 투자 금액 및 투자사 정보 수집
├── Step 2: 투자 트렌드 분석
│   ├── 2a: 연도별 투자 금액 추이 파악
│   └── 2b: 주요 투자 분야 분류 (NLP, CV, MLOps 등)
├── Step 3: 보고서 구성
│   ├── 3a: 데이터 시각화 (표, 차트)
│   └── 3b: 분석 내용 작성
└── Step 4: 보고서 검토 및 완성
    ├── 4a: 사실 확인 (출처 검증)
    └── 4b: 최종 편집
```

#### Plan-and-Execute 패턴

```
┌──────────────────────────────────────────────┐
│           Plan-and-Execute 패턴               │
│                                              │
│  ┌────────────┐                              │
│  │  Planner   │  사용자 요청을 분석하여        │
│  │  (계획자)   │  실행 계획을 수립              │
│  └─────┬──────┘                              │
│        │  계획 (Step 1, 2, 3, ...)            │
│        ▼                                      │
│  ┌────────────┐                              │
│  │  Executor  │  각 단계를 순서대로 실행        │
│  │  (실행자)   │  도구를 사용하여 작업 수행      │
│  └─────┬──────┘                              │
│        │  실행 결과                            │
│        ▼                                      │
│  ┌────────────┐                              │
│  │ Re-planner │  결과를 평가하고               │
│  │ (재계획자)  │  필요시 계획 수정              │
│  └─────┬──────┘                              │
│        │                                      │
│        ▼                                      │
│   완료 또는 수정된 계획으로 재실행              │
└──────────────────────────────────────────────┘
```

**자기 반성(Self-Reflection):**
- 에이전트가 자신의 출력을 검토하고 오류를 수정
- "이 답변이 정확한가?", "빠뜨린 정보가 있는가?" 등의 자문
- Reflexion(2023) 패턴: 실패한 시도를 메모리에 저장하고 다음 시도에 활용

### 6. 프롬프트 엔지니어링 (Prompt Engineering)

에이전트의 성능은 프롬프트 설계에 크게 의존한다. 핵심 기법을 알아보자.

#### 주요 프롬프트 기법

```
┌────────────────────────────────────────────────────────┐
│               프롬프트 엔지니어링 기법                    │
│                                                        │
│  1. Zero-shot CoT                                      │
│     "단계별로 생각해보세요" 한 줄 추가만으로 추론 능력 향상 │
│     → "Let's think step by step."                      │
│                                                        │
│  2. Few-shot                                           │
│     입력-출력 예시를 2~5개 제공하여 패턴 학습 유도        │
│     → 예시: "질문: ... 답변: ..." × 3개                 │
│                                                        │
│  3. Self-Consistency (자기 일관성)                       │
│     동일 질문에 대해 여러 번 생성 후 다수결(majority vote)│
│     → 5번 생성 → 가장 빈번한 답변 선택                  │
│                                                        │
│  4. Structured Output                                  │
│     JSON, XML 등 구조화된 형식으로 출력하도록 지시        │
│     → "다음 JSON 형식으로 답변하세요: {key: value}"      │
│                                                        │
│  5. System Prompt                                      │
│     에이전트의 역할, 규칙, 제약 조건을 사전에 정의        │
│     → "당신은 데이터 분석 전문가입니다. 항상 출처를 밝히세요"│
└────────────────────────────────────────────────────────┘
```

**에이전트 프롬프트 설계 시 주의사항:**
- 도구 사용 조건을 명확히 기술 ("정확한 수치가 필요하면 calculator를 사용하세요")
- 답변 형식을 구체적으로 지정 ("최종 답변은 반드시 Answer: 로 시작하세요")
- 종료 조건을 명시 ("충분한 정보를 얻으면 바로 답변하세요")

### 7. 에이전트 디자인 패턴

복잡한 시스템에서는 여러 에이전트를 조합하는 패턴이 사용된다.

```
[1] Router 패턴 — 하나의 LLM이 요청을 분류하여 적절한 처리 경로로 라우팅

    사용자 요청
        │
        ▼
    ┌────────┐
    │ Router │  "이 질문은 어떤 유형인가?"
    │  LLM   │
    └───┬────┘
        ├── 코드 질문 → Code Agent
        ├── 검색 필요 → Search Agent
        └── 일반 대화 → Chat Agent


[2] Supervisor 패턴 — 하나의 관리자가 여러 전문 에이전트를 조율

    사용자 요청
        │
        ▼
    ┌─────────────┐
    │  Supervisor  │  작업 분배 및 결과 통합
    └──────┬──────┘
           ├── Research Agent (조사)
           ├── Writer Agent (작성)
           └── Reviewer Agent (검토)


[3] Hierarchical 패턴 — 다계층 관리 구조

    사용자 요청
        │
        ▼
    ┌──────────────┐
    │ Top Manager  │
    └──────┬───────┘
           ├── ┌─────────────┐
           │   │ Sub Manager │
           │   └──────┬──────┘
           │          ├── Worker A
           │          └── Worker B
           └── ┌─────────────┐
               │ Sub Manager │
               └──────┬──────┘
                      ├── Worker C
                      └── Worker D


[4] Collaborative 패턴 — 에이전트들이 대등한 위치에서 협업/토론

    ┌──────────┐     ┌──────────┐     ┌──────────┐
    │ Agent A  │────→│ Agent B  │────→│ Agent C  │
    │ (분석가)  │←────│ (개발자)  │←────│ (검증자)  │
    └──────────┘     └──────────┘     └──────────┘
         ↑                                  │
         └──────────────────────────────────┘
                    피드백 루프
```
