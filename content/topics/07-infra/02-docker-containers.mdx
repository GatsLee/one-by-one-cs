## 핵심 개념

### 1. 컨테이너 vs VM (가상 머신)

컨테이너와 VM은 모두 애플리케이션을 격리하여 실행하는 기술이지만, 동작 방식이 근본적으로 다르다.

```
┌─────────────────────────────────────────────────────────────────┐
│                  VM (가상 머신)                                   │
│                                                                 │
│   ┌──────────┐  ┌──────────┐  ┌──────────┐                      │
│   │   App A  │  │   App B  │  │   App C  │                      │
│   ├──────────┤  ├──────────┤  ├──────────┤                      │
│   │  Bins/   │  │  Bins/   │  │  Bins/   │                      │
│   │  Libs    │  │  Libs    │  │  Libs    │                      │
│   ├──────────┤  ├──────────┤  ├──────────┤                      │
│   │ Guest OS │  │ Guest OS │  │ Guest OS │  ← 각각 전체 OS       │
│   │ (Ubuntu) │  │ (CentOS) │  │ (Debian) │                      │
│   └──────────┘  └──────────┘  └──────────┘                      │
│   ─────────────────────────────────────────                     │
│                  Hypervisor                    ← 가상화 계층     │
│   ─────────────────────────────────────────                     │
│                  Host OS (Linux/Windows)                        │
│   ─────────────────────────────────────────                     │
│                  Hardware                                       │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                  컨테이너 (Container)                            │
│                                                                 │
│   ┌──────────┐  ┌──────────┐  ┌──────────┐                      │
│   │   App A  │  │   App B  │  │   App C  │                      │
│   ├──────────┤  ├──────────┤  ├──────────┤                      │
│   │  Bins/   │  │  Bins/   │  │  Bins/   │                      │
│   │  Libs    │  │  Libs    │  │  Libs    │                      │
│   └──────────┘  └──────────┘  └──────────┘                      │
│   ─────────────────────────────────────────                     │
│              Container Runtime (Docker)        ← OS 커널 공유   │
│   ─────────────────────────────────────────                     │
│                  Host OS (Linux)                                │
│   ─────────────────────────────────────────                     │
│                  Hardware                                       │
└─────────────────────────────────────────────────────────────────┘
```

**핵심 차이점:**

| 항목 | VM | 컨테이너 |
|------|----|---------:|
| 격리 수준 | 하드웨어 레벨 (강력) | OS 레벨 (프로세스 격리) |
| OS | 각 VM마다 게스트 OS 전체 | 호스트 OS 커널 공유 |
| 시작 시간 | 분 단위 | 초 단위 |
| 크기 | GB 단위 (OS 포함) | MB 단위 (앱+라이브러리만) |
| 성능 | 하이퍼바이저 오버헤드 | 네이티브에 가까움 |
| 리소스 효율 | 낮음 (OS 중복) | 높음 (커널 공유) |
| 이식성 | 낮음 (이미지 크기) | 높음 (경량 이미지) |

**컨테이너 격리 기술 (리눅스):**
- **namespace**: 프로세스가 볼 수 있는 자원을 격리 (PID, Network, Mount, User 등)
- **cgroup**: CPU, 메모리, I/O 등 자원 사용량을 제한
- **Union filesystem**: 레이어 기반 파일시스템으로 이미지 효율적 관리

### 2. Docker 아키텍처

```
┌──────────────────────────────────────────────────────────────┐
│                       Docker 아키텍처                         │
│                                                              │
│  ┌────────────┐         ┌─────────────────────────────────┐  │
│  │   Docker   │  REST   │        Docker Daemon            │  │
│  │   Client   │──API──▶│        (dockerd)                │  │
│  │            │         │                                 │  │
│  │ docker build│         │  ┌───────────┐  ┌───────────┐  │  │
│  │ docker run  │         │  │ Container │  │ Container │  │  │
│  │ docker pull │         │  │     A     │  │     B     │  │  │
│  └────────────┘         │  └───────────┘  └───────────┘  │  │
│                         │                                 │  │
│                         │  ┌───────────┐  ┌───────────┐  │  │
│                         │  │  Images   │  │  Volumes  │  │  │
│                         │  └───────────┘  └───────────┘  │  │
│                         │  ┌───────────┐                 │  │
│                         │  │ Networks  │                 │  │
│                         │  └───────────┘                 │  │
│                         └─────────────────────────────────┘  │
│                                    │                         │
│                                    ▼                         │
│                         ┌─────────────────┐                  │
│                         │  Docker Registry │                 │
│                         │  (Docker Hub 등)  │                 │
│                         └─────────────────┘                  │
└──────────────────────────────────────────────────────────────┘
```

**구성 요소:**
- **Docker Client**: 사용자가 명령을 입력하는 CLI (`docker` 명령어)
- **Docker Daemon (dockerd)**: 백그라운드에서 이미지, 컨테이너, 네트워크, 볼륨을 관리
- **Docker Registry**: 이미지를 저장/배포하는 저장소 (Docker Hub, GitHub Container Registry 등)
- **Image**: 컨테이너를 만들기 위한 읽기 전용 템플릿 (레이어 구조)
- **Container**: 이미지의 실행 인스턴스 (격리된 프로세스)

#### 이미지 레이어 구조

```
┌─────────────────────────────────────────┐
│           컨테이너 레이어 (R/W)           │  ← 실행 시 생성, 쓰기 가능
├─────────────────────────────────────────┤
│  Layer 4: COPY app.py /app/             │  ← 소스 코드 복사
├─────────────────────────────────────────┤
│  Layer 3: RUN pip install flask         │  ← 의존성 설치
├─────────────────────────────────────────┤
│  Layer 2: RUN apt-get update            │  ← 패키지 업데이트
├─────────────────────────────────────────┤
│  Layer 1: FROM python:3.11-slim         │  ← 베이스 이미지
└─────────────────────────────────────────┘

각 레이어는 읽기 전용(Read-Only)이며,
Dockerfile의 각 명령어가 하나의 레이어를 생성한다.
같은 레이어를 가진 이미지는 레이어를 공유하여 디스크를 절약한다.
```

### 3. Dockerfile

Dockerfile은 이미지를 만들기 위한 설명서(레시피)다.

#### 주요 명령어

```dockerfile
# FROM: 베이스 이미지 지정 (반드시 첫 번째 명령)
FROM python:3.11-slim

# WORKDIR: 작업 디렉터리 설정 (없으면 생성)
WORKDIR /app

# COPY: 호스트 파일을 이미지에 복사
COPY requirements.txt .
COPY . .

# ADD: COPY와 유사하나 URL 다운로드, tar 자동 해제 지원
# → 일반적으로 COPY 사용 권장 (명시적)

# RUN: 이미지 빌드 시 실행할 명령어 (새 레이어 생성)
RUN pip install --no-cache-dir -r requirements.txt

# ENV: 환경변수 설정
ENV FLASK_APP=app.py
ENV FLASK_ENV=production

# ARG: 빌드 시에만 사용되는 변수 (런타임에는 없음)
ARG VERSION=1.0

# EXPOSE: 컨테이너가 사용하는 포트 문서화 (실제 포트 열기 X)
EXPOSE 5000

# CMD: 컨테이너 시작 시 실행할 기본 명령 (덮어쓰기 가능)
CMD ["python", "app.py"]

# ENTRYPOINT: 컨테이너의 메인 실행 파일 (덮어쓰기 어려움)
ENTRYPOINT ["python"]
CMD ["app.py"]    # ENTRYPOINT의 인자로 전달됨
```

#### CMD vs ENTRYPOINT

```
┌─────────────────────────────────────────────────────┐
│                 CMD vs ENTRYPOINT                    │
│                                                     │
│  CMD ["python", "app.py"]                           │
│  → docker run myimage              ▶ python app.py  │
│  → docker run myimage bash         ▶ bash            │
│    (CMD가 완전히 대체됨)                               │
│                                                     │
│  ENTRYPOINT ["python"]                              │
│  CMD ["app.py"]                                     │
│  → docker run myimage              ▶ python app.py  │
│  → docker run myimage test.py      ▶ python test.py │
│    (ENTRYPOINT는 유지, CMD 부분만 대체)               │
│                                                     │
│  정리:                                               │
│  - CMD: 기본 명령, 쉽게 덮어쓰기 가능                  │
│  - ENTRYPOINT: 고정 실행 파일, 인자만 변경 가능         │
│  - 보통 ENTRYPOINT + CMD 조합으로 사용                 │
└─────────────────────────────────────────────────────┘
```

#### 멀티 스테이지 빌드

빌드 도구는 최종 이미지에 포함시키지 않아 이미지 크기를 줄인다.

```dockerfile
# ─── Stage 1: 빌드 ───
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build          # 빌드 결과물 생성 (/app/dist)

# ─── Stage 2: 프로덕션 ───
FROM nginx:alpine
COPY --from=builder /app/dist /usr/share/nginx/html
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]

# 결과: Node.js, npm, 소스코드 없이 빌드 결과물만 포함
# → 이미지 크기: ~1GB → ~30MB
```

### 4. Docker 명령어

```bash
# ─────────── 이미지 관련 ───────────

# 이미지 빌드
docker build -t myapp:1.0 .           # 현재 디렉터리의 Dockerfile로 빌드
docker build -t myapp:1.0 -f Dockerfile.prod .  # 특정 Dockerfile 지정

# 이미지 목록
docker images                          # 로컬 이미지 목록
docker images --format "table {{.Repository}}\t{{.Tag}}\t{{.Size}}"

# 이미지 가져오기/올리기
docker pull python:3.11-slim          # 레지스트리에서 가져오기
docker push myrepo/myapp:1.0          # 레지스트리에 올리기

# 이미지 삭제
docker rmi myapp:1.0                  # 이미지 삭제
docker image prune                     # 사용하지 않는 이미지 정리

# ─────────── 컨테이너 관련 ───────────

# 컨테이너 실행
docker run myapp:1.0                   # 포그라운드 실행
docker run -d myapp:1.0                # 백그라운드 실행 (detach)
docker run -d -p 8080:5000 myapp:1.0   # 포트 매핑 (호스트:컨테이너)
docker run -d --name myapp myapp:1.0   # 이름 지정
docker run -it ubuntu bash             # 대화형 터미널 접속
docker run --rm myapp:1.0              # 종료 시 자동 삭제
docker run -e DB_HOST=localhost myapp   # 환경변수 전달
docker run -v /host/path:/container/path myapp  # 볼륨 마운트

# 컨테이너 목록/상태
docker ps                              # 실행 중인 컨테이너
docker ps -a                           # 모든 컨테이너 (중지 포함)

# 컨테이너 로그
docker logs myapp                      # 전체 로그
docker logs -f myapp                   # 실시간 로그 (follow)
docker logs --tail 100 myapp           # 마지막 100줄

# 컨테이너 내부 명령 실행
docker exec -it myapp bash             # 실행 중인 컨테이너에 셸 접속
docker exec myapp ls /app              # 단일 명령 실행

# 컨테이너 제어
docker stop myapp                      # 정상 종료 (SIGTERM)
docker start myapp                     # 중지된 컨테이너 시작
docker restart myapp                   # 재시작
docker rm myapp                        # 삭제 (중지 상태여야)
docker rm -f myapp                     # 강제 삭제 (실행 중이어도)

# ─────────── 정리 ───────────
docker system prune                    # 중지된 컨테이너, 미사용 이미지/네트워크 정리
docker system prune -a --volumes       # 전체 정리 (주의!)
docker system df                       # 디스크 사용량 확인
```

### 5. Docker Compose

여러 컨테이너를 하나의 파일로 정의하고 함께 관리하는 도구다.

#### docker-compose.yml 기본 구조

```yaml
# docker-compose.yml 구조:
#
# ┌──────────────────────────────────┐
# │  services:                       │  ← 실행할 컨테이너들
# │    web:                          │
# │      image/build                 │
# │      ports                       │
# │      environment                 │
# │      volumes                     │
# │      depends_on                  │
# │    db:                           │
# │      ...                         │
# │  volumes:                        │  ← 명명 볼륨 정의
# │  networks:                       │  ← 커스텀 네트워크 정의
# └──────────────────────────────────┘

version: "3.9"      # Compose 파일 형식 버전 (최근에는 생략 가능)

services:
  web:
    build: .                           # 현재 디렉터리의 Dockerfile로 빌드
    # 또는 image: nginx:latest         # 기존 이미지 사용
    ports:
      - "8080:5000"                    # 호스트:컨테이너
    environment:
      - FLASK_ENV=development
      - DATABASE_URL=postgresql://user:pass@db:5432/mydb
    volumes:
      - ./src:/app/src                 # 바인드 마운트 (개발용)
      - app-data:/app/data             # 명명 볼륨
    depends_on:
      - db
      - redis
    restart: unless-stopped            # 재시작 정책
    networks:
      - backend

  db:
    image: postgres:16
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: pass
      POSTGRES_DB: mydb
    volumes:
      - postgres-data:/var/lib/postgresql/data  # 데이터 영속성
    ports:
      - "5432:5432"
    networks:
      - backend
    healthcheck:                       # 헬스체크 설정
      test: ["CMD-SHELL", "pg_isready -U user"]
      interval: 10s
      timeout: 5s
      retries: 5

  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"
    volumes:
      - redis-data:/data
    networks:
      - backend

volumes:                               # 명명 볼륨 정의
  postgres-data:
  redis-data:
  app-data:

networks:                              # 커스텀 네트워크 정의
  backend:
    driver: bridge
```

#### Compose 명령어

```bash
# 서비스 시작/중지
docker compose up                  # 포그라운드 실행
docker compose up -d               # 백그라운드 실행
docker compose up --build          # 이미지 재빌드 후 실행
docker compose down                # 중지 + 컨테이너/네트워크 삭제
docker compose down -v             # 볼륨까지 삭제

# 상태 확인
docker compose ps                  # 서비스 상태
docker compose logs                # 전체 로그
docker compose logs -f web         # 특정 서비스 실시간 로그

# 개별 서비스 제어
docker compose restart web         # 특정 서비스 재시작
docker compose exec web bash       # 서비스 컨테이너에 셸 접속
docker compose run web python manage.py migrate  # 일회성 명령 실행

# 스케일링
docker compose up -d --scale web=3 # web 서비스를 3개로 확장
```

### 6. 볼륨과 네트워크

#### 볼륨 (Volume)

컨테이너의 데이터는 컨테이너가 삭제되면 사라진다. 데이터를 영속적으로 보관하려면 볼륨을 사용한다.

```
┌──────────────────────────────────────────────────────────────┐
│                     볼륨 타입 비교                             │
│                                                              │
│  1) 바인드 마운트 (Bind Mount)                                │
│     호스트의 특정 경로를 컨테이너에 직접 마운트                  │
│                                                              │
│     호스트                      컨테이너                      │
│     ┌──────────┐               ┌──────────┐                  │
│     │ ./src/   │ ─── 동기화 ──▶│ /app/src │                  │
│     └──────────┘               └──────────┘                  │
│     docker run -v ./src:/app/src myapp                       │
│     → 개발 환경에서 코드 변경 즉시 반영에 유용                  │
│                                                              │
│  2) 명명 볼륨 (Named Volume)                                  │
│     Docker가 관리하는 별도 저장 공간                            │
│                                                              │
│     Docker Volume                컨테이너                     │
│     ┌──────────────┐            ┌──────────┐                 │
│     │ postgres-data│ ── 마운트 ─▶│ /var/lib │                 │
│     │  (Docker     │            │ /pg/data │                 │
│     │   관리 영역)  │            └──────────┘                 │
│     └──────────────┘                                         │
│     docker run -v postgres-data:/var/lib/postgresql/data     │
│     → 데이터베이스 데이터 영속성에 적합                         │
│                                                              │
│  3) 익명 볼륨 (Anonymous Volume)                              │
│     이름 없이 자동 생성. 관리 어려움 → 비권장                   │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

| 항목 | 바인드 마운트 | 명명 볼륨 |
|------|-------------|-----------|
| 경로 지정 | 호스트 절대/상대 경로 | Docker가 관리하는 이름 |
| 성능 | OS에 따라 느릴 수 있음 (macOS) | 일관된 성능 |
| 용도 | 개발 환경 (코드 동기화) | 프로덕션 (데이터 영속성) |
| 이식성 | 호스트 경로에 의존 | 이식 가능 |
| 백업 | 호스트에서 직접 | `docker volume` 명령 사용 |

#### 네트워크 (Network)

```
┌─────────────────────────────────────────────────────────────┐
│                   Docker 네트워크 타입                        │
│                                                             │
│  1) bridge (기본값)                                          │
│     같은 브리지 네트워크의 컨테이너끼리 통신 가능               │
│                                                             │
│     ┌────────────────────────────────┐                      │
│     │    bridge network (172.17.0.x)  │                     │
│     │  ┌──────┐  ◀──▶  ┌──────┐     │                      │
│     │  │ web  │         │  db  │     │                      │
│     │  └──────┘         └──────┘     │                      │
│     └────────────────────────────────┘                      │
│     → 서비스 이름으로 DNS 자동 해결 (Compose 환경)            │
│     → web에서 db:5432 로 접근 가능                           │
│                                                             │
│  2) host                                                    │
│     컨테이너가 호스트의 네트워크를 직접 사용                    │
│     → 포트 매핑 불필요, 최고 성능, 격리 약함                   │
│                                                             │
│  3) overlay                                                 │
│     여러 Docker 호스트(Swarm)에 걸친 네트워크                  │
│     → 멀티 호스트 클러스터에서 사용                            │
│                                                             │
│  4) none                                                    │
│     네트워크 없음 (완전 격리)                                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

```bash
# 네트워크 관리
docker network ls                          # 네트워크 목록
docker network create my-network           # 네트워크 생성
docker network inspect my-network          # 네트워크 상세 정보
docker network connect my-network myapp    # 실행 중인 컨테이너를 네트워크에 연결
docker network rm my-network               # 네트워크 삭제
```

### 7. 컨테이너 베스트 프랙티스

#### 최소 베이스 이미지 사용

```dockerfile
# 나쁜 예 — 전체 OS 이미지 (~900MB)
FROM ubuntu:22.04
RUN apt-get update && apt-get install -y python3 python3-pip

# 좋은 예 — 슬림 이미지 (~150MB)
FROM python:3.11-slim

# 더 좋은 예 — Alpine 이미지 (~50MB)
FROM python:3.11-alpine
# 주의: Alpine은 musl libc 사용으로 일부 패키지 호환 문제 가능
```

#### 레이어 캐싱 최적화

```dockerfile
# 나쁜 예 — 코드 변경 시 매번 의존성 다시 설치
COPY . /app
RUN pip install -r requirements.txt

# 좋은 예 — 의존성 파일만 먼저 복사 → 캐시 활용
COPY requirements.txt /app/
RUN pip install --no-cache-dir -r requirements.txt
COPY . /app                 # 코드 변경 시 여기부터만 다시 빌드
```

```
레이어 캐싱 동작:

  COPY requirements.txt .   ← 파일 변경 없으면 캐시 사용
  RUN pip install ...        ← 위가 캐시면 이것도 캐시
  COPY . .                   ← 코드 변경 → 여기부터 새로 빌드
  CMD ["python", "app.py"]   ← 위가 변경되면 이것도 새로 빌드

  → 자주 변하지 않는 것을 위에, 자주 변하는 것을 아래에 배치!
```

## 코드 예시

### Python Flask Dockerfile (프로덕션용)

```dockerfile
# ─── Dockerfile ───

# 베이스 이미지: Python 3.11 슬림 버전
FROM python:3.11-slim AS base

# 환경변수 설정
ENV PYTHONDONTWRITEBYTECODE=1 \
    PYTHONUNBUFFERED=1

# 시스템 의존성 설치 (레이어 캐싱)
RUN apt-get update && \
    apt-get install -y --no-install-recommends \
        curl \
    && rm -rf /var/lib/apt/lists/*

# 비root 사용자 생성
RUN groupadd -r appuser && useradd -r -g appuser -d /app appuser

# 작업 디렉터리 설정
WORKDIR /app

# 의존성 먼저 설치 (레이어 캐싱 최적화)
COPY requirements.txt .
RUN pip install --no-cache-dir --upgrade pip && \
    pip install --no-cache-dir -r requirements.txt

# 애플리케이션 코드 복사
COPY --chown=appuser:appuser . .

# 비root 사용자로 전환
USER appuser

# 포트 문서화
EXPOSE 5000

# 헬스체크
HEALTHCHECK --interval=30s --timeout=3s --start-period=10s --retries=3 \
    CMD curl -f http://localhost:5000/health || exit 1

# Gunicorn으로 프로덕션 실행
CMD ["gunicorn", "--bind", "0.0.0.0:5000", "--workers", "4", "app:app"]
```

### docker-compose.yml (Web + DB + Redis)

```yaml
# docker-compose.yml — 전형적인 웹 애플리케이션 구성

services:
  # ─── 웹 애플리케이션 ───
  web:
    build:
      context: .
      dockerfile: Dockerfile
    container_name: flask-web
    ports:
      - "8080:5000"
    environment:
      - FLASK_ENV=production
      - DATABASE_URL=postgresql://appuser:secret123@db:5432/appdb
      - REDIS_URL=redis://redis:6379/0
      - SECRET_KEY=${SECRET_KEY:-default-dev-key}  # 호스트 환경변수 참조
    volumes:
      - ./uploads:/app/uploads           # 업로드 파일 영속화
    depends_on:
      db:
        condition: service_healthy       # DB가 준비될 때까지 대기
      redis:
        condition: service_started
    restart: unless-stopped
    networks:
      - app-network

  # ─── PostgreSQL 데이터베이스 ───
  db:
    image: postgres:16-alpine
    container_name: flask-db
    environment:
      POSTGRES_USER: appuser
      POSTGRES_PASSWORD: secret123
      POSTGRES_DB: appdb
    volumes:
      - postgres-data:/var/lib/postgresql/data
      - ./init.sql:/docker-entrypoint-initdb.d/init.sql  # 초기화 SQL
    ports:
      - "5432:5432"
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U appuser -d appdb"]
      interval: 10s
      timeout: 5s
      retries: 5
      start_period: 30s
    restart: unless-stopped
    networks:
      - app-network

  # ─── Redis 캐시/세션 ───
  redis:
    image: redis:7-alpine
    container_name: flask-redis
    command: redis-server --appendonly yes --maxmemory 256mb --maxmemory-policy allkeys-lru
    volumes:
      - redis-data:/data
    ports:
      - "6379:6379"
    restart: unless-stopped
    networks:
      - app-network

# ─── 볼륨 정의 ───
volumes:
  postgres-data:
    driver: local
  redis-data:
    driver: local

# ─── 네트워크 정의 ───
networks:
  app-network:
    driver: bridge
```

### 멀티 스테이지 빌드 (React + Nginx)

```dockerfile
# ─── Stage 1: 의존성 설치 ───
FROM node:20-alpine AS deps
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci --production=false

# ─── Stage 2: 빌드 ───
FROM node:20-alpine AS builder
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY . .
RUN npm run build

# ─── Stage 3: 프로덕션 ───
FROM nginx:1.25-alpine AS production

# 커스텀 nginx 설정
COPY nginx.conf /etc/nginx/conf.d/default.conf

# 빌드 결과물만 복사 (node_modules, 소스 코드 제외)
COPY --from=builder /app/dist /usr/share/nginx/html

# 비root 사용자 실행
RUN chown -R nginx:nginx /usr/share/nginx/html

EXPOSE 80

HEALTHCHECK --interval=30s --timeout=3s \
    CMD curl -f http://localhost/ || exit 1

CMD ["nginx", "-g", "daemon off;"]

# 최종 이미지 크기: ~40MB (vs 전체 빌드 환경 ~1.2GB)
```
