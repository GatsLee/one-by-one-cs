## 핵심 개념

### 1. CI/CD 개념

CI/CD는 코드 변경 사항을 자동으로 빌드, 테스트, 배포하는 소프트웨어 개발 방법론이다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                        CI/CD 파이프라인                               │
│                                                                     │
│  개발자                                                              │
│    │                                                                │
│    ▼                                                                │
│  ┌──────┐   ┌──────┐   ┌──────┐   ┌───────┐   ┌────────┐          │
│  │ Code │──▶│ Lint │──▶│ Test │──▶│ Build │──▶│ Deploy │          │
│  │ Push │   │      │   │      │   │       │   │        │          │
│  └──────┘   └──────┘   └──────┘   └───────┘   └────────┘          │
│              ◀──── CI (지속적 통합) ────▶  ◀── CD (지속적 배포) ──▶  │
│                                                                     │
│  ┌─────────────────────────────────────────────────────────────┐    │
│  │  CI (Continuous Integration) — 지속적 통합                   │    │
│  │  코드 변경 시 자동으로 빌드하고 테스트하여 통합 문제를 조기에     │    │
│  │  발견한다. 개발자가 메인 브랜치에 자주 merge하는 관행.           │    │
│  │                                                             │    │
│  │  CD (Continuous Delivery) — 지속적 전달                      │    │
│  │  CI를 통과한 코드를 프로덕션에 배포할 수 있는 상태로 유지한다.    │    │
│  │  실제 배포는 수동 승인 후 진행.                                │    │
│  │                                                             │    │
│  │  CD (Continuous Deployment) — 지속적 배포                    │    │
│  │  CI를 통과한 코드를 자동으로 프로덕션에 배포한다.                │    │
│  │  수동 승인 없이 완전 자동화.                                  │    │
│  └─────────────────────────────────────────────────────────────┘    │
│                                                                     │
│  Delivery vs Deployment 차이:                                       │
│  ┌──────────┐     ┌──────────┐    ┌──────────────┐                  │
│  │  CI 통과  │────▶│ Staging  │───▶│  Production  │                  │
│  └──────────┘     └──────────┘    └──────────────┘                  │
│                        │                  ▲                          │
│                        │   Delivery: 수동 승인 필요                   │
│                        │   Deployment: 자동 배포                     │
│                        └──────────────────┘                          │
└─────────────────────────────────────────────────────────────────────┘
```

**CI/CD를 하는 이유:**
- **조기 버그 발견**: 코드 변경마다 자동 테스트로 문제를 빨리 찾는다
- **통합 비용 감소**: 작은 변경을 자주 merge하여 대규모 충돌을 방지한다
- **빠른 릴리스 사이클**: 수동 과정을 자동화하여 배포 속도를 높인다
- **일관성**: 사람의 실수를 줄이고 동일한 프로세스를 보장한다

### 2. CI 파이프라인

CI 파이프라인은 코드가 push될 때마다 자동으로 실행되는 일련의 작업이다.

```
┌──────────────────────────────────────────────────────────────────┐
│                    CI 파이프라인 상세                              │
│                                                                  │
│  ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐     │
│  │  1. Lint │──▶│  2. Test │──▶│ 3. Build │──▶│4. Report │     │
│  └──────────┘   └──────────┘   └──────────┘   └──────────┘     │
│       │              │              │               │            │
│       ▼              ▼              ▼               ▼            │
│  코드 스타일     단위 테스트     빌드 성공 확인    결과 알림      │
│  문법 검사       통합 테스트     아티팩트 생성     PR에 코멘트    │
│  보안 스캔       커버리지 측정   이미지 생성      Slack 알림     │
│                                                                  │
│  실패 시 ──▶ 파이프라인 중단 + 개발자에게 알림                    │
└──────────────────────────────────────────────────────────────────┘
```

#### 각 단계 상세

**Lint (정적 분석):**
```bash
# Python
ruff check .                  # 코드 스타일 + 버그 탐지
mypy src/                     # 타입 체크
bandit -r src/                # 보안 취약점 스캔

# JavaScript/TypeScript
eslint src/                   # 코드 스타일
prettier --check src/         # 포매팅 검사
```

**Test (테스트):**
```bash
# 단위 테스트 + 커버리지
pytest --cov=src --cov-report=xml tests/

# 통합 테스트 (DB 등 외부 의존성 포함)
pytest tests/integration/ --timeout=60
```

**Build (빌드):**
```bash
# Docker 이미지 빌드
docker build -t myapp:${GITHUB_SHA} .

# 또는 패키지 빌드
python -m build                # Python 패키지
npm run build                  # JavaScript/TypeScript
```

### 3. CD 파이프라인

CD 파이프라인은 CI를 통과한 코드를 실제 환경에 배포한다.

```
┌──────────────────────────────────────────────────────────────────┐
│                    CD 파이프라인 상세                              │
│                                                                  │
│       CI 통과                                                    │
│          │                                                       │
│          ▼                                                       │
│  ┌───────────────┐     ┌───────────────┐     ┌───────────────┐  │
│  │   Staging      │────▶│   Approval    │────▶│  Production   │  │
│  │   배포         │     │   Gate        │     │   배포        │  │
│  └───────────────┘     └───────────────┘     └───────────────┘  │
│          │                     │                     │           │
│          ▼                     ▼                     ▼           │
│  스테이징 환경에       수동 승인 또는          프로덕션 환경에     │
│  자동 배포            자동 통과 조건          배포 실행           │
│  E2E 테스트 실행      (테스트 통과 등)        모니터링 시작       │
│  QA 검증                                    롤백 준비           │
│                                                                  │
│  환경 구성:                                                      │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐                   │
│  │   Dev    │───▶│ Staging  │───▶│   Prod   │                   │
│  │  개발     │    │ 검증      │    │  운영    │                   │
│  │ 자유롭게  │    │ 프로덕션  │    │ 실 사용자│                   │
│  │ 실험     │    │ 과 유사   │    │ 트래픽   │                   │
│  └──────────┘    └──────────┘    └──────────┘                   │
└──────────────────────────────────────────────────────────────────┘
```

### 4. GitHub Actions

GitHub에 내장된 CI/CD 플랫폼이다. 리포지토리에 `.github/workflows/` 디렉터리에 YAML 파일을 작성하여 파이프라인을 정의한다.

#### 기본 구조

```
GitHub Actions 구조:

  Workflow (.yml 파일)
    └── Event (on: push, pull_request 등)
         └── Job (실행 환경 단위)
              └── Step (개별 작업 단위)
                   ├── uses: 액션 사용 (재사용 가능한 단위)
                   └── run: 셸 명령 실행

  ┌──────────────────────────────────────────────────────┐
  │  Workflow: ci.yml                                    │
  │  ┌───────────────────────────────────────────────┐   │
  │  │  Event: push to main / pull_request           │   │
  │  │  ┌─────────────┐    ┌─────────────┐           │   │
  │  │  │  Job: lint   │    │  Job: test  │  ← 병렬   │   │
  │  │  │  Step 1: ... │    │  Step 1: ...│           │   │
  │  │  │  Step 2: ... │    │  Step 2: ...│           │   │
  │  │  └─────────────┘    └─────────────┘           │   │
  │  │          │                   │                 │   │
  │  │          └─────┬─────────────┘                 │   │
  │  │                ▼                               │   │
  │  │       ┌─────────────┐                          │   │
  │  │       │ Job: deploy │  ← lint, test 이후 실행  │   │
  │  │       │  Step 1: ...│                          │   │
  │  │       └─────────────┘                          │   │
  │  └───────────────────────────────────────────────┘   │
  └──────────────────────────────────────────────────────┘
```

#### Workflow YAML 문법

```yaml
# .github/workflows/ci.yml

name: CI Pipeline                    # 워크플로우 이름

# ─── 트리거 조건 ───
on:
  push:
    branches: [main, develop]        # main, develop 브랜치 push 시
  pull_request:
    branches: [main]                 # main으로의 PR 시
  workflow_dispatch:                  # 수동 실행 가능

# ─── 환경변수 (전체 워크플로우) ───
env:
  PYTHON_VERSION: "3.11"
  REGISTRY: ghcr.io

# ─── 작업 정의 ───
jobs:
  # Job 1: 코드 품질 검사
  lint:
    runs-on: ubuntu-latest           # 실행 환경 (GitHub 호스팅 러너)
    steps:
      - name: 코드 체크아웃
        uses: actions/checkout@v4    # 리포지토리 코드 가져오기

      - name: Python 설정
        uses: actions/setup-python@v5
        with:
          python-version: ${{ env.PYTHON_VERSION }}
          cache: pip                 # pip 캐시 사용

      - name: 의존성 설치
        run: |
          pip install ruff mypy

      - name: Lint 실행
        run: |
          ruff check .
          mypy src/

  # Job 2: 테스트 (matrix 전략)
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:                        # 여러 조합으로 테스트
        python-version: ["3.10", "3.11", "3.12"]
      fail-fast: false               # 하나 실패해도 나머지 계속

    steps:
      - uses: actions/checkout@v4

      - name: Python ${{ matrix.python-version }} 설정
        uses: actions/setup-python@v5
        with:
          python-version: ${{ matrix.python-version }}
          cache: pip

      - name: 의존성 설치
        run: |
          pip install -r requirements.txt
          pip install pytest pytest-cov

      - name: 테스트 실행
        run: |
          pytest --cov=src --cov-report=xml tests/

      - name: 커버리지 업로드
        if: matrix.python-version == '3.11'  # 조건부 실행
        uses: codecov/codecov-action@v4
        with:
          file: coverage.xml
          token: ${{ secrets.CODECOV_TOKEN }}

  # Job 3: 빌드 및 배포 (lint, test 완료 후)
  deploy:
    needs: [lint, test]              # 의존하는 job 지정
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'  # main 브랜치만

    steps:
      - uses: actions/checkout@v4

      - name: Docker 이미지 빌드
        run: |
          docker build -t ${{ env.REGISTRY }}/${{ github.repository }}:${{ github.sha }} .

      - name: 레지스트리 로그인
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: 이미지 푸시
        run: |
          docker push ${{ env.REGISTRY }}/${{ github.repository }}:${{ github.sha }}
          docker tag ${{ env.REGISTRY }}/${{ github.repository }}:${{ github.sha }} \
                     ${{ env.REGISTRY }}/${{ github.repository }}:latest
          docker push ${{ env.REGISTRY }}/${{ github.repository }}:latest
```

### 5. 테스트 전략

```
┌──────────────────────────────────────────────────────────────┐
│                     테스트 피라미드                            │
│                                                              │
│                         /\                                   │
│                        /  \                                  │
│                       / E2E\         적은 수, 느림, 비용 높음 │
│                      /  테스트\       UI/시나리오 테스트       │
│                     /────────\                               │
│                    /          \                               │
│                   /  통합 테스트 \     중간 수, 중간 속도      │
│                  / (Integration) \   API/DB 연동 테스트       │
│                 /────────────────\                           │
│                /                  \                           │
│               /    단위 테스트      \   많은 수, 빠름, 비용 낮음│
│              /     (Unit Test)      \  함수/클래스 단위 테스트 │
│             /________________________\                       │
│                                                              │
│  핵심 원칙:                                                   │
│  - 단위 테스트를 가장 많이 작성 (빠르고 안정적)                 │
│  - 통합 테스트로 모듈 간 연동 검증                              │
│  - E2E 테스트는 핵심 시나리오만 (느리고 깨지기 쉬움)            │
└──────────────────────────────────────────────────────────────┘
```

| 테스트 종류 | 범위 | 속도 | 예시 |
|------------|------|------|------|
| 단위 테스트 | 함수/클래스 하나 | 밀리초 | `test_calculate_total()` |
| 통합 테스트 | 모듈 간 연동 | 초 | API 엔드포인트 + DB 테스트 |
| E2E 테스트 | 전체 시스템 | 분 | 브라우저에서 로그인→구매 시나리오 |

```python
# 단위 테스트 예시 (pytest)
def calculate_discount(price: float, rate: float) -> float:
    """할인 금액 계산"""
    if rate < 0 or rate > 1:
        raise ValueError("할인율은 0~1 사이여야 합니다")
    return price * rate

def test_calculate_discount():
    assert calculate_discount(10000, 0.1) == 1000.0

def test_calculate_discount_invalid_rate():
    with pytest.raises(ValueError):
        calculate_discount(10000, 1.5)
```

### 6. 배포 전략

#### Rolling Update (순차 교체)

```
┌──────────────────────────────────────────────────────────────┐
│                    Rolling Update                             │
│                                                              │
│  시작: 모두 v1                                                │
│  ┌────┐  ┌────┐  ┌────┐  ┌────┐                              │
│  │ v1 │  │ v1 │  │ v1 │  │ v1 │                              │
│  └────┘  └────┘  └────┘  └────┘                              │
│                                                              │
│  Step 1: 하나씩 교체                                          │
│  ┌────┐  ┌────┐  ┌────┐  ┌────┐                              │
│  │ v2 │  │ v1 │  │ v1 │  │ v1 │                              │
│  └────┘  └────┘  └────┘  └────┘                              │
│                                                              │
│  Step 2:                                                     │
│  ┌────┐  ┌────┐  ┌────┐  ┌────┐                              │
│  │ v2 │  │ v2 │  │ v1 │  │ v1 │                              │
│  └────┘  └────┘  └────┘  └────┘                              │
│                                                              │
│  완료: 모두 v2                                                │
│  ┌────┐  ┌────┐  ┌────┐  ┌────┐                              │
│  │ v2 │  │ v2 │  │ v2 │  │ v2 │                              │
│  └────┘  └────┘  └────┘  └────┘                              │
│                                                              │
│  장점: 추가 인프라 불필요, 점진적 교체                          │
│  단점: 배포 중 v1+v2 혼재, 롤백 느림                           │
└──────────────────────────────────────────────────────────────┘
```

#### Blue-Green Deployment (블루-그린 배포)

```
┌──────────────────────────────────────────────────────────────┐
│                 Blue-Green Deployment                         │
│                                                              │
│  배포 전:                                                     │
│                    ┌──────────────────┐                       │
│   사용자 ─── LB ──▶│  Blue (v1) 활성  │                       │
│                    └──────────────────┘                       │
│                    ┌──────────────────┐                       │
│                    │  Green (대기)     │                       │
│                    └──────────────────┘                       │
│                                                              │
│  배포 중: Green에 v2 배포 + 검증                               │
│                    ┌──────────────────┐                       │
│   사용자 ─── LB ──▶│  Blue (v1) 활성  │                       │
│                    └──────────────────┘                       │
│                    ┌──────────────────┐                       │
│              QA ──▶│  Green (v2) 검증 │                       │
│                    └──────────────────┘                       │
│                                                              │
│  전환: 로드밸런서를 Green으로 전환                              │
│                    ┌──────────────────┐                       │
│                    │  Blue (v1) 대기  │  ← 롤백용으로 유지     │
│                    └──────────────────┘                       │
│                    ┌──────────────────┐                       │
│   사용자 ─── LB ──▶│  Green (v2) 활성 │                       │
│                    └──────────────────┘                       │
│                                                              │
│  장점: 즉각 롤백 가능 (LB만 전환), 다운타임 없음               │
│  단점: 2배의 인프라 비용                                      │
└──────────────────────────────────────────────────────────────┘
```

#### Canary Deployment (카나리 배포)

```
┌──────────────────────────────────────────────────────────────┐
│                  Canary Deployment                            │
│                                                              │
│  Phase 1: 5% 트래픽을 v2로                                    │
│                         ┌────────────────┐                   │
│                 95% ───▶│  v1 (기존)     │                   │
│   사용자 ─── LB ──┤     └────────────────┘                   │
│                 5%  ───▶┌────────────────┐                   │
│                         │  v2 (카나리)   │                   │
│                         └────────────────┘                   │
│                                                              │
│  Phase 2: 모니터링 후 트래픽 증가 (25%)                       │
│                         ┌────────────────┐                   │
│                 75% ───▶│  v1 (기존)     │                   │
│   사용자 ─── LB ──┤     └────────────────┘                   │
│                 25% ───▶┌────────────────┐                   │
│                         │  v2 (카나리)   │                   │
│                         └────────────────┘                   │
│                                                              │
│  Phase 3: 문제없으면 100%로 전환                               │
│                         ┌────────────────┐                   │
│   사용자 ─── LB ──────▶│  v2 (전체)     │                    │
│                         └────────────────┘                   │
│                                                              │
│  모니터링 지표: 에러율, 응답 시간, CPU/메모리 사용량             │
│  문제 발견 시: v2 트래픽을 0%로 → 즉시 롤백                    │
│                                                              │
│  장점: 위험을 최소화하며 점진적 검증 가능                       │
│  단점: 구현 복잡, 정교한 모니터링 필요                          │
└──────────────────────────────────────────────────────────────┘
```

| 전략 | 다운타임 | 롤백 속도 | 인프라 비용 | 복잡도 |
|------|---------|----------|-----------|--------|
| Rolling Update | 없음 | 느림 | 같음 | 낮음 |
| Blue-Green | 없음 | 즉시 | 2배 | 중간 |
| Canary | 없음 | 빠름 | 약간 추가 | 높음 |

### 7. 아티팩트 관리

```
┌──────────────────────────────────────────────────────────────┐
│                    아티팩트 관리 흐름                          │
│                                                              │
│  소스 코드 ──▶ CI 빌드 ──▶ 아티팩트 ──▶ 레지스트리 ──▶ 배포   │
│                                                              │
│  아티팩트 종류:                                               │
│  ┌────────────────────────────────────────────┐              │
│  │  Docker 이미지  → Docker Hub / GHCR / ECR  │              │
│  │  패키지         → PyPI / npm / Maven       │              │
│  │  바이너리       → GitHub Releases           │              │
│  │  정적 파일      → S3 / CloudFront           │              │
│  └────────────────────────────────────────────┘              │
│                                                              │
│  버전 관리 (Semantic Versioning):                              │
│  ┌────────────────────────────────────────────┐              │
│  │  v MAJOR . MINOR . PATCH                    │              │
│  │    ↑        ↑       ↑                       │              │
│  │    │        │       └ 버그 수정 (하위 호환)   │              │
│  │    │        └ 기능 추가 (하위 호환)           │              │
│  │    └ 호환 깨지는 변경 (Breaking Change)       │              │
│  │                                             │              │
│  │  예: v2.3.1 → v2.4.0 (기능 추가)             │              │
│  │      v2.4.0 → v3.0.0 (호환 깨지는 변경)      │              │
│  └────────────────────────────────────────────┘              │
│                                                              │
│  이미지 태깅 전략:                                            │
│  - myapp:latest          (최신, 비추천 — 불확정)              │
│  - myapp:v1.2.3          (시맨틱 버전)                       │
│  - myapp:abc1234         (커밋 해시 — 정확한 추적)            │
│  - myapp:main-20260301   (브랜치-날짜)                       │
└──────────────────────────────────────────────────────────────┘
```

**롤백 전략:**
```bash
# Docker 이미지 롤백 (이전 버전 태그로 재배포)
docker pull myapp:v1.2.2          # 이전 버전
docker stop myapp-current
docker run -d --name myapp myapp:v1.2.2

# Kubernetes 롤백
kubectl rollout undo deployment/myapp
kubectl rollout undo deployment/myapp --to-revision=3  # 특정 리비전으로
```

### GitHub Actions: Python 프로젝트 전체 CI/CD

```yaml
# .github/workflows/ci-cd.yml
# Python 프로젝트를 위한 전체 CI/CD 파이프라인

name: CI/CD Pipeline

on:
  push:
    branches: [main, develop]
    tags: ['v*']                     # v로 시작하는 태그
  pull_request:
    branches: [main]

env:
  PYTHON_VERSION: "3.11"
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}

jobs:
  # ─── 1단계: 코드 품질 검사 ───
  lint:
    name: 코드 품질 검사
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Python 설정
        uses: actions/setup-python@v5
        with:
          python-version: ${{ env.PYTHON_VERSION }}
          cache: pip

      - name: 린터 설치 및 실행
        run: |
          pip install ruff mypy
          echo "── Ruff 검사 ──"
          ruff check .
          ruff format --check .
          echo "── MyPy 타입 검사 ──"
          mypy src/ --ignore-missing-imports

  # ─── 2단계: 테스트 ───
  test:
    name: 테스트 (Python ${{ matrix.python-version }})
    runs-on: ubuntu-latest
    strategy:
      matrix:
        python-version: ["3.10", "3.11", "3.12"]

    # 서비스 컨테이너 (통합 테스트용 DB)
    services:
      postgres:
        image: postgres:16-alpine
        env:
          POSTGRES_USER: testuser
          POSTGRES_PASSWORD: testpass
          POSTGRES_DB: testdb
        ports:
          - 5432:5432
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5

    steps:
      - uses: actions/checkout@v4

      - name: Python ${{ matrix.python-version }} 설정
        uses: actions/setup-python@v5
        with:
          python-version: ${{ matrix.python-version }}
          cache: pip

      - name: 의존성 설치
        run: |
          pip install -r requirements.txt
          pip install -r requirements-dev.txt

      - name: 테스트 실행
        env:
          DATABASE_URL: postgresql://testuser:testpass@localhost:5432/testdb
        run: |
          pytest tests/ \
            --cov=src \
            --cov-report=xml \
            --cov-report=term-missing \
            -v

      - name: 커버리지 리포트 업로드
        if: matrix.python-version == '3.11'
        uses: actions/upload-artifact@v4
        with:
          name: coverage-report
          path: coverage.xml

  # ─── 3단계: Docker 이미지 빌드 & 푸시 ───
  build:
    name: Docker 이미지 빌드
    needs: [lint, test]
    runs-on: ubuntu-latest
    if: github.event_name == 'push'        # push 이벤트만

    permissions:
      contents: read
      packages: write                       # GHCR 푸시 권한

    outputs:
      image-tag: ${{ steps.meta.outputs.tags }}

    steps:
      - uses: actions/checkout@v4

      - name: Docker Buildx 설정
        uses: docker/setup-buildx-action@v3

      - name: 레지스트리 로그인
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: 메타데이터 추출
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}
          tags: |
            type=sha,prefix=
            type=ref,event=branch
            type=semver,pattern=v{{version}}

      - name: Docker 이미지 빌드 & 푸시
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          cache-from: type=gha
          cache-to: type=gha,mode=max

  # ─── 4단계: 스테이징 배포 ───
  deploy-staging:
    name: 스테이징 배포
    needs: [build]
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/develop'
    environment: staging                    # GitHub Environment 사용

    steps:
      - name: 스테이징 서버에 배포
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.STAGING_HOST }}
          username: deploy
          key: ${{ secrets.DEPLOY_SSH_KEY }}
          script: |
            cd /opt/myapp
            docker compose pull
            docker compose up -d
            echo "스테이징 배포 완료"

  # ─── 5단계: 프로덕션 배포 ───
  deploy-production:
    name: 프로덕션 배포
    needs: [build]
    runs-on: ubuntu-latest
    if: startsWith(github.ref, 'refs/tags/v')  # v 태그만
    environment: production                 # 수동 승인 필요

    steps:
      - name: 프로덕션 서버에 배포
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.PRODUCTION_HOST }}
          username: deploy
          key: ${{ secrets.DEPLOY_SSH_KEY }}
          script: |
            cd /opt/myapp
            docker compose pull
            docker compose up -d --no-deps web
            docker compose exec web python manage.py migrate
            echo "프로덕션 배포 완료: ${{ github.ref_name }}"
```

### Makefile (CI 작업 로컬 실행)

```makefile
# Makefile — CI 파이프라인의 각 단계를 로컬에서 실행
# 사용법: make lint, make test, make build 등

.PHONY: lint test build deploy clean help

PYTHON := python3
IMAGE_NAME := myapp
VERSION := $(shell git describe --tags --always --dirty)

# ─── 기본 타겟 ───
help:  ## 사용 가능한 명령어 목록 표시
	@grep -E '^[a-zA-Z_-]+:.*?## .*$$' $(MAKEFILE_LIST) | \
		awk 'BEGIN {FS = ":.*?## "}; {printf "  \033[36m%-15s\033[0m %s\n", $$1, $$2}'

# ─── 개발 환경 ───
install:  ## 의존성 설치
	pip install -r requirements.txt -r requirements-dev.txt

# ─── CI 단계 ───
lint:  ## 코드 린트 실행
	ruff check .
	ruff format --check .
	mypy src/ --ignore-missing-imports

format:  ## 코드 자동 포매팅
	ruff check --fix .
	ruff format .

test:  ## 테스트 실행 (커버리지 포함)
	pytest tests/ --cov=src --cov-report=term-missing -v

test-unit:  ## 단위 테스트만 실행
	pytest tests/unit/ -v

test-integration:  ## 통합 테스트만 실행
	pytest tests/integration/ -v --timeout=60

# ─── 빌드 ───
build:  ## Docker 이미지 빌드
	docker build -t $(IMAGE_NAME):$(VERSION) .
	docker tag $(IMAGE_NAME):$(VERSION) $(IMAGE_NAME):latest
	@echo "빌드 완료: $(IMAGE_NAME):$(VERSION)"

# ─── 배포 ───
deploy-staging:  ## 스테이징 배포
	docker compose -f docker-compose.staging.yml pull
	docker compose -f docker-compose.staging.yml up -d

deploy-prod:  ## 프로덕션 배포
	@echo "프로덕션 배포: $(VERSION)"
	@read -p "계속하시겠습니까? [y/N] " confirm && [ "$$confirm" = "y" ]
	docker compose -f docker-compose.prod.yml pull
	docker compose -f docker-compose.prod.yml up -d

# ─── 정리 ───
clean:  ## 임시 파일 및 캐시 정리
	find . -type d -name __pycache__ -exec rm -rf {} +
	find . -type f -name "*.pyc" -delete
	rm -rf .pytest_cache .mypy_cache .ruff_cache htmlcov coverage.xml
	docker image prune -f

# ─── 전체 CI 파이프라인 (로컬) ───
ci: lint test build  ## 전체 CI 파이프라인 로컬 실행
	@echo "전체 CI 파이프라인 완료!"
```
