## 핵심 개념

### 1. 데이터 모델링: 관계형 vs 비정규화

데이터베이스 설계의 출발점은 **데이터를 어떤 구조로 저장할 것인가**이다. 관계형 모델과 비정규화는 서로 다른 트레이드오프를 갖는다.

#### 정규화 (Normalization)

정규화란 **데이터 중복을 최소화**하도록 테이블을 분리하는 설계 방식이다. 데이터 일관성을 보장하지만, 읽기 시 JOIN이 많아진다.

```
정규화된 설계 (3NF):

┌──────────────────┐     ┌──────────────────┐     ┌──────────────────┐
│    users          │     │    posts          │     │    comments      │
├──────────────────┤     ├──────────────────┤     ├──────────────────┤
│ id (PK)          │←─┐  │ id (PK)          │←─┐  │ id (PK)          │
│ name             │  │  │ user_id (FK) ────│──┘  │ post_id (FK) ────│──┘
│ email            │  │  │ title            │     │ user_id (FK) ────│──┐
│ created_at       │  │  │ content          │     │ body             │  │
└──────────────────┘  │  │ created_at       │     │ created_at       │  │
                      │  └──────────────────┘     └──────────────────┘  │
                      └─────────────────────────────────────────────────┘

- 장점: 데이터 중복 없음, 수정 시 한 곳만 변경
- 단점: 읽기 시 여러 테이블 JOIN 필요 → 대규모에서 느림
```

#### 비정규화 (Denormalization)

비정규화란 **읽기 성능을 위해 의도적으로 데이터를 중복 저장**하는 설계이다. JOIN을 줄여 읽기가 빨라지지만, 쓰기 시 여러 곳을 갱신해야 한다.

```
비정규화된 설계:

┌──────────────────────────────────┐
│         posts_denormalized        │
├──────────────────────────────────┤
│ id (PK)                          │
│ title                            │
│ content                          │
│ author_name      ← 중복 저장!     │
│ author_avatar    ← 중복 저장!     │
│ comment_count    ← 미리 계산!     │
│ created_at                       │
└──────────────────────────────────┘

- 장점: 단일 테이블 조회 → 매우 빠른 읽기
- 단점: author_name 변경 시 모든 관련 행 갱신 필요
```

**비유**: 정규화는 "원본 문서는 하나만 보관하고, 필요할 때마다 원본을 찾아보는 것"이다. 비정규화는 "복사본을 여기저기 붙여두는 것"이다. 복사본이 많으면 찾기는 빠르지만, 원본이 바뀌면 복사본을 전부 고쳐야 한다.

| 비교 항목 | 정규화 | 비정규화 |
|-----------|--------|----------|
| 읽기 성능 | 느림 (JOIN 필요) | 빠름 (단일 조회) |
| 쓰기 성능 | 빠름 (한 곳만 수정) | 느림 (여러 곳 갱신) |
| 데이터 일관성 | 높음 | 불일치 위험 |
| 스토리지 | 적음 | 중복으로 증가 |
| 적합 상황 | 쓰기 중심, 트랜잭션 필수 | 읽기 중심, 대규모 트래픽 |

### 2. 샤딩 전략 (Sharding)

샤딩이란 **하나의 대규모 데이터베이스를 여러 개의 작은 조각(샤드)으로 분할**하여 서로 다른 서버에 저장하는 기법이다. 단일 서버의 용량/성능 한계를 극복한다.

```
샤딩 전:
┌──────────────────────────────────────┐
│           단일 DB 서버                 │
│   1억 건의 사용자 데이터               │
│   읽기/쓰기 모두 한 서버에 집중         │
│   CPU/RAM/디스크 한계 도달             │
└──────────────────────────────────────┘

샤딩 후:
┌─────────────┐  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐
│   샤드 1     │  │   샤드 2     │  │   샤드 3     │  │   샤드 4     │
│  user_id    │  │  user_id    │  │  user_id    │  │  user_id    │
│  1 ~ 25M    │  │  25M ~ 50M  │  │  50M ~ 75M  │  │  75M ~ 100M │
│  서버 A      │  │  서버 B      │  │  서버 C      │  │  서버 D      │
└─────────────┘  └─────────────┘  └─────────────┘  └─────────────┘
```

#### (1) 범위 기반 샤딩 (Range-Based Sharding)

샤딩 키의 **값 범위**에 따라 데이터를 분배한다.

```
범위 기반 샤딩 예시 (user_id 기준):

  user_id 1~999        → 샤드 A
  user_id 1000~1999    → 샤드 B
  user_id 2000~2999    → 샤드 C

      요청: user_id = 1500
         ↓
  ┌──────────────────┐
  │   라우팅 로직      │  → 1000 ≤ 1500 < 2000 → 샤드 B
  └──────────────────┘
         ↓
      샤드 B에 쿼리
```

- **장점**: 구현이 단순, 범위 쿼리(날짜 범위 검색 등)가 효율적
- **단점**: 핫스팟 위험 (최근 가입 사용자가 한 샤드에 몰림)

#### (2) 해시 기반 샤딩 (Hash-Based Sharding)

샤딩 키를 **해시 함수**에 넣어 어떤 샤드로 갈지 결정한다.

```
해시 기반 샤딩:

  shard_index = hash(user_id) % N   (N = 샤드 수)

  hash("user_123") % 4 = 2  → 샤드 C
  hash("user_456") % 4 = 0  → 샤드 A
  hash("user_789") % 4 = 3  → 샤드 D

┌────────┐  ┌────────┐  ┌────────┐  ┌────────┐
│ 샤드 A  │  │ 샤드 B  │  │ 샤드 C  │  │ 샤드 D  │
│ (idx 0) │  │ (idx 1) │  │ (idx 2) │  │ (idx 3) │
└────────┘  └────────┘  └────────┘  └────────┘
```

- **장점**: 데이터가 균등 분배, 핫스팟 방지
- **단점**: 범위 쿼리 불가능, 샤드 추가/제거 시 대량 재분배 (일관된 해싱으로 완화)

#### (3) 디렉토리 기반 샤딩 (Directory-Based Sharding)

**별도의 매핑 테이블(디렉토리)**이 각 레코드가 어느 샤드에 있는지 관리한다.

```
디렉토리 기반 샤딩:

  ┌────────────────────────┐
  │    Lookup Service       │
  │ (디렉토리 / 매핑 테이블)  │
  │                        │
  │  user_id → shard_id    │
  │  ──────────────────    │
  │  user_1   → 샤드 A     │
  │  user_2   → 샤드 C     │
  │  user_3   → 샤드 B     │
  └──────────┬─────────────┘
             │ 라우팅
    ┌────────┼────────┐
    ▼        ▼        ▼
  샤드 A   샤드 B   샤드 C
```

- **장점**: 유연함 (아무 기준으로나 분배 가능), 샤드 추가/이동 용이
- **단점**: 디렉토리가 SPOF, 모든 쿼리가 디렉토리를 경유 → 병목 가능

### 3. 읽기/쓰기 분리와 CQRS

대규모 서비스에서 읽기와 쓰기의 비율은 보통 **읽기 80~95% : 쓰기 5~20%**이다. 이 차이를 활용하여 읽기 경로와 쓰기 경로를 분리하면 성능을 최적화할 수 있다.

#### 읽기 복제 (Read Replica)

```
기본 읽기/쓰기 분리:

  ┌──────────┐
  │ 애플리케이션 │
  └─────┬────┘
        │
    ┌───┴───┐
    ▼       ▼
  쓰기    읽기
    │       │
    ▼       ▼
┌────────┐  ┌────────────────────────────────┐
│ Master │→→│ Replica 1 │ Replica 2 │ Replica 3│
│ (쓰기) │  │  (읽기)   │   (읽기)   │  (읽기)  │
└────────┘  └────────────────────────────────┘
     │              ↑ 비동기 복제
     └──────────────┘
```

- Master: 모든 쓰기(INSERT, UPDATE, DELETE) 처리
- Replica: 읽기(SELECT)만 처리, Master로부터 비동기 복제
- **주의**: 복제 지연(Replication Lag)으로 인해 쓰기 직후 읽기 시 오래된 데이터가 반환될 수 있음

#### CQRS (Command Query Responsibility Segregation)

CQRS는 **명령(Command, 쓰기)과 조회(Query, 읽기)의 모델을 완전히 분리**하는 패턴이다.

```
CQRS 아키텍처:

  ┌───────────────────────────────────────────────┐
  │                  API Gateway                   │
  └──────────┬──────────────────┬─────────────────┘
             │ 쓰기 요청          │ 읽기 요청
             ▼                  ▼
  ┌───────────────┐   ┌───────────────┐
  │ Command 서비스  │   │ Query 서비스   │
  │ (쓰기 모델)     │   │ (읽기 모델)    │
  └───────┬───────┘   └───────┬───────┘
          │                   │
          ▼                   ▼
  ┌───────────────┐   ┌───────────────┐
  │ 쓰기 DB        │   │ 읽기 DB        │
  │ (정규화, RDB)   │   │ (비정규화, NoSQL)│
  │ PostgreSQL     │   │ Elasticsearch  │
  └───────┬───────┘   └───────────────┘
          │                   ↑
          └── 이벤트 발행 ────→ │
              (Kafka 등)       │ 이벤트 소비 → 읽기 모델 갱신
```

- **장점**: 읽기/쓰기 각각에 최적화된 데이터 모델 사용 가능, 독립적 확장
- **단점**: 최종 일관성(Eventual Consistency), 구현 복잡도 증가

### 4. 인덱싱 전략

인덱스란 **데이터 검색 속도를 높이기 위한 자료구조**이다. 책의 목차처럼, 전체를 훑지 않고 원하는 데이터를 빠르게 찾을 수 있게 해준다.

#### B+트리 (B+ Tree)

대부분의 관계형 DB(MySQL, PostgreSQL)의 기본 인덱스 구조이다.

```
B+트리 구조 (차수 3):

                    ┌──────────┐
                    │  [30, 60] │        ← 내부 노드 (키만 저장)
                    └─┬──┬──┬──┘
                   ╱   │    ╲
                  ╱    │     ╲
    ┌──────────┐ ┌──────────┐ ┌──────────┐
    │ [10, 20] │ │ [40, 50] │ │ [70, 80] │  ← 내부 노드
    └─┬──┬──┬──┘ └─┬──┬──┬──┘ └─┬──┬──┬──┘
     ╱   │   ╲   ╱   │   ╲   ╱   │   ╲
    ▼    ▼    ▼ ▼    ▼    ▼ ▼    ▼    ▼
  [5,8][12,15][22,28][35,38][42,48][55,58][65,68][72,78][82,90]
    ↔      ↔      ↔      ↔      ↔      ↔      ↔      ↔
                리프 노드 (실제 데이터 포인터, 연결 리스트)

특징:
  - 리프 노드끼리 연결 → 범위 검색(WHERE age BETWEEN 20 AND 50) 효율적
  - O(log N) 검색, 삽입, 삭제
  - 읽기 최적화 (디스크 I/O 최소화)
```

#### LSM 트리 (Log-Structured Merge-Tree)

Cassandra, RocksDB, LevelDB 등 **쓰기 집약적** 워크로드에 사용되는 구조이다.

```
LSM 트리 동작 원리:

  쓰기 요청
     │
     ▼
  ┌─────────────┐
  │  MemTable    │  ← 메모리 (Red-Black Tree / Skip List)
  │  (정렬 저장)  │     빠른 쓰기 O(log N)
  └──────┬──────┘
         │ (크기 초과 시 플러시)
         ▼
  ┌─────────────┐
  │  SSTable L0  │  ← 디스크 (정렬된 불변 파일)
  └──────┬──────┘
         │ (컴팩션)
         ▼
  ┌─────────────┐
  │  SSTable L1  │  ← 더 큰 정렬 파일로 병합
  └──────┬──────┘
         │
         ▼
  ┌─────────────┐
  │  SSTable L2  │  ← 최종 단계
  └─────────────┘

읽기: MemTable → L0 → L1 → L2 순서로 검색
      (Bloom Filter로 불필요한 SSTable 검색 건너뛰기)
```

| 비교 | B+트리 | LSM 트리 |
|------|--------|----------|
| 쓰기 성능 | 보통 (랜덤 I/O) | 우수 (순차 I/O) |
| 읽기 성능 | 우수 (직접 검색) | 보통 (여러 레벨 탐색) |
| 공간 효율 | 보통 | 우수 (컴팩션으로 정리) |
| 범위 쿼리 | 우수 (리프 연결) | 보통 |
| 대표 DB | MySQL, PostgreSQL | Cassandra, RocksDB |
| 적합 워크로드 | 읽기 중심, OLTP | 쓰기 중심, 로그/시계열 |

### 5. 데이터 파티셔닝과 복제

#### 파티셔닝 (Partitioning)

파티셔닝은 하나의 **논리적 테이블을 물리적으로 여러 조각**으로 나누는 것이다. 샤딩과 비슷하지만, 파티셔닝은 보통 단일 DB 인스턴스 내에서의 분할을 의미한다.

```
수평 파티셔닝 (행 단위 분할):

  orders 테이블 전체:
  ┌──────────────────────────────┐
  │ 2024년 주문 | 2025년 주문 | 2026년 주문 │
  └──────────────────────────────┘
                 ↓ created_at 기준 파티셔닝
  ┌────────────┐ ┌────────────┐ ┌────────────┐
  │ orders_2024 │ │ orders_2025 │ │ orders_2026 │
  │  (120만 건)  │ │  (200만 건)  │ │  (50만 건)   │
  └────────────┘ └────────────┘ └────────────┘

  SELECT * FROM orders WHERE created_at = '2026-02-15'
  → orders_2026 파티션만 검색 (파티션 프루닝)

수직 파티셔닝 (열 단위 분할):

  users 테이블:
  ┌──────────────────────────────────────────┐
  │ id | name | email | bio | profile_image  │
  └──────────────────────────────────────────┘
                 ↓ 자주 쓰는 열 / 드물게 쓰는 열 분리
  ┌──────────────────────┐  ┌──────────────────────┐
  │ users_core            │  │ users_profile         │
  │ id | name | email     │  │ id | bio | profile_image │
  │ (자주 접근, 작은 크기)  │  │ (드물게 접근, 큰 크기)    │
  └──────────────────────┘  └──────────────────────┘
```

#### 복제 (Replication)

복제란 **동일한 데이터를 여러 서버에 복사**하여 가용성과 읽기 성능을 높이는 기법이다.

```
복제 방식:

1) 단일 리더 (Single-Leader)
   ┌────────┐
   │ Leader  │──→ Follower 1
   │ (쓰기)  │──→ Follower 2
   └────────┘──→ Follower 3

2) 다중 리더 (Multi-Leader)
   ┌────────┐ ←──→ ┌────────┐
   │Leader A │     │Leader B │
   └────┬───┘     └────┬───┘
        │              │
   Follower A      Follower B
   (각 리더가 독립적으로 쓰기 수용 → 충돌 해결 필요)

3) 리더 없음 (Leaderless) — Dynamo 방식
   ┌────────┐  ┌────────┐  ┌────────┐
   │ Node A  │  │ Node B  │  │ Node C  │
   └────────┘  └────────┘  └────────┘
   W=2, R=2 (정족수: W+R > N이면 최신 데이터 보장)
```

### 6. 실전 사례

#### 사례 1: 트위터 타임라인 DB 설계

```
핵심 문제: 사용자의 타임라인(홈 피드)을 어떻게 구성할 것인가?

테이블 설계:
┌─────────────┐  ┌─────────────┐  ┌──────────────────┐
│   users      │  │   tweets     │  │   follows         │
├─────────────┤  ├─────────────┤  ├──────────────────┤
│ id (PK)     │  │ id (PK)     │  │ follower_id (FK) │
│ username    │  │ user_id(FK) │  │ followee_id (FK) │
│ display_name│  │ content     │  │ created_at       │
└─────────────┘  │ created_at  │  └──────────────────┘
                 │ media_urls  │
                 └─────────────┘

타임라인 구성 방법:
┌────────────────────────────────────────────────────┐
│ 방법 1 (Fan-out on Read, Pull 모델):                 │
│                                                    │
│  타임라인 요청 시 실시간으로 쿼리:                      │
│  SELECT t.* FROM tweets t                          │
│  JOIN follows f ON t.user_id = f.followee_id       │
│  WHERE f.follower_id = :me                         │
│  ORDER BY t.created_at DESC LIMIT 100              │
│                                                    │
│  → 대규모에서 매우 느림 (팔로잉 수 × 트윗 수 조인)     │
├────────────────────────────────────────────────────┤
│ 방법 2 (Fan-out on Write, Push 모델):                │
│                                                    │
│  트윗 작성 시 팔로워들의 타임라인 캐시에 미리 삽입:      │
│                                                    │
│  timeline_cache (Redis Sorted Set):                │
│    timeline:{user_id} → [(tweet_id, timestamp), ...]│
│                                                    │
│  → 읽기 O(1), 쓰기 시 팔로워 수만큼 삽입 필요         │
│  → 팔로워가 수백만인 셀럽 계정은 쓰기 폭발              │
├────────────────────────────────────────────────────┤
│ 트위터의 실제 하이브리드 접근:                          │
│  - 일반 사용자: Fan-out on Write                     │
│  - 셀럽 (팔로워 > 100만): Fan-out on Read            │
│  - 타임라인 조회 시 두 결과를 병합                     │
└────────────────────────────────────────────────────┘
```

#### 사례 2: 채팅 시스템 DB 설계

```
채팅 시스템의 메시지 저장:

┌─────────────┐  ┌──────────────────┐  ┌─────────────────┐
│   users      │  │   conversations   │  │   messages       │
├─────────────┤  ├──────────────────┤  ├─────────────────┤
│ id (PK)     │  │ id (PK)          │  │ id (PK)         │
│ username    │  │ type (1:1/group) │  │ conversation_id  │
│ status      │  │ created_at       │  │ sender_id       │
└─────────────┘  └──────────────────┘  │ content         │
                                       │ type (text/img) │
┌──────────────────┐                   │ created_at      │
│ conversation_     │                   │ status          │
│ participants      │                   │ (sent/delivered │
├──────────────────┤                   │  /read)         │
│ conversation_id   │                   └─────────────────┘
│ user_id           │
│ joined_at         │
│ last_read_at      │    ← 읽음 확인 기준
└──────────────────┘

인덱스 전략:
  messages: (conversation_id, created_at) 복합 인덱스
           → 특정 대화의 최신 메시지 빠르게 조회

  대규모 시 고려사항:
  - 메시지 테이블을 conversation_id 기준으로 샤딩
  - 오래된 메시지는 Cold Storage로 이동 (티어링)
  - 최근 메시지만 핫 스토리지(SSD)에 유지
```
## 정리

| 개념 | 핵심 포인트 |
|------|------------|
| 정규화 | 데이터 중복 최소화, 일관성 보장, 읽기 시 JOIN 필요 |
| 비정규화 | 읽기 최적화를 위한 의도적 중복, 쓰기 시 갱신 비용 증가 |
| 범위 기반 샤딩 | 키 범위로 분배, 범위 쿼리 효율적이지만 핫스팟 위험 |
| 해시 기반 샤딩 | 해시 함수로 균등 분배, 범위 쿼리 불가 |
| 디렉토리 기반 샤딩 | 매핑 테이블로 유연한 분배, 디렉토리가 SPOF |
| 일관된 해싱 | 노드 추가/제거 시 최소한의 데이터만 이동 |
| CQRS | 읽기/쓰기 모델 분리, 각각 최적화 가능, 최종 일관성 |
| B+트리 | 읽기 최적화 인덱스, 범위 쿼리 효율적 (MySQL, PostgreSQL) |
| LSM 트리 | 쓰기 최적화 인덱스, 순차 I/O 활용 (Cassandra, RocksDB) |
| 읽기 복제 | Master-Replica로 읽기 확장, 복제 지연 주의 |
