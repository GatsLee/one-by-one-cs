## 핵심 개념

### 1. 확장성(Scalability)이란?

확장성이란 시스템이 **증가하는 부하(트래픽, 데이터, 사용자)에 대응하여 성능을 유지하는 능력**이다. 서비스가 성장하면 단일 서버로는 한계가 온다. 이때 두 가지 확장 전략이 있다.

### 2. 수직 확장 vs 수평 확장

```
┌─────────────────────────────────────────────────────────┐
│                    수직 확장 (Scale Up)                    │
│                                                         │
│   ┌──────────┐         ┌──────────────┐                 │
│   │  서버 1   │   →→→   │   서버 1      │                │
│   │ 4GB RAM  │         │  64GB RAM    │                 │
│   │ 2 CPU    │         │  32 CPU      │                 │
│   └──────────┘         └──────────────┘                 │
│   (기존 서버)           (더 강력한 서버)                   │
│                                                         │
│   - 하드웨어 업그레이드 (CPU, RAM, SSD 추가)               │
│   - 단일 서버의 한계 존재 (물리적 상한)                     │
│   - 단일 장애점(SPOF) 위험                                │
│   - 구현이 간단 (코드 수정 불필요)                         │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│                    수평 확장 (Scale Out)                   │
│                                                         │
│                   ┌──────────┐                          │
│                   │ 로드밸런서 │                          │
│                   └────┬─────┘                          │
│              ┌─────────┼─────────┐                      │
│              ▼         ▼         ▼                      │
│         ┌────────┐┌────────┐┌────────┐                  │
│         │ 서버 1  ││ 서버 2  ││ 서버 3  │                 │
│         │ 4GB    ││ 4GB    ││ 4GB    │                  │
│         └────────┘└────────┘└────────┘                  │
│                                                         │
│   - 동일한 사양의 서버를 여러 대 추가                       │
│   - 이론적으로 무한 확장 가능                              │
│   - 장애 발생 시 다른 서버가 대체 (고가용성)                │
│   - 상태 관리, 세션 동기화 등 복잡도 증가                   │
└─────────────────────────────────────────────────────────┘
```

**비유**: 수직 확장은 "더 큰 트럭을 사는 것"이고, 수평 확장은 "트럭 여러 대를 운행하는 것"이다.

| 비교 항목 | 수직 확장 (Scale Up) | 수평 확장 (Scale Out) |
|-----------|---------------------|----------------------|
| 방법 | 하드웨어 성능 향상 | 서버 수 증가 |
| 비용 | 고성능 장비일수록 기하급수적 | 저렴한 장비 여러 대 |
| 한계 | 물리적 상한 존재 | 이론적 무한 확장 |
| 복잡도 | 낮음 (기존 코드 유지) | 높음 (분산 시스템 설계) |
| 가용성 | SPOF 위험 | 높은 가용성 |
| 적합 시점 | 초기 서비스, 소규모 | 대규모 트래픽 서비스 |

### 3. 캐싱(Caching)이란?

캐싱은 **자주 요청되는 데이터를 빠른 저장소에 미리 저장해두어 응답 속도를 높이는 기법**이다.

```
캐시 없는 경우:
  클라이언트 ──요청──→ 서버 ──쿼리──→ DB (100ms)
  클라이언트 ←─응답── 서버 ←─결과── DB

캐시 있는 경우:
  클라이언트 ──요청──→ 서버 ──조회──→ 캐시 (1ms) ← Cache Hit!
  클라이언트 ←─응답── 서버 ←─결과── 캐시

  (캐시에 없으면)
  클라이언트 ──요청──→ 서버 ──쿼리──→ DB (100ms) ← Cache Miss
  클라이언트 ←─응답── 서버 ←─결과── DB
                              └──저장──→ 캐시
```

**비유**: 도서관에서 책을 빌릴 때, 자주 빌리는 책은 책상 위에 올려두는 것과 같다. 서가까지 갈 필요 없이 바로 집으면 된다.

### 4. 캐시 전략

#### (1) Cache-Aside (Lazy Loading)

가장 일반적인 전략. 애플리케이션이 캐시를 직접 관리한다.

```
읽기 흐름:
1. 캐시에서 데이터 조회
2-a. Cache Hit → 캐시 데이터 반환
2-b. Cache Miss → DB에서 조회 → 캐시에 저장 → 반환

┌──────┐    1.조회     ┌──────┐
│ 앱   │───────────→ │캐시   │  Hit → 반환
│ 서버  │←──────────  │(Redis)│
│      │  2-b.Miss   │      │
│      │─────────→   └──────┘
│      │    DB 조회    ┌──────┐
│      │───────────→ │  DB   │
│      │←──────────  │      │
│      │   결과 반환   └──────┘
│      │─── 캐시 저장 → 캐시
└──────┘
```

- **장점**: 필요한 데이터만 캐싱, 캐시 장애 시에도 DB로 동작 가능
- **단점**: 첫 요청은 항상 느림(Cold Start), 데이터 불일치 가능

#### (2) Write-Through

쓰기 시 캐시와 DB에 동시에 저장한다.

```
쓰기 흐름:
1. 앱 → 캐시에 쓰기
2. 캐시 → DB에 쓰기 (동기)
3. 완료 응답

┌──────┐   1.쓰기    ┌──────┐   2.쓰기    ┌──────┐
│  앱   │──────────→│ 캐시  │──────────→│  DB   │
│      │←──────────│      │←──────────│      │
│      │   3.완료   │      │   동기완료  │      │
└──────┘           └──────┘           └──────┘
```

- **장점**: 캐시와 DB가 항상 일관성 유지
- **단점**: 쓰기 지연 증가 (캐시 + DB 두 번 쓰기), 읽히지 않는 데이터도 캐싱

#### (3) Write-Back (Write-Behind)

쓰기 시 캐시에만 저장하고, DB에는 나중에 비동기적으로 반영한다.

```
쓰기 흐름:
1. 앱 → 캐시에 쓰기 (즉시 완료)
2. 캐시 → DB에 비동기 쓰기 (나중에)

┌──────┐   1.쓰기    ┌──────┐ ···비동기··→ ┌──────┐
│  앱   │──────────→│ 캐시  │             │  DB   │
│      │←──────────│      │             │      │
│      │   즉시완료  │      │──2.배치쓰기──→│      │
└──────┘           └──────┘             └──────┘
```

- **장점**: 쓰기 성능 최고, DB 부하 감소
- **단점**: 캐시 장애 시 데이터 유실 위험, 복잡한 구현

### 5. CDN (Content Delivery Network)

CDN은 **전 세계에 분산된 서버 네트워크**를 통해 정적 콘텐츠(이미지, CSS, JS)를 사용자와 가까운 위치에서 제공한다.

```
CDN 없이:
  한국 사용자 ──────────────→ 미국 원본 서버 (지연 200ms)
  일본 사용자 ──────────────→ 미국 원본 서버 (지연 150ms)

CDN 사용:
  한국 사용자 ──→ 서울 CDN 엣지 (지연 10ms)
  일본 사용자 ──→ 도쿄 CDN 엣지 (지연 10ms)
                       ↕ (주기적 동기화)
                  미국 원본 서버
```

**CDN 동작 원리**:
1. 사용자가 리소스 요청
2. 가장 가까운 CDN 엣지 서버로 라우팅
3. 엣지에 캐시 있으면 → 바로 반환
4. 캐시 없으면 → 원본 서버에서 가져와서 캐시 후 반환

**대표 CDN 서비스**: CloudFlare, AWS CloudFront, Akamai

### 6. Redis 기본 개념과 사용 패턴

Redis는 **인메모리(In-Memory) 키-값(Key-Value) 저장소**로, 캐시, 세션 저장, 메시지 큐 등 다양한 용도로 사용된다.

**핵심 특징**:
- 메모리 기반 → 읽기/쓰기 매우 빠름 (마이크로초 단위)
- 다양한 자료구조 지원: String, List, Set, Hash, Sorted Set
- 단일 스레드 기반 (명령 순서 보장)
- 영속성 옵션: RDB 스냅샷, AOF 로그
- TTL(Time-To-Live) 설정으로 자동 만료

**Redis 주요 자료구조**:

```
String:  "user:1:name" → "김철수"        (단순 키-값)
Hash:    "user:1" → {name: "김철수", age: 25}  (객체 저장)
List:    "queue:tasks" → [task1, task2, task3]  (큐/스택)
Set:     "user:1:tags" → {python, java, go}    (중복 없는 집합)
Sorted Set: "leaderboard" → {(김철수,100), (이영희,95)}  (랭킹)
```

### 7. 캐시 무효화 문제

> "컴퓨터 과학에서 어려운 문제는 두 가지뿐이다: 캐시 무효화와 이름 짓기." — Phil Karlton

캐시 무효화(Cache Invalidation)란 **캐시된 데이터가 더 이상 유효하지 않을 때 이를 갱신하거나 제거하는 것**이다.

**주요 전략**:

| 전략 | 설명 | 장점 | 단점 |
|------|------|------|------|
| TTL | 일정 시간 후 자동 만료 | 간단한 구현 | 만료 전 불일치 가능 |
| 이벤트 기반 | DB 변경 시 캐시 삭제 | 즉각적 갱신 | 이벤트 누락 위험 |
| 버전 태그 | URL에 버전/해시 포함 | 정확한 무효화 | 관리 복잡 |

**캐시 스탬피드(Cache Stampede)**: TTL 만료 직후 동시에 수많은 요청이 DB로 몰리는 현상. 해결책으로 **잠금(Locking)**, **사전 갱신(Preemptive Refresh)**, **확률적 조기 만료** 등이 있다.

## 코드 예시

### Cache-Aside 패턴 구현 (Python + Redis)

```python
import redis
import json
import time

# Redis 연결
cache = redis.Redis(host='localhost', port=6379, db=0, decode_responses=True)

# 가상의 데이터베이스 (딕셔너리로 시뮬레이션)
fake_db = {
    "user:1": {"name": "김철수", "age": 25, "email": "kim@example.com"},
    "user:2": {"name": "이영희", "age": 30, "email": "lee@example.com"},
}


def get_from_db(key: str) -> dict | None:
    """데이터베이스에서 조회하는 함수 (시뮬레이션)"""
    time.sleep(0.1)  # DB 조회 지연 시뮬레이션 (100ms)
    return fake_db.get(key)


def get_user_cache_aside(user_id: str) -> dict | None:
    """
    Cache-Aside 패턴으로 사용자 정보를 조회한다.
    1. 먼저 캐시에서 찾는다 (빠름)
    2. 없으면 DB에서 가져와서 캐시에 저장한다
    """
    cache_key = f"user:{user_id}"

    # 1단계: 캐시에서 조회
    cached = cache.get(cache_key)
    if cached:
        print(f"  [Cache HIT] {cache_key}")
        return json.loads(cached)

    # 2단계: Cache Miss → DB에서 조회
    print(f"  [Cache MISS] {cache_key} → DB 조회 중...")
    data = get_from_db(cache_key)

    if data:
        # 3단계: 캐시에 저장 (TTL 300초 = 5분)
        cache.setex(cache_key, 300, json.dumps(data))
        print(f"  [캐시 저장] {cache_key} (TTL: 300초)")

    return data


def update_user(user_id: str, new_data: dict) -> None:
    """
    사용자 정보 수정 시 캐시를 무효화한다.
    DB를 먼저 수정한 뒤 캐시를 삭제한다 (Cache Invalidation).
    """
    cache_key = f"user:{user_id}"

    # 1단계: DB 업데이트
    fake_db[cache_key] = new_data
    print(f"  [DB 업데이트] {cache_key}")

    # 2단계: 캐시 무효화 (삭제)
    cache.delete(cache_key)
    print(f"  [캐시 삭제] {cache_key}")


# === 실행 예시 ===
if __name__ == "__main__":
    print("=== 첫 번째 조회 (Cache Miss) ===")
    user = get_user_cache_aside("1")
    print(f"  결과: {user}\n")

    print("=== 두 번째 조회 (Cache Hit) ===")
    user = get_user_cache_aside("1")
    print(f"  결과: {user}\n")

    print("=== 사용자 정보 수정 ===")
    update_user("1", {"name": "김철수", "age": 26, "email": "kim_new@example.com"})
    print()

    print("=== 수정 후 조회 (Cache Miss → 새 데이터) ===")
    user = get_user_cache_aside("1")
    print(f"  결과: {user}")
```

### Redis 자료구조 활용 예시

```python
import redis

r = redis.Redis(host='localhost', port=6379, db=0, decode_responses=True)

# === 1. String: 단순 캐싱 ===
r.setex("session:abc123", 3600, "user_id:42")  # 1시간 TTL
session = r.get("session:abc123")

# === 2. Hash: 사용자 프로필 저장 ===
r.hset("user:1", mapping={
    "name": "김철수",
    "age": "25",
    "city": "서울"
})
name = r.hget("user:1", "name")      # "김철수"
all_info = r.hgetall("user:1")        # {"name": "김철수", "age": "25", ...}

# === 3. List: 작업 큐 ===
r.rpush("queue:emails", "email_1", "email_2", "email_3")  # 오른쪽에 추가
task = r.lpop("queue:emails")  # 왼쪽에서 꺼냄 → "email_1" (FIFO)

# === 4. Set: 태그 관리 ===
r.sadd("post:1:tags", "python", "redis", "caching")
r.sadd("post:2:tags", "python", "django", "web")
# 교집합: 두 게시글의 공통 태그
common = r.sinter("post:1:tags", "post:2:tags")  # {"python"}

# === 5. Sorted Set: 리더보드 ===
r.zadd("leaderboard", {"김철수": 100, "이영희": 95, "박민수": 120})
# 상위 3명 (점수 높은 순)
top3 = r.zrevrange("leaderboard", 0, 2, withscores=True)
# [("박민수", 120.0), ("김철수", 100.0), ("이영희", 95.0)]
```

### Write-Through 패턴 간단 구현

```python
import redis
import json

cache = redis.Redis(host='localhost', port=6379, db=0, decode_responses=True)
fake_db = {}


def write_through(key: str, value: dict, ttl: int = 300) -> None:
    """
    Write-Through: 캐시와 DB에 동시에 쓴다.
    항상 일관성을 보장하지만, 쓰기 지연이 늘어난다.
    """
    # 1단계: 캐시에 저장
    cache.setex(key, ttl, json.dumps(value))
    print(f"  [캐시 저장] {key}")

    # 2단계: DB에 저장 (동기)
    fake_db[key] = value
    print(f"  [DB 저장] {key}")


def read_through(key: str) -> dict | None:
    """항상 캐시에서 읽는다 (Write-Through에서는 캐시가 항상 최신)"""
    cached = cache.get(key)
    if cached:
        return json.loads(cached)
    return None


# 사용 예시
write_through("product:1", {"name": "노트북", "price": 1500000})
product = read_through("product:1")
print(f"  조회 결과: {product}")
```

## 정리

| 개념 | 핵심 포인트 |
|------|------------|
| 수직 확장 | 서버 한 대의 성능을 높임. 간단하지만 물리적 한계 존재 |
| 수평 확장 | 서버 수를 늘림. 이론적 무한 확장 가능하나 분산 시스템 복잡도 증가 |
| Cache-Aside | 읽기 시 캐시 확인 후 없으면 DB 조회. 가장 일반적인 패턴 |
| Write-Through | 쓰기 시 캐시+DB 동시 저장. 일관성 보장, 쓰기 느림 |
| Write-Back | 캐시에만 쓰고 나중에 DB 반영. 빠르지만 유실 위험 |
| CDN | 전세계 엣지 서버에 정적 콘텐츠 캐싱. 지연 시간 대폭 감소 |
| Redis | 인메모리 KV 저장소. String/Hash/List/Set/Sorted Set 지원 |
| 캐시 무효화 | TTL, 이벤트 기반, 버전 태그 방식. 캐시 스탬피드 주의 |
