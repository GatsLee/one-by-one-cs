## 핵심 개념

### 1. 분산 시스템이란?

분산 시스템은 네트워크로 연결된 여러 대의 컴퓨터(노드)가 협력하여 하나의 시스템처럼 동작하는 구조다. 사용자 입장에서는 하나의 서비스로 보이지만, 내부적으로는 여러 노드가 데이터를 나누어 저장하고 처리한다.

<Mermaid chart="graph TD
  client[&quot;클라이언트
하나의 시스템으로 보임&quot;]
  client --> nodeA[&quot;노드 A - 서울
데이터 1,2&quot;]
  client --> nodeB[&quot;노드 B - 부산
데이터 2,3&quot;]
  nodeA <-->|동기화| nodeB
  nodeA <-->|동기화| nodeC[&quot;노드 C - 대전
데이터 1,3&quot;]
  nodeB <-->|동기화| nodeC" />

**왜 분산 시스템을 쓰는가?**
- **확장성(Scalability)**: 트래픽이 늘면 노드를 추가하면 된다
- **가용성(Availability)**: 한 노드가 죽어도 다른 노드가 서비스를 계속한다
- **지리적 분산**: 사용자와 가까운 노드에서 빠르게 응답한다

하지만 분산 시스템에는 근본적인 트레이드오프가 존재한다. 이것이 바로 CAP 정리다.

### 2. CAP 정리 (Brewer's Theorem)

2000년 에릭 브루어(Eric Brewer)가 제안하고, 2002년 세스 길버트와 낸시 린치가 증명한 정리다.

> **분산 시스템은 다음 세 가지 속성 중 최대 두 가지만 동시에 보장할 수 있다.**

#### C — Consistency (일관성)
모든 노드가 같은 시점에 같은 데이터를 보는 것. 노드 A에 쓴 데이터를 즉시 노드 B에서 읽을 수 있어야 한다.

```
[일관성 보장]
클라이언트 → 노드A에 "x=5" 기록
클라이언트 → 노드B에서 "x" 읽기 → 반드시 "5" 반환

[일관성 미보장]
클라이언트 → 노드A에 "x=5" 기록
클라이언트 → 노드B에서 "x" 읽기 → "3" 반환 (이전 값)
```

#### A — Availability (가용성)
장애가 나지 않은 모든 노드는 합리적인 시간 안에 응답을 반환해야 한다. "잠시 기다려 주세요"라는 무한 대기가 아니라, 반드시 성공 또는 실패 응답을 준다.

#### P — Partition Tolerance (분단 허용성)
네트워크 분단(partition)이 발생해도 시스템이 계속 동작해야 한다. 분단이란 노드 간 통신이 끊기는 상황이다.

<Mermaid chart="graph LR
  nodeA[&quot;노드 A
서울&quot;] -.-x|네트워크 끊김| nodeB[&quot;노드 B
부산&quot;]" />

서로 통신 불가 → 각자 독립적으로 동작해야 함

#### 왜 세 가지를 동시에 못 하는가?

실제 네트워크에서 분단(P)은 **반드시 발생**한다. 케이블이 끊기거나, 라우터가 고장나거나, 데이터센터가 격리되는 상황은 피할 수 없다. 따라서 P는 선택이 아닌 필수다.

P가 발생한 상황에서:
- **C를 선택하면 (CP)**: 분단된 노드 간 데이터 불일치를 막기 위해, 일부 요청을 거부한다 → A 포기
- **A를 선택하면 (AP)**: 모든 노드가 응답하지만, 분단된 노드 간 데이터가 다를 수 있다 → C 포기

<Mermaid chart="graph TD
  P[&quot;P 분단 허용
실제 시스템에서 필수&quot;]
  P --> CP[&quot;CP
일관성 우선&quot;]
  P --> AP[&quot;AP
가용성 우선&quot;]" />

### 3. CP 시스템 vs AP 시스템

#### CP 시스템 (일관성 + 분단 허용)

분단 발생 시 일관성을 보장하기 위해 일부 요청을 거부한다.

| 시스템 | 설명 |
|--------|------|
| **ZooKeeper** | 분산 코디네이터. 과반수 노드 응답 필요 |
| **etcd** | Kubernetes의 설정 저장소. Raft 합의 사용 |
| **HBase** | 강한 일관성의 분산 데이터베이스 |
| **MongoDB (기본 설정)** | Primary 노드 기반 읽기/쓰기 |

**사용 사례**: 금융 거래, 재고 관리 등 데이터 정확성이 핵심인 시스템

#### AP 시스템 (가용성 + 분단 허용)

분단 발생 시에도 모든 노드가 응답하지만, 노드 간 데이터가 일시적으로 다를 수 있다.

| 시스템 | 설명 |
|--------|------|
| **Cassandra** | 링 구조. 최종 일관성 |
| **DynamoDB** | AWS의 Key-Value 저장소 |
| **CouchDB** | 문서 기반 DB. 충돌 해결 메커니즘 |
| **DNS** | 도메인 이름 시스템 |

**사용 사례**: 소셜 미디어 타임라인, 상품 추천, 로그 수집 등 약간의 불일치가 허용되는 시스템

```
[은행 계좌 이체 — CP 필요]
잔액 100만원 → 50만원 이체
노드A: 잔액 50만원 (정확)
노드B: 잔액 100만원 (불일치!)  ← 이러면 안 됨!
→ CP: 분단 시 이체 거부 (안전)

[SNS 좋아요 수 — AP 가능]
게시글 좋아요 1000개 → +1
노드A: 1001개
노드B: 1000개 (잠시 불일치)  ← 잠깐은 괜찮음
→ AP: 항상 응답, 나중에 동기화
```

### 4. 일관성 모델 (Consistency Models)

분산 시스템에서 "일관성"은 단순한 이분법이 아니라 스펙트럼이다.

#### 강한 일관성 (Strong Consistency / Linearizability)
- 쓰기 완료 후 어떤 노드에서든 최신 값을 읽을 수 있다
- 마치 단일 서버처럼 동작
- 대가: 높은 지연 시간 (모든 노드 동기화 필요)

#### 순차 일관성 (Sequential Consistency)
- 모든 노드가 같은 순서로 연산을 관찰
- 실시간 순서는 보장하지 않음 (약간의 지연 허용)

#### 인과적 일관성 (Causal Consistency)
- 인과 관계가 있는 연산만 순서를 보장
- "A가 B의 원인이면, A가 먼저 보여야 한다"

#### 최종 일관성 (Eventual Consistency)
- 더 이상 쓰기가 없으면, 결국 모든 노드가 같은 값으로 수렴
- 수렴까지 시간이 걸릴 수 있음
- 가장 약하지만 가장 빠르고 가용성 높음

```
강한 일관성 ←─────────────────────→ 최종 일관성
  (느림, 정확)                        (빠름, 느슨)
   │                                      │
   ├─ 은행 거래                    ├─ DNS 전파
   ├─ 재고 관리                    ├─ SNS 좋아요
   └─ 분산 락                      └─ 추천 시스템
```

### 5. Paxos 합의 알고리즘

레슬리 램포트(Leslie Lamport)가 1989년에 제안한 합의 알고리즘이다. 분산 시스템에서 여러 노드가 하나의 값에 동의하는 방법을 정의한다.

#### 핵심 역할
- **Proposer**: 값을 제안하는 노드
- **Acceptor**: 제안을 수락/거부하는 노드
- **Learner**: 합의된 값을 학습하는 노드

#### 2단계 프로토콜

```
[Phase 1: Prepare/Promise]
Proposer                    Acceptor (3개)
   │                         │
   ├── Prepare(n=1) ────────►│ Acceptor1: Promise(n=1) ──►
   ├── Prepare(n=1) ────────►│ Acceptor2: Promise(n=1) ──►
   ├── Prepare(n=1) ────────►│ Acceptor3: Promise(n=1) ──►
   │                         │
   │  과반수(2개 이상) Promise 받음 → Phase 2 진행

[Phase 2: Accept/Accepted]
Proposer                    Acceptor
   │                         │
   ├── Accept(n=1, v="X") ──►│ Acceptor1: Accepted ──►
   ├── Accept(n=1, v="X") ──►│ Acceptor2: Accepted ──►
   ├── Accept(n=1, v="X") ──►│ Acceptor3: (장애) ──X
   │                         │
   │  과반수 Accepted → 합의 완료! 값 "X" 확정
```

**Paxos의 문제점**: 이해하기 어렵고, 구현이 복잡하며, 리더가 없어 여러 Proposer가 충돌할 수 있다(라이브락). 이 문제를 해결하기 위해 Raft가 등장했다.

### 6. Raft 합의 알고리즘

디에고 옹가로(Diego Ongaro)가 2014년에 발표한 합의 알고리즘으로, Paxos와 동등한 안전성을 보장하면서도 이해하기 쉽게 설계되었다. etcd, Consul, CockroachDB 등에서 사용된다.

#### 노드의 세 가지 상태

<Mermaid chart="graph LR
  follower[Follower] -->|타임아웃
heartbeat 없음| candidate[Candidate]
  candidate -->|선거 승리
과반수 득표| leader[Leader]
  leader -->|새 리더 발견 시
Follower로 강등| follower
  candidate -->|투표 과반수 획득 실패| follower" />

#### (1) 리더 선출 (Leader Election)

```
시간 흐름 →

Follower A: ─────────[타임아웃!]──► Candidate ──[투표 요청]──►
Follower B: ─────────────────────── 투표 응답(찬성) ──►
Follower C: ─────────────────────── 투표 응답(찬성) ──►

A가 과반수(2/3) 득표 → Leader로 승격!

Leader A:   ──[heartbeat]──[heartbeat]──[heartbeat]──►
Follower B: ──────────────────────────────────────────►
Follower C: ──────────────────────────────────────────►
```

**선거 규칙:**
- 각 노드는 무작위 타임아웃(150~300ms)을 가짐
- 타임아웃이 먼저 끝난 노드가 후보(Candidate)가 됨
- 후보는 자기 자신에게 투표하고 다른 노드에 투표 요청
- 과반수 득표하면 리더가 됨
- 각 임기(Term)에서 한 노드는 한 번만 투표 가능

#### (2) 로그 복제 (Log Replication)

리더가 클라이언트의 쓰기 요청을 받아 다른 노드에 복제하는 과정이다.

```
클라이언트: "x=5 저장해줘"
       │
       ▼
┌─────────────┐
│  Leader (A)  │ ① 로그에 "x=5" 기록 (uncommitted)
│  로그: [x=5] │
└──────┬──────┘
       │ ② 로그 복제 요청 (AppendEntries RPC)
       ├──────────────────────┐
       ▼                      ▼
┌─────────────┐        ┌─────────────┐
│ Follower (B) │        │ Follower (C) │
│ 로그: [x=5]  │        │ 로그: [x=5]  │
└──────┬──────┘        └──────┬──────┘
       │ ③ 복제 성공 응답      │ ③ 복제 성공 응답
       └──────┬───────────────┘
              ▼
┌─────────────┐
│  Leader (A)  │ ④ 과반수 확인 → commit!
│  x=5 확정    │ ⑤ 클라이언트에 성공 응답
└─────────────┘
```

**커밋 규칙:**
- 과반수(N/2 + 1) 노드에 복제되면 커밋
- 커밋된 로그는 절대 삭제되지 않음
- 리더는 주기적 heartbeat로 팔로워에 커밋 정보 전달

#### (3) 안전성 보장

- **선거 제한**: 로그가 최신이 아닌 후보에게는 투표하지 않음
- **리더 완전성**: 커밋된 로그를 가진 노드만 리더가 될 수 있음
- **로그 매칭**: 같은 인덱스와 같은 Term의 로그는 같은 명령을 가짐

## 정리

| 개념 | 핵심 내용 | 사용 사례 |
|------|-----------|-----------|
| **CAP 정리** | C, A, P 중 2개만 동시 보장 가능 | 분산 시스템 설계의 기본 원칙 |
| **CP 시스템** | 일관성 우선, 분단 시 일부 요청 거부 | 금융, 재고 (ZooKeeper, etcd) |
| **AP 시스템** | 가용성 우선, 일시적 불일치 허용 | SNS, 추천 (Cassandra, DynamoDB) |
| **강한 일관성** | 쓰기 후 어디서든 최신 값 보장 | 계좌 잔액, 분산 락 |
| **최종 일관성** | 시간이 지나면 결국 같은 값으로 수렴 | DNS, 캐시, 로그 수집 |
| **Paxos** | 최초의 합의 알고리즘, 이해 어려움 | 학술적 기반, Google Chubby |
| **Raft** | 이해 쉬운 합의 알고리즘, 리더 기반 | etcd, Consul, CockroachDB |
| **Raft 리더 선출** | 무작위 타임아웃, 과반수 투표 | 리더 장애 시 자동 복구 |
| **Raft 로그 복제** | 리더→팔로워 복제, 과반수 커밋 | 분산 상태 머신 복제 |
