## 핵심 개념

채팅 시스템은 실시간 통신, 메시지 전달 보장, 대규모 동시 접속 처리 등 분산 시스템의 핵심 과제를 종합적으로 다루는 대표적인 설계 문제이다. WhatsApp, Slack, 카카오톡 같은 서비스가 어떻게 동작하는지 이해하는 데 초점을 맞춘다.

### 1. 요구사항 정의

```
┌─────────────────────────────────────────────────────────┐
│                  기능적 요구사항 (FR)                       │
├─────────────────────────────────────────────────────────┤
│ 1. 1:1 채팅: 두 사용자 간 실시간 메시지 송수신             │
│ 2. 그룹 채팅: 최대 500명의 그룹에서 메시지 송수신          │
│ 3. 온라인 상태 표시: 사용자의 접속 여부 표시               │
│ 4. 읽음 확인: 상대방이 메시지를 읽었는지 표시              │
│ 5. 푸시 알림: 오프라인 사용자에게 새 메시지 알림            │
│ 6. 멀티 디바이스: 여러 기기에서 동시 접속 지원              │
│ 7. 미디어 전송: 이미지, 파일 전송 (선택)                   │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│                비기능적 요구사항 (NFR)                       │
├─────────────────────────────────────────────────────────┤
│ 1. 저지연: 메시지 전달 < 200ms (동일 리전 내)              │
│ 2. 메시지 유실 방지: 전달 보장 (at-least-once)            │
│ 3. 메시지 순서 보장: 발송 순서대로 표시                    │
│ 4. 고가용성: 99.99% 이상 업타임                           │
│ 5. 확장성: 수천만 동시 접속 지원                           │
└─────────────────────────────────────────────────────────┘

용량 추산:
  - DAU: 5,000만 명
  - 평균 메시지/일/사용자: 40건
  - 일일 메시지: 20억 건
  - 메시지 QPS: 20억 / 86,400 ≈ 23,000 QPS
  - 피크: ~70,000 QPS
  - 평균 메시지 크기: 200 bytes
  - 일일 저장: 20억 × 200B ≈ 400 GB
  - 5년 저장: 400GB × 365 × 5 ≈ 730 TB
```

### 2. 통신 프로토콜: WebSocket vs HTTP Long Polling

채팅 시스템의 핵심 기술 결정은 **클라이언트와 서버 간의 실시간 통신 방식**이다.

#### HTTP Polling (단순 폴링)

```
HTTP Polling (비효율적):

  클라이언트                    서버
     │                          │
     │── "새 메시지 있나요?" ──→  │  (없음)
     │←── "없습니다" ──────────  │
     │          ... 5초 대기 ...  │
     │── "새 메시지 있나요?" ──→  │  (없음)
     │←── "없습니다" ──────────  │
     │          ... 5초 대기 ...  │
     │── "새 메시지 있나요?" ──→  │  (있음!)
     │←── "메시지 3건" ────────  │
     │                          │

  문제: 대부분의 요청에 새 메시지가 없음 → 리소스 낭비
  5,000만 DAU × 분당 12회 = 초당 1,000만 무의미한 요청
```

#### HTTP Long Polling

```
HTTP Long Polling:

  클라이언트                    서버
     │                          │
     │── "새 메시지 있나요?" ──→  │
     │        (서버가 대기...)    │  ← 메시지가 올 때까지 연결 유지
     │        (30초간 대기...)    │
     │                          │  ← 새 메시지 도착!
     │←── "메시지 1건" ────────  │
     │                          │
     │── "새 메시지 있나요?" ──→  │  ← 즉시 다시 연결
     │        (서버가 대기...)    │
     │                          │

  개선: 불필요한 요청 감소
  한계: 서버가 많은 대기 연결을 유지해야 함
        메시지 발신자와 수신자가 다른 서버에 연결될 수 있음
        HTTP 헤더 오버헤드가 여전히 존재
```

#### WebSocket

```
WebSocket (양방향 실시간 통신):

  클라이언트                    서버
     │                          │
     │── HTTP Upgrade 요청 ──→  │
     │←── 101 Switching ──────  │  ← 프로토콜 업그레이드
     │                          │
     │ ═══ WebSocket 연결 수립 ═══ │  ← 전이중(Full-Duplex) 연결
     │                          │
     │── "안녕하세요" ──────→    │  ← 클라이언트 → 서버
     │                          │
     │←── "반갑습니다" ────────  │  ← 서버 → 클라이언트 (서버 푸시!)
     │                          │
     │── "뭐해요?" ──────→      │
     │←── "공부 중이에요" ────   │
     │                          │
     │ ═══ 연결 유지 (heartbeat) ═══ │
     │                          │

  장점:
  - 양방향 실시간 통신 (서버가 능동적으로 푸시)
  - HTTP 헤더 오버헤드 없음 (초기 핸드셰이크 후)
  - 메시지별 2~6 bytes 프레임 헤더만 필요
  - 지연 시간 최소화
```

| 비교 | HTTP Polling | Long Polling | WebSocket |
|------|-------------|--------------|-----------|
| 방향 | 단방향 (클라이언트→서버) | 단방향 | 양방향 (전이중) |
| 실시간성 | 낮음 (폴링 간격) | 보통 | 높음 |
| 서버 부하 | 높음 (불필요한 요청) | 보통 | 낮음 (이벤트 기반) |
| 헤더 오버헤드 | 매 요청마다 | 매 요청마다 | 초기 핸드셰이크만 |
| 구현 복잡도 | 낮음 | 보통 | 높음 |
| 적합 사례 | 대시보드 갱신 | 알림 시스템 | 채팅, 게임, 주식 |

**결론**: 채팅 시스템에서는 **WebSocket이 가장 적합**하다. 단, 서버→클라이언트 방향이 필요 없는 기능(메시지 발송, 사용자 인증 등)은 일반 HTTP API를 사용해도 된다.

### 3. 메시지 전달 보장 (Message Delivery Guarantees)

네트워크는 불안정하다. 메시지가 유실되지 않도록 **전달 보장 메커니즘**이 필요하다.

```
메시지 전달 흐름 (at-least-once):

  발신자 A                Chat 서버              수신자 B
     │                       │                      │
     │── ① msg 전송 ──→      │                      │
     │                       │── DB에 저장 (msg_id) ──│
     │←── ② ACK (msg_id) ──  │                      │
     │                       │                      │
     │                       │── ③ msg 전달 ──→      │ (B가 온라인)
     │                       │                      │
     │                       │←── ④ ACK (msg_id) ──  │
     │                       │                      │
     │                       │── 전달 완료 표시 ──    │

  B가 오프라인인 경우:
     │                       │                      │ (오프라인)
     │── ① msg 전송 ──→      │                      │
     │                       │── DB에 저장 ──────    │
     │←── ② ACK ──────────   │                      │
     │                       │                      │
     │                       │ (B가 온라인 복귀)      │
     │                       │                      │
     │                       │←── ⑤ 접속 알림 ──── │
     │                       │── ⑥ 미전달 메시지 ──→ │
     │                       │←── ⑦ ACK ──────────  │

전달 보장 수준:
┌────────────────────────────────────────────────────┐
│ At-Most-Once  : 최대 1번 전달 (유실 가능)            │
│                 → 알림 등 유실 허용 시                │
│                                                    │
│ At-Least-Once : 최소 1번 전달 (중복 가능)            │
│                 → 채팅에 적합 (중복은 msg_id로 제거)  │
│                                                    │
│ Exactly-Once  : 정확히 1번 전달                     │
│                 → 구현 매우 어려움, 성능 비용 높음     │
└────────────────────────────────────────────────────┘

중복 제거 (Idempotency):
  클라이언트가 동일 msg_id를 재전송하면, 서버가 무시한다.
  msg_id = {sender_id}_{timestamp}_{sequence}
```

### 4. 메시지 저장소 설계

채팅 메시지는 **쓰기가 빈번하고, 최근 메시지 위주로 읽기**하는 특성을 가진다. 이에 적합한 저장소를 선택해야 한다.

```
메시지 데이터 모델:

┌──────────────────────────────────────────┐
│              messages                     │
├──────────────────────────────────────────┤
│ message_id      BIGINT PK               │  ← Snowflake ID (시간순 정렬)
│ conversation_id BIGINT                   │  ← 대화방 ID
│ sender_id       BIGINT                   │  ← 발신자
│ content         TEXT                     │  ← 메시지 내용
│ type            ENUM(text,image,file)    │  ← 메시지 유형
│ status          ENUM(sent,delivered,read)│  ← 전달 상태
│ created_at      TIMESTAMP               │  ← 발송 시각
└──────────────────────────────────────────┘

┌──────────────────────────────────────────┐
│            conversations                  │
├──────────────────────────────────────────┤
│ conversation_id  BIGINT PK              │
│ type             ENUM(direct,group)      │
│ name             VARCHAR (그룹명)         │
│ created_at       TIMESTAMP              │
│ last_message_at  TIMESTAMP              │  ← 대화 목록 정렬용
└──────────────────────────────────────────┘

┌──────────────────────────────────────────┐
│       conversation_members                │
├──────────────────────────────────────────┤
│ conversation_id  BIGINT                  │
│ user_id          BIGINT                  │
│ role             ENUM(member,admin)       │
│ joined_at        TIMESTAMP              │
│ last_read_msg_id BIGINT                  │  ← 읽음 확인 기준
│ muted_until      TIMESTAMP (nullable)    │  ← 알림 음소거
│ PK: (conversation_id, user_id)           │
└──────────────────────────────────────────┘

DB 선택:
┌────────────────────────────────────────────────────────┐
│ 관계형 DB (MySQL, PostgreSQL):                          │
│   - 소규모에 적합, ACID 보장                             │
│   - 대규모에서 샤딩 필요                                 │
│                                                        │
│ 와이드 컬럼 DB (Cassandra, HBase):                       │
│   - 쓰기 최적화 (LSM 트리)                               │
│   - 시간순 데이터에 강함                                  │
│   - conversation_id를 파티션 키로 사용                    │
│   → 카카오톡, Discord가 Cassandra 사용                   │
│                                                        │
│ 최적 선택: 메시지 → Cassandra/ScyllaDB                   │
│           사용자/대화방 메타데이터 → MySQL/PostgreSQL       │
└────────────────────────────────────────────────────────┘

Cassandra 스키마 예시:
  CREATE TABLE messages (
    conversation_id BIGINT,
    message_id      TIMEUUID,    -- 시간 기반 UUID
    sender_id       BIGINT,
    content         TEXT,
    type            TEXT,
    PRIMARY KEY (conversation_id, message_id)
  ) WITH CLUSTERING ORDER BY (message_id DESC);

  → conversation_id로 파티셔닝
  → message_id(시간순)로 클러스터링 → 최신 메시지부터 조회
```

### 5. 읽음 확인과 푸시 알림

#### 읽음 확인 (Read Receipt)

```
읽음 확인 동작 흐름:

  사용자 A                서버               사용자 B
     │                     │                    │
     │── msg_42 전송 ──→   │                    │
     │                     │── msg_42 전달 ──→  │
     │                     │                    │
     │                     │                    │── 대화방 열기
     │                     │                    │
     │                     │←── read_ack ──── │
     │                     │    (conv_id,       │
     │                     │     last_read=42)  │
     │                     │                    │
     │←── 읽음 표시 ──────  │                    │
     │   "msg_42 읽음 ✓✓"  │                    │

구현 방식:
  - conversation_members 테이블의 last_read_msg_id를 갱신
  - 읽음 이벤트를 발신자에게 WebSocket으로 전달
  - 그룹 채팅: 각 멤버의 last_read_msg_id 중 최솟값 추적

  주의: 읽음 확인은 100% 정확하지 않아도 됨 (최종 일관성)
        네트워크 지연이나 오프라인 시 약간의 지연 허용
```

#### 푸시 알림 (Push Notification)

```
푸시 알림 흐름:

  ┌──────────┐   ┌──────────────┐   ┌───────────────┐
  │ Chat 서버 │──→│ 알림 서비스    │──→│ FCM / APNs    │
  └──────────┘   │              │   │ (Firebase/    │
                 │ 조건 확인:    │   │  Apple Push)  │
                 │ - 수신자 오프라인?│   └───────┬───────┘
                 │ - 음소거 아님?  │           │
                 │ - 방해금지 아님?│           ▼
                 └──────────────┘   ┌───────────────┐
                                    │ 사용자 디바이스  │
                                    │ (스마트폰)      │
                                    └───────────────┘

알림 판단 로직:
  1. 수신자가 WebSocket 연결 중인가?
     → YES: 푸시 알림 불필요 (실시간 전달 완료)
     → NO: 푸시 알림 발송
  2. 수신자가 해당 대화방을 음소거했는가?
     → YES: 알림 생략
  3. 수신자가 방해금지 모드인가?
     → YES: 알림 큐잉 (나중에 전달)

알림 내용:
  - 발신자 이름
  - 메시지 미리보기 (암호화된 메시지면 "새 메시지가 있습니다")
  - 대화방 정보
  - 딥링크 (앱의 해당 대화방으로 바로 이동)
```

### 6. 온라인 상태 (Presence)

```
온라인 상태 관리:

  ┌─────────────────────────────────────────────────┐
  │                Presence Service                  │
  │                                                 │
  │  구현 방식 1: Heartbeat                           │
  │                                                 │
  │  클라이언트가 30초마다 heartbeat 전송:              │
  │    → 수신: "온라인" 상태 유지                      │
  │    → 60초간 미수신: "오프라인"으로 변경              │
  │                                                 │
  │  ┌────────┐  heartbeat  ┌────────────┐          │
  │  │클라이언트│────────→   │ Presence   │           │
  │  │        │  (30초마다)  │ Service    │           │
  │  └────────┘            │            │           │
  │                        │ Redis:     │           │
  │                        │ user:123   │           │
  │                        │ = "online" │           │
  │                        │ TTL: 60s   │           │
  │                        └────────────┘           │
  │                                                 │
  │  구현 방식 2: WebSocket 연결 상태 기반              │
  │                                                 │
  │  WebSocket 연결 시: "온라인"                       │
  │  WebSocket 끊김 시: "오프라인"                     │
  │  → 네트워크 불안정 시 빈번한 상태 전환 발생           │
  │  → Heartbeat와 혼합하여 사용                      │
  └─────────────────────────────────────────────────┘

상태 전파:
  사용자 A가 온라인 → A의 친구 목록 조회
  → 각 친구에게 "A가 온라인입니다" WebSocket 이벤트 전송

  ⚠ 친구가 매우 많으면 (SNS 서비스) 부하 큼
  → 현재 활성 대화 상대에게만 전파
  → 대화방 진입 시 상대의 상태를 조회 (Lazy Loading)
```

### 7. 전체 아키텍처 다이어그램

```
┌──────────────────────────────────────────────────────────────────┐
│                    채팅 시스템 전체 아키텍처                         │
└──────────────────────────────────────────────────────────────────┘

  ┌────────┐  ┌────────┐  ┌────────┐
  │디바이스1 │  │디바이스2 │  │디바이스3 │
  └───┬────┘  └───┬────┘  └───┬────┘
      │           │           │
      └─────────┬─┴───────────┘
                │
          ┌─────▼──────┐
          │ API Gateway │ (인증, Rate Limiting)
          │ / LB        │
          └──────┬─────┘
                 │
     ┌───────────┼──────────────────────┐
     │           │                      │
     ▼           ▼                      ▼
┌─────────┐ ┌──────────┐         ┌──────────┐
│HTTP API  │ │WebSocket │         │WebSocket │
│서버      │ │서버 1     │         │서버 2     │
│(인증,    │ │(실시간    │         │(실시간    │
│ 그룹관리, │ │ 메시지)   │         │ 메시지)   │
│ 프로필)  │ └────┬─────┘         └────┬─────┘
└─────────┘      │                    │
                 │                    │
     ┌───────────┴────────────────────┘
     │
     ▼
┌────────────────────────────────────────────┐
│           Message Routing Service           │
│                                            │
│  "user_A는 WebSocket 서버 2에 연결 중"       │
│  → 메시지를 WebSocket 서버 2로 라우팅         │
│                                            │
│  라우팅 정보: Redis Pub/Sub 또는 ZooKeeper    │
│  user_id → websocket_server_id 매핑          │
└──────────┬─────────────────────────────────┘
           │
     ┌─────┴─────────────────────────┐
     │                               │
     ▼                               ▼
┌──────────────┐            ┌──────────────┐
│ Message DB    │            │ Presence     │
│ (Cassandra)   │            │ Service      │
│               │            │ (Redis)      │
│ 메시지 영구   │            │              │
│ 저장          │            │ 온라인 상태   │
│               │            │ Heartbeat    │
└──────────────┘            └──────────────┘
           │
     ┌─────┴─────────────────────────┐
     │                               │
     ▼                               ▼
┌──────────────┐            ┌──────────────┐
│ Push         │            │ Media        │
│ Notification │            │ Service      │
│ Service      │            │              │
│              │            │ S3/CDN       │
│ FCM, APNs   │            │ 이미지/파일   │
└──────────────┘            └──────────────┘

메시지 전송 흐름 (A → B):
  ① A가 WebSocket으로 메시지 전송 (WS서버1에 연결)
  ② WS서버1이 메시지를 Message Routing Service에 전달
  ③ Routing Service가 메시지를 Cassandra에 저장
  ④ Routing Service가 B의 연결 서버 확인 (Redis: B → WS서버2)
  ⑤-a B가 온라인 → WS서버2를 통해 실시간 전달
  ⑤-b B가 오프라인 → Push Notification Service로 알림 발송
  ⑥ B가 ACK 전송 → 전달 완료 상태 갱신

그룹 메시지 흐름:
  ① A가 그룹에 메시지 전송
  ② 그룹 멤버 목록 조회 (MySQL: conversation_members)
  ③ 각 멤버의 WebSocket 서버로 Fan-out
  ④ 오프라인 멤버에게 푸시 알림
```

## 코드 예시

### WebSocket 채팅 서버 (Python + FastAPI)

```python
import asyncio
import json
import time
from dataclasses import dataclass, field
from collections import defaultdict

from fastapi import FastAPI, WebSocket, WebSocketDisconnect

app = FastAPI()


@dataclass
class Message:
    """채팅 메시지 데이터 모델"""
    id: str
    conversation_id: str
    sender_id: str
    content: str
    msg_type: str = "text"
    status: str = "sent"           # sent → delivered → read
    created_at: float = field(default_factory=time.time)


class ConnectionManager:
    """
    WebSocket 연결 관리자.
    실제 서비스에서는 Redis Pub/Sub로 서버 간 메시지를 전달한다.
    여기서는 단일 서버 기준으로 구현한다.
    """

    def __init__(self):
        # user_id → WebSocket 연결 목록 (멀티 디바이스 지원)
        self.active_connections: dict[str, list[WebSocket]] = defaultdict(list)
        # 메시지 저장소 (실제로는 Cassandra)
        self.message_store: dict[str, list[Message]] = defaultdict(list)
        # 대화방 멤버 (실제로는 MySQL)
        self.conversation_members: dict[str, set[str]] = {}
        # 읽음 위치 (실제로는 Redis)
        self.read_positions: dict[str, dict[str, str]] = defaultdict(dict)

    async def connect(self, websocket: WebSocket, user_id: str):
        """사용자의 WebSocket 연결을 등록한다."""
        await websocket.accept()
        self.active_connections[user_id].append(websocket)
        print(f"[연결] {user_id} 접속 (디바이스 수: "
              f"{len(self.active_connections[user_id])})")

        # 온라인 상태 알림 (친구들에게)
        await self._broadcast_presence(user_id, "online")

    def disconnect(self, user_id: str, websocket: WebSocket):
        """사용자의 WebSocket 연결을 해제한다."""
        self.active_connections[user_id].remove(websocket)
        if not self.active_connections[user_id]:
            del self.active_connections[user_id]
            print(f"[연결 해제] {user_id} 오프라인")

    def is_online(self, user_id: str) -> bool:
        """사용자가 온라인인지 확인한다."""
        return user_id in self.active_connections

    async def send_to_user(self, user_id: str, data: dict):
        """
        특정 사용자에게 메시지를 전송한다.
        멀티 디바이스: 모든 연결된 디바이스에 전송한다.
        """
        if user_id in self.active_connections:
            disconnected = []
            for ws in self.active_connections[user_id]:
                try:
                    await ws.send_json(data)
                except Exception:
                    disconnected.append(ws)
            # 끊어진 연결 정리
            for ws in disconnected:
                self.active_connections[user_id].remove(ws)

    async def handle_message(self, sender_id: str, data: dict):
        """
        수신된 메시지를 처리한다.
        1. 메시지 저장
        2. 수신자에게 전달 (온라인) 또는 알림 (오프라인)
        """
        conv_id = data["conversation_id"]
        msg = Message(
            id=f"msg_{int(time.time()*1000)}_{sender_id}",
            conversation_id=conv_id,
            sender_id=sender_id,
            content=data["content"],
        )

        # 1. 메시지 저장
        self.message_store[conv_id].append(msg)

        # 2. 발신자에게 ACK 전송
        await self.send_to_user(sender_id, {
            "type": "ack",
            "message_id": msg.id,
            "status": "sent",
        })

        # 3. 대화방의 다른 멤버에게 전달
        members = self.conversation_members.get(conv_id, set())
        for member_id in members:
            if member_id == sender_id:
                continue

            if self.is_online(member_id):
                # 온라인: 실시간 전달
                await self.send_to_user(member_id, {
                    "type": "new_message",
                    "message": {
                        "id": msg.id,
                        "conversation_id": conv_id,
                        "sender_id": sender_id,
                        "content": msg.content,
                        "created_at": msg.created_at,
                    }
                })
                msg.status = "delivered"
            else:
                # 오프라인: 푸시 알림 전송 (시뮬레이션)
                print(f"  [푸시 알림] {member_id}에게 알림: "
                      f"'{msg.content[:20]}...'")

    async def handle_read_ack(self, user_id: str, data: dict):
        """
        읽음 확인 처리.
        상대방에게 읽음 표시 이벤트를 전달한다.
        """
        conv_id = data["conversation_id"]
        last_read_msg_id = data["last_read_msg_id"]

        # 읽음 위치 갱신
        self.read_positions[conv_id][user_id] = last_read_msg_id

        # 대화 상대에게 읽음 표시 이벤트 전달
        members = self.conversation_members.get(conv_id, set())
        for member_id in members:
            if member_id != user_id and self.is_online(member_id):
                await self.send_to_user(member_id, {
                    "type": "read_receipt",
                    "conversation_id": conv_id,
                    "reader_id": user_id,
                    "last_read_msg_id": last_read_msg_id,
                })

    async def _broadcast_presence(self, user_id: str, status: str):
        """온라인 상태를 관련 사용자들에게 알린다."""
        # 실제로는 친구 목록/최근 대화 상대에게만 전달
        for uid in list(self.active_connections.keys()):
            if uid != user_id:
                await self.send_to_user(uid, {
                    "type": "presence",
                    "user_id": user_id,
                    "status": status,
                })


manager = ConnectionManager()

# 테스트용 대화방 설정
manager.conversation_members["conv_1"] = {"alice", "bob"}
manager.conversation_members["conv_group_1"] = {"alice", "bob", "charlie"}


@app.websocket("/ws/{user_id}")
async def websocket_endpoint(websocket: WebSocket, user_id: str):
    """WebSocket 엔드포인트"""
    await manager.connect(websocket, user_id)
    try:
        while True:
            raw = await websocket.receive_text()
            data = json.loads(raw)

            if data["type"] == "message":
                await manager.handle_message(user_id, data)
            elif data["type"] == "read_ack":
                await manager.handle_read_ack(user_id, data)
            elif data["type"] == "ping":
                await websocket.send_json({"type": "pong"})

    except WebSocketDisconnect:
        manager.disconnect(user_id, websocket)
        await manager._broadcast_presence(user_id, "offline")
```

### 메시지 순서 보장을 위한 Snowflake ID 생성기

```python
import time
import threading


class SnowflakeIDGenerator:
    """
    Twitter Snowflake 방식의 분산 고유 ID 생성기.
    시간순 정렬 가능하며, 초당 약 400만 개의 고유 ID를 생성할 수 있다.

    ID 구조 (64비트):
    ┌─────────────┬──────────┬────────────┐
    │ 41비트       │ 10비트    │ 12비트      │
    │ 타임스탬프    │ 머신 ID   │ 시퀀스 번호  │
    │ (밀리초)     │ (1024대)  │ (4096/ms)  │
    └─────────────┴──────────┴────────────┘
    """

    # 기준 시각: 2024-01-01 00:00:00 UTC (밀리초)
    EPOCH = 1704067200000

    def __init__(self, machine_id: int):
        if machine_id < 0 or machine_id > 1023:
            raise ValueError("machine_id는 0~1023 범위여야 합니다")

        self.machine_id = machine_id
        self.sequence = 0
        self.last_timestamp = -1
        self.lock = threading.Lock()

    def _current_millis(self) -> int:
        return int(time.time() * 1000)

    def generate(self) -> int:
        """고유한 ID를 생성한다 (스레드 안전)."""
        with self.lock:
            timestamp = self._current_millis()

            if timestamp == self.last_timestamp:
                # 같은 밀리초 내에서 시퀀스 증가
                self.sequence = (self.sequence + 1) & 0xFFF  # 12비트 마스크
                if self.sequence == 0:
                    # 시퀀스 초과 → 다음 밀리초까지 대기
                    while timestamp <= self.last_timestamp:
                        timestamp = self._current_millis()
            else:
                self.sequence = 0

            self.last_timestamp = timestamp

            # ID 조합: 타임스탬프(41) | 머신ID(10) | 시퀀스(12)
            snowflake_id = (
                ((timestamp - self.EPOCH) << 22) |
                (self.machine_id << 12) |
                self.sequence
            )
            return snowflake_id

    @staticmethod
    def extract_timestamp(snowflake_id: int) -> float:
        """Snowflake ID에서 생성 시각을 추출한다."""
        timestamp_ms = (snowflake_id >> 22) + SnowflakeIDGenerator.EPOCH
        return timestamp_ms / 1000


# === 실행 예시 ===
if __name__ == "__main__":
    gen = SnowflakeIDGenerator(machine_id=1)

    print("=== Snowflake ID 생성 ===")
    ids = [gen.generate() for _ in range(5)]
    for sid in ids:
        ts = SnowflakeIDGenerator.extract_timestamp(sid)
        print(f"  ID: {sid}  생성시각: {time.ctime(ts)}")

    print(f"\n=== ID가 시간순 정렬되는지 확인 ===")
    print(f"  정렬됨: {ids == sorted(ids)}")

    print(f"\n=== ID 크기: {ids[0].bit_length()} 비트 ===")
```

## 정리

| 개념 | 핵심 포인트 |
|------|------------|
| WebSocket | 양방향 전이중 통신, 채팅의 핵심 프로토콜, 초기 HTTP 핸드셰이크 후 업그레이드 |
| Long Polling | 서버가 응답을 대기시키는 방식, WebSocket보다 덜 효율적이나 방화벽 통과 용이 |
| At-Least-Once | 최소 1번 전달 보장, 중복은 msg_id로 제거 (Idempotency) |
| Snowflake ID | 타임스탬프+머신ID+시퀀스로 시간순 정렬 가능한 분산 고유 ID 생성 |
| 메시지 저장 | Cassandra/ScyllaDB 추천 (쓰기 최적화, conversation_id 파티셔닝) |
| 읽음 확인 | 수신자의 last_read_msg_id를 갱신, 발신자에게 WebSocket 이벤트 전달 |
| 푸시 알림 | 오프라인 사용자에게 FCM/APNs로 알림, 음소거/방해금지 조건 확인 필요 |
| 온라인 상태 | Heartbeat(30초) + Redis TTL(60초)로 관리, 관련 사용자에게만 전파 |
| 멀티 디바이스 | user_id별 여러 WebSocket 연결 유지, 모든 디바이스에 동일 메시지 전달 |
