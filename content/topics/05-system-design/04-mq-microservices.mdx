## 핵심 개념

### 1. 메시지 큐(Message Queue)란?

메시지 큐는 **비동기 통신**을 위한 미들웨어다. 프로듀서(생산자)가 메시지를 큐에 넣으면, 컨슈머(소비자)가 나중에 꺼내서 처리한다. 두 시스템이 직접 통신하지 않고, 중간 버퍼를 통해 느슨하게 결합(loose coupling)된다.

```
[동기 통신 — 직접 호출]
서비스A ──요청──► 서비스B (B가 죽으면 A도 실패!)
         ◄──응답──

[비동기 통신 — 메시지 큐 사용]
서비스A ──메시지──► [  큐  ] ──메시지──► 서비스B
  (즉시 반환)       (버퍼)       (자기 속도로 처리)
```

**비유**: 식당의 주문 전표 시스템
- 웨이터(프로듀서)가 주문서를 전표 걸이(큐)에 건다
- 주방(컨슈머)이 순서대로 전표를 가져가서 요리한다
- 웨이터는 요리가 끝날 때까지 기다리지 않고 다음 테이블로 간다

#### 메시지 큐의 핵심 장점

| 장점 | 설명 |
|------|------|
| **비동기 처리** | 프로듀서는 메시지를 보내고 즉시 다른 일을 할 수 있음 |
| **부하 분산** | 급격한 트래픽을 큐가 버퍼링하여 컨슈머가 감당 가능한 속도로 처리 |
| **장애 격리** | 컨슈머가 죽어도 메시지는 큐에 남아 있어 데이터 손실 없음 |
| **확장성** | 컨슈머를 늘려 처리 속도를 높일 수 있음 |

### 2. Pub/Sub vs Point-to-Point

메시지 큐의 두 가지 기본 패턴이다.

#### Point-to-Point (큐 모델)

하나의 메시지는 **하나의 컨슈머만** 처리한다. 작업 분배(work distribution)에 적합하다.

```
                    ┌──────────┐
Producer ──msg1──►  │          │ ──msg1──► Consumer A
Producer ──msg2──►  │   큐     │ ──msg2──► Consumer B
Producer ──msg3──►  │          │ ──msg3──► Consumer A
                    └──────────┘
  msg1은 A만, msg2는 B만, msg3은 A만 처리
  → 각 메시지는 정확히 한 번만 처리됨
```

**사용 사례**: 주문 처리, 이메일 발송, 이미지 변환 등

#### Pub/Sub (발행/구독 모델)

하나의 메시지가 **모든 구독자**에게 전달된다. 이벤트 브로드캐스트에 적합하다.

```
                    ┌──────────┐ ──msg──► Subscriber A (알림 서비스)
Publisher ──msg──►  │  토픽    │ ──msg──► Subscriber B (로그 서비스)
                    │ (topic)  │ ──msg──► Subscriber C (분석 서비스)
                    └──────────┘
  모든 구독자가 같은 메시지를 받음
```

**사용 사례**: 이벤트 알림, 로그 수집, 실시간 데이터 스트리밍

```
[비교]
                Point-to-Point         Pub/Sub
메시지 수신자    1개 컨슈머             모든 구독자
메시지 소비      소비 후 큐에서 삭제     토픽에 유지 (기간 설정)
목적             작업 분배              이벤트 브로드캐스트
예시             작업 큐                뉴스 피드, 알림
```

### 3. Apache Kafka

링크드인에서 개발한 분산 이벤트 스트리밍 플랫폼이다. **로그 기반** 아키텍처로, 메시지를 디스크에 순차적으로 저장한다.

```
[Kafka 아키텍처]

Producer ──►┌──────────────────────────────────┐
Producer ──►│         Kafka 클러스터             │
Producer ──►│                                   │
            │  Topic: "orders"                  │
            │  ┌──────────────────────────┐     │
            │  │ Partition 0: [m1][m2][m5] │     │──► Consumer Group A
            │  │ Partition 1: [m3][m4][m6] │     │    (주문 처리)
            │  │ Partition 2: [m7][m8]     │     │
            │  └──────────────────────────┘     │──► Consumer Group B
            │                                   │    (분석)
            │  Broker 1  |  Broker 2  | Broker 3│
            └──────────────────────────────────┘
```

#### Kafka 핵심 개념

| 개념 | 설명 |
|------|------|
| **Topic** | 메시지의 카테고리 (예: "orders", "user-events") |
| **Partition** | 토픽을 여러 조각으로 분리. 병렬 처리의 단위 |
| **Offset** | 파티션 내 메시지의 고유 순번. 컨슈머가 어디까지 읽었는지 추적 |
| **Broker** | Kafka 서버 노드. 여러 브로커가 클러스터를 구성 |
| **Consumer Group** | 같은 그룹의 컨슈머는 파티션을 나눠 읽음 (병렬 처리) |

#### Kafka의 특징
- **높은 처리량**: 디스크 순차 쓰기 + 제로카피(zero-copy)로 초당 수백만 메시지 처리
- **내구성**: 메시지를 디스크에 저장하고 복제(replication)
- **메시지 보존**: 처리 후에도 메시지를 일정 기간 보관 (기본 7일)
- **순서 보장**: 같은 파티션 내에서 메시지 순서 보장
- **Pull 방식**: 컨슈머가 자기 속도에 맞게 메시지를 가져감

### 4. RabbitMQ

Erlang으로 개발된 전통적인 메시지 브로커다. AMQP(Advanced Message Queuing Protocol)를 구현한다.

```
[RabbitMQ 아키텍처]

Producer ──► Exchange ──라우팅──► Queue1 ──► Consumer A
                │                Queue2 ──► Consumer B
                │                Queue3 ──► Consumer C
                │
     Exchange Types:
     - Direct:  라우팅 키 정확히 매칭
     - Fanout:  모든 큐에 복사 (브로드캐스트)
     - Topic:   패턴 매칭 (orders.* → orders.create, orders.cancel)
     - Headers: 메시지 헤더 기반 라우팅
```

#### RabbitMQ의 특징
- **유연한 라우팅**: Exchange 타입으로 다양한 라우팅 패턴 지원
- **Push 방식**: 브로커가 컨슈머에게 메시지를 보냄 (설정으로 Pull도 가능)
- **메시지 확인**: 컨슈머가 처리 완료를 ACK로 알려야 큐에서 삭제
- **낮은 지연시간**: 메모리 기반 처리로 밀리초 단위 지연
- **프로토콜 지원**: AMQP, MQTT, STOMP 등 다양한 프로토콜 지원

#### Kafka vs RabbitMQ 비교

```
┌─────────────┬──────────────────────┬──────────────────────┐
│             │       Kafka          │      RabbitMQ        │
├─────────────┼──────────────────────┼──────────────────────┤
│ 모델        │ 분산 로그(스트리밍)    │ 메시지 브로커(큐잉)    │
│ 처리량      │ 매우 높음 (100만+/s)  │ 높음 (수만/s)         │
│ 지연시간    │ 밀리초~수십밀리초      │ 마이크로초~밀리초      │
│ 메시지 보존 │ 설정 기간까지 보관     │ 소비 후 삭제          │
│ 순서 보장   │ 파티션 내 보장        │ 큐 내 보장            │
│ 소비 방식   │ Pull                 │ Push (기본)           │
│ 재처리      │ 오프셋 되돌려 재처리   │ 별도 설정 필요        │
│ 적합한 용도 │ 이벤트 스트리밍, 로그  │ 작업 큐, RPC          │
│ 확장성      │ 파티션 추가로 수평확장  │ 클러스터링 가능        │
└─────────────┴──────────────────────┴──────────────────────┘
```

### 5. 마이크로서비스 아키텍처 vs 모놀리식

#### 모놀리식 아키텍처

하나의 코드베이스에 모든 기능이 포함된 단일 애플리케이션이다.

```
[모놀리식]
┌────────────────────────────────────┐
│            하나의 애플리케이션         │
│  ┌────────┐ ┌────────┐ ┌────────┐ │
│  │ 사용자  │ │ 주문   │ │ 결제   │ │
│  │ 모듈   │ │ 모듈   │ │ 모듈   │ │
│  └────┬───┘ └────┬───┘ └────┬───┘ │
│       └──────────┼──────────┘     │
│            ┌─────▼─────┐          │
│            │ 공유 DB    │          │
│            └───────────┘          │
└────────────────────────────────────┘
  하나의 프로세스로 실행, 하나의 DB 공유
```

**장점**: 개발 단순, 디버깅 쉬움, 트랜잭션 관리 간단
**단점**: 규모가 커지면 빌드/배포 느림, 한 모듈 장애가 전체 영향, 기술 스택 변경 어려움

#### 마이크로서비스 아키텍처

기능 단위로 독립된 서비스로 분리하여 각각 배포, 확장, 관리한다.

```
[마이크로서비스]
┌─────────┐   ┌─────────┐   ┌─────────┐
│ 사용자   │   │ 주문    │   │ 결제    │
│ 서비스   │◄─►│ 서비스  │◄─►│ 서비스  │
│ (Python) │   │ (Java)  │   │ (Go)    │
│ [DB1]    │   │ [DB2]   │   │ [DB3]   │
└─────────┘   └─────────┘   └─────────┘
  각각 독립된 프로세스, 독립된 DB, 독립된 기술 스택
```

**장점**: 독립 배포, 독립 확장, 장애 격리, 팀별 자율성
**단점**: 운영 복잡성, 분산 트랜잭션, 네트워크 지연, 디버깅 어려움

### 6. 서비스 디스커버리 (Service Discovery)

마이크로서비스 환경에서 서비스 인스턴스의 위치(IP, 포트)를 동적으로 찾는 메커니즘이다. 서비스가 수시로 생성/삭제되는 클라우드 환경에서 필수적이다.

```
[클라이언트 사이드 디스커버리]
                  ┌────────────────┐
                  │  Service       │
                  │  Registry      │
                  │  (Consul/etcd) │
                  └───────┬────────┘
                    ②조회  │  ①등록
        ┌─────────────────┼─────────────────┐
        │                 │                 │
   ┌────▼────┐      ┌────▼────┐      ┌────▼────┐
   │ 주문    │      │ 주문    │      │ 주문    │
   │서비스 A │      │서비스 B │      │서비스 C │
   │:8001   │      │:8002   │      │:8003   │
   └─────────┘      └─────────┘      └─────────┘

클라이언트가 Registry에서 서비스 목록을 조회한 후 직접 호출

[서버 사이드 디스커버리]
클라이언트 ──► Load Balancer ──► 서비스 인스턴스
                    │
              Service Registry
  로드 밸런서가 Registry를 조회하여 라우팅
  (AWS ELB, Kubernetes Service 등)
```

### 7. API 게이트웨이

마이크로서비스 앞단에서 모든 요청을 받아 적절한 서비스로 라우팅하는 진입점이다.

```
                    ┌──────────────────┐
클라이언트 ────────►│  API Gateway     │
  (웹/앱)          │                  │
                    │ - 인증/인가       │
                    │ - 라우팅          │
                    │ - 속도 제한       │
                    │ - 로깅            │
                    │ - 응답 캐싱       │
                    │ - 응답 집계       │
                    └───┬──────┬───────┘
                        │      │
              ┌─────────▼┐  ┌──▼────────┐
              │ 사용자    │  │  주문     │  ...
              │ 서비스    │  │  서비스   │
              └──────────┘  └──────────┘
```

**대표 도구**: Kong, AWS API Gateway, Netflix Zuul, Nginx

**API 게이트웨이의 역할:**
1. **요청 라우팅**: `/users/*` → 사용자 서비스, `/orders/*` → 주문 서비스
2. **인증/인가**: JWT 토큰 검증, API 키 관리
3. **속도 제한(Rate Limiting)**: IP당 요청 수 제한
4. **응답 집계**: 여러 서비스의 응답을 모아서 하나로 반환
5. **프로토콜 변환**: 외부 REST ↔ 내부 gRPC 변환

### 8. 사가 패턴 (Saga Pattern) — 분산 트랜잭션

모놀리식에서는 하나의 DB 트랜잭션으로 여러 테이블을 한 번에 변경할 수 있다. 하지만 마이크로서비스에서는 각 서비스가 자체 DB를 가지므로, 여러 서비스에 걸친 트랜잭션이 필요하다. 이때 사가 패턴을 사용한다.

> **사가(Saga)**: 여러 로컬 트랜잭션의 연쇄로, 하나가 실패하면 보상 트랜잭션(compensating transaction)을 실행하여 이전 단계를 되돌린다.

```
[주문 사가 — 성공 시나리오]
① 주문 생성 ──► ② 재고 차감 ──► ③ 결제 처리 ──► ④ 배송 요청
   (주문DB)       (재고DB)        (결제DB)        (배송DB)

[주문 사가 — 결제 실패 시나리오]
① 주문 생성 ──► ② 재고 차감 ──► ③ 결제 실패! ──X
   (주문DB)       (재고DB)        (결제DB)
       ◄──────────── ◄──────────
④ 재고 복구      ⑤ 주문 취소
  (보상 트랜잭션)   (보상 트랜잭션)
```

#### 사가 구현 방식

**1. 코레오그래피 (Choreography)** — 이벤트 기반

각 서비스가 이벤트를 발행하고, 다음 서비스가 이를 구독하여 처리한다. 중앙 관리자가 없다.

```
주문 서비스                  재고 서비스                결제 서비스
    │                          │                        │
    ├── "주문생성" 이벤트 ────►│                        │
    │                          ├── "재고차감" 이벤트 ──►│
    │                          │                        ├── "결제완료" 이벤트
    │                          │                        │
    ◄──────────────────────────┼────────────────────────┘
    주문 확정
```

장점: 단순, 느슨한 결합
단점: 흐름 파악 어려움, 순환 의존성 위험

**2. 오케스트레이션 (Orchestration)** — 중앙 조율자

하나의 오케스트레이터가 전체 흐름을 관리한다.

```
                ┌──────────────┐
                │ 사가          │
                │ 오케스트레이터 │
                └──────┬───────┘
                       │
       ┌───────────────┼───────────────┐
       ▼               ▼               ▼
  ┌─────────┐    ┌─────────┐    ┌─────────┐
  │ 주문    │    │ 재고    │    │ 결제    │
  │ 서비스  │    │ 서비스  │    │ 서비스  │
  └─────────┘    └─────────┘    └─────────┘
```

장점: 흐름이 명확, 복잡한 로직 관리 용이
단점: 오케스트레이터가 단일 장애점이 될 수 있음

## 코드 예시

### 메시지 큐 시뮬레이션 (Point-to-Point & Pub/Sub)

```python
import threading
import time
import queue
from collections import defaultdict
from typing import Callable, List

# === 1. Point-to-Point 큐 ===

class PointToPointQueue:
    """
    Point-to-Point 메시지 큐.
    하나의 메시지는 정확히 하나의 컨슈머만 처리한다.
    내부적으로 Python의 thread-safe queue.Queue를 사용한다.
    """

    def __init__(self, name: str):
        self.name = name
        self._queue = queue.Queue()

    def send(self, message: str):
        """프로듀서가 메시지를 큐에 넣는다"""
        self._queue.put(message)
        print(f"[P2P:{self.name}] 전송: '{message}'")

    def receive(self, consumer_id: str, timeout: float = 1.0) -> str:
        """컨슈머가 메시지를 큐에서 가져간다 (블로킹)"""
        try:
            message = self._queue.get(timeout=timeout)
            print(f"[P2P:{self.name}] 컨슈머 {consumer_id} 수신: '{message}'")
            return message
        except queue.Empty:
            return None


# === 2. Pub/Sub 시스템 ===

class PubSubBroker:
    """
    Pub/Sub 메시지 브로커.
    토픽에 발행된 메시지는 모든 구독자에게 전달된다.
    """

    def __init__(self):
        # 토픽별 구독자 목록
        self._subscribers: dict[str, List[Callable]] = defaultdict(list)

    def subscribe(self, topic: str, callback: Callable, subscriber_name: str):
        """토픽을 구독한다. 메시지가 오면 callback이 호출된다."""
        self._subscribers[topic].append((subscriber_name, callback))
        print(f"[PubSub] '{subscriber_name}'이 토픽 '{topic}' 구독")

    def publish(self, topic: str, message: str):
        """토픽에 메시지를 발행한다. 모든 구독자에게 전달된다."""
        print(f"\n[PubSub] 토픽 '{topic}'에 발행: '{message}'")
        subscribers = self._subscribers.get(topic, [])

        if not subscribers:
            print(f"  구독자 없음!")
            return

        for subscriber_name, callback in subscribers:
            callback(message)
            print(f"  → '{subscriber_name}' 수신 완료")


# === 3. 사가 패턴 시뮬레이션 ===

class SagaOrchestrator:
    """
    사가 오케스트레이터.
    여러 단계(step)를 순서대로 실행하고,
    실패 시 보상 트랜잭션을 역순으로 실행한다.
    """

    def __init__(self, name: str):
        self.name = name
        self.steps: list[dict] = []
        self.completed_steps: list[dict] = []

    def add_step(self, name: str, action: Callable, compensation: Callable):
        """
        사가에 단계를 추가한다.
        action: 정방향 트랜잭션
        compensation: 보상(롤백) 트랜잭션
        """
        self.steps.append({
            "name": name,
            "action": action,
            "compensation": compensation
        })

    def execute(self) -> bool:
        """사가를 실행한다. 실패 시 보상 트랜잭션을 실행한다."""
        print(f"\n{'='*50}")
        print(f"사가 '{self.name}' 실행 시작")
        print(f"{'='*50}")

        for step in self.steps:
            print(f"\n[단계] {step['name']} 실행 중...")
            try:
                result = step["action"]()
                if not result:
                    raise Exception(f"{step['name']} 실패")

                self.completed_steps.append(step)
                print(f"  ✓ {step['name']} 성공")
            except Exception as e:
                print(f"  ✗ {step['name']} 실패: {e}")
                self._compensate()
                return False

        print(f"\n사가 '{self.name}' 완료!")
        return True

    def _compensate(self):
        """완료된 단계를 역순으로 보상한다."""
        print(f"\n[보상] 롤백 시작 (완료된 {len(self.completed_steps)}단계)")

        for step in reversed(self.completed_steps):
            print(f"  [보상] {step['name']} 되돌리기...")
            try:
                step["compensation"]()
                print(f"  ✓ {step['name']} 보상 완료")
            except Exception as e:
                print(f"  ✗ {step['name']} 보상 실패: {e} (수동 처리 필요!)")

    )
    saga.add_step(
        "결제 처리",
        action=lambda: (state.update({"payment": 50000}), True)[1],

    saga.execute()

    state2 = {"order": None, "inventory": 10, "payment": 0}

    saga2.add_step(

        action=lambda: (state2.update({"order": "CREATED"}), True)[1],

    )
    saga2.add_step(
        "재고 차감",
        action=lambda: (state2.update({"inventory": state2["inventory"] - 1}), True)[1],
        compensation=lambda: state2.update({"inventory": state2["inventory"] + 1})
    )
    saga2.add_step(
        "결제 처리",
        action=lambda: False,  # 결제 실패!
        compensation=lambda: None
    )

    saga2.execute()
    print(f"최종 상태: {state2}")
    # 보상 트랜잭션으로 재고가 복구되고 주문이 취소됨
```

```python
class SimpleKafkaSimulator:
    """Kafka의 파티셔닝 동작을 인메모리로 시뮬레이션"""

    def __init__(self, topic: str, num_partitions: int = 3):
        self.topic = topic
        self.partitions = {i: [] for i in range(num_partitions)}
        self.num_partitions = num_partitions
        self.consumer_offsets = {}  # (group, partition) → offset

    def send(self, key: str, value: dict):
        """키를 해싱하여 파티션을 결정하고 메시지 저장"""
        partition_id = hash(key) % self.num_partitions
        offset = len(self.partitions[partition_id])
        self.partitions[partition_id].append({
            "offset": offset,
            "key": key,
            "value": value
        })
        print(f"[Send] key='{key}' → partition={partition_id}, offset={offset}")

    def consume(self, group_id: str, partition_id: int) -> list:
        """특정 파티션에서 아직 안 읽은 메시지를 가져옴"""
        offset_key = (group_id, partition_id)
        current_offset = self.consumer_offsets.get(offset_key, 0)
        messages = self.partitions[partition_id][current_offset:]
        self.consumer_offsets[offset_key] = len(self.partitions[partition_id])
        return messages

    def show_partitions(self):
        """파티션 상태 출력"""
        print(f"\n[토픽: {self.topic}]")
        for pid, messages in self.partitions.items():
            print(f"  Partition {pid}: {len(messages)}개 메시지")
            for msg in messages:
                print(f"    [{msg['offset']}] {msg['key']}: {msg['value']}")
```

## 정리

| 개념 | 핵심 내용 | 사용 사례 |
|------|-----------|-----------|
| **메시지 큐** | 비동기 통신용 미들웨어, 느슨한 결합 | 서비스 간 비동기 통신 |
| **Point-to-Point** | 메시지를 하나의 컨슈머만 처리 | 작업 분배 (주문 처리) |
| **Pub/Sub** | 메시지를 모든 구독자에게 전달 | 이벤트 브로드캐스트 |
| **Kafka** | 분산 로그 기반, 높은 처리량, 메시지 보존 | 이벤트 스트리밍, 로그 수집 |
| **RabbitMQ** | AMQP 기반, 유연한 라우팅, 낮은 지연 | 작업 큐, RPC |
| **마이크로서비스** | 기능 단위 독립 서비스, 각자 DB 보유 | 대규모 시스템 |
| **서비스 디스커버리** | 서비스 위치를 동적으로 찾는 메커니즘 | Consul, etcd, K8s Service |
| **API 게이트웨이** | 모든 요청의 단일 진입점 | Kong, AWS API Gateway |
| **사가 패턴** | 분산 트랜잭션을 로컬 트랜잭션 연쇄로 처리 | 주문-재고-결제 흐름 |
| **코레오그래피** | 이벤트 기반 사가, 중앙 관리자 없음 | 단순한 워크플로우 |
| **오케스트레이션** | 중앙 조율자가 흐름 관리 | 복잡한 워크플로우 |
