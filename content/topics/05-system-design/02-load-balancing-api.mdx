## 핵심 개념

### 1. 로드 밸런싱이란?

로드 밸런싱(부하 분산)은 **들어오는 네트워크 트래픽을 여러 서버에 분배하여 하나의 서버에 과부하가 걸리지 않도록 하는 기술**이다.

  end
  subgraph WithLB[&quot;로드 밸런서 사용&quot;]

    LB -->|33% 트래픽| S2[서버 2]
    LB -->|33% 트래픽| S3[서버 3]
  end
" />

**비유**: 은행 창구. 손님이 많을 때 한 창구에만 줄을 서면 비효율적이다. 안내원(로드 밸런서)이 빈 창구로 안내해주면 전체 대기 시간이 줄어든다.

### 2. L4 vs L7 로드 밸런서

로드 밸런서는 OSI 모델의 어느 계층에서 동작하느냐에 따라 나뉜다.

#### L4 로드 밸런서 (전송 계층)

```
┌─────────────────────────────────────────────────────┐
│  L4 로드 밸런서 (Transport Layer)                     │
│                                                     │
│  확인하는 정보:                                       │
│  ┌──────────────────────────────────────┐           │
│  │ IP 헤더: 출발지 IP, 목적지 IP          │           │
│  │ TCP 헤더: 출발지 포트, 목적지 포트       │           │
│  └──────────────────────────────────────┘           │
│                                                     │
│  → HTTP 내용(URL, 헤더)은 확인하지 않음                │
│  → 패킷 수준에서 빠르게 라우팅                          │
│  → TCP 연결을 그대로 백엔드에 전달                      │
└─────────────────────────────────────────────────────┘
```

- IP 주소와 포트 번호만 보고 라우팅
- 빠르고 단순하며 오버헤드가 적음
- SSL 종료, URL 기반 라우팅 불가능
- 예: AWS NLB, HAProxy (L4 모드)

#### L7 로드 밸런서 (애플리케이션 계층)

```
┌─────────────────────────────────────────────────────┐
│  L7 로드 밸런서 (Application Layer)                   │
│                                                     │
│  확인하는 정보:                                       │
│  ┌──────────────────────────────────────┐           │
│  │ HTTP 메서드: GET, POST, PUT...        │           │
│  │ URL 경로: /api/users, /static/img     │           │
│  │ HTTP 헤더: Host, Cookie, Auth         │           │
│  │ 요청 본문(body)까지 검사 가능           │           │
│  └──────────────────────────────────────┘           │
│                                                     │
│  → 콘텐츠 기반 라우팅 가능                             │
│  → SSL 종료(termination) 가능                        │
│  → 요청 수정, 캐싱, 압축 가능                          │
└─────────────────────────────────────────────────────┘
```

- HTTP 내용을 파싱하여 라우팅 결정
- URL 경로별, 헤더별 다른 서버로 라우팅 가능
- SSL 종료, 쿠키 기반 세션 고정, 캐싱 가능
- L4보다 느리지만 더 유연함
- 예: AWS ALB, Nginx, HAProxy (L7 모드)

| 비교 항목 | L4 로드 밸런서 | L7 로드 밸런서 |
|-----------|--------------|--------------|
| 동작 계층 | 전송 계층 (TCP/UDP) | 애플리케이션 계층 (HTTP) |
| 판단 기준 | IP, 포트 | URL, 헤더, 쿠키, 본문 |
| 속도 | 매우 빠름 | 상대적으로 느림 |
| 기능 | 단순 분산 | 콘텐츠 기반 라우팅, SSL 종료 |
| SSL 처리 | 불가 (패스스루) | SSL 종료 가능 |
| 적합 용도 | 대용량 TCP 트래픽 | 웹 서비스, API 서비스 |

### 3. 로드 밸런싱 알고리즘

#### (1) Round Robin (라운드 로빈)

요청을 순서대로 돌아가며 서버에 배분한다.

```
요청 1 → 서버 A
요청 2 → 서버 B
요청 3 → 서버 C
요청 4 → 서버 A  (다시 처음부터)
요청 5 → 서버 B
요청 6 → 서버 C
```

- **장점**: 가장 단순하고 공평한 분배
- **단점**: 서버 성능 차이를 고려하지 않음, 현재 부하 무시
- **변형**: Weighted Round Robin — 서버에 가중치를 부여 (성능 좋은 서버에 더 많이)

#### (2) Least Connections (최소 연결)

현재 활성 연결이 가장 적은 서버에 요청을 보낸다.

```
서버 A: 현재 연결 5개
서버 B: 현재 연결 2개  ← 새 요청은 여기로!
서버 C: 현재 연결 8개
```

- **장점**: 실제 서버 부하를 반영, 처리 시간이 다양한 요청에 적합
- **단점**: 연결 수 추적 오버헤드, 새로 추가된 서버에 요청 집중 가능

#### (3) IP Hash (IP 해시)

클라이언트 IP를 해시하여 항상 같은 서버로 보낸다.

```
해시 함수: hash(클라이언트_IP) % 서버_수

192.168.1.10 → hash → 0 → 서버 A (항상)
192.168.1.20 → hash → 1 → 서버 B (항상)
192.168.1.30 → hash → 2 → 서버 C (항상)
```

- **장점**: 세션 고정(Session Affinity) 보장 — 같은 사용자가 항상 같은 서버
- **단점**: 서버 수 변경 시 해시 재분배, 불균등 분배 가능

#### 알고리즘 비교

| 알고리즘 | 기준 | 세션 유지 | 부하 반영 | 복잡도 |
|---------|------|----------|----------|-------|
| Round Robin | 순서 | X | X | 매우 낮음 |
| Weighted RR | 순서 + 가중치 | X | 부분적 | 낮음 |
| Least Connections | 활성 연결 수 | X | O | 중간 |
| IP Hash | 클라이언트 IP | O | X | 낮음 |

### 4. RESTful API 설계 원칙

REST(Representational State Transfer)는 **HTTP 프로토콜을 활용하여 리소스를 관리하는 아키텍처 스타일**이다.

#### REST 6가지 제약 조건

1. **클라이언트-서버 분리**: UI와 데이터 저장 관심사 분리
2. **무상태(Stateless)**: 각 요청은 모든 필요 정보를 포함, 서버가 세션 상태 저장 안 함
3. **캐시 가능(Cacheable)**: 응답에 캐시 가능 여부 명시
4. **계층화 시스템(Layered)**: 중간 계층(프록시, LB) 투명하게 사용 가능
5. **균일 인터페이스(Uniform Interface)**: 리소스는 URI로 식별, HTTP 메서드로 조작
6. **Code on Demand (선택)**: 서버가 클라이언트에 실행 가능한 코드 전송 가능

#### HTTP 메서드와 CRUD

| HTTP 메서드 | 작업 | CRUD | 멱등성 | 설명 |
|------------|------|------|-------|------|
| GET | 조회 | Read | O | 리소스 조회, 부수 효과 없음 |
| POST | 생성 | Create | X | 새 리소스 생성 |
| PUT | 전체 수정 | Update | O | 리소스 전체 교체 |
| PATCH | 부분 수정 | Update | X | 리소스 일부 수정 |
| DELETE | 삭제 | Delete | O | 리소스 삭제 |

**멱등성(Idempotency)**: 같은 요청을 여러 번 보내도 결과가 동일한 성질. GET, PUT, DELETE는 멱등적이다.

#### RESTful URL 설계

```
좋은 설계 (명사, 복수형, 계층 구조):
  GET    /api/users              → 사용자 목록 조회
  GET    /api/users/42           → 42번 사용자 조회
  POST   /api/users              → 사용자 생성
  PUT    /api/users/42           → 42번 사용자 수정
  DELETE /api/users/42           → 42번 사용자 삭제
  GET    /api/users/42/posts     → 42번 사용자의 게시글 목록

나쁜 설계 (동사 사용, 비일관적):
  GET    /api/getUser?id=42      ← 동사 사용
  POST   /api/createUser         ← 동사 사용
  GET    /api/user/42/delete     ← GET으로 삭제?!
```

### 5. API 버전 관리

API가 변경될 때 기존 클라이언트를 깨뜨리지 않기 위해 버전을 관리한다.

```
방법 1 — URI 경로 (가장 일반적):
  /api/v1/users
  /api/v2/users

방법 2 — 쿼리 파라미터:
  /api/users?version=1
  /api/users?version=2

방법 3 — HTTP 헤더:
  Accept: application/vnd.myapi.v1+json
  Accept: application/vnd.myapi.v2+json
```

| 방법 | 장점 | 단점 |
|------|------|------|
| URI 경로 | 직관적, 캐싱 용이 | URL 변경 필요 |
| 쿼리 파라미터 | 선택적 버전 지정 | 라우팅 복잡 |
| HTTP 헤더 | URL 깔끔 | 테스트 어려움, 발견성 낮음 |

### 6. 페이지네이션 (Pagination)

대량 데이터를 한 번에 반환하면 서버와 클라이언트 모두 부담이다. 페이지 단위로 나누어 전달한다.

#### 오프셋 기반 (Offset-Based)

```
GET /api/posts?page=2&size=10

의미: 2페이지, 페이지당 10개
→ OFFSET 10, LIMIT 10 (11번째~20번째 항목)

응답:
{
  "data": [...],
  "page": 2,
  "size": 10,
  "total": 150,
  "total_pages": 15
}
```

- **장점**: 직관적, 특정 페이지로 바로 이동 가능
- **단점**: 데이터 추가/삭제 시 중복 또는 누락 발생, 큰 OFFSET은 DB 성능 저하

#### 커서 기반 (Cursor-Based)

```
GET /api/posts?cursor=eyJpZCI6MTB9&size=10

의미: 커서(마지막으로 본 항목의 ID) 이후 10개
→ WHERE id > 10 LIMIT 10

응답:
{
  "data": [...],
  "next_cursor": "eyJpZCI6MjB9",
  "has_more": true
}
```

- **장점**: 데이터 변경에 안전, 대용량 데이터에 성능 우수
- **단점**: 특정 페이지 직접 이동 불가, 구현 복잡

### 7. Rate Limiting (요청 제한)

API 남용을 방지하고 서버를 보호하기 위해 **클라이언트별 요청 횟수를 제한**한다.

```
제한 초과 시:
  HTTP 429 Too Many Requests

응답 헤더:
  X-RateLimit-Limit: 100       ← 윈도우당 최대 요청 수
  X-RateLimit-Remaining: 0     ← 남은 요청 수
  X-RateLimit-Reset: 1609459200 ← 리셋 시각 (Unix timestamp)
  Retry-After: 60              ← 재시도까지 대기 시간(초)
```

**주요 알고리즘**:

| 알고리즘 | 설명 |
|---------|------|
| Fixed Window | 고정 시간 창(예: 1분)마다 카운트 리셋. 경계에서 2배 허용 문제 |
| Sliding Window Log | 각 요청의 타임스탬프 기록, 현재 윈도우 내 요청 수 계산 |
| Sliding Window Counter | Fixed Window의 장점 + 이전 윈도우 비율 고려. 효율적 |
| Token Bucket | 일정 속도로 토큰 충전, 요청마다 토큰 소모. 버스트 허용 |
| Leaky Bucket | 요청을 큐에 넣고 일정 속도로 처리. 균일한 처리율 보장 |
## 정리

| 개념 | 핵심 포인트 |
|------|------------|
| 로드 밸런서 | 트래픽을 여러 서버에 분배하여 부하 분산 및 고가용성 확보 |
| L4 LB | IP/포트 기반, 빠르고 단순, 콘텐츠 분석 불가 |
| L7 LB | HTTP 내용 기반, URL/헤더별 라우팅 가능, SSL 종료 |
| Round Robin | 순서대로 배분, 가장 단순 |
| Least Connections | 연결 적은 서버에 배분, 실제 부하 반영 |
| IP Hash | 같은 IP → 같은 서버, 세션 고정 |
| RESTful API | 리소스 중심, HTTP 메서드로 CRUD, 무상태 |
| 페이지네이션 | 오프셋(직관적, 성능이슈) vs 커서(안전, 고성능) |
| Rate Limiting | API 남용 방지, Token Bucket이 대표적 |
