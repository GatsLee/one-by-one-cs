## 핵심 개념

뉴스피드(타임라인)는 Facebook, Instagram, Twitter, LinkedIn 등 대부분의 SNS 서비스의 핵심 기능이다. 사용자가 팔로우하는 사람들의 게시물을 **시간순 또는 관련성 순으로 조합**하여 보여주는 시스템을 설계한다.

### 1. 요구사항 정의

```
┌─────────────────────────────────────────────────────────┐
│                  기능적 요구사항 (FR)                       │
├─────────────────────────────────────────────────────────┤
│ 1. 피드 발행: 사용자가 텍스트/이미지/동영상 게시            │
│ 2. 피드 조회: 팔로우하는 사람들의 게시물을 시간순으로 표시   │
│ 3. 팔로우/언팔로우: 다른 사용자를 구독/구독 해제            │
│ 4. 좋아요, 댓글 (선택)                                    │
│ 5. 무한 스크롤 (Pagination)                              │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│                비기능적 요구사항 (NFR)                       │
├─────────────────────────────────────────────────────────┤
│ 1. 피드 조회 지연 < 200ms                                │
│ 2. 새 게시물이 팔로워에게 5초 이내에 반영                   │
│ 3. 피크 시 초당 수십만 피드 요청 처리                       │
│ 4. 고가용성: 99.99% 업타임                                │
│ 5. 최종 일관성 허용 (약간의 지연은 OK)                     │
└─────────────────────────────────────────────────────────┘

용량 추산:
  - DAU: 3억 명
  - 평균 팔로잉: 300명
  - 하루 게시물: 사용자당 평균 0.5건 → 1.5억 건/일
  - 하루 피드 조회: 사용자당 10회 → 30억 회/일
  - 피드 조회 QPS: 30억 / 86,400 ≈ 35,000 QPS
  - 피크: ~100,000 QPS
  - 게시물 크기: 텍스트 1KB + 메타데이터 500B ≈ 1.5KB
  - 미디어: 이미지 평균 200KB, 동영상 평균 5MB (별도 저장)
```

### 2. Fan-out on Write vs Fan-out on Read

뉴스피드 시스템의 **가장 핵심적인 설계 결정**이다. 게시물이 팔로워들의 피드에 어떻게 도달하는가?

#### Fan-out on Write (Push 모델)

```
Fan-out on Write: 게시물 작성 시점에 팔로워들의 피드에 미리 배포

  사용자 A가 게시물 작성
     │
     ▼
  ┌──────────────┐
  │ Feed Service  │
  │               │
  │ 1. 게시물 저장 │
  │ 2. A의 팔로워  │
  │    목록 조회   │
  │ 3. 각 팔로워의 │
  │    피드에 삽입  │
  └──────┬───────┘
         │
    ┌────┴────────────────────────┐
    ▼           ▼           ▼     ▼
  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐
  │ B의   │  │ C의   │  │ D의   │  │ E의   │
  │ 피드   │  │ 피드   │  │ 피드   │  │ 피드   │
  │ 캐시   │  │ 캐시   │  │ 캐시   │  │ 캐시   │
  │[A:p1] │  │[A:p1] │  │[A:p1] │  │[A:p1] │
  │[F:p3] │  │[G:p2] │  │[A:p0] │  │[H:p5] │
  │[G:p2] │  │[F:p3] │  │[F:p3] │  │[A:p0] │
  └──────┘  └──────┘  └──────┘  └──────┘

  피드 조회 시: 자신의 피드 캐시를 읽기만 하면 됨 → O(1) 조회

  장점:
  - 피드 조회가 매우 빠름 (미리 준비된 캐시 조회)
  - 읽기 부하가 거의 없음
  - 구현이 직관적

  단점:
  - 쓰기 증폭 (Write Amplification): 팔로워가 100만이면 100만 번 쓰기
  - 비활성 사용자의 피드에도 불필요하게 쓰기
  - 게시물 삭제 시 모든 팔로워 피드에서 제거 필요
  - 팔로우 관계 변경 시 피드 갱신 복잡
```

#### Fan-out on Read (Pull 모델)

```
Fan-out on Read: 피드 조회 시점에 팔로잉한 사용자들의 게시물을 실시간 조합

  사용자 B가 피드 조회
     │
     ▼
  ┌──────────────┐
  │ Feed Service  │
  │               │
  │ 1. B가 팔로우  │
  │    하는 사용자 │
  │    목록 조회   │
  │ 2. 각 사용자의 │
  │    최근 게시물 │
  │    조회        │
  │ 3. 병합+정렬  │
  └──────┬───────┘
         │
    ┌────┴────────────────────┐
    ▼           ▼              ▼
  ┌──────┐  ┌──────┐  ┌──────┐
  │ A의   │  │ F의   │  │ G의   │
  │게시물 │  │게시물 │  │게시물 │
  │테이블 │  │테이블 │  │테이블 │
  └──────┘  └──────┘  └──────┘
       │         │         │
       └─────────┴─────────┘
                 │
           병합 + 시간순 정렬
                 │
              B의 피드

  장점:
  - 쓰기가 간단 (게시물만 저장)
  - 비활성 사용자에게 불필요한 처리 없음
  - 게시물 삭제/수정이 즉시 반영

  단점:
  - 읽기가 느림 (팔로잉 300명 × 최근 게시물 쿼리 + 병합)
  - 높은 읽기 부하 (피드 조회마다 다수의 DB 쿼리)
  - 피크 시간에 DB 과부하
```

#### 하이브리드 모델 (실전 접근법)

```
하이브리드 모델: 사용자 특성에 따라 전략을 분리

  ┌─────────────────────────────────────────────────┐
  │              게시물 작성 시 판단                    │
  │                                                 │
  │  작성자의 팔로워 수 확인                            │
  │                                                 │
  │  팔로워 < 10,000 (일반 사용자):                    │
  │    → Fan-out on Write                           │
  │    → 팔로워들의 피드 캐시에 즉시 배포               │
  │                                                 │
  │  팔로워 ≥ 10,000 (인플루언서/셀럽):                │
  │    → Fan-out on Read                            │
  │    → 게시물만 저장, 피드 조회 시 병합               │
  └─────────────────────────────────────────────────┘

  피드 조회 시:
  ┌──────────────────────────────────────────────┐
  │                                              │
  │  1. 미리 배포된 피드 캐시 조회 (일반 사용자 게시물)│
  │  2. 팔로우 중인 인플루언서의 최근 게시물 조회      │
  │  3. 두 결과를 병합 + 시간순 정렬                  │
  │  4. 상위 N개 반환                              │
  │                                              │
  └──────────────────────────────────────────────┘
```

| 비교 | Fan-out on Write | Fan-out on Read | 하이브리드 |
|------|-----------------|-----------------|-----------|
| 피드 조회 속도 | 매우 빠름 | 느림 | 빠름 |
| 쓰기 비용 | 높음 (팔로워 수 비례) | 낮음 | 중간 |
| 셀럽 문제 | 심각 (쓰기 폭발) | 없음 | 해결 |
| 실시간성 | 즉각적 | 조회 시점 기준 | 즉각적 |
| 비활성 사용자 | 낭비 | 낭비 없음 | 최적화 가능 |
| 사용 서비스 | Instagram 초기 | Twitter 초기 | 대부분의 대형 SNS |

### 3. 피드 생성 서비스 (Feed Generation Service)

게시물이 작성되면 팔로워들의 피드에 배포하는 서비스이다.

```
피드 생성 파이프라인:

  ┌──────────┐
  │ 사용자가   │
  │ 게시물 작성│
  └────┬─────┘
       │
       ▼
  ┌──────────────┐
  │ Post Service  │
  │               │
  │ 1. 게시물 DB  │
  │    저장       │
  │ 2. 미디어     │
  │    업로드     │
  └──────┬───────┘
         │ 이벤트 발행
         ▼
  ┌──────────────┐
  │    Kafka      │  (PostCreated 이벤트)
  └──────┬───────┘
         │
         ▼
  ┌──────────────────────────────────────────┐
  │        Feed Fanout Workers               │
  │                                          │
  │  1. 작성자의 팔로워 목록 조회               │
  │     (Graph DB 또는 Follow 테이블)          │
  │                                          │
  │  2. 팔로워 분류:                           │
  │     - 활성 사용자: 피드 캐시에 즉시 삽입    │
  │     - 비활성 사용자: 건너뛰기 (조회 시 조합) │
  │                                          │
  │  3. 각 팔로워의 피드 캐시에 post_id 삽입     │
  │     Redis Sorted Set:                    │
  │     feed:{user_id} → [(post_id, score)]  │
  │     score = 타임스탬프 (시간순 정렬)        │
  │                                          │
  │  4. 피드 크기 제한 (최근 1000개만 유지)      │
  │     ZREMRANGEBYRANK로 오래된 항목 제거      │
  └──────────────────────────────────────────┘

비동기 처리의 이유:
  - 팔로워 수만큼의 쓰기를 동기로 하면 API 응답이 느려짐
  - Kafka 큐를 통해 백그라운드 워커가 처리
  - 피크 시에도 큐가 버퍼 역할을 하여 부하를 평탄화
```

### 4. 피드 조회 서비스 (Feed Retrieval Service)

사용자가 앱을 열면 뉴스피드를 조회하는 서비스이다.

```
피드 조회 흐름:

  사용자 앱 열기
     │
     ▼
  ┌────────────────┐
  │ Feed Service    │
  │                │
  │ 1단계: 피드 캐시 조회                              │
  │   Redis: ZREVRANGE feed:{user_id} 0 19             │
  │   → 최근 20개의 post_id 목록                       │
  │   → [post_101, post_98, post_95, ...]             │
  │                                                    │
  │ 2단계: 인플루언서 게시물 조회 (하이브리드)              │
  │   팔로우 중인 인플루언서의 최근 게시물을 별도 조회      │
  │   → [celeb_post_200, celeb_post_199]              │
  │                                                    │
  │ 3단계: 병합 및 정렬                                  │
  │   두 목록을 시간순으로 병합 (Merge Sort)               │
  │   → 상위 20개 선택                                  │
  │                                                    │
  │ 4단계: 게시물 상세 정보 조회                           │
  │   Post Service에서 post_id들의 상세 정보 일괄 조회    │
  │   (작성자 이름, 프로필 사진, 본문, 미디어 URL,        │
  │    좋아요 수, 댓글 수 등)                            │
  │                                                    │
  │ 5단계: 응답 반환                                     │
  │   [                                                │
  │     {post_id, author, content, media, likes, ...}, │
  │     {post_id, author, content, media, likes, ...}, │
  │     ...                                            │
  │   ]                                                │
  └────────────────┘

Pagination (무한 스크롤):
  ┌────────────────────────────────────────────────┐
  │ Cursor 기반 페이지네이션:                         │
  │                                                │
  │ 첫 페이지: GET /feed?limit=20                   │
  │ 응답: {posts: [...], next_cursor: "post_81"}    │
  │                                                │
  │ 다음 페이지: GET /feed?limit=20&cursor=post_81  │
  │ → post_81 이전의 게시물 20개 반환                 │
  │                                                │
  │ 왜 offset이 아닌 cursor인가?                     │
  │ - offset: 새 게시물 추가 시 중복/누락 발생         │
  │ - cursor: 특정 게시물 기준이므로 안정적            │
  └────────────────────────────────────────────────┘
```

### 5. 캐시 전략

```
다계층 캐시 아키텍처:

  ┌────────────────────────────────────────────────────────┐
  │                    캐시 계층 구조                         │
  │                                                        │
  │  Layer 1: 피드 캐시 (Feed Cache)                        │
  │  ┌──────────────────────────────────────────┐          │
  │  │ Redis Sorted Set                         │          │
  │  │ feed:{user_id} → [(post_id, timestamp)]  │          │
  │  │ 각 사용자의 피드 (최근 1000개)              │          │
  │  │ TTL: 7일 (비활성 사용자 자동 제거)          │          │
  │  └──────────────────────────────────────────┘          │
  │                                                        │
  │  Layer 2: 게시물 캐시 (Post Cache)                      │
  │  ┌──────────────────────────────────────────┐          │
  │  │ Redis Hash                               │          │
  │  │ post:{post_id} → {author, content,       │          │
  │  │                    media_urls, likes, ...}│          │
  │  │ 인기 게시물의 상세 정보                     │          │
  │  │ TTL: 24시간                               │          │
  │  └──────────────────────────────────────────┘          │
  │                                                        │
  │  Layer 3: 사용자 캐시 (User Cache)                      │
  │  ┌──────────────────────────────────────────┐          │
  │  │ Redis Hash                               │          │
  │  │ user:{user_id} → {name, avatar_url,      │          │
  │  │                    bio, follower_count}   │          │
  │  │ 작성자 프로필 정보                          │          │
  │  │ TTL: 1시간                                │          │
  │  └──────────────────────────────────────────┘          │
  │                                                        │
  │  Layer 4: 소셜 그래프 캐시 (Follow Cache)               │
  │  ┌──────────────────────────────────────────┐          │
  │  │ Redis Set                                │          │
  │  │ followers:{user_id} → {follower1, ...}   │          │
  │  │ following:{user_id} → {followee1, ...}   │          │
  │  │ Fan-out 시 팔로워 목록 빠르게 조회          │          │
  │  │ TTL: 없음 (실시간 갱신)                    │          │
  │  └──────────────────────────────────────────┘          │
  └────────────────────────────────────────────────────────┘

캐시 무효화 전략:
  - 게시물 수정/삭제 → Post Cache 갱신 + 피드에서 제거
  - 프로필 변경 → User Cache 갱신
  - 팔로우/언팔로우 → Follow Cache 갱신 + 피드 재구성
```

### 6. 미디어 처리 (이미지/동영상)

```
미디어 업로드 및 처리 파이프라인:

  사용자가 이미지/동영상 첨부하여 게시
     │
     ▼
  ┌────────────────┐
  │ Upload Service  │
  │                │
  │ 1. Pre-signed  │  ← 클라이언트가 S3에 직접 업로드
  │    URL 발급    │     (서버 대역폭 절약)
  └──────┬─────────┘
         │
         ▼
  ┌────────────────┐
  │   Object Store  │  (AWS S3, GCS)
  │   (원본 저장)   │
  └──────┬─────────┘
         │ 업로드 완료 이벤트
         ▼
  ┌────────────────────────────────────────────┐
  │          Media Processing Pipeline          │
  │                                            │
  │  이미지 처리:                                │
  │  ┌──────────────────────────────────┐      │
  │  │ 원본 (4000×3000)                  │      │
  │  │   ├→ 썸네일 (150×150)             │      │
  │  │   ├→ 중간 (600×450)               │      │
  │  │   ├→ 피드용 (1080×810)            │      │
  │  │   └→ WebP 변환 (용량 30% 절약)    │      │
  │  └──────────────────────────────────┘      │
  │                                            │
  │  동영상 처리:                                │
  │  ┌──────────────────────────────────┐      │
  │  │ 원본 (4K, 500MB)                  │      │
  │  │   ├→ 1080p (H.264)               │      │
  │  │   ├→ 720p (H.264)                │      │
  │  │   ├→ 480p (H.264)                │      │
  │  │   ├→ HLS/DASH 세그먼트            │      │
  │  │   └→ 썸네일 추출 (0.5초 지점)      │      │
  │  └──────────────────────────────────┘      │
  └──────────────┬─────────────────────────────┘
                 │
                 ▼
  ┌────────────────┐
  │      CDN       │ (CloudFront, CloudFlare)
  │                │
  │ 전세계 엣지에   │
  │ 미디어 캐싱     │
  │                │
  │ URL 예시:      │
  │ cdn.example.com│
  │ /media/thumb/  │
  │  abc123.webp   │
  └────────────────┘

Pre-signed URL 업로드 흐름:
  ① 클라이언트 → 서버: "이미지 업로드하려고 합니다"
  ② 서버 → S3: Pre-signed URL 생성 요청
  ③ 서버 → 클라이언트: Pre-signed URL 반환
  ④ 클라이언트 → S3: 이미지 직접 업로드 (서버 미경유!)
  ⑤ S3 → 서버: 업로드 완료 이벤트 (Lambda/SQS)
  ⑥ 서버: 미디어 처리 파이프라인 시작

  장점: 대용량 파일이 서버를 경유하지 않으므로
        서버 대역폭과 CPU를 절약
```

### 피드 Fan-out 서비스 구현

```python
import time
import json
from dataclasses import dataclass, field
from collections import defaultdict
from typing import Any


@dataclass
class Post:
    """게시물 데이터 모델"""
    id: str
    author_id: str
    content: str
    media_urls: list[str] = field(default_factory=list)
    like_count: int = 0
    comment_count: int = 0
    created_at: float = field(default_factory=time.time)


class NewsFeedService:
    """
    뉴스피드 시스템의 핵심 로직.
    Redis와 DB를 딕셔너리로 시뮬레이션한다.
    """

    # 인플루언서 기준 (팔로워 수)
    INFLUENCER_THRESHOLD = 10000

    # 피드 캐시의 최대 크기
    FEED_CACHE_SIZE = 1000

    def __init__(self):
        # DB 시뮬레이션
        self.posts_db: dict[str, Post] = {}
        self.followers_db: dict[str, set[str]] = defaultdict(set)
        self.following_db: dict[str, set[str]] = defaultdict(set)

        # Redis 캐시 시뮬레이션
        # feed:{user_id} → [(post_id, timestamp)] 정렬된 리스트
        self.feed_cache: dict[str, list[tuple[str, float]]] = defaultdict(list)
        self.post_cache: dict[str, dict] = {}
        self.user_cache: dict[str, dict] = {}

    def follow(self, follower_id: str, followee_id: str) -> None:
        """팔로우 관계를 설정한다."""
        self.followers_db[followee_id].add(follower_id)
        self.following_db[follower_id].add(followee_id)
        print(f"  [팔로우] {follower_id} → {followee_id}")

    def _is_influencer(self, user_id: str) -> bool:
        """인플루언서 여부를 판단한다."""
        return len(self.followers_db[user_id]) >= self.INFLUENCER_THRESHOLD

    def create_post(self, author_id: str, content: str,
                    media_urls: list[str] = None) -> Post:
        """
        게시물을 작성하고 팔로워에게 배포한다.
        1. 게시물 DB 저장
        2. Fan-out: 팔로워 피드 캐시에 삽입 (일반 사용자만)
        """
        post = Post(
            id=f"post_{int(time.time()*1000)}",
            author_id=author_id,
            content=content,
            media_urls=media_urls or [],
        )

        # 1. DB 저장
        self.posts_db[post.id] = post
        self.post_cache[post.id] = {
            "id": post.id,
            "author_id": post.author_id,
            "content": post.content,
            "media_urls": post.media_urls,
            "like_count": post.like_count,
            "created_at": post.created_at,
        }
        print(f"  [게시물 생성] {post.id} by {author_id}")

        # 2. Fan-out (인플루언서가 아닌 경우만)
        if self._is_influencer(author_id):
            print(f"  [인플루언서] Fan-out 건너뛰기 "
                  f"(팔로워: {len(self.followers_db[author_id])}명)")
        else:
            self._fanout_on_write(post)

        return post

    def _fanout_on_write(self, post: Post) -> None:
        """팔로워들의 피드 캐시에 게시물을 배포한다."""
        followers = self.followers_db[post.author_id]
        print(f"  [Fan-out] {len(followers)}명에게 배포 중...")

        for follower_id in followers:
            # 피드 캐시에 (post_id, timestamp) 삽입
            feed = self.feed_cache[follower_id]
            feed.append((post.id, post.created_at))
            # 시간순 내림차순 정렬
            feed.sort(key=lambda x: x[1], reverse=True)
            # 캐시 크기 제한
            if len(feed) > self.FEED_CACHE_SIZE:
                self.feed_cache[follower_id] = feed[:self.FEED_CACHE_SIZE]

        print(f"  [Fan-out 완료] {len(followers)}명 배포 완료")

    def get_feed(self, user_id: str, limit: int = 20,
                 cursor: str = None) -> list[dict]:
        """
        사용자의 뉴스피드를 조회한다 (하이브리드 모델).
        1. 피드 캐시에서 미리 배포된 게시물 조회
        2. 팔로우 중인 인플루언서의 최근 게시물 조회
        3. 병합 + 정렬 + 페이지네이션
        """
        # 1단계: 피드 캐시에서 조회
        cached_feed = self.feed_cache.get(user_id, [])
        cached_post_ids = [(pid, ts) for pid, ts in cached_feed]

        # 2단계: 인플루언서 게시물 조회 (Fan-out on Read)
        influencer_posts = []
        following = self.following_db.get(user_id, set())
        for followee_id in following:
            if self._is_influencer(followee_id):
                # 인플루언서의 최근 게시물 조회
                for post_id, post in self.posts_db.items():
                    if post.author_id == followee_id:
                        influencer_posts.append((post_id, post.created_at))

        # 3단계: 병합 + 정렬
        all_posts = cached_post_ids + influencer_posts
        all_posts.sort(key=lambda x: x[1], reverse=True)

        # 4단계: 커서 기반 페이지네이션
        if cursor:
            cursor_ts = None
            for pid, ts in all_posts:
                if pid == cursor:
                    cursor_ts = ts
                    break
            if cursor_ts:
                all_posts = [(pid, ts) for pid, ts in all_posts
                            if ts < cursor_ts]

        # 5단계: 게시물 상세 정보 조회
        result = []
        for post_id, _ in all_posts[:limit]:
            post_detail = self.post_cache.get(post_id)
            if post_detail:
                result.append(post_detail)

        return result

    def delete_post(self, post_id: str, author_id: str) -> None:
        """
        게시물을 삭제하고 팔로워 피드에서 제거한다.
        """
        # DB에서 삭제
        if post_id in self.posts_db:
            del self.posts_db[post_id]
        if post_id in self.post_cache:
            del self.post_cache[post_id]

        # 팔로워 피드에서 제거
        followers = self.followers_db[author_id]
        for follower_id in followers:
            feed = self.feed_cache.get(follower_id, [])
            self.feed_cache[follower_id] = [
                (pid, ts) for pid, ts in feed if pid != post_id
            ]

        print(f"  [삭제] {post_id} 삭제 완료 "
              f"({len(followers)}명 피드에서 제거)")


# === 실행 예시 ===
if __name__ == "__main__":
    service = NewsFeedService()

    # 사용자 관계 설정
    service.follow("bob", "alice")
    service.follow("charlie", "alice")
    service.follow("bob", "charlie")
    print()

    # 게시물 작성 (일반 사용자 → Fan-out on Write)
    post1 = service.create_post("alice", "오늘 날씨가 좋아요!")
    print()

    import time as t; t.sleep(0.01)  # 타임스탬프 차이를 위해

    post2 = service.create_post("charlie", "맛있는 점심 먹었습니다",
                                 media_urls=["cdn.example.com/img/lunch.jpg"])
    print()

    # bob의 피드 조회 (alice와 charlie 모두 팔로우)
    print("=== bob의 뉴스피드 ===")
    feed = service.get_feed("bob", limit=10)
    for item in feed:
        print(f"  [{item['author_id']}] {item['content']}")
        if item['media_urls']:
            print(f"    미디어: {item['media_urls']}")

    print()

    # charlie의 피드 조회 (alice만 팔로우)
    print("=== charlie의 뉴스피드 ===")
    feed = service.get_feed("charlie", limit=10)
    for item in feed:
        print(f"  [{item['author_id']}] {item['content']}")
```

### 미디어 업로드 Pre-signed URL 생성 예시

```python
import hashlib
import time
import uuid
from dataclasses import dataclass


@dataclass
class MediaUploadRequest:
    """미디어 업로드 요청"""
    user_id: str
    file_name: str
    file_size: int       # bytes
    content_type: str    # image/jpeg, video/mp4 등


@dataclass
class PresignedUploadResponse:
    """Pre-signed URL 응답"""
    upload_url: str
    media_id: str
    cdn_url: str
    expires_in: int      # 초


class MediaService:
    """
    미디어 업로드/처리 서비스.
    실제 서비스에서는 AWS S3 SDK를 사용한다.
    """

    MAX_IMAGE_SIZE = 10 * 1024 * 1024      # 10MB
    MAX_VIDEO_SIZE = 500 * 1024 * 1024     # 500MB
    ALLOWED_IMAGE_TYPES = {"image/jpeg", "image/png", "image/webp", "image/gif"}
    ALLOWED_VIDEO_TYPES = {"video/mp4", "video/mov", "video/webm"}

    def __init__(self):
        self.media_store: dict[str, dict] = {}

    def request_upload(self, req: MediaUploadRequest) -> PresignedUploadResponse:
        """
        Pre-signed URL을 생성하여 반환한다.
        클라이언트는 이 URL로 S3에 직접 업로드한다.
        """
        # 1. 파일 유형 검증
        all_allowed = self.ALLOWED_IMAGE_TYPES | self.ALLOWED_VIDEO_TYPES
        if req.content_type not in all_allowed:
            raise ValueError(f"지원하지 않는 파일 유형: {req.content_type}")

        # 2. 파일 크기 검증
        is_video = req.content_type in self.ALLOWED_VIDEO_TYPES
        max_size = self.MAX_VIDEO_SIZE if is_video else self.MAX_IMAGE_SIZE
        if req.file_size > max_size:
            raise ValueError(
                f"파일 크기 초과: {req.file_size} > {max_size} bytes"
            )

        # 3. 고유 미디어 ID 생성
        media_id = str(uuid.uuid4())
        ext = req.file_name.rsplit(".", 1)[-1] if "." in req.file_name else "bin"

        # 4. S3 키 생성 (날짜별 분류)
        date_prefix = time.strftime("%Y/%m/%d")
        s3_key = f"media/{date_prefix}/{media_id}.{ext}"

        # 5. Pre-signed URL 생성 (실제로는 boto3.generate_presigned_url)
        upload_url = (
            f"https://s3.amazonaws.com/my-bucket/{s3_key}"
            f"?X-Amz-Signature=SIMULATED"
            f"&X-Amz-Expires=3600"
        )

        # 6. CDN URL 생성
        cdn_url = f"https://cdn.example.com/{s3_key}"

        # 7. 메타데이터 저장
        self.media_store[media_id] = {
            "id": media_id,
            "user_id": req.user_id,
            "s3_key": s3_key,
            "cdn_url": cdn_url,
            "content_type": req.content_type,
            "status": "pending_upload",  # 업로드 대기
        }

        return PresignedUploadResponse(
            upload_url=upload_url,
            media_id=media_id,
            cdn_url=cdn_url,
            expires_in=3600,
        )

    def on_upload_complete(self, media_id: str) -> dict:
        """
        S3 업로드 완료 콜백.
        미디어 처리 파이프라인을 시작한다.
        """
        media = self.media_store.get(media_id)
        if not media:
            raise ValueError(f"미디어를 찾을 수 없습니다: {media_id}")

        media["status"] = "processing"

        # 이미지 처리 작업 생성 (실제로는 Celery/SQS)
        is_video = media["content_type"] in self.ALLOWED_VIDEO_TYPES

        if is_video:
            tasks = ["transcode_720p", "transcode_480p",
                     "generate_thumbnail", "create_hls_segments"]
        else:
            tasks = ["resize_thumbnail_150", "resize_feed_1080",
                     "convert_webp"]

        print(f"  [미디어 처리] {media_id}")
        print(f"  작업 목록: {tasks}")

        # 처리 완료 시뮬레이션
        media["status"] = "completed"
        media["variants"] = {
            "thumbnail": media["cdn_url"].replace("/media/", "/media/thumb/"),
            "feed": media["cdn_url"].replace("/media/", "/media/feed/"),
        }

        return media


# === 실행 예시 ===
if __name__ == "__main__":
    service = MediaService()

    # 이미지 업로드 요청
    print("=== 이미지 업로드 ===")
    req = MediaUploadRequest(
        user_id="user_1",
        file_name="lunch_photo.jpg",
        file_size=2_500_000,  # 2.5MB
        content_type="image/jpeg",
    )
    response = service.request_upload(req)
    print(f"  업로드 URL: {response.upload_url[:60]}...")
    print(f"  CDN URL: {response.cdn_url}")
    print(f"  만료: {response.expires_in}초")

    # 업로드 완료 처리
    print(f"\n=== 업로드 완료 처리 ===")
    media = service.on_upload_complete(response.media_id)
    print(f"  상태: {media['status']}")
    print(f"  변환 결과: {json.dumps(media['variants'], indent=4)}"
          if 'json' in dir() else f"  변환 결과: {media['variants']}")
```

## 정리

| 개념 | 핵심 포인트 |
|------|------------|
| Fan-out on Write | 게시 시 팔로워 피드에 미리 배포, 읽기 O(1), 셀럽 계정에서 쓰기 폭발 |
| Fan-out on Read | 조회 시 실시간 조합, 쓰기 간단, 읽기 느림 |
| 하이브리드 모델 | 일반 사용자는 Push, 인플루언서는 Pull로 장점 결합 |
| 피드 캐시 | Redis Sorted Set으로 user별 피드 저장, 최근 1000개 유지 |
| 다계층 캐시 | 피드/게시물/사용자/팔로우 4계층으로 DB 부하 최소화 |
| Cursor 페이지네이션 | offset 대신 cursor(마지막 post_id)로 안정적 무한 스크롤 |
| Pre-signed URL | 클라이언트가 S3에 직접 업로드하여 서버 대역폭 절약 |
| 미디어 파이프라인 | 이미지 리사이즈, 동영상 트랜스코딩, CDN 배포를 비동기 처리 |
| Kafka Fan-out | 게시물 이벤트를 Kafka로 발행하여 워커가 비동기로 팔로워 피드 배포 |
