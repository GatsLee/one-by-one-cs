## 핵심 개념

### 왜 계층 모델이 필요한가?

네트워크 통신은 매우 복잡하다. 물리적 전기 신호부터 웹 페이지 렌더링까지 수많은 과정이 필요하다. 이것을 **계층(Layer)**으로 나누면:

- 각 계층이 독립적으로 발전 가능 (하드웨어 변경이 소프트웨어에 영향 없음)
- 문제 발생 시 어느 계층의 문제인지 빠르게 파악 가능
- 표준화 용이

> **비유**: 우편 시스템도 계층적이다. 편지 작성(응용) → 봉투에 넣기(표현) → 우체국 접수(세션) → 배송 경로 결정(네트워크) → 트럭으로 운반(전송/데이터링크) → 도로(물리)

### OSI 7계층 모델

ISO(국제 표준화 기구)에서 정의한 통신 표준 모델. **이론적 참조 모델**.

| 계층 | 이름 | 프로토콜/기술 | 역할 |
|------|------|-------------|------|
| 7 | 응용 (Application) | HTTP, FTP, SMTP, DNS | 사용자와 직접 상호작용 |
| 6 | 표현 (Presentation) | SSL/TLS, JPEG, 인코딩 | 데이터 형식 변환, 암호화, 압축 |
| 5 | 세션 (Session) | NetBIOS, RPC | 세션 설정/유지/종료 |
| 4 | 전송 (Transport) | TCP, UDP / 포트 번호 | 종단 간 통신 |
| 3 | 네트워크 (Network) | IP, ICMP, ARP / IP 주소 | 라우팅, 경로 결정 |
| 2 | 데이터 링크 (Data Link) | 이더넷, Wi-Fi / MAC 주소 | 인접 노드 간 전송 |
| 1 | 물리 (Physical) | 전기/광 신호, 케이블, 허브 | 비트 전송 |

### 각 계층 상세 설명

#### 7계층 — 응용 계층 (Application Layer)

사용자가 직접 사용하는 응용 프로그램과 네트워크 사이의 인터페이스.

| 프로토콜 | 용도 | 포트 |
|---------|------|------|
| HTTP/HTTPS | 웹 페이지 전송 | 80/443 |
| FTP | 파일 전송 | 20, 21 |
| SMTP | 이메일 발송 | 25, 587 |
| DNS | 도메인 → IP 변환 | 53 |
| SSH | 원격 접속 (암호화) | 22 |

#### 6계층 — 표현 계층 (Presentation Layer)

데이터의 **형식 변환, 암호화, 압축**을 담당.

- 데이터 인코딩: ASCII, UTF-8, EBCDIC 변환
- 암호화/복호화: SSL/TLS (실질적으로 4~7계층에 걸침)
- 압축: 데이터 크기 줄이기

#### 5계층 — 세션 계층 (Session Layer)

통신 **세션의 설정, 유지, 종료**를 관리.

- 대화 제어: 반이중(Half-duplex), 전이중(Full-duplex)
- 동기화: 체크포인트 설정으로 오류 시 복구 지점 제공
- 실제로는 응용 계층에 통합되는 경우가 많음

#### 4계층 — 전송 계층 (Transport Layer)

**종단 간(End-to-End)** 신뢰성 있는 데이터 전송. 포트 번호로 프로세스 식별.

- **TCP**: 연결 지향, 신뢰성, 순서 보장, 흐름/혼잡 제어
- **UDP**: 비연결, 빠름, 순서/신뢰성 보장 안 됨

#### 3계층 — 네트워크 계층 (Network Layer)

**라우팅**: 패킷이 출발지에서 목적지까지 최적 경로를 따라 전달.

- **IP 주소**: 네트워크상 장치 식별 (논리적 주소)
- **라우터**: 경로 결정 장비
- **IP 프로토콜**: 패킷 전달 (비연결, 최선형 전달)
- **ICMP**: 에러 보고 (ping 명령어)
- **ARP**: IP 주소 → MAC 주소 변환

#### 2계층 — 데이터 링크 계층 (Data Link Layer)

**인접한 노드 간**의 데이터 전송. 프레임 단위.

- **MAC 주소**: 하드웨어 고유 식별자 (48비트, AA:BB:CC:DD:EE:FF)
- **프레임**: 데이터 링크 계층의 PDU (Protocol Data Unit)
- **이더넷**: 유선 LAN 표준
- **Wi-Fi (802.11)**: 무선 LAN 표준
- **스위치**: 이 계층에서 동작하는 장비

#### 1계층 — 물리 계층 (Physical Layer)

**비트를 물리적 신호로 변환**하여 전송.

- 전기 신호 (구리선), 광 신호 (광섬유), 전파 (무선)
- 케이블 종류, 커넥터, 핀 배치
- 허브, 리피터: 신호 증폭/재생

### TCP/IP 4계층 모델

실제 인터넷에서 사용하는 모델. OSI 7계층보다 **실용적**.

| OSI 7계층 | TCP/IP 4계층 | 프로토콜 |
|-----------|------------|---------|
| 7. 응용 | **4. 응용** | HTTP, DNS, FTP, SMTP |
| 6. 표현 | ↑ | SSL/TLS, JPEG |
| 5. 세션 | ↑ | RPC, NetBIOS |
| 4. 전송 | **3. 전송** | TCP, UDP |
| 3. 네트워크 | **2. 인터넷** | IP, ICMP, ARP |
| 2. 데이터 링크 | **1. 네트워크 인터페이스** | 이더넷, Wi-Fi |
| 1. 물리 | ↑ | 전기/광 신호 |

**TCP/IP 모델의 특징:**
- OSI의 5, 6, 7계층을 **응용 계층** 하나로 통합
- OSI의 1, 2계층을 **네트워크 인터페이스 계층**으로 통합
- 실제 인터넷 프로토콜에 기반한 실용적 모델

### 패킷 캡슐화 / 역캡슐화

데이터가 각 계층을 지나면서 **헤더(Header)**가 추가(캡슐화)되고, 수신 측에서 역순으로 제거(역캡슐화)된다.

**송신 측 — 캡슐화 (위에서 아래로):**

| 계층 | 캡슐화 결과 |
|------|-----------|
| 응용 | `데이터` |
| 전송 | `TCP 헤더` \| 데이터 → **세그먼트** |
| 네트워크 | `IP 헤더` \| TCP 헤더 \| 데이터 → **패킷** |
| 데이터 링크 | `ETH 헤더` \| IP \| TCP \| 데이터 \| `FCS` → **프레임** |
| 물리 | `01001101001011...` → **비트 스트림** |

**수신 측 — 역캡슐화 (아래에서 위로):**
각 계층에서 해당 헤더를 제거하며 위로 올려보내, 최종적으로 응용 계층에서 원본 데이터를 획득한다.

**각 계층의 PDU (Protocol Data Unit):**

| 계층 | PDU 이름 | 내용 |
|------|---------|------|
| 응용 | 메시지/데이터 | 순수 애플리케이션 데이터 |
| 전송 | **세그먼트** (TCP) / 데이터그램 (UDP) | + 포트 번호, 시퀀스 번호 |
| 네트워크 | **패킷** | + 출발지/목적지 IP 주소 |
| 데이터 링크 | **프레임** | + 출발지/목적지 MAC 주소 |
| 물리 | **비트** | 0과 1의 전기/광 신호 |

### 실제 네트워크 장비와 계층

| 장비 | 계층 | 역할 |
|------|------|------|
| **허브** | 1 | 신호를 모든 포트로 전달 (브로드캐스트) |
| **스위치** | 2 | MAC 주소 테이블로 해당 포트로만 전달 |
| **라우터** | 3 | IP 주소 기반 패킷 경로 결정 |
| **방화벽** | 3-7 | 패킷 필터링, 접근 제어 |
| **로드밸런서** | 4-7 | 트래픽 분산 |

## 정리

| 항목 | 내용 |
|------|------|
| OSI 7계층 | 이론적 참조 모델 (물리→데이터링크→네트워크→전송→세션→표현→응용) |
| TCP/IP 4계층 | 실용적 모델 (네트워크 인터페이스→인터넷→전송→응용) |
| 캡슐화 | 데이터가 하위 계층으로 내려가면서 헤더가 추가되는 과정 |
| 역캡슐화 | 수신 측에서 헤더를 역순으로 제거하는 과정 |
| PDU | 세그먼트(전송), 패킷(네트워크), 프레임(데이터링크), 비트(물리) |
| 주소 체계 | MAC 주소(2계층), IP 주소(3계층), 포트 번호(4계층) |
| 장비 | 허브(1계층), 스위치(2계층), 라우터(3계층) |
