> 현대 네트워크 통신은 단순한 요청-응답을 넘어 실시간 양방향 통신, 스트리밍, 고성능 RPC로 진화했다.
> 이 문서에서는 HTTP/1.1의 한계부터 HTTP/2, HTTP/3, WebSocket, SSE, gRPC까지 핵심 프로토콜을 깊이 있게 다룬다.

---

## 핵심 개념

### 1. HTTP/1.1의 한계

HTTP/1.1은 웹의 근간이지만 현대 애플리케이션에서 심각한 성능 병목을 일으킨다.

**주요 한계점:**

- **요청-응답(Request-Response) 전용**: 클라이언트가 항상 먼저 요청해야 서버가 응답 가능. 서버가 능동적으로 데이터를 보낼 수 없음
- **Head-of-Line(HOL) Blocking**: 하나의 TCP 연결에서 이전 요청이 완료될 때까지 다음 요청이 대기
- **연결 오버헤드**: 매 요청마다 TCP 핸드셰이크 + 헤더 전송 반복 (Keep-Alive로 일부 완화)
- **중복 헤더**: 동일한 헤더(Cookie, User-Agent 등)를 매 요청마다 반복 전송

```
HTTP/1.1 Head-of-Line Blocking 문제
=====================================

  클라이언트                          서버
     |                                 |
     |--- 요청 1 (index.html) -------->|
     |                                 |  처리 중...
     |<-- 응답 1 (index.html) ---------|
     |                                 |
     |--- 요청 2 (style.css) --------->|  <-- 요청 1이 끝나야
     |                                 |      요청 2 전송 가능
     |<-- 응답 2 (style.css) ----------|
     |                                 |
     |--- 요청 3 (app.js) ------------>|  <-- 요청 2가 끝나야
     |                                 |      요청 3 전송 가능
     |<-- 응답 3 (app.js) -------------|

  * 파이프라이닝으로 일부 완화 가능하지만, 응답 순서는 여전히 보장 필요
  * 실무에서는 브라우저가 6~8개의 병렬 TCP 연결로 우회
```

**HTTP/1.1 시대의 우회 기법들:**
- Domain Sharding: 여러 도메인으로 리소스 분산 (병렬 연결 수 증가)
- 이미지 스프라이트: 여러 이미지를 하나로 합쳐 요청 수 감소
- 파일 번들링/인라이닝: JS/CSS를 하나로 합치거나 HTML에 인라인

### 2. HTTP/2 — 멀티플렉싱과 바이너리 프레이밍

HTTP/2는 HTTP/1.1의 성능 한계를 근본적으로 해결한다. Google의 SPDY 프로토콜을 기반으로 2015년 표준화되었다.

**핵심 특징:**

| 기능 | 설명 |
|------|------|
| **멀티플렉싱** | 하나의 TCP 연결에서 여러 스트림을 동시에 처리 |
| **헤더 압축 (HPACK)** | 중복 헤더를 인덱스 테이블로 압축, 대역폭 절약 |
| **서버 푸시** | 클라이언트 요청 없이 서버가 리소스를 미리 전송 |
| **바이너리 프레이밍** | 텍스트 대신 바이너리 프레임으로 파싱 효율 향상 |
| **스트림 우선순위** | 중요한 리소스에 높은 우선순위 부여 가능 |

```
HTTP/1.1 vs HTTP/2 연결 처리 비교
===================================

[HTTP/1.1 - 여러 TCP 연결 필요]

  클라이언트 ----TCP 연결 1----> 서버    요청 A → 응답 A
            ----TCP 연결 2----> 서버    요청 B → 응답 B
            ----TCP 연결 3----> 서버    요청 C → 응답 C
            ----TCP 연결 4----> 서버    요청 D → 응답 D

  * 각 연결마다 TCP 핸드셰이크 오버헤드 발생
  * 브라우저 제한: 도메인당 최대 6~8개 연결


[HTTP/2 - 단일 TCP 연결에서 멀티플렉싱]

  클라이언트 ====단일 TCP 연결====> 서버
               |                    |
               | Stream 1: 요청 A   |  ──┐
               | Stream 2: 요청 B   |    │ 동시에
               | Stream 3: 요청 C   |    │ 처리됨
               | Stream 4: 요청 D   |  ──┘
               |                    |
               | Stream 1: 응답 A   |  ──┐
               | Stream 3: 응답 C   |    │ 순서와 무관하게
               | Stream 2: 응답 B   |    │ 도착 가능
               | Stream 4: 응답 D   |  ──┘

  * HOL Blocking 해소 (애플리케이션 레벨)
  * 하나의 연결로 모든 요청 처리 → 연결 오버헤드 최소화
```

**바이너리 프레이밍 레이어:**

```
HTTP/2 프레임 구조
==================

+-----------------------------------------------+
|                Frame Header (9 bytes)          |
+-------+--------+---------+--------------------+
| Length | Type   | Flags   | Stream Identifier  |
| (24)  | (8)    | (8)     | (31 bits)          |
+-------+--------+---------+--------------------+
|                                               |
|              Frame Payload                     |
|           (가변 길이, 최대 16KB)               |
|                                               |
+-----------------------------------------------+

프레임 타입:
  - HEADERS  : HTTP 헤더 전송
  - DATA     : HTTP 본문 전송
  - SETTINGS : 연결 설정
  - PUSH_PROMISE : 서버 푸시 알림
  - PING     : 연결 상태 확인
  - GOAWAY   : 연결 종료 알림
```

### 3. HTTP/3 & QUIC — UDP 기반의 차세대 프로토콜

HTTP/3는 TCP 대신 UDP 기반의 QUIC 프로토콜 위에 구축된다.

**TCP의 근본적 한계:**
- TCP 레벨에서의 HOL Blocking (HTTP/2에서도 해결 불가)
- TCP + TLS 핸드셰이크에 최소 2~3 RTT 소요
- 네트워크 전환 시 연결 재수립 필요 (모바일 환경 문제)

```
연결 수립 비교: TCP+TLS vs QUIC
=================================

[TCP + TLS 1.3]                    [QUIC (HTTP/3)]

Client          Server             Client          Server
  |                |                 |                |
  |---SYN--------->|  1 RTT         |---Initial----->|  0-RTT
  |<--SYN+ACK------|                 |  (암호화 포함)  |  (재연결 시)
  |---ACK--------->|                 |                |
  |                |                 |<--Handshake----|
  |---ClientHello->|  2 RTT         |                |
  |<--ServerHello--|                 |---Data-------->|  즉시 데이터
  |---Finished---->|                 |<--Data---------|  전송 가능
  |<--Finished-----|                 |                |
  |                |
  |---Data-------->|  3 RTT         총: 0~1 RTT
  |<--Data---------|                 (첫 연결 1 RTT,
  |                |                  재연결 0 RTT)
  총: 2~3 RTT

  * QUIC은 전송 계층에 암호화가 내장됨
  * 연결 ID 기반 → 네트워크 전환 시에도 연결 유지
```

**QUIC의 핵심 장점:**
- **0-RTT 연결**: 이전에 연결한 적 있는 서버에 즉시 데이터 전송 가능
- **독립 스트림**: 한 스트림의 패킷 손실이 다른 스트림에 영향 없음
- **내장 암호화**: TLS 1.3이 프로토콜에 통합됨
- **연결 마이그레이션**: IP 변경 시에도 Connection ID로 연결 유지 (Wi-Fi ↔ LTE)

### 4. WebSocket — 양방향 실시간 통신

WebSocket은 HTTP 업그레이드 핸드셰이크를 통해 전이중(full-duplex) 양방향 통신 채널을 제공한다.

**핵심 특징:**
- 클라이언트와 서버 모두 자유롭게 메시지 전송 가능
- 연결 한 번 수립 후 지속 유지 (persistent connection)
- 프레임 오버헤드가 매우 작음 (2~14 바이트)
- 텍스트(UTF-8)와 바이너리 데이터 모두 지원

```
WebSocket 연결 생명주기
========================

  클라이언트                              서버
     |                                     |
     |  1) HTTP 업그레이드 핸드셰이크       |
     |--- GET /chat HTTP/1.1 ------------>|
     |    Upgrade: websocket               |
     |    Connection: Upgrade              |
     |    Sec-WebSocket-Key: dGhlIH...     |
     |    Sec-WebSocket-Version: 13        |
     |                                     |
     |<-- HTTP/1.1 101 Switching ---------|
     |    Protocols                        |
     |    Upgrade: websocket               |
     |    Connection: Upgrade              |
     |    Sec-WebSocket-Accept: s3pPL...   |
     |                                     |
     |  2) WebSocket 연결 수립 완료         |
     |=========== 양방향 통신 =============|
     |                                     |
     |--- 텍스트 프레임: "안녕하세요" ----->|
     |<-- 텍스트 프레임: "반갑습니다" ------|
     |<-- 바이너리 프레임: [이미지] --------|
     |--- 텍스트 프레임: "감사합니다" ----->|
     |                                     |
     |  3) Ping/Pong (연결 유지 확인)       |
     |<-- Ping 프레임 ---------------------|
     |--- Pong 프레임 -------------------->|
     |                                     |
     |  4) 연결 종료                        |
     |--- Close 프레임 (1000) ------------>|
     |<-- Close 프레임 (1000) -------------|
     |                                     |

  * ws://  — 암호화 없는 WebSocket (포트 80)
  * wss:// — TLS 암호화 WebSocket (포트 443)
  * 핸드셰이크 이후에는 HTTP가 아닌 WebSocket 프로토콜 사용
```

**WebSocket vs HTTP 폴링 비교:**

```
실시간 데이터 수신 방식 비교
============================

[HTTP 폴링 (Polling)]
  Client: 새 메시지 있어? → Server: 없음
  Client: 새 메시지 있어? → Server: 없음
  Client: 새 메시지 있어? → Server: 1개 있음!
  Client: 새 메시지 있어? → Server: 없음
  → 불필요한 요청 반복, 서버 부하 증가, 지연 발생

[HTTP 롱 폴링 (Long Polling)]
  Client: 새 메시지 있어? → Server: (대기 중...)
                                    → 있음! 응답 전송
  Client: 새 메시지 있어? → Server: (대기 중...)
  → 폴링보다 낫지만 여전히 연결 재수립 필요

[WebSocket]
  Client ←→ Server  (상시 연결 유지)
  Server → Client: 새 메시지 도착!   (즉시 전달)
  Client → Server: 읽음 확인         (즉시 전달)
  → 오버헤드 최소, 지연 최소, 양방향 자유
```

**주요 사용 사례:**
- 채팅 애플리케이션 (Slack, Discord)
- 실시간 게임 (좌표/상태 동기화)
- 주식/암호화폐 실시간 시세
- 협업 도구 (Google Docs 공동 편집)
- 라이브 스포츠 스코어보드

### 5. Server-Sent Events (SSE) — 서버→클라이언트 단방향 스트림

SSE는 서버에서 클라이언트로 단방향 실시간 데이터를 스트리밍하는 기술이다. HTTP 위에서 동작하며 WebSocket보다 단순하다.

**SSE의 특징:**
- 서버 → 클라이언트 단방향 통신만 가능
- 일반 HTTP 연결을 사용 (별도 프로토콜 불필요)
- 자동 재연결 기능 내장
- 텍스트 데이터만 지원 (바이너리 불가)
- 브라우저의 EventSource API로 간편하게 사용

```
SSE 통신 흐름
==============

  클라이언트 (브라우저)                    서버
     |                                     |
     |--- GET /events HTTP/1.1 ---------->|
     |    Accept: text/event-stream        |
     |                                     |
     |<-- HTTP/1.1 200 OK ----------------|
     |    Content-Type: text/event-stream  |
     |    Connection: keep-alive           |
     |                                     |
     |<-- data: {"temp": 22.5}\n\n --------|  이벤트 1
     |                                     |
     |<-- data: {"temp": 23.1}\n\n --------|  이벤트 2
     |                                     |
     |<-- event: alert\n                ---|  이벤트 3
     |    data: {"msg": "고온 경고"}\n\n --|  (이름 지정)
     |                                     |
     |    (연결 끊어짐)                     |
     |--- GET /events (자동 재연결) ------>|
     |    Last-Event-ID: 3                 |
     |                                     |
     |<-- 이벤트 4부터 이어서 전송... ------|

  SSE 데이터 형식:
    data: 메시지 내용\n\n        (기본)
    event: 이벤트명\n            (이벤트 타입 지정)
    id: 이벤트ID\n              (재연결 시 복구용)
    retry: 3000\n               (재연결 대기 시간 ms)
```

**WebSocket vs SSE 비교:**

| 항목 | WebSocket | SSE |
|------|-----------|-----|
| 방향 | 양방향 (full-duplex) | 단방향 (서버→클라이언트) |
| 프로토콜 | ws:// / wss:// | HTTP |
| 데이터 형식 | 텍스트 + 바이너리 | 텍스트만 |
| 자동 재연결 | 직접 구현 필요 | 내장 |
| 브라우저 지원 | 모든 브라우저 | IE 제외 모두 |
| 프록시 호환 | 문제 가능 | HTTP 호환 |
| 적합한 상황 | 채팅, 게임, 협업 | 알림, 피드, 모니터링 |

### 6. gRPC — 고성능 원격 프로시저 호출

gRPC는 Google이 개발한 고성능 RPC(Remote Procedure Call) 프레임워크이다. Protocol Buffers(protobuf)를 인터페이스 정의 언어(IDL)이자 직렬화 형식으로 사용하며, HTTP/2 위에서 동작한다.

**gRPC의 핵심 특징:**
- HTTP/2 기반 → 멀티플렉싱, 헤더 압축 자동 적용
- Protocol Buffers → 바이너리 직렬화로 JSON보다 최대 10배 빠른 파싱
- 강타입(strong typing) → .proto 파일로 서비스 인터페이스 명확히 정의
- 다중 언어 지원 → 하나의 .proto로 Go, Python, Java 등 코드 자동 생성
- 양방향 스트리밍 지원

```
gRPC 아키텍처
==============

  .proto 파일 (서비스 정의)
  ┌─────────────────────────────────────┐
  │ service Greeter {                   │
  │   rpc SayHello (HelloReq)           │
  │       returns (HelloReply);         │
  │ }                                   │
  └──────────┬──────────────────────────┘
             │ protoc 컴파일러
     ┌───────┴───────┐
     ▼               ▼
  Python 코드     Go 코드        ... 다른 언어
  (자동 생성)    (자동 생성)

  ┌────────────┐     HTTP/2      ┌────────────┐
  │  gRPC      │ ◄═══════════► │  gRPC      │
  │  Client    │  바이너리 통신   │  Server    │
  │  (Python)  │  (protobuf)    │  (Go)      │
  └────────────┘                └────────────┘

  * 클라이언트는 마치 로컬 함수를 호출하듯 서버의 메서드 호출
  * 네트워크 통신의 복잡성이 프레임워크에 의해 추상화됨
```

**gRPC의 4가지 통신 패턴:**

```
gRPC 통신 패턴
===============

1) Unary (단항)
   Client ---[Request]---> Server
   Client <--[Response]--- Server
   * 일반적인 요청-응답, REST API와 유사

2) Server Streaming (서버 스트리밍)
   Client ---[Request]---> Server
   Client <--[Response 1]- Server
   Client <--[Response 2]- Server
   Client <--[Response 3]- Server
   * 서버가 여러 응답을 스트리밍으로 전송
   * 예: 검색 결과를 점진적으로 반환

3) Client Streaming (클라이언트 스트리밍)
   Client ---[Request 1]-> Server
   Client ---[Request 2]-> Server
   Client ---[Request 3]-> Server
   Client <--[Response]--- Server
   * 클라이언트가 여러 요청을 보낸 후 서버가 단일 응답
   * 예: 파일 업로드 (청크 단위 전송)

4) Bidirectional Streaming (양방향 스트리밍)
   Client ---[Request 1]-> Server
   Client <--[Response 1]- Server
   Client ---[Request 2]-> Server
   Client ---[Request 3]-> Server
   Client <--[Response 2]- Server
   Client <--[Response 3]- Server
   * 클라이언트와 서버가 독립적으로 스트리밍
   * 예: 실시간 채팅, 음성 인식
```

**Protocol Buffers (protobuf):**

```
Protocol Buffers vs JSON 비교
==============================

[JSON] (텍스트 기반, 사람이 읽기 쉬움)
{
  "name": "김철수",
  "age": 25,
  "email": "kim@example.com"
}
→ 크기: ~65 bytes (텍스트)
→ 파싱: 문자열 처리 필요 (느림)

[Protobuf] (바이너리, 기계에 최적화)
0A 09 EA B9 80 EC B2 A0 EC 88 98 10 19 1A 0F ...
→ 크기: ~30 bytes (바이너리)
→ 파싱: 미리 생성된 코드로 직접 디코딩 (빠름)

장점: 크기 작음, 파싱 빠름, 스키마 강제
단점: 사람이 읽을 수 없음, 디버깅 어려움
```

```python
# websocket_server.py
# WebSocket 에코 서버 + 브로드캐스트 채팅 서버
import asyncio
import websockets
import json
from datetime import datetime

# 연결된 클라이언트를 추적하는 집합
connected_clients = set()


async def handle_client(websocket):
    """각 WebSocket 클라이언트를 처리하는 핸들러"""
    # 새 클라이언트 등록
    connected_clients.add(websocket)
    client_addr = websocket.remote_address
    print(f"[연결] 새 클라이언트: {client_addr} (총 {len(connected_clients)}명)")

    try:
        # 연결 환영 메시지 전송
        welcome = json.dumps({
            "type": "system",
            "message": "채팅방에 입장했습니다.",
            "timestamp": datetime.now().isoformat()
        })
        await websocket.send(welcome)

        # 메시지 수신 루프
        async for raw_message in websocket:
            data = json.loads(raw_message)
            print(f"[수신] {client_addr}: {data.get('message', '')}")

            # 브로드캐스트: 모든 연결된 클라이언트에게 메시지 전달
            broadcast_msg = json.dumps({
                "type": "chat",
                "sender": data.get("sender", "익명"),
                "message": data.get("message", ""),
                "timestamp": datetime.now().isoformat()
            })

            # 자기 자신을 제외한 모든 클라이언트에게 전송
            others = connected_clients - {websocket}
            if others:
                await asyncio.gather(
                    *(client.send(broadcast_msg) for client in others)
                )

    except websockets.exceptions.ConnectionClosed as e:
        print(f"[종료] 클라이언트 연결 종료: {e.code} {e.reason}")
    finally:
        # 클라이언트 제거
        connected_clients.discard(websocket)
        print(f"[제거] {client_addr} 제거됨 (남은 클라이언트: {len(connected_clients)}명)")


async def main():
    """WebSocket 서버 시작"""
    # localhost:8765에서 WebSocket 서버 실행
    async with websockets.serve(handle_client, "localhost", 8765):
        print("WebSocket 서버 시작: ws://localhost:8765")
        await asyncio.Future()  # 무한 대기 (서버 유지)


if __name__ == "__main__":
    asyncio.run(main())
```

```python
# WebSocket 채팅 클라이언트
import asyncio
import websockets
import json


async def chat_client(username: str):
    """WebSocket 채팅 클라이언트"""
    uri = "ws://localhost:8765"

    async with websockets.connect(uri) as websocket:
        print(f"서버에 연결됨: {uri}")

        async def receive_messages():
            """서버로부터 메시지를 수신하는 태스크"""
            try:
                async for raw_message in websocket:
                    data = json.loads(raw_message)
                    if data["type"] == "system":
                        print(f"[시스템] {data['message']}")
                    elif data["type"] == "chat":
                        print(f"[{data['sender']}] {data['message']}")
            except websockets.exceptions.ConnectionClosed:
                print("서버 연결이 종료되었습니다.")

        # 수신 태스크를 백그라운드에서 실행
        receive_task = asyncio.create_task(receive_messages())

        # 사용자 입력 처리 루프
        try:
            while True:
                message = await asyncio.to_thread(
                    input, ""  # 블로킹 input을 비동기로 처리
                )
                if message.lower() == "/quit":
                    break

                # JSON 형태로 메시지 전송
                payload = json.dumps({
                    "sender": username,
                    "message": message
                })
                await websocket.send(payload)

        except (KeyboardInterrupt, EOFError):
            pass
        finally:
            receive_task.cancel()
            print("채팅을 종료합니다.")


if __name__ == "__main__":
    name = input("닉네임을 입력하세요: ")
    asyncio.run(chat_client(name))
```

```python
# sse_server.py
# FastAPI를 사용한 Server-Sent Events 엔드포인트
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import asyncio
import json
import random
from datetime import datetime

app = FastAPI()


async def generate_sensor_data():
    """
    센서 데이터를 생성하는 SSE 스트림 제너레이터.
    SSE 형식: "data: {json}\n\n"
    """
    event_id = 0
    while True:
        event_id += 1
        # 가상의 온도/습도 센서 데이터 생성
        sensor_data = {
            "id": event_id,
            "temperature": round(20 + random.uniform(-5, 10), 1),
            "humidity": round(40 + random.uniform(-10, 20), 1),
            "timestamp": datetime.now().isoformat()
        }

        # SSE 형식으로 데이터 전송
        # id: 재연결 시 마지막 이벤트 추적용
        # event: 이벤트 타입 (클라이언트에서 구분용)
        # data: 실제 데이터 (JSON 문자열)
        yield f"id: {event_id}\n"
        yield f"event: sensor_update\n"
        yield f"data: {json.dumps(sensor_data)}\n\n"

        # 1초 간격으로 데이터 전송
        await asyncio.sleep(1)


async def generate_alert_stream():
    """
    경고 알림 SSE 스트림.
    온도가 임계값을 초과하면 alert 이벤트를 전송한다.
    """
    while True:
        # 30% 확률로 경고 발생
        if random.random() < 0.3:
            alert = {
                "level": random.choice(["warning", "critical"]),
                "message": "온도 임계값 초과!",
                "value": round(30 + random.uniform(0, 10), 1),
                "timestamp": datetime.now().isoformat()
            }
            yield f"event: alert\n"
            yield f"data: {json.dumps(alert)}\n\n"

        await asyncio.sleep(3)


@app.get("/events/sensors")
async def sse_sensors():
    """센서 데이터 SSE 엔드포인트"""
    return StreamingResponse(
        generate_sensor_data(),
        media_type="text/event-stream",
        headers={
            "Cache-Control": "no-cache",        # 캐시 비활성화
            "Connection": "keep-alive",          # 연결 유지
            "X-Accel-Buffering": "no",           # Nginx 버퍼링 비활성화
        }
    )


@app.get("/events/alerts")
async def sse_alerts():
    """경고 알림 SSE 엔드포인트"""
    return StreamingResponse(
        generate_alert_stream(),
        media_type="text/event-stream",
        headers={
            "Cache-Control": "no-cache",
            "Connection": "keep-alive",
        }
    )


# 클라이언트 측 JavaScript (브라우저)
"""

const sensorSource = new EventSource('/events/sensors');

// 기본 message 이벤트 (event 필드 없을 때)

    console.log('메시지:', event.data);
};

// 이름이 지정된 이벤트 수신
sensorSource.addEventListener('sensor_update', (event) => {

// 에러 처리 (자동 재연결 시도됨)

};

sensorSource.close();
"""
```

### gRPC .proto 파일 정의 및 Python 구현

```protobuf
// greeter.proto
// gRPC 서비스 정의 — Protocol Buffers IDL
syntax = "proto3";

package greeter;

// 서비스 정의: 4가지 통신 패턴 모두 포함
service GreeterService {
    // 1) Unary: 단일 요청 → 단일 응답
    rpc SayHello (HelloRequest) returns (HelloReply);

    // 2) Server Streaming: 단일 요청 → 다중 응답 스트림
    rpc SayHelloStream (HelloRequest) returns (stream HelloReply);

    // 3) Client Streaming: 다중 요청 스트림 → 단일 응답
    rpc CollectGreetings (stream HelloRequest) returns (GreetingSummary);

    // 4) Bidirectional Streaming: 다중 요청 ↔ 다중 응답
    rpc Chat (stream ChatMessage) returns (stream ChatMessage);
}

// 메시지 타입 정의
message HelloRequest {
    string name = 1;       // 필드 번호 1
    string language = 2;   // 필드 번호 2
}

message HelloReply {
    string message = 1;
    int64 timestamp = 2;
}

message GreetingSummary {
    int32 total_greetings = 1;
    repeated string names = 2;    // repeated = 리스트(배열)
}

message ChatMessage {
    string sender = 1;
    string content = 2;
    int64 timestamp = 3;
}
```

```python
# grpc_server.py
# gRPC 서버 구현 (Python)
# 사전 준비: pip install grpcio grpcio-tools
# 코드 생성: python -m grpc_tools.protoc -I. --python_out=. --grpc_python_out=. greeter.proto

import grpc
from concurrent import futures
import time

# protoc로 자동 생성된 파일 import
import greeter_pb2
import greeter_pb2_grpc


class GreeterServicer(greeter_pb2_grpc.GreeterServiceServicer):
    """gRPC 서비스 구현체"""

    def SayHello(self, request, context):
        """1) Unary RPC: 단일 요청-응답"""
        # 언어에 따른 인사말 생성
        greetings = {
            "ko": f"안녕하세요, {request.name}님!",
            "en": f"Hello, {request.name}!",
            "ja": f"こんにちは、{request.name}さん！",
        }
        message = greetings.get(
            request.language, f"Hello, {request.name}!"
        )
        return greeter_pb2.HelloReply(
            message=message,
            timestamp=int(time.time())
        )

    def SayHelloStream(self, request, context):
        """2) Server Streaming: 서버가 여러 응답을 스트리밍"""
        languages = ["ko", "en", "ja"]
        greetings = {
            "ko": "안녕하세요",
            "en": "Hello",
            "ja": "こんにちは",
        }
        # 3개 언어로 인사를 차례로 스트리밍
        for lang in languages:
            greeting = f"{greetings[lang]}, {request.name}!"
            yield greeter_pb2.HelloReply(
                message=greeting,
                timestamp=int(time.time())
            )
            time.sleep(1)  # 1초 간격으로 전송

    def CollectGreetings(self, request_iterator, context):
        """3) Client Streaming: 클라이언트 여러 요청 → 단일 응답"""
        names = []
        for request in request_iterator:
            names.append(request.name)
            print(f"  수신: {request.name}")

        return greeter_pb2.GreetingSummary(
            total_greetings=len(names),
            names=names
        )

    def Chat(self, request_iterator, context):
        """4) Bidirectional Streaming: 양방향 스트리밍 채팅"""
        for message in request_iterator:
            print(f"  [{message.sender}] {message.content}")
            # 에코 응답 전송
            reply = greeter_pb2.ChatMessage(
                sender="서버",
                content=f"'{message.content}' 수신 완료",
                timestamp=int(time.time())
            )
            yield reply


def serve():
    """gRPC 서버 시작"""
    # 워커 스레드 풀로 서버 생성
    server = grpc.server(futures.ThreadPoolExecutor(max_workers=10))

    # 서비스 등록
    greeter_pb2_grpc.add_GreeterServiceServicer_to_server(
        GreeterServicer(), server
    )

    # 포트 바인딩 및 서버 시작
    server.add_insecure_port("[::]:50051")
    server.start()
    print("gRPC 서버 시작: localhost:50051")
    server.wait_for_termination()


if __name__ == "__main__":
    serve()
```

```python
# grpc_client.py
# gRPC 클라이언트 구현 (Python)
import grpc
import time
import greeter_pb2
import greeter_pb2_grpc


def run():
    """gRPC 클라이언트 실행"""
    # 서버에 연결 (비암호화 채널)
    channel = grpc.insecure_channel("localhost:50051")
    stub = greeter_pb2_grpc.GreeterServiceStub(channel)

    # === 1) Unary RPC ===
    print("=== Unary RPC ===")
    response = stub.SayHello(
        greeter_pb2.HelloRequest(name="철수", language="ko")
    )
    print(f"응답: {response.message}")

    # === 2) Server Streaming ===
    print("\n=== Server Streaming ===")
    responses = stub.SayHelloStream(
        greeter_pb2.HelloRequest(name="영희")
    )
    for resp in responses:
        print(f"스트림 응답: {resp.message}")

    # === 3) Client Streaming ===
    print("\n=== Client Streaming ===")

    def name_generator():
        """이름을 하나씩 전송하는 제너레이터"""
        names = ["Alice", "Bob", "Charlie", "김철수"]
        for name in names:
            print(f"  전송: {name}")
            yield greeter_pb2.HelloRequest(name=name)
            time.sleep(0.5)

    summary = stub.CollectGreetings(name_generator())
    print(f"요약: 총 {summary.total_greetings}명 - {list(summary.names)}")

    # === 4) Bidirectional Streaming ===
    print("\n=== Bidirectional Streaming ===")

    def message_generator():
        """채팅 메시지를 생성하는 제너레이터"""
        messages = ["안녕하세요", "gRPC 테스트 중입니다", "감사합니다"]
        for msg in messages:
            yield greeter_pb2.ChatMessage(
                sender="클라이언트",
                content=msg,
                timestamp=int(time.time())
            )
            time.sleep(1)

    responses = stub.Chat(message_generator())
    for resp in responses:
        print(f"  [{resp.sender}] {resp.content}")


if __name__ == "__main__":
    run()
```
