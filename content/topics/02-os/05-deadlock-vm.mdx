## 핵심 개념

### 1. 데드락(Deadlock)이란?

데드락(교착 상태)은 **두 개 이상의 프로세스가 서로가 점유한 자원을 기다리며 무한히 대기하는 상태**이다. 어떤 프로세스도 진행할 수 없다.

```
데드락 상황 비유 — 좁은 다리:

  차 A →→→→→→→ ┃        ┃ ←←←←←←← 차 B
               ┃  다리  ┃
               ┃        ┃

  차 A: "B가 비켜줘야 내가 건너간다"
  차 B: "A가 비켜줘야 내가 건너간다"
  → 아무도 못 움직임 = 데드락!
```

**프로세스 수준 예시**:

```
프로세스 P1: 자원 R1을 점유, 자원 R2를 요청
프로세스 P2: 자원 R2를 점유, 자원 R1을 요청

  P1 ──점유──→ R1          P1 ──요청──→ R2
                   ↗                      ↘
  순환 대기!                                순환 대기!
                   ↖                      ↙
  P2 ──요청──→ R1          P2 ──점유──→ R2

  자원 할당 그래프 (순환 존재 = 데드락 가능):

    P1 ───→ R2 ───→ P2
    ↑                 │
    └───── R1 ←──────┘
```

### 2. 데드락 발생의 4가지 필요 조건 (Coffman 조건)

데드락이 발생하려면 다음 **4가지 조건이 동시에** 성립해야 한다.

| 조건 | 이름 | 설명 | 비유 |
|------|------|------|------|
| 1 | 상호배제 (Mutual Exclusion) | 자원을 한 번에 하나의 프로세스만 사용 가능 | 화장실 칸은 1명만 사용 |
| 2 | 점유대기 (Hold and Wait) | 자원을 점유한 채로 다른 자원을 기다림 | 포크 하나 들고 다른 포크 기다림 |
| 3 | 비선점 (No Preemption) | 점유 중인 자원을 강제로 빼앗을 수 없음 | 사용 중인 포크를 뺏을 수 없음 |
| 4 | 순환대기 (Circular Wait) | 프로세스들이 원형으로 서로의 자원을 기다림 | A→B→C→A 순환 고리 |

**중요**: 4가지 조건 중 하나라도 깨면 데드락은 발생하지 않는다.

### 3. 데드락 처리 전략

#### (1) 데드락 예방 (Prevention)

4가지 조건 중 **최소 하나를 원천적으로 불가능하게** 만든다.

| 깨는 조건 | 방법 | 부작용 |
|-----------|------|-------|
| 상호배제 | 자원 공유 가능하게 (예: 읽기 전용) | 본질적으로 공유 불가능한 자원 존재 |
| 점유대기 | 시작 시 필요한 자원 모두 할당 | 자원 낭비, 기아(starvation) |
| 비선점 | 자원 요청 실패 시 점유 자원 반납 | 작업 손실 가능 |
| 순환대기 | 자원에 번호 부여, 오름차순으로만 요청 | 유연성 감소 |

#### (2) 데드락 회피 (Avoidance)

시스템이 **안전 상태(Safe State)**를 유지하도록 자원 할당을 결정한다.

```
안전 상태 vs 불안전 상태:

┌──────────────────────────────────────────────┐
│              전체 상태 공간                    │
│  ┌────────────────────────────────────┐      │
│  │         안전 상태                   │      │
│  │   (데드락 없이 모든 프로세스 완료 가능) │      │
│  │                                    │      │
│  └────────────────────────────────────┘      │
│  ┌─────────────┐                             │
│  │ 불안전 상태   │  ← 데드락 가능 (반드시는 아님) │
│  │ ┌─────────┐ │                             │
│  │ │ 데드락   │ │                             │
│  │ └─────────┘ │                             │
│  └─────────────┘                             │
└──────────────────────────────────────────────┘
```

**은행원 알고리즘(Banker's Algorithm)**:
은행이 대출할 때 "모든 고객이 최대 대출을 받아도 파산하지 않는가?"를 확인하듯, OS가 자원 할당 전에 "모든 프로세스가 완료될 수 있는 순서(안전 순서)가 존재하는가?"를 확인한다.

```
은행원 알고리즘 예시:
총 자원: 12개
프로세스: P0, P1, P2

           최대 필요   현재 할당   추가 필요
  P0         10          5          5
  P1          4          2          2
  P2          9          2          7
                   가용 자원: 3개 (12 - 5 - 2 - 2)

안전 순서 확인:
  1. P1 실행 가능 (추가 필요 2 ≤ 가용 3) → P1 완료 → 가용 = 3+2 = 5
  2. P0 실행 가능 (추가 필요 5 ≤ 가용 5) → P0 완료 → 가용 = 5+5 = 10
  3. P2 실행 가능 (추가 필요 7 ≤ 가용 10) → P2 완료
  → 안전 순서: <P1, P0, P2> → 안전 상태!
```

#### (3) 데드락 탐지 (Detection) 및 회복 (Recovery)

데드락을 허용하되, 주기적으로 탐지하고 발견 시 회복한다.

**탐지 방법**: 자원 할당 그래프에서 사이클(순환) 탐색
**회복 방법**:
1. **프로세스 종료**: 데드락에 관련된 프로세스 중 하나를 강제 종료
2. **자원 선점**: 프로세스로부터 자원을 강제로 빼앗아 다른 프로세스에 할당
3. **롤백**: 체크포인트로 프로세스를 되돌림

### 4. 가상 메모리(Virtual Memory)

가상 메모리는 **물리 메모리(RAM)보다 큰 주소 공간을 프로세스에 제공하는 메모리 관리 기법**이다. 디스크의 일부를 메모리처럼 사용한다.

```
가상 메모리 개념:

프로세스가 보는 메모리       실제 물리 메모리(RAM)      디스크(스왑)
┌───────────────┐          ┌───────────────┐       ┌──────────┐
│ 페이지 0      │ ────────→│ 프레임 2      │       │          │
│ 페이지 1      │ ────────→│ 프레임 5      │       │          │
│ 페이지 2      │ ·····································→│ 스왑 영역 │
│ 페이지 3      │ ────────→│ 프레임 0      │       │          │
│ 페이지 4      │ ·····································→│ 스왑 영역 │
│  ...          │          │  ...          │       │          │
│ 페이지 N      │          │ (제한된 크기)  │       │          │
└───────────────┘          └───────────────┘       └──────────┘
  (매우 큼 4GB)              (작음 2GB)              (큼)

  ─────→ : RAM에 있는 페이지 (빠름)
  ·····→ : 디스크에 있는 페이지 (느림, 필요 시 RAM으로)
```

### 5. 요구 페이징(Demand Paging)

요구 페이징은 **프로세스가 실제로 접근하는 페이지만 메모리에 적재하는 방식**이다. 처음부터 모든 페이지를 로드하지 않는다.

```
요구 페이징 동작 흐름:

  CPU가 가상 주소 접근
          │
          ▼
  ┌─────────────────┐
  │   페이지 테이블    │
  │   참조           │
  └────────┬────────┘
           │
     ┌─────┴─────┐
     │ 유효 비트  │
     │ 확인      │
     └─────┬─────┘
       ┌───┴───┐
       │       │
    유효(1)  무효(0)
       │       │
       ▼       ▼
   RAM에서   페이지 폴트!
   바로접근    │
              ▼
         ┌──────────────────────┐
         │ 1. 디스크에서 페이지 찾기 │
         │ 2. 빈 프레임에 적재     │
         │ 3. 페이지 테이블 갱신   │
         │ 4. 명령어 재실행        │
         └──────────────────────┘
```

**페이지 폴트(Page Fault)**: 접근하려는 페이지가 RAM에 없을 때 발생하는 인터럽트. 디스크에서 해당 페이지를 가져와야 한다 (매우 느림: ~10ms).

### 6. 페이지 교체 알고리즘

RAM이 가득 찼을 때 새 페이지를 적재하려면 **기존 페이지 하나를 내보내야** 한다. 어떤 페이지를 내보낼지 결정하는 것이 페이지 교체 알고리즘이다.

#### (1) FIFO (First-In, First-Out)

가장 먼저 들어온 페이지를 교체한다.

```
참조 문자열: 7 0 1 2 0 3 0 4 2 3
프레임 수: 3

  시간  참조  프레임 상태      폴트?
  ──── ──── ───────────── ─────
   1    7   [7, -, -]       O
   2    0   [7, 0, -]       O
   3    1   [7, 0, 1]       O
   4    2   [2, 0, 1]       O (7 교체 ← 가장 오래됨)
   5    0   [2, 0, 1]       X (히트)
   6    3   [2, 3, 1]       O (0 교체)
   7    0   [2, 3, 0]       O (1 교체)
   8    4   [4, 3, 0]       O (2 교체)
   9    2   [4, 2, 0]       O (3 교체)
  10    3   [4, 2, 3]       O (0 교체)

  페이지 폴트: 9번
```

- **장점**: 구현이 매우 단순 (큐 사용)
- **단점**: Belady의 모순 — 프레임 수를 늘려도 폴트가 증가할 수 있음
- **문제**: "오래됐지만 자주 쓰이는 페이지"도 내보냄

#### (2) LRU (Least Recently Used)

**가장 오랫동안 사용되지 않은** 페이지를 교체한다.

```
참조 문자열: 7 0 1 2 0 3 0 4 2 3
프레임 수: 3

  시간  참조  프레임 상태       폴트?
  ──── ──── ──────────────  ─────
   1    7   [7, -, -]        O
   2    0   [7, 0, -]        O
   3    1   [7, 0, 1]        O
   4    2   [2, 0, 1]        O (7 교체 ← 가장 오래전에 사용)
   5    0   [2, 0, 1]        X (히트, 0의 사용 시간 갱신)
   6    3   [2, 0, 3]        O (1 교체 ← 1이 가장 오래전)
   7    0   [2, 0, 3]        X (히트)
   8    4   [4, 0, 3]        O (2 교체)
   9    2   [4, 2, 3]        O (0 교체 ← 0이 가장 오래전? 아니, 3이 6번에 사용, 0이 7번에 사용 → 3이 가장 오래전)
              정정: [4, 2, 3] → 3을 교체?
              0은 시간 7에 사용, 3은 시간 6에 사용, 4는 시간 8에 사용
              가장 오래전 = 3(시간6) → [4, 2, 0] → 아니, 잠깐.
              시간 8에서 프레임: [4, 0, 3], 사용시간: 4→8, 0→7, 3→6
              시간 9에서 2를 참조, 가장 오래전 사용 = 3(시간6)
              → [4, 2, 0] 아니라 3을 교체: [4, 2, 0]?
              아니 페이지 0은 프레임에 있으니 유지.
              3을 빼고 2를 넣음: [4, 0, 2]
   9    2   [4, 0, 2]        O (3 교체)
  10    3   [3, 0, 2]        O (4 교체 ← 4가 가장 오래전(시간8), 0은 시간7, 2는 시간9)
              0은 시간7이 가장 오래전 → 0을 교체: [4, 3, 2]?
              4: 시간8, 0: 시간7, 2: 시간9
              가장 오래전 = 0(시간7) → 0 교체: [4, 3, 2]

  페이지 폴트: 8번 (FIFO의 9번보다 적음)
```

- **장점**: 실용적으로 좋은 성능, Belady의 모순 없음
- **단점**: 구현 비용이 높음 (매 접근마다 시간 기록 필요)
- **구현**: 카운터, 스택, 또는 이중 연결 리스트 + 해시맵

#### (3) LFU (Least Frequently Used)

**사용 빈도가 가장 낮은** 페이지를 교체한다.

- 각 페이지의 참조 횟수를 카운트
- 참조 횟수가 같으면 LRU 적용 (가장 오래된 것 교체)
- **단점**: 과거에 많이 사용되었지만 지금은 안 쓰이는 페이지가 계속 남음

#### (4) Optimal (최적, OPT)

**앞으로 가장 오랫동안 사용되지 않을** 페이지를 교체한다.

- 이론적 최적해 (가장 적은 페이지 폴트)
- **실현 불가능**: 미래의 참조를 알 수 없기 때문
- **용도**: 다른 알고리즘의 성능 비교 기준(벤치마크)

#### 알고리즘 비교

| 알고리즘 | 교체 기준 | 장점 | 단점 |
|---------|----------|------|------|
| FIFO | 가장 먼저 들어온 것 | 구현 단순 | Belady 모순, 성능 나쁨 |
| LRU | 가장 오래전에 사용된 것 | 좋은 성능 | 구현 비용 높음 |
| LFU | 사용 빈도 가장 낮은 것 | 자주 쓰는 것 보존 | 오래된 인기 페이지 문제 |
| OPT | 미래에 가장 늦게 쓸 것 | 이론적 최적 | 구현 불가능 |

### 7. 스래싱(Thrashing)

스래싱은 **프로세스가 실제 작업보다 페이지 교체에 더 많은 시간을 소비하는 현상**이다.

```
CPU 이용률 vs 다중 프로그래밍 정도:

  CPU 이용률
  100% │          ╱╲
       │        ╱    ╲
       │      ╱        ╲  ← 스래싱 시작!
       │    ╱             ╲
       │  ╱                 ╲ (급격한 하락)
       │╱                     ╲
   0%  └──────────────────────────
       프로세스 수 →

  프로세스가 너무 많으면:
  → 각 프로세스에 할당된 프레임 부족
  → 페이지 폴트 폭증
  → 디스크 I/O 폭증
  → CPU 대기 시간 증가
  → OS가 "CPU가 놀고 있네" → 프로세스 더 추가
  → 악순환 → CPU 이용률 급락!
```

**해결 방법**:
1. **워킹 셋 모델(Working Set Model)**: 각 프로세스의 워킹 셋(최근 사용 페이지 집합) 크기만큼 프레임 보장
2. **PFF(Page Fault Frequency)**: 페이지 폴트율이 임계값을 초과하면 프레임 추가 할당, 낮으면 회수
3. **프로세스 수 제한**: 다중 프로그래밍 정도를 줄임

|------|------------|
| 데드락 | 프로세스들이 서로의 자원을 기다리며 무한 대기하는 상태 |
| 4가지 조건 | 상호배제, 점유대기, 비선점, 순환대기 (모두 충족 시 발생) |
| 데드락 예방 | 4가지 조건 중 하나를 원천 차단 |

| 데드락 탐지/회복 | 허용 후 사이클 탐지, 프로세스 종료 또는 자원 선점 |
| 가상 메모리 | 디스크를 활용해 물리 메모리보다 큰 주소 공간 제공 |
| 요구 페이징 | 실제 접근하는 페이지만 메모리에 적재 |

| OPT | 미래 참조를 알면 최적. 구현 불가, 벤치마크용 |
